# Workflow Instruction: Prepare Tasks (Main)

This is the central workflow for transforming structured requirements, derived from various inputs, into actionable tasks, project artifacts, and updated documentation within the project structure.

## Goal
To take the structured analysis and requirements generated by one of the specific input-processing workflows (listed below) and create the corresponding task files, directory structures (if needed), and update relevant documentation (like release READMEs or core project files).

## Process Overview (Two Phases)

1.  **Phase 1: Analysis & Requirements Gathering (Separate Workflows)**
    *   Choose the workflow corresponding to your input source to analyze it and generate structured requirements.
    *   Available Analysis Workflows:
        *   [From Concepts in Backlog](./from-concepts-in-backlog.md): Refining vague ideas.
        *   [From Git Diff](./from-diff.md): Analyzing code changes.
        *   [From Feature Requirements Document (FRD)](./from-frd.md): Processing formal requirements.
        *   [From Product Requirements Document (PRD)](./from-prd.md): Analyzing high-level product specs.
        *   [From PR Comments (API Fetch)](./from-pr-comments-api.md): Processing feedback fetched via GitHub API.
        *   [From PR Comments (MCP Fetch)](./from-pr-comments-mcp.md): Processing feedback fetched via MCP functions.
        *   [From Release-Specific Backlog](./from-release-backlog.md): Processing informal notes within a release.
    *   **Output of Phase 1:** A structured set of requirements, potentially including a proposed release plan, dependencies, and associated source information (e.g., comment IDs).

2.  **Phase 2: Task & Artifact Generation (This Workflow)**
    *   Use the structured requirements from Phase 1 as **input** to this workflow.
    *   This workflow handles the creation of concrete project elements.

## Input (for This Workflow)

*   The structured analysis/requirements output from one of the Phase 1 workflows.
*   Target release path(s) (often determined during Phase 1, e.g., `docs-project/backlog/vX.Y.Z-Codename/` or `docs-project/current/...`).
*   Confirmation of any proposed release plans (especially from PRD analysis).

## Process Steps (This Workflow)

1.  **Receive Structured Requirements:**
    *   Accept the structured analysis output from the preceding Phase 1 workflow.

2.  **Confirm/Establish Target Release(s):**
    *   Based on the input analysis, confirm the target release directory (e.g., for PR feedback, FRD features, release backlog items).
    *   If the input came from a PRD analysis that proposed *new* releases, confirm the plan and create the necessary release directory structures in `docs-project/backlog/` according to the [Project Management Guide](docs-dev/guides/project-management.md) (including standard subdirectories like `tasks/`, `docs/`, `decisions/`, etc.).

3.  **Generate Task Files:**
    *   For each identified requirement:
        *   Create a structured task file (`NN-task-name.md`) within the appropriate target release's `tasks/` directory.
        *   Follow the standard task format defined in the [Write Actionable Task Guide](docs-dev/guides/write-actionable-task.md).
        *   Include details from the analysis (e.g., source comment IDs, dependencies).
        *   Ensure correct numerical sequencing of tasks within the release.

4.  **Generate/Update Documentation & Artifacts:**
    *   **Release README:** Update the `README.md` file within each affected target release directory to list the newly created tasks.
    *   **Core Docs (if applicable, e.g., from PRD):**
        *   Generate or update `docs-project/what-do-we-build.md` based on the product vision.
        *   Generate or update `docs-project/architecture.md` based on identified components.
        *   Update the main `docs-project/README.md` to reflect any new multi-release plans.
    *   **ADRs (if applicable):** Create placeholder files for any critical architectural decisions identified during analysis in the relevant release's `decisions/` directory.

5.  **Communicate Results:**
    *   Report the number of tasks created and their location(s).
    *   List any new release directories created.
    *   Mention any core project documents updated.
    *   Suggest next steps (e.g., review created tasks, begin implementation).

## Output / Success Criteria

**Output:**
*   Structured task files (`.md`) created in the appropriate release `tasks/` directories.
*   Release directory structures created or updated in `docs-project/backlog/` or `docs-project/current/`.
*   Release `README.md` files updated.
*   Potentially updated core project documents (`what-do-we-build.md`, `architecture.md`, `docs-project/README.md`).
*   Placeholder ADR files created if needed.
*   Confirmation message summarizing actions taken.

**Success Criteria:**
*   All requirements from the input analysis are transformed into tasks or artifacts.
*   Tasks are correctly formatted and placed in the designated release directories.
*   Directory structures are created according to standards.
*   Relevant documentation is updated accurately.
