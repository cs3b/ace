# Address Extremely Long Filename Issues

## Intention

To identify and implement solutions for handling extremely long filenames that may cause issues across different operating systems and file systems within the Coding Agent Workflow Toolkit.

## Problem It Solves

**Observed Issues:**
- Certain generated filenames within the toolkit are excessively long, potentially exceeding operating system limits.
- This can lead to errors during file creation, modification, or access, particularly on systems with stricter filename length restrictions (e.g., older Windows versions, some network file systems).
- The repetition of the phrase "This is an extremely long idea that would cause filename length issues" suggests a potential pattern in the generation of such names, possibly related to descriptive naming or dynamic content embedding that has gone unchecked.

**Impact:**
- Files may become inaccessible or unmanageable, leading to workflow failures and data loss.
- Inconsistent behavior across different development environments and deployment targets.
- Difficulty in performing operations like Git commits, file searches, or backups on these long filenames.
- Potential for unexpected truncation or corruption of filenames, leading to confusion and errors.

## Key Patterns from Reflections

- **File System Compatibility**: The project relies on a multi-repository structure coordinated via Git submodules, and tools operate within developer environments. Filename length limitations are a known cross-platform issue.
- **Dynamic Content Generation**: ADR-014 mentions "long, long, long, long, long, long, long, long, long, long, long, long, long, long, long" in a filename example (`very-long-long-long-long-long-long-long-long-long-long-long-long-long-long-long.md`), indicating that dynamic content generation could be a source of extremely long names.
- **Workflow Self-Containment**: ADR-001 mandates self-contained workflows, which might lead to longer descriptions or embedded content within filenames if not managed carefully.
- **Template Embedding**: ADR-002 and ADR-005 discuss template embedding; if template paths or identifiers become excessively long, this could contribute to filename length issues.
- **Project Organization**: The use of `.ace/handbook/templates/` and `.ace/handbook/guides/` with potentially long descriptive subdirectories could exacerbate the problem.

## Solution Direction

1.  **Filename Sanitization/Shortening Strategy**: Implement a mechanism to automatically shorten or sanitize generated filenames that exceed a defined safe threshold.
2.  **Configuration for Max Filename Length**: Introduce configuration options to set a maximum allowed filename length, triggering sanitization when exceeded.
3.  **Adoption of Hashing or ID-Based Naming**: For dynamically generated files where descriptive names are not critical for immediate human readability, consider using hashes or unique IDs for filenames.

## Critical Questions

**Before proceeding, we need to answer:**
1.  What is the maximum filename length supported by the target operating systems and file systems (e.g., Linux, macOS, Windows, common network file systems)?
2.  What is the current average and maximum filename length observed in the project's generated files, and which specific tools or workflows are generating the longest names?
3.  What is the acceptable trade-off between filename readability and length? Should we prioritize shortening, hashing, or a combination?

**Open Questions:**
- How should shortened or hashed filenames be mapped back to their original, more descriptive intent for debugging or user understanding?
- What is the impact of filename shortening on Git's ability to track file history and diffs?
- Are there existing libraries or best practices for managing long filenames in Ruby or cross-platform CLI tools?

- Operating system limitations are the primary cause of filename length issues. - *Needs validation*
- A programmatic solution is feasible and desirable rather than manual renaming. - *Needs validation*
- Developers will accept shortened or ID-based filenames if the mapping or context is provided. - *Needs validation*

## Expected Benefits

- Increased robustness and reliability of the toolkit across various development environments.
- Prevention of file system errors related to filename length limitations.
- Improved manageability of files generated by dynamic content and workflow processes.
- Enhanced compatibility with different operating systems and network configurations.

## Big Unknowns

**Technical Unknowns:**
- The precise implementation strategy for filename sanitization (e.g., truncation, hashing, encoding) and its impact on file system operations and Git history.
- The best approach for mapping shortened/hashed filenames back to their original context.

**User/Market Unknowns:**
- How end-users (developers, AI agents) will perceive and interact with shortened or hashed filenames.
- Whether specific OS environments encountered by users will have unusually strict or lenient filename length limits.

**Implementation Unknowns:**

- The potential for unintended side effects of sanitization on file linking, searching, or external tool integrations.
