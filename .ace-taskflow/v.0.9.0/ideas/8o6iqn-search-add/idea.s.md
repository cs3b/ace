# Standardize ACE (Agent Coding Environment) Naming and Terminology Across Mono-Repo

## Problem
Inconsistent usage of the project name and acronym (ACE, Ace, Agent Coding Environment, Coding Agent Workflow Toolkit) across documentation, code comments, and internal strings creates ambiguity. This lack of standardization complicates context loading for AI agents and reduces the professional clarity of the project documentation, especially following the recent focus on CLI and ID standardization (Tasks 149, 179).

## Solution
Execute a project-wide refactoring to enforce standardized terminology:
1. **Acronym**: Always use `ACE` (all caps) when referring to the system.
2. **Full Name**: Use `Agent Coding Environment` for the full expansion.
3. **Meta Context**: Use terms like `ACE Mono-Repo` or `ACE Ecosystem` when referring to the collection of gems.

This standardization will be applied to all `ace-*` gem READMEs, internal documentation (`docs/`), code comments, and user-facing CLI output.

## Implementation Approach
This task requires orchestration across multiple components:

1. **`ace-docs` Integration**: Use `ace-docs` to scan and enforce consistency in all Markdown files, specifically targeting `docs/what-do-we-build.md`, `docs/architecture.md`, and all gem `README.md` files.
2. **Code Refactoring**: Utilize `ace-search` to locate instances of inconsistent naming within `lib/` directories of all `ace-*` gems (e.g., searching for 'Agent Coding Environment' vs 'Agent Coding Environment').
3. **`ace-core` Review**: Ensure core constants and configuration loading mechanisms in `ace-core` and `ace-support-core` use the standardized names where necessary, particularly in error messages and configuration summaries.
4. **`ace-lint` Preset**: Create a temporary `ace-lint` preset to validate the naming conventions during the cleanup phase, ensuring new contributions adhere to the standard.

This involves **Molecules** (file scanning, string replacement) and **Organisms** (orchestrating the refactoring across the mono-repo).

## Considerations
- **CLI Interface Design**: Ensure that user-facing help text generated by Thor (or dry-cli, pending Task 179 completion) uses the standardized `ACE` acronym.
- **Configuration Cascade**: Verify that configuration keys remain lowercase and hyphenated, separating the technical key from the human-readable description.
- **Impact on LLMs**: The primary goal is to provide a clean, unambiguous vocabulary for agents, reducing the cognitive load when processing project context.

## Benefits
- **Improved Context Awareness**: Provides agents (like Claude Code) with a deterministic and consistent project name, enhancing the reliability of context-aware operations.
- **Professional Documentation**: Elevates the quality and consistency of all user-facing and internal documentation.
- **Reduced Ambiguity**: Eliminates confusion arising from multiple project names, especially for external contributors or new team members.
- **Clean Foundation**: Prepares the ecosystem for future expansion, such as the `ace-handbook` gem, with established naming conventions.
