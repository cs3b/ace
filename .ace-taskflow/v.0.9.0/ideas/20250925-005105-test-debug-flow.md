## Problem
Currently, when tests fail within the ACE project, `ace-test-runner` provides results but lacks an integrated, structured debugging flow. Human developers must manually investigate failures by adding `puts` statements, re-running tests, or launching an external debugger. For AI agents, this manual process is a significant bottleneck, as they lack a deterministic mechanism to diagnose the root cause of test failures, hindering their ability to autonomously fix bugs and reducing overall development efficiency.

## Solution
Introduce a "debug flow" feature within the `ace-test-runner` gem. This feature will provide structured diagnostics for failed tests, allowing human developers to launch an interactive debugging session directly from a failed test. Crucially, for AI agents, it will provide a deterministic, parseable output of relevant context, variable states, and execution paths leading to the failure, enabling more intelligent autonomous diagnostics.

## Implementation Approach
- **ace-test-runner extension:** Add a new subcommand or flag (e.g., `ace-test debug <test_path>:<line>`) to trigger the debug flow for a specific test or failure point.
- **ATOM Architecture:**
    - A new `Organism` (e.g., `Ace::TestRunner::Organisms::DebugOrchestrator`) would coordinate the entire debugging process.
    - `Molecules` could include `Ace::TestRunner::Molecules::TestRerunner` (configured for verbose output/debug mode), `Ace::TestRunner::Molecules::ContextDumper` (to capture relevant scope and variables), and `Ace::TestRunner::Molecules::InteractiveDebuggerLauncher` (for human interaction).
    - `Atoms` would consist of pure functions for parsing test output, extracting variable states, formatting debug logs, and sanitizing paths (`path_expander`, `path_validator` from `ace-core`).
- **AI-Native Output:** For AI agents, the debug output should be a structured format (e.g., JSON or YAML) containing the stack trace, relevant variable values at the point of failure, a diff of expected vs. actual outputs, and potentially suggested next steps.
- **Human-Friendly Interface:** For human developers, the command could optionally launch a Ruby `debug` gem session or similar interactive debugger directly at the point of failure.
- **Context Loading:** Leverage `ace-context` to load relevant project files, configuration, and documentation during the debugging process, ensuring the agent has all necessary information as per ADR-001.

## Considerations
- **Deterministic vs. Interactive:** Clearly distinguish between the deterministic, structured output designed for AI agents and the interactive session provided for human developers.
- **Security & Path Validation:** Ensure that running tests in debug mode adheres to `path_validation` principles and does not expose sensitive information or create vulnerabilities. Implement `input_sanitization` for any user-provided debug parameters.
- **Performance:** The debug flow should be efficient and not significantly impact normal test execution performance.
- **Integration with `ace-test-support`:** Share common debugging utilities, assertions, and reporting mechanisms through `ace-test-support`.
- **Configuration Cascade:** Allow `.ace/` configuration (via `ace-core`) to define preferred debuggers, output formats, and verbosity levels.
- **Error Reporting:** Utilize the `ErrorReporter` module (ADR-009) for consistent error handling within the debug flow.

## Benefits
- **Accelerated Debugging:** Significantly reduces the time and effort required to diagnose and fix test failures for both human developers and AI agents.
- **Enhanced Agent Autonomy:** Equips AI agents with the ability to understand and react to test failures more intelligently, moving closer to autonomous bug fixing capabilities.
- **Improved Code Quality:** Faster feedback loops lead to quicker resolution of issues and higher quality code within the ACE ecosystem.
- **Standardized Diagnostics:** Provides a consistent, predictable method for reporting and analyzing test failures across all `ace-*` gems and associated workflows.
- **Reduced Manual Intervention:** Minimizes the need for human oversight during test failure analysis, freeing up developer time.
