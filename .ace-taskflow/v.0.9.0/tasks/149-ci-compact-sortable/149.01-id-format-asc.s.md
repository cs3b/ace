# Create ace-support-timestamp Package with Base36 Compact IDs

## Objective

Create a dedicated `ace-support-timestamp` Ruby gem that provides configurable Base36 compact ID generation (6 characters) as a replacement for 14-character timestamps. The package must support configurable year_zero, provide CLI tools for encode/decode/migrate operations, and ensure case-insensitive filesystem compatibility.

## Behavioral Specification

### User Experience

**Input**: Time objects (Time.now), configuration options (year_zero, alphabet, precision)

**Process**:
- User configures encoding options via `.ace/timestamp/config.yml`
- User calls `Ace::Timestamp::Atoms::CompactIdEncoder.encode(time)` or uses CLI
- System normalizes to UTC, packs timestamp components, encodes as Base36
- User receives 6-character sortable ID

**Output**: 6-character lowercase Base36 ID (e.g., `000000`, `0buzn`, `8hei9`)

### Expected Behavior

**Core Encoding Behavior**:
- Accept Time object and optional config hash
- Normalize input time to UTC
- Pack timestamp components into 6-digit Base36 string
- Return lowercase string for case-insensitive safety

**Core Decoding Behavior**:
- Accept 6-character Base36 string and optional config hash
- Validate format (length, characters)
- Decode and return Time object in UTC
- Raise descriptive errors for invalid input

**Configuration Behavior**:
- Load year_zero from config (default: 2000)
- Support 108-year coverage from configured base year
- Allow runtime config override for flexibility

**CLI Behavior**:
- `ace-timestamp encode "2025-01-01 00:00:00"` → outputs compact ID
- `ace-timestamp decode "000000"` → outputs timestamp
- `ace-timestamp migrate [package]` → migrates existing files

### Interface Contract

**Ruby API**:

```ruby
# Encode time to compact ID
Ace::Timestamp::Atoms::CompactIdEncoder.encode(time)
# => "000000"

# With config override
Ace::Timestamp::Atoms::CompactIdEncoder.encode(time, year_zero: 2025)
# => "1o0000"

# Decode compact ID to time
Ace::Timestamp::Atoms::CompactIdEncoder.decode("000000")
# => 2000-01-01 00:00:00 UTC

# Validate format
Ace::Timestamp::Atoms::CompactIdEncoder.valid?("000000")
# => true

Ace::Timestamp::Atoms::CompactIdEncoder.valid?("invalid")
# => false
```

**CLI Interface**:

```bash
# Encode timestamp
ace-timestamp encode "2025-01-01 00:00:00"
# Output: 000000

# Decode compact ID
ace-timestamp decode "000000"
# Output: 2000-01-01 00:00:00 UTC

# Migrate package files
ace-timestamp migrate ace-taskflow
# Output: Migrated 25 idea directories

# Show configuration
ace-timestamp config
# Output: year_zero: 2000
#         alphabet: base36
#         precision: second
```

**Error Handling**:

- `Ace::Timestamp::InvalidIdError`: Non-Base36 characters or wrong length
- `Ace::Timestamp::RangeError`: Year outside configured coverage (e.g., before 2000 with default config)
- `Ace::Timestamp::ConfigError`: Invalid configuration values

**Edge Cases**:

- Year boundary transitions (1999→2000, 2107→2108)
- Leap year dates (Feb 29 handling)
- End of month/day rollovers
- Minimum and maximum representable timestamps

### Success Criteria

- [ ] **6-Character IDs**: All generated IDs are exactly 6 lowercase Base36 characters
- [ ] **Sortability**: Lexicographic string sort matches chronological order
- [ ] **Roundtrip**: `encode(decode(id)) == id` for all valid IDs
- [ ] **Configurable Year Zero**: Default 2000, overridable via config and parameter
- [ ] **CLI Tools**: All encode/decode/migrate commands work correctly
- [ ] **108-Year Coverage**: Covers year_zero to year_zero + 108 years
- [ ] **Case-Insensitive Safe**: Uses only 0-9a-z characters
- [ ] **UTC Normalization**: All timestamps normalized to UTC internally

### Validation Questions

- [ ] **Year Zero Configuration**: Should year_zero be a global config or per-package setting?
- [ ] **Migration Strategy**: Should migrations be in-place or create new files?
- [ ] **Backward Compatibility**: Should old timestamp format still be supported alongside new format?
- [ ] **Error Handling**: Should decoding gracefully fallback to timestamp format for compact ID failures?
