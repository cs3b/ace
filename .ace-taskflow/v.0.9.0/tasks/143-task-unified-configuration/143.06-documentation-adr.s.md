## ADR Template

```markdown
# ADR-023: Unified Configuration Loading Pattern

## Status
Accepted

## Context
ace-* packages developed with varying configuration approaches:
- Some used `Ace::Core.config.get()` with `.ace.example/` defaults
- Some embedded defaults in Ruby code (inline YAML, DEFAULT_CONFIG hashes)
- Some used custom YAML loading

This inconsistency made it difficult to:
- Understand how to configure each package
- Maintain configuration across packages
- Ensure user overrides work consistently

## Decision
All ace-* packages MUST:
1. Define defaults in `.ace.example/<package>/config.yml`
2. Load configuration via `Ace::Core.config.get('ace', '<package>')`
3. Not embed default values in Ruby code

## Consequences

### Positive
- Consistent configuration experience across all packages
- Users can override any setting via `.ace/<package>/config.yml`
- Single source of truth for defaults
- Easier maintenance and discovery

### Negative
- Migration work required for non-compliant packages
- Slightly more complex initial setup for new packages

### Migration
Packages migrated as part of Task 143:
- ace-taskflow
- ace-git-worktree
- ace-nav
- ace-test-runner
- ace-search (hardcoded values)
- ace-docs (hardcoded values)
- ace-context (hardcoded values)

Deferred:
- ace-llm (ENV-based, separate task)
- ace-llm-providers-cli (ENV-based, separate task)
```
