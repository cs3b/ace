# Core Workflow Transformations

## Behavioral Specification

### User Experience

- **Input**: Agent or user invokes `/ace:review-task 230` on a draft task
- **Process**: Workflow validates behavioral spec completeness, researches answers to open questions, promotes to pending when ready
- **Output**: Task promoted to pending status (ready for JIT planning + implementation) OR blocking questions documented

### Expected Behavior

**Current Problems:**
- plan-task.wf.md (540 lines) produces heavyweight implementation plans that go stale before execution
- No clear gate between "spec is ready" and "start implementing"
- review-task.wf.md reviews any status without making lifecycle decisions
- work-on-task.wf.md requires pre-existing implementation plan in task file

**Desired State:**

1. **review-task.wf.md** becomes the draft→pending readiness gate:
   - Validates behavioral spec (UX, interface contracts, success criteria, scope)
   - Uses existing autonomous research + question generation pattern
   - Promotes validated drafts to `status: pending`
   - Blocks with `needs_review: true` when questions remain

2. **plan-task.wf.md** becomes a JIT planning workflow:
   - Output goes to stdout/ace-assign phase reports (ephemeral)
   - Does NOT modify task spec files or change task status
   - Keeps technical research, file modification planning, test strategy
   - Removes: pending template, status promotion, UX docs, heavy templates

3. **draft-task.wf.md** minor update:
   - "Next step" references plan-task → review-task

4. **work-on-task.wf.md** minor update:
   - Accepts tasks with behavioral specs only (no implementation plan required)
   - Notes JIT planning as prior step when via ace-assign

### Interface Contract

```bash
# New lifecycle flow
/ace:draft-task          # Creates draft with behavioral spec
/ace:review-task 230     # Validates spec → promotes to pending
# Then via ace-assign:
# onboard → plan-task (JIT) → work-on-task → ...
```
