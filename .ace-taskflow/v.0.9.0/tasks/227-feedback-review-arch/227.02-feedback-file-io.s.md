# Feedback File I/O

## Behavioral Specification

### User Experience
- **Input**: FeedbackItem objects and directory paths
- **Process**: System reads/writes/archives feedback files with proper locking
- **Output**: Files persisted to disk, FeedbackItems loaded from disk

### Expected Behavior

Three molecules handle all file operations:

1. **FeedbackFileWriter**: Writes FeedbackItem to disk, creates directories, handles file locking
2. **FeedbackFileReader**: Parses feedback files back into FeedbackItem objects
3. **FeedbackDirectoryManager**: Manages feedback directory structure, lists files, handles archiving

All operations should:
- Use file-level locking for concurrent agent access
- Create directories as needed
- Handle missing files gracefully
- Support the `_archived/` subdirectory pattern

**Error Handling:**
- Directory not found: Create it
- File not found on read: Return nil
- File locked by another process: 3 retries with exponential backoff (100ms, 200ms, 400ms), raise FileLockError after 700ms total
- Invalid frontmatter: Raise ParseError with file path

**Atomic Writes:**
- All write operations use temp-file-and-move pattern:
  1. Write to `{path}.tmp.{Process.pid}`
  2. Move temp file to target path
  3. Clean up temp file in ensure block

**Encoding:**
- All file operations MUST use UTF-8 encoding
- Use `File.write(path, content, encoding: 'UTF-8')`
- Use `File.read(path, encoding: 'UTF-8')`

**Edge Cases:**
- Concurrent writes: File locking prevents corruption
- Partial write failure: Leave file unchanged
- Archive of already-archived item: No-op, return success
