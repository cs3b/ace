# Feedback-Based Review Output Architecture

## Overview

Replace the monolithic `synthesis-report.md` with a structured feedback file system. Each finding from reviews becomes a tracked feedback item with its own lifecycle, enabling concurrent verification by multiple agents.

## Architecture Change

```
Current:  Review Reports -> ReportSynthesizer -> synthesis-report.md (monolithic)
New:      Review Reports -> FeedbackExtractor -> feedback/*.s.md (individual files)
                                              -> FeedbackManager -> _archived/
```

## Feedback File Format

**Filename**: `{6-char-id}-{slug}.s.md` (e.g., `8o7abc-security-sql-injection.s.md`)

```yaml
---
id: 8o7abc
title: SQL Injection in User Input Handler
files:
  - src/handlers/user.rb:42-55
  - src/db/query.rb:10
reviewer: google:gemini-2.5-flash
status: draft
priority: high
created: 2026-01-22T19:00:00Z
updated: 2026-01-22T19:00:00Z
---

## Finding
{Original finding from the review}

## Context
{Why this matters, related code analysis}

## Research
{Validation research - populated during verification phase}

## Resolution
{How it was resolved - populated when status becomes done}
```

## Status Workflow

```
draft (initial extraction)
  |
pending (verified, ready for work) | invalid (false positive) | skip (not applicable)
  |
done (addressed/fixed)

Final states: invalid, skip, done -> move to feedback/_archived/
```

## Directory Structure

```
{task_dir}/
├── feedback/
│   ├── 8o7abc-security-sql-injection.s.md
│   ├── 8o7abd-perf-n-plus-one-query.s.md
│   └── _archived/
│       └── 8o7abf-resolved-item.s.md
└── reviews/
    ├── 8o7ab0-gemini-code-pr-review.md
    ├── 8o7ab1-claude-code-pr-review.md
    └── 8o7ab2-synthesis.md  (optional, can be disabled)
```

## Subtasks

- **01**: Feedback Model and Core Atoms - FeedbackItem model, ID generator, slug generator
- **02**: Feedback File I/O - Writer, reader, directory manager molecules
- **03**: Feedback Extractor - LLM-based extraction from review reports
- **04**: Feedback Manager - Organism for full lifecycle orchestration
- **05**: Synthesizer Integration - Integration with ReportSynthesizer
- **06**: CLI Commands - Commands for feedback list/show/verify/resolve
- **07**: Task Integration - Save feedback to task directories
- **08**: Documentation - docs/feedback-workflow.md, CHANGELOG updates

## Key Design Decisions

1. **JSON intermediate format** for LLM extraction (reliable parsing)
2. **Archive via directory move** (clear separation, easy restore)
3. **File-level locking** for concurrent agent access (3 retries, exponential backoff: 100ms, 200ms, 400ms)
4. **Synthesis preserved** (feedback is additive, `--no-feedback` to disable)
5. **Default storage location**: When no `--task` is specified, feedback files are saved to the session cache directory: `.cache/ace-review/sessions/{session_id}/feedback/`
6. **Task context detection order**: 1) `--task` flag, 2) Branch name pattern (e.g., `task-227`), 3) Current directory within .ace-taskflow, 4) Session cache fallback

## Verification

1. Run review with feedback extraction:
   ```bash
   ace-review --preset code-pr --task test-task --auto-execute
   ls .ace-taskflow/.../test-task/feedback/
   ```

2. List and verify feedback:
   ```bash
   ace-review feedback list --status draft
   ace-review feedback verify {id} --valid --research "Confirmed issue"
   ```

3. Resolve and check archive:
   ```bash
   ace-review feedback resolve {id} --resolution "Fixed in commit abc123"
   ls .ace-taskflow/.../test-task/feedback/_archived/
   ```

4. Run existing tests:
   ```bash
   ace-test ace-review
   ```
