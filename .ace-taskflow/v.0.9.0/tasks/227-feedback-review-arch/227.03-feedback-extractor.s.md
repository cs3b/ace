# Feedback Extractor

## Behavioral Specification

### User Experience
- **Input**: Review report content (markdown) from various reviewers
- **Process**: LLM extracts individual findings as structured JSON, converts to FeedbackItems
- **Output**: Array of FeedbackItem objects ready for persistence

### Expected Behavior

The FeedbackExtractor molecule uses an LLM to extract individual findings from review reports. It:

1. **Sends report to LLM** with extraction prompt
2. **Receives JSON array** of findings with structured fields
3. **Converts to FeedbackItems** with generated IDs and slugs
4. **Groups similar findings** to avoid duplicates
5. **Assigns priorities** based on finding severity

The JSON intermediate format ensures reliable parsing even with complex markdown content.

```ruby
# Extraction result structure (JSON from LLM)
{
  "findings": [
    {
      "title": "SQL Injection in User Input Handler",
      "files": ["src/handlers/user.rb:42-55"],
      "priority": "high",
      "finding": "The user input is passed directly to...",
      "context": "This is critical because..."
    }
  ]
}
```

**Extraction Prompt** (in `handbook/prompts/extract-feedback.system.md`):

```markdown
Extract individual actionable findings from this code review report.

For each finding, provide:
- title: Brief descriptive title (max 60 chars)
- files: Array of file paths with line numbers if available
- priority: critical|high|medium|low|info
- finding: The original finding text
- context: Why this matters, impact analysis

Output as JSON array. Group similar findings. Skip purely informational notes.
```

**Error Handling:**
- LLM returns invalid JSON: Retry once, then raise ParseError
- LLM returns empty findings: Return empty array
- LLM timeout: Raise TimeoutError with message

**Edge Cases:**
- Report with no findings: Return empty array
- Very long report: Chunk if needed (> 100k tokens)
- Duplicate findings across reports: Dedupe by title similarity
