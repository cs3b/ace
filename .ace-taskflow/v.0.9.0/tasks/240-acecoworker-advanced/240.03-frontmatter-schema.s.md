# Frontmatter Schema

## Objective

Add an `assign:` frontmatter schema that allows task/workflow files to define assignment configuration, workflow phases, and variables declaratively, plus show assignment hierarchy via `list --tree`.

## Scope of Work

### Included
- Parse `assign:` frontmatter block from `.wf.md` and task files
- Support assignment config (`required`, `parent`)
- Support workflow definition via `preset` or inline `phases`
- Support `variables` block for interpolation (passed through to prepare)
- Parent-child assignment linking
- `ace-assign list --tree` hierarchy view
- Update shared `ux/usage.md` with frontmatter examples

### Excluded
- Full 234.08 schema (context, delegation, gates, report blocks)
- Variable interpolation inside frontmatter (use prepare --var)
- Cross-directory hierarchy discovery
- Automatic child assignment creation

## Technical Approach

### Architecture Pattern
- [ ] Introduce a dedicated atom for parsing assign config from YAML frontmatter
- [ ] Reuse existing prepare/list command plumbing from 240.02
- [ ] Model assignment parent-child relationships via assignment metadata

### Technology Stack
- [ ] Use YAML frontmatter parsing already in `PhaseFileParser`
- [ ] Keep assign config as a plain hash with validation results
- [ ] Extend list rendering with a tree formatter

### Implementation Strategy
- [ ] Implement `AssignConfigParser.parse` returning `config`, `valid`, and `errors`
- [ ] Make prepare command load frontmatter, merge CLI overrides, then emit job.yaml
- [ ] Add `--tree` to list command to render parent-child assignments

## Tool Selection

| Criteria | Option A | Option B | Option C | Selected |
|----------|----------|----------|----------|----------|
| Parse location | New atom | Reuse PhaseFileParser directly | Inline in CLI | Option A |
| Validation | Structured errors | Raise exceptions | No validation | Option A |
| Tree output | CLI formatter | External script | JSON only | Option A |
| Backward compatibility | High | Medium | Low | Option A |
| Maintainability | High | Medium | Low | Option A |

**Selection Rationale:** A dedicated atom isolates parsing and keeps CLI code thin while enabling consistent validation.

## Risk Assessment

### Technical Risks
- **Risk:** Frontmatter parsing conflicts with existing workflow formats
  - **Probability:** Medium
  - **Impact:** Medium
  - **Mitigation:** Only parse `assign:` key; ignore others
  - **Rollback:** Disable frontmatter parse path and rely on CLI flags

- **Risk:** Parent metadata not present for older assignments
  - **Probability:** High
  - **Impact:** Low
  - **Mitigation:** Treat missing parent as root in tree rendering
  - **Rollback:** Fallback to flat list
