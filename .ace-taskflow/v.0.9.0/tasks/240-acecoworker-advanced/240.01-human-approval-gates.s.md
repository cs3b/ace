# Human Approval Gates

## Objective

Pause workflow execution for explicit human approval at critical steps, persisting gate state for resume/decision handling.

## Scope of Work

### Included
- `gate: true` + optional `gate_prompt:` in step frontmatter
- Session paused state persisted in `session.yaml`
- `ace-coworker resume --approve` and `--reject --reason "..."` commands
- Status output shows gate prompt and resume instructions

### Excluded
- Multiple gate types (only human gate)
- Timeout/conditional gates
- Auto-rewind on rejection
- External approvals/webhooks

## Technical Approach

### Architecture Pattern
- [ ] Extend the queue state machine with a session-level paused state
- [ ] Treat gate steps as non-executable markers that block advancement
- [ ] Persist gate metadata in `session.yaml` for resume operations

### Technology Stack
- [ ] Use existing YAML serialization via `Session#to_h` and `SessionManager`
- [ ] Preserve current step file schema with minimal new fields
- [ ] Ensure compatibility with queue scanning and status rendering

### Implementation Strategy
- [ ] Add gate metadata to `Step` model and parsing pipeline
- [ ] Store session `status` and `gate` state with backward-compatible defaults
- [ ] Introduce a `resume` command that clears or rejects gate state

## Tool Selection

| Criteria | Option A | Option B | Option C | Selected |
|----------|----------|----------|----------|----------|
| Persistence | Extend session.yaml | New gate.yaml file | In-memory only | Option A |
| Integration | Centralized | Split state | Volatile | Option A |
| Maintenance | Low | Medium | High risk | Option A |
| Backward compatibility | High | Medium | Low | Option A |
| UX clarity | High | Medium | Low | Option A |

**Selection Rationale:** Extend `session.yaml` because session metadata already drives status selection and is the natural place to persist pause state.

### Dependencies
- [ ] No new external gems required
- [ ] Compatibility verified with existing YAML parsing

## Risk Assessment

### Technical Risks
- **Risk:** Session schema change breaks existing session loads
  - **Probability:** Medium
  - **Impact:** Medium
  - **Mitigation:** Provide defaults for new fields and tolerate unknown keys
  - **Rollback:** Revert schema additions and ignore gate fields

- **Risk:** Paused sessions still show an in_progress step
  - **Probability:** Medium
  - **Impact:** Low
  - **Mitigation:** Gate logic clears in-progress step before pausing
  - **Rollback:** Restore previous status rendering

### Integration Risks
- **Risk:** Status command lacks gate context after multi-session changes
  - **Mitigation:** Use session status and gate data over queue-only view
  - **Monitoring:** Add a status test that includes gate prompt
