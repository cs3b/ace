# Independent Verifier Agent Pattern

## Scope

Introduce a dual-agent pattern where a separate verifier agent independently examines sandbox state after test execution. The verifier has no access to executor metadata — it forms its own judgment from scenario definition and sandbox artifacts. This reduces confirmation bias in agent self-assessment.

## Behavioral Spec

### Architecture

```
┌─────────────┐     ┌───────────┐     ┌──────────────┐
│   Executor   │────▶│  Sandbox  │◀────│   Verifier   │
│    Agent     │     │   State   │     │    Agent     │
└─────────────┘     └───────────┘     └──────────────┘
       │                                      │
       ▼                                      ▼
  executor-report/                     verifier-report/
  - summary.r.md                       - verification.r.md
  - experience.r.md                    - discrepancies.r.md
  - metadata.yml
```

### Verifier Context (What It Sees)

The verifier agent receives ONLY:
- Sandbox directory path (post-execution state)
- `scenario.yml` content
- TC files (the test case definitions)
- Success criteria (from goal-mode TCs) or expected outcomes (from procedural TCs)

The verifier does NOT see:
- Executor's reports (summary.r.md, experience.r.md)
- Executor's metadata (pass/fail counts, timing)
- Executor's observations or reasoning
- Run configuration (provider, model, etc.)

### Verifier Workflow

1. **Receive context**: Sandbox path + scenario + TCs
2. **Examine sandbox**: Inspect files, git state, command outputs, artifacts
3. **Evaluate independently**: For each TC, determine if success criteria are met
4. **Report findings**: Write `verification.r.md` with per-TC assessment
5. **Flag discrepancies**: If verifier disagrees with any expected outcome, document in `discrepancies.r.md`

### Discrepancy Handling

After both agents complete, the orchestrator compares reports:
- **Agreement**: Both pass or both fail → high confidence result
- **Executor PASS / Verifier FAIL**: Potential false positive — flag for review
- **Executor FAIL / Verifier PASS**: Potential false negative — flag for review
- **Both FAIL with different reasons**: Document both failure analyses

### Isolation Mechanism

Use `ace-assign` fork context for verifier isolation:
- Verifier runs as a separate assignment/agent invocation
- No shared memory or conversation context with executor
- Only shared artifact is the sandbox directory (read-only for verifier)

### CLI Interface

```bash
# Run with verification (dual-agent)
ace-test-e2e run ace-lint TS-LINT-001 --verify

# Run suite with verification
ace-test-e2e suite --tags smoke --verify

# Verification is optional — default remains single-agent
ace-test-e2e run ace-lint TS-LINT-001  # no --verify, existing behavior
```

### Opt-In Design

- **Default**: Single-agent execution (existing behavior, unchanged)
- **--verify flag**: Enables dual-agent mode
- Verification doubles LLM cost per test — appropriate for release gates, not pre-commit
- Recommended pairing: `ace-test-e2e suite --verify` for release validation
