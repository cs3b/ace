# Goal-Based Test Case Format

## Scope

Introduce a `mode: goal` option for TC files where the agent plans its own approach to satisfy success criteria, rather than following procedural steps. This evaluates agent capability (problem-solving) rather than instruction-following.

## Behavioral Spec

### TC Frontmatter Extension

```yaml
---
tc-id: TC-001
title: Lint a Ruby file and produce report
mode: goal          # NEW: "goal" or "procedural" (default: procedural)
tags: [smoke]
---
```

- `mode: procedural` (default) — existing behavior, agent follows Steps exactly
- `mode: goal` — agent receives objective + success criteria, plans own approach

### Goal-Mode TC Structure

```markdown
---
tc-id: TC-003
title: Generate commit message from staged changes
mode: goal
tags: [happy-path, use-case:commit]
---

## Objective

Generate an intelligent commit message for staged Ruby file changes using ace-git-commit.

## Available Tools

- `ace-git-commit` — CLI tool for generating commit messages
- `git` — Standard git operations
- Standard shell tools (bash, fish)

## Success Criteria

- [ ] A git commit is created in the sandbox repository
- [ ] The commit message references the nature of the change (not generic "update files")
- [ ] The commit message follows conventional commit format or the repo's commit style
- [ ] `git log --oneline -1` shows the new commit

## Hints

- The sandbox has a git repo with staged changes (from setup)
- ace-git-commit can operate on staged changes with `--staged`
- Check the commit was actually created, not just that the tool ran
```

### Structural Requirements

**Required sections for goal-mode**:
- `## Objective` — 1-3 sentences describing what to accomplish
- `## Success Criteria` — Checkboxes that can be verified against sandbox state
- `## Available Tools` — What the agent can use (tool discovery, not prescription)

**Optional sections**:
- `## Hints` — Non-prescriptive guidance to reduce wasted exploration
- `## Constraints` — Boundaries (e.g., "do not modify files outside sandbox/src/")

**Prohibited in goal-mode**:
- `## Steps` section — contradicts goal-based approach
- Exact commands in success criteria — criteria describe outcomes, not methods

### Execute Workflow Changes

The execute workflow (`execute.wf.md`) must handle both modes:

**Procedural mode** (existing): Execute steps as written, verify outputs
**Goal mode** (new):
1. Read objective and success criteria
2. Agent plans approach using available tools
3. Agent executes plan, adapting as needed
4. Agent self-evaluates against each success criterion
5. Report includes: approach taken, criteria met/unmet, observations

### Result Evaluation

For goal-mode TCs, the result parser evaluates:
- Each success criterion: PASS/FAIL with evidence
- Agent's approach: was it reasonable? (informational, not scored)
- Unexpected findings: side effects, alternative solutions discovered

### Conversion Candidates

Convert 3+ existing procedural TCs to goal-mode as examples:
1. **TS-LINT-001/TC-001** — "Lint a valid file and produce report" → goal: verify lint produces correct output
2. **TS-COMMIT-001/TC-001** — "Generate commit from staged changes" → goal: create meaningful commit
3. **TS-BUNDLE-001/TC-001** — "Load project context" → goal: resolve bundle and verify content

Selection criteria for conversion: TCs where the procedure is straightforward but the outcome is what matters.

## Deliverables

- [ ] `mode` field supported in TC frontmatter (`goal` | `procedural`, default: `procedural`)
- [ ] Goal-mode TC structure documented (Objective, Success Criteria, Available Tools)
- [ ] Execute workflow handles both modes
- [ ] Result parser evaluates success criteria for goal-mode
- [ ] 3+ existing TCs converted to goal-mode as reference examples
- [ ] TC authoring guide updated with goal-mode section

## Acceptance Criteria

- [ ] `mode: procedural` TCs execute identically to current behavior
- [ ] `mode: goal` TCs present objective + criteria to agent without prescriptive steps
- [ ] Goal-mode reports include per-criterion PASS/FAIL evaluation
- [ ] Converted example TCs pass when executed in goal-mode
- [ ] TC parser validates goal-mode structure (rejects goal TCs with ## Steps)
