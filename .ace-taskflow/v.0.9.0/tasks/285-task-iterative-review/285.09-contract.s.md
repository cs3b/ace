# Add verification evidence contract for simulation claims

## Behavioral Specification

### User Experience
- **Input**: Reviewer/implementer completes a task that claims runnable simulation behavior.
- **Process**: Task spec includes a required evidence section showing the command run and artifact proof.
- **Output**: Completion/review decisions are grounded in explicit execution evidence.

### Expected Behavior
- Standardize a required section for simulation/execution-focused tasks:
  - `## Verification Evidence`
- Section format must include:
  - command(s) executed
  - artifact path(s) inspected
  - expected vs observed outcome summary
- Update relevant workflow docs/templates so new tasks include this section by default when behavior claims are executable.
- Integrate with completion/review policy from 285.08/285.10.

### Interface Contract

```markdown
## Verification Evidence

- Command: `mise exec -- ace-taskflow review-next-phase --source <ref> --modes draft,plan --no-writeback --verbose`
- Artifacts: `.cache/ace-taskflow/simulations/<run-id>/synthesis.yml`, `writeback-preview.md`
- Result: [pass/fail + short rationale]
```

### Success Criteria
- [x] Verification evidence section contract is documented in task workflow/template guidance
- [x] Evidence contract is applied to task-285 simulation subtasks as reference implementation
- [x] Completion/review logic can consume evidence section consistently
- [x] Documentation clearly distinguishes live-provider failures from implementation failures

### Validation Questions
- [x] Should evidence be mandatory only for selected task classes (simulation/automation), or globally? -> Selected runnable classes (simulation/automation/runtime execution), not global.
- [x] Should artifact references require existing files at completion time or allow historical evidence links? -> Require existing artifacts at completion by default; allow historical links only when immutable and explicitly justified.

## Scope of Work

- **In scope**: task spec contract, workflow/template updates, reference application to 285 lineage.
- **Out of scope**: provider reliability changes, new runtime commands.

## Deliverables

- Updated task drafting/review guidance for `Verification Evidence`
- Example evidence blocks in task 285 follow-up specs
- Policy notes for handling provider-unavailable outcomes

## Policy Notes

- `Verification Evidence` is required for runnable behavior claims (simulation/automation/runtime), optional for purely descriptive tasks.
- Review/completion checks consume the same evidence fields:
  - command(s)
  - artifact path(s)
  - expected vs observed summary
  - failure classification
- Failure classification must distinguish:
  - `implementation failure`
  - `provider unavailable`
  - `environment issue`
