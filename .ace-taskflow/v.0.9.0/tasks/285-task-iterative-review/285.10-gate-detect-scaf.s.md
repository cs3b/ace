# Harden review gate to detect scaffold-vs-runnable gaps

## Behavioral Specification

### User Experience
- **Input**: Reviewer runs task review workflow for draft tasks claiming runnable behavior.
- **Process**: Review gate checks that runnable claims have explicit execution-path proof and test/evidence mapping.
- **Output**: Task remains draft with `needs_review: true` when runnable proof is missing; only promotable when evidence is complete.

### Expected Behavior
- Extend review checklist to include runnable-path checks:
  - claim exists ("runs simulation", "executes automation", etc.)
  - execution command defined
  - artifact/result proof present
  - failure-mode interpretation documented (implementation failure vs provider outage)
- Add explicit blocking question category for scaffold-vs-runnable gaps.
- Update review output to emit actionable blocking message and required evidence fields.

### Interface Contract
- `task/review` workflow must emit one of:
  - `ready` (runnable proof complete), or
  - `blocked` with `needs_review: true` and missing evidence items
- Blocking reason format includes:
  - missing command proof
  - missing artifact proof
  - missing test/evidence mapping

### Success Criteria
- [x] Review workflow detects scaffold-vs-runnable mismatch and blocks promotion
- [x] Blocking output provides deterministic remediation steps
- [x] Tasks with complete runnable evidence pass review without manual interpretation gaps
- [x] Review checklist updates are documented and testable

### Validation Questions
- [x] Should runnable-claim detection be explicit via task metadata tag, or heuristic via text matching? -> Hybrid: explicit classification preferred, heuristic detection allowed as fallback.
- [x] Should review gate require both test proof and runtime artifact proof, or allow one sufficient evidence path? -> Require command/artifact/result/failure evidence for runnable claims; require explicit claim-to-evidence mapping where test evidence is used when available.

## Scope of Work

- **In scope**: review workflow checklist/rules/output enhancements.
- **Out of scope**: implementing runtime simulation logic itself.

## Deliverables

- Updated `task/review` runnable-proof checklist rules
- Blocking output contract for scaffold-vs-runnable issues
- Reference examples based on task-285 simulation pipeline
