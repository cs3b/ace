# Enforce task completion gate for unresolved Success Criteria

## Behavioral Specification

### User Experience
- **Input**: Developer runs `ace-task done <task-ref>`.
- **Process**: CLI validates required checklist sections in task markdown before allowing completion.
- **Output**: Task is marked done only when required items are resolved, unless explicit override is provided.

### Expected Behavior
- Add completion validation in task done path for required sections:
  - `Success Criteria`
  - `Validation Questions` (configurable policy; default non-blocking)
- If unresolved required items exist:
  - fail completion with actionable summary listing unresolved items
  - preserve task status (no move to archive)
- Add explicit override:
  - `ace-task done <task-ref> --allow-incomplete`
  - override logs warning in output
- Keep existing completion semantics unchanged when checklist is complete.

### Interface Contract

```bash
ace-task done 285.08
ace-task done 285.08 --allow-incomplete
```

Validation failure response must include:
- blocked section name
- unresolved item count
- task path/reference
- override hint (`--allow-incomplete`)

### Success Criteria
- [x] `ace-task done` blocks completion when required Success Criteria are unresolved
- [x] `--allow-incomplete` bypasses gate with explicit warning
- [x] Non-gated tasks remain compatible with existing behavior
- [x] Command and organism tests cover pass/fail/override scenarios

### Validation Questions
- [ ] Should `Validation Questions` be required by default or opt-in via config?
- [ ] Should override usage be persisted in task metadata for auditability?

## Scope of Work

- **In scope**: CLI done flag, validation molecule, task-manager integration, tests.
- **Out of scope**: automatic checklist mutation, retro workflow changes.

## Deliverables

- Completion validation molecule for markdown checklist sections
- `ace-task done` integration with `--allow-incomplete`
- Regression and edge-case test coverage
