# Multiple --subject Flags Usage

## Overview

Allow multiple `--subject` flags on a single `ace-review` command, merging them into a combined ace-context configuration for comprehensive code reviews.

## Key Benefits

- Combine different source types (diffs, files, PRs, tasks) in one review
- Build complex review scopes without verbose YAML
- Reuse presets while customizing the subject scope
- Natural command-line composition: `--subject A --subject B`

## Command Structure

### Basic Pattern

```bash
ace-review --preset <preset-name> --subject <type:value> --subject <type:value> [--auto-execute]
```

### Subject Types

- `diff:RANGE` - Git diff range (e.g., `HEAD~3..HEAD`, `origin/main..HEAD`)
- `files:PATTERN` - File glob pattern (e.g., `lib/**/*.rb`, `docs/**/*.md`)
- `pr:IDENTIFIER` - Pull request (e.g., `123`, `owner/repo#456`)
- `task:REFERENCE` - Task files (e.g., `145`, `task.145.03`)
- Keywords: `staged`, `working` - Git staged/working changes

### Merge Behavior

| Scenario | Input | Merged Configuration |
|----------|-------|---------------------|
| Same type (arrays) | `--subject files:a.rb --subject files:b.rb` | `files: ["a.rb", "b.rb"]` |
| Same type (scalars) | `--subject pr:123 --subject pr:456` | `pr: [123, 456]` |
| Different types | `--subject diff:HEAD~3 --subject files:*.md` | `diffs: ["HEAD~3"], files: ["*.md"]` |
| Mixed | `--subject pr:123 --subject diff:HEAD~1` | `pr: [123], diffs: ["HEAD~1"]` |
| With keywords | `--subject staged --subject files:docs/*.md` | `diffs: [<staged>], files: ["docs/*.md"]` |

**Rule:** Arrays concatenate, scalars become arrays when multiple values provided.

## Usage Scenarios

### Scenario 1: Review Diff + Related Files

**Goal:** Review code changes plus documentation that might need updates

```bash
ace-review --preset code \
  --subject diff:origin/main..HEAD \
  --subject files:docs/**/*.md \
  --auto-execute
```

**Expected behavior:**
- Loads git diff from `origin/main..HEAD`
- Includes all markdown files in `docs/` directory
- Applies code preset's review style to combined content
- Executes review automatically with LLM

**Equivalent YAML (what gets passed to ace-context):**
```yaml
diffs: ["origin/main..HEAD"]
files: ["docs/**/*.md"]
```

### Scenario 2: Multiple File Patterns

**Goal:** Review specific file sets across different directories

```bash
ace-review --preset security \
  --subject files:lib/**/*.rb \
  --subject files:test/**/*_test.rb \
  --subject files:config/*.yml \
  --auto-execute
```

**Expected behavior:**
- Combines three file glob patterns
- Applies security preset to all matched files
- Single review covering lib, test, and config files

**Merged configuration:**
```yaml
files: ["lib/**/*.rb", "test/**/*_test.rb", "config/*.yml"]
```

### Scenario 3: PR + Additional Context

**Goal:** Review a PR plus specific files for broader context

```bash
ace-review --preset code-pr \
  --subject pr:123 \
  --subject files:docs/CHANGELOG.md \
  --auto-execute
```

**Expected behavior:**
- Fetches PR #123 diff via ace-context
- Includes CHANGELOG.md for release context
- Applies PR code review preset

**Merged configuration:**
```yaml
pr: [123]
files: ["docs/CHANGELOG.md"]
```

### Scenario 4: Multiple PRs (Batch Review)

**Goal:** Review multiple related PRs together

```bash
ace-review --preset security \
  --subject pr:123 \
  --subject pr:456 \
  --subject pr:789 \
  --auto-execute
```

**Expected behavior:**
- Fetches diffs for PRs 123, 456, and 789
- Combines all changes into single review
- Applies security analysis to all PRs

**Merged configuration:**
```yaml
pr: [123, 456, 789]
```

### Scenario 5: Staged Changes + Specific Files

**Goal:** Review staged changes plus documentation

```bash
ace-review --preset docs \
  --subject staged \
  --subject files:README.md \
  --auto-execute
```

**Expected behavior:**
- Includes git staged changes (keyword resolved to diff)
- Adds README.md for context
- Applies documentation review preset

**Merged configuration:**
```yaml
diffs: ["<staged-diff-content>"]  # staged keyword resolved
files: ["README.md"]
```

### Scenario 6: Task Files + Additional Diff

**Goal:** Review task directory plus recent changes

```bash
ace-review --preset code \
  --subject task:145.03 \
  --subject diff:HEAD~5..HEAD \
  --auto-execute
```

**Expected behavior:**
- Resolves task 145.03 to its directory files
- Includes last 5 commits of diff
- Single review combining task context and recent changes

**Merged configuration:**
```yaml
files: [".ace-taskflow/v.0.9.0/tasks/145-feat-review-ace/**/*"]  # task resolved
diffs: ["HEAD~5..HEAD"]
```

## Command Reference

### Basic Syntax

```bash
ace-review --subject <type:value> [--subject <type:value>]... [options]
```

### Options

- `--preset NAME` - Review preset to apply (required or set in config)
- `--subject CONFIG` - Subject configuration (repeatable)
- `--auto-execute` - Execute LLM query automatically
- `--output FILE` - Specific output file path
- `--verbose` - Verbose output

### Subject Types

| Type | Format | Example |
|------|--------|---------|
| diff | `diff:RANGE` | `diff:HEAD~3..HEAD` |
| files | `files:PATTERN` | `files:lib/**/*.rb` |
| pr | `pr:IDENTIFIER` | `pr:123`, `pr:owner/repo#456` |
| task | `task:REFERENCE` | `task:145`, `task:145.03` |
| keyword | `staged`, `working` | `staged` |

### Internal Implementation

When you use multiple `--subject` flags:

1. **CLI collection:** Each `--subject` value appended to `@options[:subjects]` array
2. **SubjectExtractor processing:** Detects array input and processes each subject individually
3. **Merge logic:** `deep_merge_arrays` combines subjects:
   - Arrays concatenate: `[a] + [b] = [a, b]`
   - Scalars to array: `a + b = [a, b]`
4. **ace-context delegation:** Final merged config passed as YAML to ace-context
5. **Content resolution:** ace-context loads all sources and aggregates content

## Tips and Best Practices

### Combining Source Types

**Do:** Combine complementary sources
```bash
ace-review --subject diff:HEAD~3 --subject files:docs/*.md  # ✓ Changes + docs
```

**Avoid:** Redundant overlapping sources
```bash
ace-review --subject diff:HEAD~3 --subject diff:HEAD~5..HEAD  # ✗ Overlapping ranges
```

### Order Independence

Subject flags can be specified in any order:
```bash
# These are equivalent:
ace-review --subject files:*.rb --subject diff:HEAD~3
ace-review --subject diff:HEAD~3 --subject files:*.rb
```

### Comma-Separated Alternative

For simple cases, comma separation may be clearer (if supported):
```bash
# Multiple flags
ace-review --subject files:a.rb --subject files:b.rb

# Potential future: comma-separated (same type)
# ace-review --subject files:a.rb,b.rb
```

### Performance Considerations

- Each subject type may trigger external calls (gh CLI for PRs, git commands for diffs)
- Combine subjects thoughtfully to avoid unnecessary overhead
- Use `--verbose` to see what's being loaded

### Troubleshooting

**Issue:** "No code to review" error

**Cause:** No subjects resolved to actual content

**Solution:** Verify subject patterns match files/ranges
```bash
ace-review --subject files:nonexistent/*.rb --verbose  # Shows empty match
```

**Issue:** Unexpected content in review

**Cause:** Subjects merging in unexpected ways

**Solution:** Use `--dry-run` to preview content
```bash
ace-review --subject A --subject B --dry-run  # Preview merged config
```

## Backward Compatibility

### Single Subject (Existing Behavior)

All existing commands continue to work:

```bash
# Single typed subject
ace-review --preset code --subject diff:HEAD~3

# YAML subject
ace-review --preset code --subject 'diffs: ["HEAD~3"], files: ["*.rb"]'

# Keyword subject
ace-review --preset code --subject staged

# Auto-detect git range
ace-review --preset code --subject origin/main..HEAD
```

### Migration from YAML

Instead of complex YAML:
```bash
# Old way (still works)
ace-review --preset code --subject 'diffs: ["HEAD~3"], files: ["docs/*.md"]'

# New way (cleaner)
ace-review --preset code --subject diff:HEAD~3 --subject files:docs/*.md
```
