# Enhance ace-review with Unified Subject Definition (Orchestrator)

## Overview

Enable composable code reviews with unified `--subject type:value` syntax, separating "what to review" (subject) from "how to review" (preset).

## The Core Problem

Users want to reuse a preset's review style but change what they're reviewing. Currently requires verbose YAML: `--subject 'diffs: ["range"]'`

## Solution

Simple `type:value` syntax:
```bash
ace-review --preset security --subject diff:origin/main..HEAD
ace-review --preset code --subject pr:123
ace-review --preset docs --subject files:docs/**/*.md
```

## Architecture Decision

**ace-context handles all content resolution**, including PR diffs. This keeps ace-review's SubjectExtractor simple - it just parses the type and delegates to ace-context.

## Out of Scope (Phase 2)

**Subject Presets** - Composable named subject definitions for reuse across commands.

```bash
# Phase 2 - NOT in scope for this task
ace-review --subject-preset lib-changes
```

This feature will be tracked separately. See: [future idea file or backlog task when created]

## Subtasks

### 145.01 - Add pr: support to ace-context
Add `pr:` as a new source type in ace-context configuration schema.

**Scope:**
- New `pr:` key in ace-context config (alongside files:, diffs:, commands:)
- Integration with `gh` CLI for PR diff fetching
- Support formats: `pr: 123`, `pr: "owner/repo#456"`, `pr: "https://..."`

**Deliverable:** ace-context can load PR content via `pr:` configuration key

### 145.02 - Unified subject resolution in ace-review
Enhance SubjectExtractor to parse `type:value` syntax and delegate content resolution to ace-context.

**Scope:**
- Parse `type:value` format: `diff:`, `pr:`, `files:`, `task:`
- Convert to ace-context configuration
- Backward compatibility with existing YAML/auto-detect
- Error messages with examples

**Deliverable:** `ace-review --subject diff:range` works cleanly

## Terminology

Use "resolution" consistently across all specs:
- "Subject resolution" - parsing and converting `--subject` input
- "Content resolution" - ace-context loading content from sources
- "Task resolution" - finding task path from reference

Avoid: "content aggregation", "subject extraction" (use "subject resolution")
