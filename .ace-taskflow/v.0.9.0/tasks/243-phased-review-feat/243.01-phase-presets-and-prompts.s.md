# 243.01: Phase Presets + Prompts

## Objective

Create three phase-focused review presets for ace-review with corresponding system prompts that guide LLM focus to specific concerns.

## Behavioral Specification

### User Experience

- **Input**: User runs `ace-review --preset correctness` (or quality, polish)
- **Process**: Preset loads phase-specific focus areas, blocking threshold, and system prompt
- **Output**: Review focused on phase-appropriate concerns with severity labels matching threshold

### Preset Configuration

```yaml
# correctness.yml
name: correctness
description: "Phase 1: Make it work - focus on bugs and functionality"
focus:
  - correctness
  - error_handling
  - test_coverage
  - security
ignore:
  - style
  - performance
  - refactoring
blocking_threshold: medium

# quality.yml
name: quality
description: "Phase 2: Make it fast - focus on performance and structure"
focus:
  - performance
  - architecture
  - standards
  - security
ignore:
  - style_minor
  - refactoring_cosmetic
blocking_threshold: high

# polish.yml
name: polish
description: "Phase 3: Make it beautiful - focus on simplicity and elegance"
focus:
  - readability
  - simplification
  - naming
  - documentation
ignore: []
blocking_threshold: none
```

## System Prompt Design

### Correctness Prompt

```markdown
You are reviewing code for CORRECTNESS ONLY.

Focus exclusively on:
- Bugs and logic errors
- Missing functionality vs requirements
- Error handling gaps
- Security vulnerabilities

DO NOT comment on:
- Code style or formatting
- Performance (unless it causes bugs)
- Refactoring opportunities
- Naming conventions

Your findings should answer: "Does this code work correctly?"
```

### Quality Prompt

```markdown
You are reviewing code for QUALITY AND PERFORMANCE.

Assume the code is functionally correct. Focus on:
- Performance issues (N+1 queries, unnecessary allocations, inefficient algorithms)
- Architecture compliance (patterns, layering, dependencies)
- Code standards adherence (project conventions)
- Security best practices
- API design quality

DO NOT comment on:
- Minor style improvements
- Cosmetic refactoring
- Alternative approaches (unless significantly better)

Your findings should answer: "Is this code well-structured and performant?"
```

### Polish Prompt

```markdown
You are reviewing code for POLISH AND ELEGANCE.

Assume the code is correct and well-structured. Focus on:
- Simplification opportunities (reduce complexity)
- Naming improvements (clarity, consistency)
- Dead code removal
- Documentation gaps
- Code duplication
- Readability enhancements

ALL findings in this phase are NON-BLOCKING suggestions.

Your findings should answer: "How can we make this code more elegant?"
```

## Implementation Notes

### Preset Schema

Extend existing preset schema with:
- `focus`: Array of concern categories to emphasize
- `ignore`: Array of concern categories to skip
- `blocking_threshold`: `high` | `medium` | `low` | `none`
- `system_prompt`: Path to phase-specific prompt (optional, defaults to standard)

### Category Mapping

Map focus/ignore categories to concrete guidance:
- `correctness` → bugs, logic errors, missing functionality
- `security` → vulnerabilities, injection, authentication
- `performance` → N+1, allocations, algorithms
- `architecture` → patterns, layering, dependencies
- `standards` → project conventions, naming
- `style` → formatting, whitespace
- `refactoring` → simplification, deduplication
