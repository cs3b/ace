# Phased Review Architecture (Work → Fast → Beautiful)

## Overview

Apply Kent Beck's iterative development principle to code review: "Make it work, make it right, make it fast" becomes a phased review cycle where each phase focuses on specific concerns with appropriate blocking thresholds.

## Problem Statement

Current review approach:
- Single pass tries to catch everything (correctness, performance, style)
- All findings treated equally regardless of phase-appropriateness
- Reviewers get overwhelmed trying to evaluate all dimensions at once

Phased approach:
1. **Phase 1 (Correctness)**: Does it work? Fix bugs first
2. **Phase 2 (Quality)**: Is it well-structured? Performance, standards
3. **Phase 3 (Polish)**: Can we simplify? Non-blocking refinements

## Design: Phase-Focused Review Presets

### Phase 1: Correctness (`preset: correctness`)

**Focal Point**: Does it work? Does it deliver what it promises?

| Reviewer Focus | Skip in This Phase |
|----------------|-------------------|
| Logic errors and bugs | Style/formatting |
| Missing functionality | Performance opts |
| Error handling gaps | Refactoring |
| Security vulnerabilities | Naming improvements |
| Test coverage for critical paths | |

**Blocking Threshold**: `medium` - Medium+ findings block

### Phase 2: Quality (`preset: quality`)

**Focal Point**: Is it well-structured? Follows standards? Performs well?

| Reviewer Focus | Skip in This Phase |
|----------------|-------------------|
| Performance issues (N+1, allocations) | Minor refactoring |
| Architecture compliance (ATOM pattern) | Cosmetic improvements |
| Code standards adherence | Alternative approaches |
| Security best practices | |
| API design quality | |

**Blocking Threshold**: `high` - Only High+ findings block

### Phase 3: Polish (`preset: polish`)

**Focal Point**: Can we simplify? Make it cleaner? More elegant?

| Reviewer Focus | Blocking Threshold |
|----------------|-------------------|
| Simplification opportunities | None (non-blocking) |
| Naming improvements | All suggestions only |
| Dead code removal | |
| Documentation gaps | |
| Code duplication | |
| Readability enhancements | |

## Integration with Task 233

**Phases become reviewer dimensions** in Task 233's Multi-Dimensional Review architecture:

```yaml
reviewers:
  - id: correctness-reviewer
    focus: correctness
    phase: 1
    trust_level: high
    blocking_threshold: medium
    prompt: phase/correctness.system.md

  - id: quality-reviewer
    focus: quality
    phase: 2
    trust_level: high
    blocking_threshold: high
    prompt: phase/quality.system.md

  - id: polish-reviewer
    focus: polish
    phase: 3
    trust_level: medium
    blocking_threshold: none
    prompt: phase/polish.system.md
```

## Design Decisions

1. **Separate LLM reviews per phase** - Each phase gets dedicated prompt optimized for that focus
2. **Phases ARE dimensions** in Task 233's Multi-Dimensional Review architecture
3. **ace-coworker handles cycle logic** (not ace-review):
   - ace-review = one review per call (stays simple)
   - ace-coworker = orchestrates cycles, decides when to continue
4. **Configurable threshold for cycle continuation**:
   - `max_cycles`: Hard limit per phase
   - `min_improvement`: Stop if findings don't decrease significantly

## Out of Scope

- Real-time phase switching during review (phases are sequential)
- Automated phase progression without configuration (always explicit)
- Heuristic-based phase skipping (user configures which phases to run)
