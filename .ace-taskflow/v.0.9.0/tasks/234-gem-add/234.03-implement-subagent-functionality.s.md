# Subagent support

## Objective

Enable workflow instructions to spawn subagents via Task tool for isolated execution.

## Scope

Enhance workflow instructions to support subagent spawning:
- Instructions can include Task tool invocation syntax
- Agent follows instructions which may include spawning subagents
- Context isolation between parent and subagent

## Deliverables

- [ ] Document subagent instruction pattern
- [ ] Example workflow using subagent spawning
- [ ] Guide for when to use subagents vs direct execution

**Workflow example:**
```yaml
steps:
  - name: implement
    instructions: |
      Use Task tool with subagent_type=general-purpose:
      "Work on task $task following wfi://work-on-task"

  - name: release
    instructions: ace-bundle wfi://draft-release
```

## Context Isolation Definition

"Context isolation" means:
1. **Separate session scope**: Subagent operates with its own instruction context, not the parent's full workflow state
2. **No shared working directory requirements**: Subagent may use same working directory but does not inherit parent's internal state
3. **Report boundary**: Subagent output is captured as a discrete report file, not merged into parent's session log

**How subagent reports are captured:**
1. Parent workflow instruction tells agent to spawn Task tool with specific prompt
2. Task tool returns result to agent
3. Agent includes relevant output in its report via `ace-coworker report`
4. Parent session log records: `{"event": "subagent_completed", "step": "implement", "summary": "..."}`

**Not in scope for isolation:**
- File system isolation (subagent can modify same files)
- Environment variable isolation
- Git state isolation

## Acceptance Criteria

- [ ] Instructions can include Task tool spawning
- [ ] Subagent reports captured correctly via parent's `ace-coworker report`
- [ ] Context isolation demonstrated (subagent has own instruction context)
- [ ] Documentation explains when to use subagents
