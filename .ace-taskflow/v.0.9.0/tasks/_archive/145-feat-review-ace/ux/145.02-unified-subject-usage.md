# Unified Subject Syntax Usage Guide

## Overview

The unified subject syntax provides a clean, type-safe way to specify what code to review in `ace-review`. It replaces complex YAML configurations with simple `type:value` syntax while maintaining full backward compatibility.

**Available Subject Types:**
- `diff:` - Git diff ranges
- `pr:` - GitHub pull requests
- `files:` - File patterns and globs
- `task:` - Task references (includes all files in task directory)
- Keywords: `staged`, `working` (convenience shortcuts)

**Benefits:**
- Cleaner command-line syntax (no YAML quoting)
- Explicit intent (type prefix shows what you're reviewing)
- Consistent resolution (all types flow through ace-context)
- Type-safe parsing (validates format before processing)

## Command Types

All examples use the `ace-review` bash CLI command (not Claude Code commands).

## Command Structure

```bash
ace-review --preset <preset> --subject <type:value>
```

**Subject Format:**
- Type prefix: `diff:`, `pr:`, `files:`, `task:`
- Value: Type-specific identifier
- No spaces around colon
- Quote if value contains special characters

## Usage Scenarios

### Scenario 1: Review Git Diff Range

**Goal:** Review changes between two git commits or branches

```bash
# Review changes since origin/main
ace-review --preset code --subject diff:origin/main..HEAD

# Review last 5 commits
ace-review --preset code --subject diff:HEAD~5..HEAD

# Review specific commit range
ace-review --preset security --subject diff:abc123..def456

# Review branch changes
ace-review --preset code-pr --subject diff:feature-branch..main
```

**Expected Output:**
- ace-context fetches diff via git
- Diff wrapped in `<diff source="diff:range">` tags
- LLM analyzes changes according to preset

### Scenario 2: Review GitHub Pull Request

**Goal:** Review PR diff content (not full PR metadata, just the changes)

```bash
# Review PR by number (current repo)
ace-review --preset code --subject pr:123

# Review PR from another repo
ace-review --preset security --subject pr:owner/repo#456

# Review PR from full URL
ace-review --preset docs --subject pr:https://github.com/owner/repo/pull/789
```

**Expected Output:**
- ace-context delegates to task 145.01's PR fetcher
- PR diff wrapped in `<diff source="pr:123">` tags
- LLM analyzes PR changes according to preset

**Note:** This is subject-only mode (diff content via ace-context). For full PR mode with metadata and comments, use `--pr` flag instead:
```bash
# Full PR mode (not using subject syntax)
ace-review --pr 123 --preset code-pr --auto-execute
```

### Scenario 3: Review Specific Files

**Goal:** Review specific files or file patterns

```bash
# Review all Ruby files
ace-review --preset code --subject files:lib/**/*.rb

# Review multiple file patterns (comma-separated)
ace-review --preset docs --subject files:docs/**/*.md,README.md

# Review specific file
ace-review --preset security --subject files:config/secrets.yml

# Review all test files
ace-review --preset code --subject files:test/**/*_test.rb
```

**Expected Output:**
- ace-context reads files matching pattern
- Files wrapped in `<file path="...">` tags
- LLM analyzes file contents according to preset

### Scenario 4: Review Task Files

**Goal:** Review all files associated with a task

```bash
# Review main task files
ace-review --preset spec --subject task:145

# Review subtask files (ONLY subtask directory, not parent)
ace-review --preset code --subject task:145.02

# Review with full task ID
ace-review --preset comprehensive --subject task:v.0.9.0+145.01
```

**Expected Output:**
- ace-taskflow resolves task to directory path
- Glob pattern: `{task_dir}/**/*` for all files in task directory
- For subtasks: ONLY includes subtask directory (e.g., `145.02*/**/*`)
- Files wrapped in `<file path="...">` tags
- LLM analyzes task implementation according to preset

**Subtask Behavior:**
When reviewing `task:145.02`:
- ✅ Includes: `.ace-taskflow/v.0.9.0/tasks/145-feat-review-ace/145.02*/**/*`
- ❌ Excludes: Parent task files, sibling subtasks
- Glob pattern: `{subtask_dir}/**/*` (only files within subtask directory)

### Scenario 5: Keyword Shortcuts (Existing Behavior)

**Goal:** Review staged or working changes using convenience keywords

```bash
# Review staged changes (ready to commit)
ace-review --preset code --subject staged

# Review unstaged changes (working directory)
ace-review --preset code --subject working

# Case insensitive
ace-review --preset code --subject STAGED
```

**Expected Output:**
- `staged` → ace-context diffs config: `{ "diffs" => ["--staged"] }`
- `working` → ace-context diffs config: `{ "diffs" => [""] }` (unstaged changes)
- Consistent with typed subjects (flows through ace-context)

### Scenario 6: Backward Compatibility (YAML Syntax)

**Goal:** Use existing YAML syntax for complex configurations

```bash
# Legacy YAML format (still works)
ace-review --subject 'diffs: ["origin/main..HEAD"]' --preset code

# Complex YAML config
ace-review --subject '{
  "diffs": ["HEAD~5..HEAD"],
  "files": ["docs/**/*.md"]
}' --preset comprehensive

# Auto-detected git range (no type prefix)
ace-review --subject HEAD~1..HEAD --preset code

# Auto-detected file pattern
ace-review --subject lib/**/*.rb --preset code
```

**Expected Output:**
- YAML parsed and passed to ace-context
- Auto-detection based on pattern heuristics
- Same behavior as before upgrade

## Command Reference

### Subject Type: diff:

**Syntax:** `diff:<git-range>`

**Parameters:**
- `<git-range>`: Any valid git diff range (commits, branches, HEAD~N, etc.)

**Internal Implementation:**
- Converts to: `{ "diffs" => ["<git-range>"] }`
- ace-context uses GitExtractor to fetch diff
- Output wrapped in `<diff source="diff:range">` tags

**Examples:**
```bash
diff:origin/main..HEAD
diff:HEAD~5..HEAD
diff:abc123..def456
diff:feature-branch..main
```

### Subject Type: pr:

**Syntax:** `pr:<identifier>`

**Parameters:**
- `<identifier>`: PR number, qualified reference, or URL
  - Number: `123`
  - Qualified: `owner/repo#456`
  - URL: `https://github.com/owner/repo/pull/789`

**Internal Implementation:**
- Converts to: `{ "pr" => "<identifier>" }`
- ace-context delegates to task 145.01 PR fetcher
- Uses `gh pr diff` command
- Output wrapped in `<diff source="pr:123">` tags

**Examples:**
```bash
pr:123
pr:owner/repo#456
pr:https://github.com/owner/repo/pull/789
```

**Error Cases:**
- `gh` CLI not installed → Clear error with install instructions
- Not authenticated → Suggests `gh auth login`
- PR not found → Error message with PR identifier

### Subject Type: files:

**Syntax:** `files:<pattern>[,<pattern>...]`

**Parameters:**
- `<pattern>`: File path or glob pattern
- Multiple patterns: Comma-separated (no spaces)

**Internal Implementation:**
- Splits on commas: `pattern.split(",").map(&:strip)`
- Converts to: `{ "files" => [array of patterns] }`
- ace-context reads files matching patterns
- Output wrapped in `<file path="...">` tags

**Examples:**
```bash
files:lib/**/*.rb
files:docs/**/*.md,README.md
files:config/secrets.yml
files:test/**/*_test.rb,lib/**/*.rb
```

### Subject Type: task:

**Syntax:** `task:<reference>`

**Parameters:**
- `<reference>`: Task identifier (any ace-taskflow format)
  - Number: `145`
  - Prefix: `task.145`
  - Full ID: `v.0.9.0+145`
  - Subtask: `145.02`

**Internal Implementation:**
- Calls `ace-taskflow task <reference> --path` to resolve task directory
- For subtasks: Uses only subtask directory (not parent)
- Converts to: `{ "files" => ["{task_dir}/**/*"] }`
- ace-context reads all files in task directory
- Output wrapped in `<file path="...">` tags

**Examples:**
```bash
task:145
task:145.02
task:task.145
task:v.0.9.0+145.01
```

**Error Cases:**
- ace-taskflow not installed → Error with installation instructions
- Task not found → Error message with task reference
- Task has no path → Error message (check task status)

### Keywords: staged, working

**Syntax:** `staged` or `working`

**Parameters:** None (keywords only)

**Internal Implementation:**
- `staged` → `{ "diffs" => ["--staged"] }`
- `working` → `{ "diffs" => [""] }` (unstaged changes)
- Both flow through ace-context GitExtractor
- Output wrapped in `<diff source="git:staged">` or `<diff source="git:working">` tags

**Examples:**
```bash
staged
working
STAGED  # Case insensitive
```

## Parsing Precedence

When SubjectExtractor receives input, it tries these patterns in order (first match wins):

1. **Typed subjects** (`type:value`) - Highest priority
   - Example: `diff:HEAD~1..HEAD` → diff type
   - Example: `pr:123` → pr type
   - Example: `files:lib/**/*.rb` → files type
   - Example: `task:145` → task type

2. **Keywords** (`staged`, `working`) - Convenience shortcuts
   - Example: `staged` → staged keyword
   - Example: `working` → working keyword

3. **YAML detection** - Starts with `{` or contains valid YAML keys
   - Example: `diffs: ["HEAD..main"]` → YAML parsed
   - Example: `{ "files": ["lib/**/*.rb"] }` → YAML parsed

4. **Auto-detect** - Git range patterns, file globs
   - Example: `HEAD~5..HEAD` → detected as git range
   - Example: `lib/**/*.rb` → detected as file pattern

## Tips and Best Practices

### Use Explicit Types for Clarity

**Good:**
```bash
ace-review --subject diff:HEAD~1..HEAD    # Clear intent
ace-review --subject files:lib/**/*.rb    # Explicit type
```

**Acceptable (auto-detect):**
```bash
ace-review --subject HEAD~1..HEAD         # Works, but less clear
ace-review --subject lib/**/*.rb          # Works, but ambiguous
```

### Quote Values with Special Characters

```bash
# Quote if value contains spaces or special shell characters
ace-review --subject "pr:https://github.com/owner/repo/pull/123"
ace-review --subject "files:docs/**/*.md,README.md"
```

### Combine Subject with Context

```bash
# Subject: what to review
# Context: background information
ace-review \
  --subject diff:HEAD~5..HEAD \
  --context 'presets: [project]' \
  --preset code
```

### Use Task Reviews for Focused Analysis

```bash
# Review only subtask files (not entire parent task)
ace-review --subject task:145.02 --preset spec

# Save review back to task directory
ace-review --subject task:145.02 --task 145.02 --preset code --auto-execute
```

## Troubleshooting

### "Unknown subject type" Error

**Error:**
```
Error: Unknown subject type 'commit'.

Supported types: diff, pr, files, task
Did you mean: diff:abc123 (for commit range)?
```

**Solution:** Use correct type prefix from supported list

### "Invalid subject format" Error

**Error:**
```
Error: Invalid subject format. Expected 'type:value' or keyword.

Valid formats:
  --subject diff:<range>     Git diff range
  --subject pr:<number>      GitHub PR
  --subject files:<pattern>  File pattern
  --subject task:<ref>       Task reference
  --subject staged           Staged changes
  --subject working          Unstaged changes
```

**Solution:** Check format matches `type:value` pattern

### "Task not found" Error

**Error:**
```
Error: Task '145.99' not found. Run `ace-taskflow task 145.99` for details.
```

**Solution:** Verify task exists with `ace-taskflow task <ref>`

### ace-context Extraction Failed

**Warning (non-fatal):**
```
ace-context extraction failed: <error message>
```

**What happens:** SubjectExtractor returns empty string, review continues with empty subject

**Solutions:**
- Check ace-context is installed: `gem list ace-context`
- Verify syntax: Test config with `ace-context -f config.yml`
- Enable debug mode: `ACE_REVIEW_DEBUG=1 ace-review ...`

## Migration Notes

### From `--pr` Flag to `--subject pr:`

**Old (Full PR Mode):**
```bash
ace-review --pr 123 --preset code-pr --auto-execute
```

**New (Subject-Only Mode):**
```bash
ace-review --subject pr:123 --preset code --auto-execute
```

**Key Difference:**
- `--pr` flag: Full PR mode (diff + metadata + comments via GhPrFetcher)
- `--subject pr:`: Subject-only mode (diff content only via ace-context)

**When to use each:**
- Use `--pr` for comprehensive PR reviews (recommended)
- Use `--subject pr:` when you only need diff content in a multi-source review

### From YAML to Typed Syntax

**Old (YAML):**
```bash
ace-review --subject 'diffs: ["origin/main..HEAD"]' --preset code
```

**New (Typed):**
```bash
ace-review --subject diff:origin/main..HEAD --preset code
```

**Migration:**
1. Identify subject type (diffs → diff, files → files, pr → pr)
2. Use type prefix with colon
3. Remove YAML array syntax for single values
4. Use comma-separated values for multiple files

**Both still work** - choose based on complexity:
- Simple cases: Use typed syntax
- Complex multi-source: Use YAML

## Examples by Use Case

### Daily Development Workflow

```bash
# Review staged changes before commit
ace-review --subject staged --preset code

# Review working changes (unstaged)
ace-review --subject working --preset code

# Review last commit
ace-review --subject diff:HEAD~1..HEAD --preset code
```

### Pull Request Reviews

```bash
# Review current branch vs main
ace-review --subject diff:origin/main..HEAD --preset code-pr

# Review PR diff only
ace-review --subject pr:123 --preset code

# Full PR review (metadata + comments)
ace-review --pr 123 --preset code-pr --auto-execute
```

### Task-Based Reviews

```bash
# Review task implementation
ace-review --subject task:145 --preset spec --auto-execute

# Review subtask files only
ace-review --subject task:145.02 --preset code

# Save review to task directory
ace-review --subject task:145.02 --task 145.02 --preset code --auto-execute
```

### Security Reviews

```bash
# Review security-sensitive files
ace-review --subject files:config/**/*.yml,lib/auth/**/*.rb --preset security

# Review PR for security issues
ace-review --pr 123 --preset security --auto-execute

# Review recent changes for security
ace-review --subject diff:origin/main..HEAD --preset security
```
