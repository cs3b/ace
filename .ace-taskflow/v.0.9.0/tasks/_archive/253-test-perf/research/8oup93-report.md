## 1. The Core Problem: The "Subprocess Leak"

In Task 251 and 252, we identified that the primary source of slowness in our "Fast Loop" (Unit/Integration tests) was the accidental execution of real shell commands.

### Key Finding: Stubbing the Inner implementation is NOT enough.
If code follows this pattern:
```ruby
def run(cmd)
  return unless available? # system("cmd --version")
  execute(cmd)           # Open3.capture3
end
```
Stubbing `execute` still allows `available?` to trigger a subprocess call (~500ms).

### Solution: "Stub the Boundary"
We must stub the outermost check that leads to I/O.
```ruby
# Correct Stubbing
Runner.stub(:available?, true) do
  Runner.stub(:execute, mock_result) do
    # 60ms loop instead of 600ms
  end
end
```

## 2. Fast Loop vs. Slow Loop Architecture

We have formalized the separation of testing responsibilities:

| Loop | Layer | Speed Goal | Rules |
|------|-------|------------|-------|
| **Fast** | Unit/Atoms | < 10ms | Zero I/O, Mock all collaborators. |
| **Fast** | Integration/Molecules | < 100ms | Stubbed I/O, Test component wiring. |
| **Slow** | E2E/Systems | Seconds | Real CLI, Sandboxed FS, Real 3rd party APIs. |

## 3. Avoiding "Testing the Mock"

### The Principle
- **Stub for Data**: Use to provide return values (e.g., config, file content). Focus on testing how the subject *uses* that data.
- **Mock for Interaction**: Use `Minitest::Mock` expectations only for side-effect-only methods (e.g., `Notifier.send`, `Git.commit`).

### Red Flag
If a test breaks when you rename a private method, you are testing implementation details (the mock), not the behavior.

## 4. Automated Verification

We have implemented the `verify-test-suite` workflow which uses `ace-test --profile 10` to enforce the **100ms Rule**:
- Any unit/integration test taking > 100ms is treated as a bug (a "mocking leak").
- Remediation involves identifying the I/O call and moving the real execution to the E2E layer while stubbing it in the unit layer.

## 5. Integrated Recommendations (Extended Strategy)

Based on cross-analysis with concurrent research (`8oup7s`, `8oupdg`), we are expanding the strategy to include:

### 5.1 The Test Responsibility Map
We will adopt the **Test Responsibility Map** (`8oupdg`) as a mandatory planning artifact.
- **Rule**: Every behavior must be assigned to the *lowest possible layer*.
- **Prevention**: Explicitly prevents "E2E Creep" (testing edge cases in slow loops).

### 5.2 "Zombie Mock" Hunting
We identified **Zombie Mocks** (`8oup7s`) as a critical regression risk.
- **Definition**: Mocks that stub methods no longer called, while the test secretly passes via real IO.
- **Detection**: Added to `verify-test-suite`: "If a 'mocked' test > 100ms, break the mock. If it still passes, it's a Zombie."

### 5.3 Composite Helpers
We will standardize on **Composite Helpers** (`8oup7s`) to improve maintainability.
- **Pattern**: `with_mock_environment(repo: true)` instead of 6 levels of `mock.expect`.

### 5.4 Automated Gates
Manual verification is insufficient. We recommend adding a **Test Performance Gate** script:
- `ace-test --profile 20 | awk '$NF > 0.1 { fail=1 }'` to fail CI on slow unit tests.
