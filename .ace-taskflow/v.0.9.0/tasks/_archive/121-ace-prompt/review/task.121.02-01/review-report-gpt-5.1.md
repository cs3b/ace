### Deep Diff Analysis

- **Intent (ace-git-worktree)**: Centralize hierarchical task ID handling via `TaskIDExtractor`, add `TaskPusher` to push task metadata commits, and wire these into the worktree orchestration so parent tasks and subtasks are handled distinctly (`ace-git-worktree/lib/ace/git/worktree/atoms/task_id_extractor.rb:1`, `.../molecules/task_pusher.rb:1`, `.../organisms/task_worktree_orchestrator.rb:25`).
- **Impact (ace-git-worktree)**: Fixes real bugs where operations on `121.01` affected `121`; introduces new behavior where task worktree creation can also push to a remote by default, and multiple components now depend on `TaskIDExtractor` and `TaskManager` (organism) instead of local regexes and `TaskLoader`.
- **Alternatives (ace-git-worktree)**: Integration with `ace-taskflow` could eventually be funneled through a dedicated integration layer (the proposed `i10t` modules) rather than calling organisms directly from molecules, and `auto_push_task` could be opt‑in instead of defaulting on.
- **Intent (ace-prompt gem)**: Introduce a new CLI gem providing a prompt workspace with archiving, plus setup/reset commands driven by templates and `tmpl://` resolution (`ace-prompt/lib/ace/prompt.rb:1`, `ace-prompt/lib/ace/prompt/cli.rb:1`).
- **Impact (ace-prompt)**: Adds a new runtime dependency to the mono-repo (`Gemfile:14`, `Gemfile.lock`), new CLI (`bin/ace-prompt`, `ace-prompt/exe/ace-prompt:1`), ATOM-structured implementation, and a significant number of tests and docs specific to prompt workflows.
- **Alternatives (ace-prompt)**: Template resolution could be made fully configuration-driven (through `ace-support-core` config cascade) instead of hardcoding `tmpl://ace-prompt/base-prompt`, and TemplateResolver could avoid shelling out entirely by delegating to an `ace-nav` Ruby API when available.
- **Intent (task/idea/retro docs)**: Capture the subtask workflow learnings, multi-model review synthesis, and future architectural ideas (integration modules, unified review subjects/presets) under `.ace-taskflow/v.0.9.0`, informing future design rather than affecting runtime.
- **Impact (task/idea docs)**: Clarifies workflows and future feature direction for `ace-review` and integration patterns; no direct code impact but helpful context for maintainers and future tasks.

### Code Quality Assessment

- **Maintainability (TaskIDExtractor and usages)**: Extracting ID logic into `Ace::Git::Worktree::Atoms::TaskIDExtractor` is a strong win; its API (`extract`, `normalize`) is clear, and its reuse across models, molecules, and organisms eliminates a lot of ad‑hoc regex handling (`ace-git-worktree/lib/ace/git/worktree/atoms/task_id_extractor.rb:1`, `.../models/worktree_config.rb:399`, `.../commands/remove_command.rb:483`, `.../organisms/task_worktree_orchestrator.rb:349`).
- **Potential robustness improvement (TaskIDExtractor.normalize)**: The fallback `elsif match = ref.match(/\b(\d{3})\b/)` may match incidental 3‑digit numbers inside longer strings (`ace-git-worktree/lib/ace/git/worktree/atoms/task_id_extractor.rb:82`); anchoring to `\A(\d{3})\z` would better reflect ACE task ID semantics and avoid accidental matches.
- **Behavioral complexity (TaskWorktreeOrchestrator)**: The orchestration method has grown to handle status update, metadata injection, commit, optional push, worktree creation, and hooks (`ace-git-worktree/lib/ace/git/worktree/organisms/task_worktree_orchestrator.rb:82`); the step recording is explicit and readable, but this class is approaching high cognitive load and may benefit from extracting smaller helper objects for “task status + metadata + push” and “worktree creation + hooks”.
- **Default behavior risk (auto_push_task)**: `WorktreeConfig` introduces `auto_push_task?` and `push_remote`, defaulting `auto_push_task` to true (`ace-git-worktree/lib/ace/git/worktree/models/worktree_config.rb:33–41, 115–123`). This is clear in code but surprising as a default (automatic network side effects) and is worth revisiting or calling out prominently in documentation.
- **TemplateResolver correctness and robustness**: Overall structure is reasonable, but `resolve_via_ace_nav` uses `uri.shellescape` without requiring `shellwords`, and rescues any error to return `nil` (`ace-prompt/lib/ace/prompt/molecules/template_resolver.rb:60`). This silently disables `ace-nav` resolution when `ace-nav` is installed and `shellescape` is missing; adding `require "shellwords"` and using `Shellwords.shellescape(uri)` would make behavior explicit and robust.
- **PromptInitializer/TemplateManager clarity**: These pieces are well‑factored and have clear, small methods; error handling returns structured result hashes with `success`/`error` and is easy to consume (`ace-prompt/lib/ace/prompt/molecules/template_manager.rb:11`, `ace-prompt/lib/ace/prompt/organisms/prompt_initializer.rb:22`). The use of `Time.now` inside `TemplateManager.archive_file` is acceptable but makes deterministic testing slightly harder; the tests currently check only the shape of filenames, which is sufficient.
- **CLI implementation (ace-prompt)**: The Thor-based CLI now correctly returns status codes and leaves `exit` to the executable wrapper (`ace-prompt/lib/ace/prompt/cli.rb:7`, `ace-prompt/exe/ace-prompt:5`); option handling is idiomatic, and error paths consistently return `1`. Complexity is moderate but well‑structured via dedicated organisms/molecules.

### Architectural Analysis

- **ATOM compliance (ace-git-worktree)**: The new `TaskIDExtractor` atom and `TaskPusher` molecule are placed correctly in `atoms/` and `molecules/`, and their usage by organisms (`TaskWorktreeOrchestrator`, `WorktreeManager`, etc.) respects ATOM layering (`ace-git-worktree/lib/ace/git/worktree/atoms/task_id_extractor.rb:1`, `.../molecules/task_pusher.rb:1`, `.../organisms/task_worktree_orchestrator.rb:25`).
- **Cross-gem interactions**: `TaskFetcher` and `TaskStatusUpdater` now depend on `Ace::Taskflow::Organisms::TaskManager` (organism) rather than `TaskLoader` (molecule) (`ace-git-worktree/lib/ace/git/worktree/molecules/task_fetcher.rb:23`, `.../molecules/task_status_updater.rb:205`). This is architecturally cleaner in terms of delegating to a higher‑level interface, but it also means a molecule is now coupled to an organism in another gem; the newly added idea document about standard integration modules (`.ace-taskflow/v.0.9.0/ideas/20251128-110638...i10t...s.md`) points to a more principled future direction.
- **Push behavior in orchestrator**: The orchestrator’s push step currently calls `@task_pusher.push` using whatever `current_branch` reports (`ace-git-worktree/lib/ace/git/worktree/organisms/task_worktree_orchestrator.rb:437`). In most workflows this will be the task worktree branch, but if commands are invoked from an unexpected context, there is a risk of pushing the wrong branch; passing the specific branch name resolved earlier into `push_task_changes` would be safer.
- **ace-prompt ATOM structure**: The new gem follows the documented ATOM layout: pure atoms (`TimestampGenerator`, `ContentHasher`), I/O‑focused molecules (`PromptReader`, `PromptArchiver`, `TemplateResolver`, `TemplateManager`), higher‑level orchestrators (`PromptProcessor`, `PromptInitializer`), and a thin CLI (`ace-prompt/lib/ace/prompt/cli.rb:1`). This aligns well with the architecture docs in `docs/architecture.md`.
- **Configuration and path handling**: For archive/process flows, `ace-prompt` correctly uses `Ace::Core::Molecules::ProjectRootFinder` to locate project roots (`ace-prompt/lib/ace/prompt/molecules/prompt_archiver.rb:19`, `.../prompt_reader.rb:12`), while setup/reset intentionally operate in a user‑global directory (`Dir.home`), clearly separated in `PromptInitializer` (`ace-prompt/lib/ace/prompt/organisms/prompt_initializer.rb:8–13`). This separation is sensible and explicit.
- **Integration pattern alignment**: The `.ace.example/nav/protocols/tmpl-sources/ace-prompt.yml` file and the bundled `handbook/templates/base-prompt.template.md` demonstrate good alignment with the mono-repo’s protocol and template patterns, even though the proposed `i10t` modules are not yet implemented.

### Security Review

- **Command execution and shell usage (TemplateResolver)**: `TemplateResolver` uses backticks and `which` to locate and call `ace-nav`, but the only variable injected into the shell is the URI, and it’s intended to be passed through `shellescape` (`ace-prompt/lib/ace/prompt/molecules/template_resolver.rb:60, 73`). Because `shellescape` is currently undefined and any error in `resolve_via_ace_nav` is rescued and treated as “no result”, `ace-nav` execution is effectively disabled rather than becoming a security risk; nonetheless, adding `require "shellwords"` and using `Shellwords.shellescape(uri)` would both restore the desired behavior and make command injection protection explicit.
- **Open3 usage (TaskFetcher)**: `TaskFetcher` still shells out via `Open3.capture3` for the CLI fallback, but arguments are passed as an array (`cmd = ["bundle", "exec", "ace-taskflow", "task", "show", ...]`), which avoids shell interpolation (`ace-git-worktree/lib/ace/git/worktree/molecules/task_fetcher.rb:130`); `valid_task_reference?` also restricts characters to a safe subset (`task_fetcher.rb:103–115`), reducing risk.
- **File system operations (ace-prompt)**: Prompt archiving, template copying, and path handling are all scoped to either the project root or a controlled `~/.cache/ace-prompt` directory (`ace-prompt/lib/ace/prompt/molecules/prompt_archiver.rb:19`, `ace-prompt/lib/ace/prompt/organisms/prompt_initializer.rb:10`). There are no externally supplied arbitrary paths being used without validation, so the surface for directory traversal or unexpected overwrite is minimal.
- **Network operations (ace-git-worktree)**: The new `TaskPusher` molecule wraps `git push` via `GitCommand` (`ace-git-worktree/lib/ace/git/worktree/molecules/task_pusher.rb:23`). While this does perform network operations, inputs are either defaults (`origin`/current branch) or CLI options parsed by `create_command`, not raw user strings passed into a shell, so injection risks are low; the main security concern here is the behavioral default of auto‑pushing, not injection.

### Refactoring Opportunities

- **Centralize branch selection for pushes**: Change `TaskWorktreeOrchestrator#push_task_changes` to accept and use the specific branch name intended for the worktree, rather than relying on `TaskPusher`’s `current_branch` helper (`ace-git-worktree/lib/ace/git/worktree/organisms/task_worktree_orchestrator.rb:437`). This will make behavior more deterministic and easier to test.
- **Adjust `auto_push_task` default and docs**: Consider defaulting `auto_push_task` to false in `WorktreeConfig` (`ace-git-worktree/lib/ace/git/worktree/models/worktree_config.rb:35`) and documenting the push behavior and configuration in a README section; this would better align with the principle of least surprise while still supporting automation for users who opt in.
- **Tighten TaskIDExtractor fallback regex**: Update the fallback branch in `TaskIDExtractor.normalize` to anchor bare numeric IDs (`/\A(\d{3})\z/`) and possibly log when regex fallback is used instead of `TaskReferenceParser` (`ace-git-worktree/lib/ace/git/worktree/atoms/task_id_extractor.rb:82`). This would reduce accidental matches and improve observability of cross‑gem parsing failures.
- **Harden TemplateResolver shell integration**: Add `require "shellwords"` and use `Shellwords.shellescape` for URIs in `resolve_via_ace_nav`, and consider factoring the backtick call into a small helper that can be stubbed in tests (`ace-prompt/lib/ace/prompt/molecules/template_resolver.rb:60–73`). This will make the integration both safer and easier to test.
- **Consider a future integration layer (`i10t`)**: As the ideas document suggests, moving all cross‑gem and external‑tool integrations (e.g., `TaskManager`, `ace-taskflow` CLI, `ace-nav`) into a dedicated integration module per gem would further clarify boundaries, make test doubles easier to manage, and reduce the architectural tension of molecules calling organisms across gem boundaries.
