# Create ace-test package and consolidate testing documentation

## Behavioral Specification

### User Experience

- **Input**: Developer runs `ace-bundle project` or `ace-nav guide://testing-*`
- **Process**: Testing knowledge is discoverable via protocols; no 1104-line file in bundle
- **Output**: Lean bundle (~1000 lines), testing guides accessible via `ace-test` package

### Expected Behavior

**Current State Problem**:
- `docs/testing-patterns.md` is 1104 lines (52% of 2135-line bundle)
- Testing workflows scattered in `ace-taskflow` (create-test-cases, fix-tests)
- Testing guides scattered in `ace-test-runner`, `ace-docs`
- No single source of truth for testing best practices

**Desired State**:
1. New `ace-test` package exists (similar structure to `ace-handbook`)
2. All testing knowledge consolidated in one location
3. Testing resources discoverable via `guide://testing-*`, `agent://test*`, `wfi://create-test-cases` protocols
4. `docs/testing-patterns.md` deleted entirely
5. Project bundle reduced from 2135 to <1000 lines (~50% reduction)

### Interface Contract

**Protocol Discovery**:
```bash
# Discover all testing guides
ace-nav guide:/// | grep test
# Expected output:
# guide://testing-philosophy
# guide://test-organization
# guide://mocking-patterns
# guide://test-performance
# guide://testable-code-patterns
# guide://testing-tdd-cycle
# guide://embedded-testing-guide
# guide://quick-reference

# Discover testing agents
ace-nav agent:/// | grep test
# Expected output:
# agent://test
# agent://mock
# agent://profile-tests

# Discover testing workflows
ace-nav wfi:/// | grep -E "(test|create-test|fix-test)"
# Expected output:
# wfi://create-test-cases
# wfi://fix-tests
```

**Bundle Size Verification**:
```bash
ace-bundle project
# Expected: Bundle saved (~1000 lines) in 2 chunks
# Previous: Bundle saved (2135 lines) in 2 chunks
```

## Objective

Reduce project bundle size by 50% while consolidating scattered testing knowledge into a single, discoverable package. This improves onboarding experience (leaner context) and provides a single source of truth for testing best practices.

## Scope of Work

- **User Experience Scope**:
  - Running `ace-bundle project` returns ~1000 lines instead of 2135
  - Testing resources discoverable via `ace-nav guide://testing-*`, `agent://test*`, `wfi://*test*`
  - No testing documentation in main `docs/` directory

- **System Behavior Scope**:
  - New `ace-test` package installed and discoverable
  - Protocol sources configured for guides, agents, workflows, templates
  - All testing knowledge accessible without bundle bloat

- **Interface Scope**:
  - Protocol discovery (`ace-nav guide://`, `agent://`, `wfi://`)
  - Bundle generation (`ace-bundle project`)
  - Package installation via RubyGems

### Deliverables

#### Behavioral Specifications
- `ace-test` package with agents, workflows, guides, templates
- Protocol source configurations for all resource types
- README explaining package usage

#### Validation Artifacts
- Bundle size verification (<1000 lines)
- Protocol discovery tests (all resources accessible)
- Cross-reference validation (no broken links)

## Out of Scope

- ❌ **Implementation Details**: How to split testing-patterns.md content into guides
- ❌ **Content Creation**: Writing new testing documentation (consolidation only)
- ❌ **CLI Tools**: ace-test is a pure workflow package (no CLI binary)
- ❌ **Test Runner**: ace-test-runner package remains unchanged for CLI execution
- ❌ **E2E Testing**: ace-test-e2e-runner package scope not affected
