# Migrate ace-coworker E2E Tests

## Scope

Convert 6 `.mt.md` files (28 TCs) in `ace-coworker/test/e2e/` to the new directory-based per-TC format. **This is the most complex migration** due to sequential test dependencies requiring composite lifecycle TCs.

## Sequential Dependency Resolution

### MT-COWORKER-001 (11 TCs → 1 composite)

Linear workflow lifecycle: error checks → session creation → status → step completion → failure → retry → finish.

**Chain:** TC-001 (error: no config) → TC-002 (error: no session) → TC-003 (error: no report) → TC-004 (deprecated start) → TC-004b (cache auto-creation) → TC-005 (create session) → TC-006 (verify structure) → TC-007 (check status) → TC-008 (complete step) → TC-009 (mark failed) → TC-010 (dynamic step + retry) → TC-011 (complete remaining)

**Result:** `TC-001-full-workflow-lifecycle.tc.md` organized in phases:
- Phase 1: Error Cases (clean state)
- Phase 2: Session Creation
- Phase 3: Status and Completion
- Phase 4: Failure Handling
- Phase 5: Dynamic Steps and Retry
- Phase 6: Workflow Completion

### MT-COWORKER-002 (8 TCs → 1 composite)

Linear fork context workflow: parse frontmatter → regular step → fork steps → transitions → complete.

**Result:** `TC-001-full-fork-context-workflow.tc.md`

### MT-COWORKER-003b (3 TCs → 1 composite)

Shared session with progressive hierarchy manipulation: create child → inject sibling (renumber) → cascade renumber.

**Result:** `TC-001-hierarchy-injection-renumbering.tc.md`

### MT-COWORKER-003c (2 TCs → 1 composite)

Progressive auto-completion: single-level → multi-level hierarchy completion.

**Result:** `TC-001-hierarchy-auto-completion.tc.md`

## Technical Approach

### Composite TC Design

For merged sequential TCs, use phased structure preserving all verification points:

```markdown
## Steps

### Phase 1: [Logical group name]
[Steps from original TC-001, TC-002, ...]

### Phase 2: [Next group]
[Steps from original TC-00X, ...]

## Expected

### Phase 1
[Combined expectations]

### Phase 2
[Combined expectations]
```

### Session Discovery Pattern

Tests dynamically locate session directories — preserve in TC steps:
```bash
SESSION_DIR=$(find "$CACHE_BASE" -maxdepth 1 -mindepth 1 -type d | sort | tail -1)
```

### Fixture Extraction

| Scenario | Fixtures |
|----------|----------|
| 001 | job.yaml (3-step workflow), report.md, dummy-report.md, verify/implement/fix-report.md |
| 002 | job.yaml (4-step with forks), prepare/implement/verify/document-report.md |
| 003a | job.yaml (2-step flat) |
| 003b | job.yaml (2-step, children created in Test Data) |
| 003c | job2.yaml, job3.yaml, various report files |
| 003d | job4.yaml, job5.yaml |

### Setup Pattern

All scenarios:
```yaml
setup:
  - env:
      PROJECT_ROOT_PATH: .
      CACHE_BASE: .cache/ace-coworker
  - run: mkdir -p .cache/ace-coworker
  - copy-fixtures
```
