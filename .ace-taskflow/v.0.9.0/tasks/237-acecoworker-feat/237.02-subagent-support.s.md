# Fork Context Support for Jobs

## Overview

Enable job files to declare `context: fork` in frontmatter, instructing ace-coworker to execute them via Task tool in an isolated agent context.

## Objective

Allow jobs to run in forked agent contexts with rich structure:
- `context: fork` in job frontmatter triggers Task tool execution
- Job files can have structured sections (Onboard / Work / Report)
- Presets stay simple; job files can be rich
- Verification remains a separate job (worker shouldn't verify own work)

## Key Insight

> Jobs that run in isolation don't need predefined skills. The job file IS the full agent prompt.

This enables dynamic workflows where the job file itself describes the complete forked agent task.

## Scope of Work

### Two Layers

**Layer 1: Presets (job.yml)** - Stay simple, no changes
```yaml
steps:
  - name: implement
    instructions: "Implement {{taskref}}"
```

**Layer 2: Session Jobs (jobs/*.j.md)** - Can be rich
```markdown
---
context: fork
---
## Onboard
- ace-bundle project
- ace-bundle task://{{taskref}}

## Work
Implement the task following the specification...

## Report
Return: status, changes, commits, issues
```

## Technical Approach

### Job File Structure

```markdown
---
status: pending
context: fork
---

## Onboard

Load context before starting work:
- `ace-bundle project`
- `ace-bundle task://{{taskref}}`

## Work

[Main instructions for the forked agent]

## Report

Return structured summary:
- **Status**: completed | partial | blocked
- **Changes**: files modified
- **Issues**: problems encountered
```

### Execution Flow

```
ace-coworker advance
  │
  ├── 1. Read current job file
  ├── 2. Check frontmatter for `context: fork`
  ├── 3. If fork:
  │     ├── Build Task tool prompt from job content
  │     ├── Execute via Task tool (subagent_type: general-purpose)
  │     ├── Capture subagent response
  │     └── Store response as report
  ├── 4. If not fork:
  │     └── Display instructions (existing behavior)
  └── 5. Advance to next job
```

### Task Tool Integration

When `context: fork` is detected:

```ruby
def execute_forked_job(job)
  prompt = build_fork_prompt(job)

  # The orchestrating agent uses Task tool
  # ace-coworker outputs instructions for this
  puts "Execute this job in a forked context:"
  puts "```"
  puts "Use Task tool with subagent_type: general-purpose"
  puts "Prompt:"
  puts prompt
  puts "```"
end

def build_fork_prompt(job)
  <<~PROMPT
    #{job.content}

    Working directory: #{Dir.pwd}
    Session: #{@session.id}

    After completing, return a structured report.
  PROMPT
end
```

### Job File Parser Extension

```ruby
def self.extract_fields(parsed)
  fm = parsed[:frontmatter]
  {
    status: fm["status"],
    context: fm["context"],  # NEW: "fork" or nil
    # ... other fields
  }
end
```

### Separation of Concerns

**Work job** - Does the implementation:
```markdown
---
context: fork
---
## Onboard
- ace-bundle project

## Work
Implement the feature...

## Report
Return changes made
```

**Verify job** - Separate job, validates the work:
```markdown
---
status: pending
verification: ace-test
---
Run verification and report results.
```

Worker should not verify own work. Keep these as separate jobs.
