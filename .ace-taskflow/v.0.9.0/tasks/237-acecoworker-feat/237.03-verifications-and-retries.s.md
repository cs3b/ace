# Verifications and Retries

## Overview

Run verification commands after steps to catch failures. Simplified from 234.04.

## Objective

Add lightweight verification support:
- `verification:` field in step frontmatter
- `ace-coworker report --verify` runs verification command
- If verification fails, auto-mark step as failed with output
- Manual retry via existing `ace-coworker retry`

## Scope of Work

### Simplified from 234.04

**Included:**
- Single verification command per step (in frontmatter)
- `--verify` flag on report command
- Verification output stored in step error field
- Uses existing retry command for recovery

**Excluded (from original 234.04):**
- Multiple verification commands per step
- Error normalization heuristics
- Repeated failure detection
- Auto-retry loops
- verification_retries/on_repeated_failure config

## Technical Approach

### Step Frontmatter Extension

```yaml
---
status: in_progress
verification: ace-test
---
```

### Verification Flow

```
ace-coworker report report.md --verify
  │
  ├── Store report content in step file
  ├── Parse verification command from frontmatter
  ├── Execute: ace-test
  │     ├── Exit 0 → mark step done, advance
  │     └── Exit !0 → mark step failed, store output
  └── Display result
```

### CLI Interface

```bash
# Report with verification
ace-coworker report changes.md --verify
# Output (success):
# Step 020 (implement) completed
# Verification: ace-test passed
# Advancing to step 030

# Output (failure):
# Step 020 (implement) failed verification
# Verification: ace-test failed (exit 1)
# Error output stored in step file
# Use 'ace-coworker retry 020' to retry
```
