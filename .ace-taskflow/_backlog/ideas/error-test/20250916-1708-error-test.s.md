# Error Testing for LLM Query Tool

## Intention

To ensure robust error handling and reporting for the `llm-query` tool by simulating and testing various error conditions.

## Problem It Solves

**Observed Issues:**
- The `llm-query` tool might fail due to network issues, invalid API keys, malformed requests, or unexpected responses from LLM providers.
- Current error handling might be inconsistent across different LLM providers.
- Debugging failures in `llm-query` can be difficult without clear error messages and context.
- Potential for unexpected behavior when LLM providers return non-standard errors or malformed data.

**Impact:**
- Users (human or AI agents) may receive cryptic or unhelpful error messages, hindering their ability to resolve issues.
- Inconsistent error handling can lead to fragile workflows that break unpredictably.
- Difficulty in diagnosing the root cause of `llm-query` failures, increasing debugging time.
- Potential for security vulnerabilities if errors expose sensitive information.

## Key Patterns from Reflections

- **Centralized CLI Error Reporting (ADR-009)**: All CLI commands should route errors through a central `ErrorReporter` module, supporting a `--debug` flag for verbose output.
- **ATOM Architecture**: Error handling logic should be encapsulated within appropriate layers (e.g., Molecules for request-level errors, Organisms for provider-specific errors).
- **LLM Integration Architecture**: Need to handle errors from multiple providers gracefully, normalizing them where possible.
- **HTTP Client Strategy (ADR-010)**: Faraday's middleware (like `response :raise_error` and custom `RetryMiddleware`) can catch and report HTTP-level errors.
- **Security-First Development**: Error messages should not expose sensitive information like API keys or internal system details unless explicitly requested via a debug flag.

## Solution Direction

1. **Establish Error Scenarios**: Define a comprehensive list of error conditions to test, covering network, provider API, authentication, request, and response errors.
2. **Implement Provider-Specific Error Normalization**: Develop mechanisms within each provider's integration (e.g., `OpenAIClient`, `GoogleClient`) to catch provider-specific errors and map them to a standardized internal error format.
3. **Integrate with Central Error Reporter**: Ensure all caught and normalized errors are reported via the `ErrorReporter` module, respecting the `--debug` flag for detailed output.

## Critical Questions

**Before proceeding, we need to answer:**
1. What are the most common and critical error conditions we need to simulate for each supported LLM provider?
2. How should provider-specific error codes or messages be normalized into a consistent internal error structure?
3. What level of detail should be exposed by default versus what requires the `--debug` flag for `llm-query`?

**Open Questions:**
- How will we handle rate limiting errors specifically across different providers?
- What is the strategy for testing errors related to malformed LLM provider responses (e.g., invalid JSON)?
- Should there be a specific error type for "model not found" or "context window exceeded"?

## Assumptions to Validate

**We assume that:**
- The `ErrorReporter` module is correctly implemented and accessible for `llm-query` to use. - *Needs validation*
- Provider APIs reliably return distinct error codes or messages that can be parsed and categorized. - *Needs validation*
- Network error conditions can be reliably simulated in a testing environment. - *Needs validation*

## Expected Benefits

- **Improved User Experience**: Users receive clear, actionable error messages when `llm-query` fails.
- **Enhanced Debuggability**: Developers can quickly diagnose and fix issues with detailed error information available via the debug flag.
- **Increased Workflow Reliability**: Consistent error handling across providers makes AI agent workflows more robust.
- **Standardized Error Reporting**: Adherence to `ErrorReporter` ensures consistent output across all CLI tools.

## Big Unknowns

**Technical Unknowns:**
- The exact structure and error codes returned by all LLM providers for various failure scenarios.
- The best way to simulate specific network conditions (e.g., intermittent connectivity, high latency) for testing.

**User/Market Unknowns:**
- How users (human or AI) prefer to receive and interpret error messages from CLI tools.
- The impact of different error reporting strategies on user satisfaction and task completion rates.

**Implementation Unknowns:**
- The effort required to implement provider-specific error normalization for all supported LLMs.
- The best approach for integrating `llm-query`'s error handling with Faraday's middleware and the `ErrorReporter`.
