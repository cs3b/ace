# CI for Testing Ideas

## Intention

Enable AI agents to test their ideas and code changes in a reproducible and automated Continuous Integration (CI) environment.

## Problem It Solves

**Observed Issues:**
- AI agents lack a reliable way to test their proposed code changes or workflow ideas in an isolated, automated environment before committing them.
- Manual testing by human developers is a bottleneck, delaying the iteration cycle for AI-generated code.
- Ensuring the consistency and correctness of AI-generated code across different development environments is challenging.
- Reproducing bugs found in AI-generated code can be difficult without a standardized testing environment.

**Impact:**
- Slower development cycles as AI agents cannot autonomously validate their work.
- Increased risk of introducing regressions or bugs into the codebase due to lack of automated validation.
- Higher burden on human developers to review and test AI-generated code, reducing their productivity.
- Difficulty in establishing a baseline for code quality and correctness for AI-assisted development.

## Key Patterns from Reflections

- **CI-Aware VCR Configuration (ADR-006)**: Tests need to run reliably in CI environments without making external API calls. VCR cassettes should be used to mock external services, and recording modes must be configured to prevent API usage in CI.
- **Test-Driven Development**: The project emphasizes TDD, implying that any new functionality or code change should be accompanied by tests.
- **Multi-Repository Coordination**: The system uses Git submodules, meaning CI pipelines might need to handle submodule updates and checkouts.
- **ATOM Architecture**: Tests should ideally be structured to test components at various layers (Atoms, Molecules, Organisms, Ecosystems).
- **Executable Tools**: The `.ace/tools` gem provides CLI executables that should be testable, potentially through CLI integration testing frameworks like Aruba.
- **Workflow Self-Containment (ADR-001)**: If workflows are to be tested, they must be self-contained, meaning their tests should not rely on external workflow dependencies.

## Solution Direction

1. **Automated CI Pipeline Setup**: Configure a CI pipeline (e.g., GitHub Actions) to automatically run tests whenever code changes are pushed.
2. **Reproducible Test Environment**: Ensure the CI environment can precisely replicate the development environment, including Ruby version, gem dependencies, and system configurations.
3. **Comprehensive Test Execution**: Execute all relevant tests, including unit tests, integration tests, CLI tests, and potentially workflow tests, within the CI pipeline.

## Critical Questions

**Before proceeding, we need to answer:**
1. What specific CI platform will be used (e.g., GitHub Actions, GitLab CI)?
2. What types of tests need to be executed in CI (unit, integration, CLI, workflow)?
3. How will VCR cassettes be managed and updated for CI runs, especially for LLM-related tests?

**Open Questions:**
- How can AI agents be given the capability to *trigger* specific tests or validation runs on their generated code?
- What is the strategy for testing AI-generated workflows themselves within CI?
- How will test coverage metrics be tracked and enforced in the CI pipeline?

- A CI platform is available and can be configured for the project. - *Needs validation*

## Expected Benefits

- Faster feedback loop for AI-generated code changes.
- Improved code quality and stability through automated validation.
- Increased confidence in deploying AI-assisted development outputs.
- Reduced manual testing burden on human developers.

## Big Unknowns

- The exact mechanism for AI agents to trigger CI tests or receive test results.
- The approach for testing self-contained workflows (ADR-001) in a CI environment, given they are designed for agent execution.

**User/Market Unknowns:**

**Implementation Unknowns:**
