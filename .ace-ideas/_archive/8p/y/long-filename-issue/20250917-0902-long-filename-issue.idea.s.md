# Handle Extremely Long Filenames

## Intention

To develop a strategy and implement tooling to manage and mitigate issues arising from extremely long filenames generated by AI workflows, ensuring compatibility and usability across different operating systems and file systems.

## Problem It Solves

**Observed Issues:**
- AI-generated workflow instructions can produce exceptionally long filenames due to extensive descriptions, parameters, or embedded content.
- Extremely long filenames can exceed operating system limits (e.g., Windows MAX_PATH), leading to file creation failures, unmanageable file paths, and broken workflows.
- Existing tools and scripts may not handle such long filenames gracefully, causing errors or unexpected behavior.
- Manual truncation or shortening of filenames is error-prone and can lead to loss of crucial context.

**Impact:**
- Workflow execution failures due to OS path length limitations.
- Inability to commit or manage files with excessively long names in Git.
- Poor developer experience when interacting with files that are difficult to reference or view.
- Potential data loss or corruption if files cannot be saved or accessed correctly.
- Increased complexity in scripting and automation that relies on predictable filename structures.

## Key Patterns from Reflections

- **Workflow Self-Containment (ADR-001)**: Workflows are designed to be independent and embed necessary content. This could lead to longer descriptions or embedded content that, when translated to filenames, become excessively long.
- **XML Template Embedding (ADR-002)**: While structured, the `path` attribute in XML could potentially point to generated files with very long names if not managed.
- **Consistent Path Standards (ADR-004)**: The adherence to relative paths and specific directory structures (e.g., `.ace/handbook/templates/`) is good, but the *content* of filenames within these structures can become problematic.
- **ATOM Architecture**: The layered approach might lead to components generating output that needs unique, descriptive filenames, potentially increasing length.
- **Multi-Repository Coordination**: Managing files across multiple submodules with long filenames could exacerbate path length issues, especially when combined.
- **Documentation-Driven Development**: The emphasis on descriptive documentation might translate into long filenames when used for generated files or workflow outputs.

## Solution Direction

1. **Filename Truncation and Hashing Strategy**: Implement a mechanism to automatically shorten excessively long filenames while preserving essential context and ensuring uniqueness.
2. **Operating System Compatibility Checks**: Develop tooling to detect and report potential filename length issues based on the target operating system's limitations.
3. **Workflow Instruction Guidelines**: Update workflow design principles to encourage shorter, more manageable filenames where possible, and provide guidance on using the new truncation/hashing strategies.

## Critical Questions

**Before proceeding, we need to answer:**
1. What are the specific maximum filename length limits for common target operating systems (Windows, macOS, Linux) and file systems (NTFS, APFS, ext4)?
2. What is the acceptable threshold for filename length before truncation/hashing is applied?
3. How should the truncation/hashing strategy balance filename readability with uniqueness and context preservation?

**Open Questions:**
- What is the optimal truncation length or hashing algorithm to use?
- How will Git handle extremely long filenames, and are there specific Git configurations or tools needed?
- Will this strategy impact the ability to search for files based on their original, longer names?
- How will this affect existing scripts or tools that rely on exact filename matching?

## Assumptions to Validate

**We assume that:**
- AI-generated content is the primary source of extremely long filenames. - *Needs validation*
- Operating system limits are the main technical constraint being hit. - *Needs validation*
- A programmatic solution (truncation/hashing) is preferable to manual intervention. - *Needs validation*

## Expected Benefits

- **Reduced Workflow Failures**: Prevent errors caused by exceeding OS filename length limits.
- **Improved Cross-Platform Compatibility**: Ensure workflows function reliably across different development environments.
- **Enhanced Usability**: Make it easier for developers and agents to interact with generated files.
- **Increased Automation Robustness**: Make scripts and tools more resilient to unpredictable filename lengths.
- **Context Preservation**: Retain essential information even in shortened filenames.

## Big Unknowns

**Technical Unknowns:**
- The exact implementation details of the truncation/hashing algorithm and its impact on performance.
- Potential conflicts with existing file naming conventions or tools within the ecosystem.

**User/Market Unknowns:**
- How users (developers or AI agents) will perceive shortened filenames if context is lost.
- User tolerance for potential disambiguation mechanisms if hashes are used.

**Implementation Unknowns:**
- The effort required to update all relevant tools, scripts, and workflow instructions to incorporate the new strategy.
- The best approach for migrating existing files with long names if necessary.
