# Onboarding friction: new project setup needs a clearer path

## Context

I just bootstrapped ACE from scratch in a new project (Astro website, Ruby 4.0.2 via mise). Sharing the friction points from the real first-time experience — everything below is intended as constructive feedback.

## 1. Gem dependency resolution is broken on fresh install

`bundle install` fails due to stale RubyGems API metadata. Multiple gems produce warnings like:

```
Local specification for ace-git-worktree-0.19.0 has different dependencies than the remote gem, ignoring it
```

Required `bundle install --full-index` to work around. This is a first-impression killer for new users who just want to get started.

## 2. No one-liner Gemfile snippet in README

Had to manually discover and list 40+ gems individually. There's no meta-gem and no recommended Gemfile snippet anywhere in the docs.

Even a simple "add this to your Gemfile" block in the README would save significant time:

```ruby
# Gemfile
gem "ace-framework"  # or a curated list of top-level gems
```

## 3. `ace-framework init` and `ace-handbook sync` are the golden path but undiscoverable

Once I found them, everything clicked:

```
$ ace-framework init --force
Configuration initialization complete:
  Files copied: 496
  Files skipped: 0

$ ace-handbook sync
synced claude -> .claude/skills (27 skills, 0 updated, 0 removed)
synced codex -> .codex/skills (27 skills, 27 updated, 0 removed)
```

These two commands are *exactly* what a new project needs. But they're not mentioned in:
- README.md
- docs/quick-start.md
- DEVELOPMENT.md

I discovered them by accident after manually creating broken config files and empty `.claude/` directories. These should be steps 1-2 in any getting-started guide.

## 4. LLM provider config crashes instead of guiding

Running `ace-git-commit` after setup:

```
$ ace-git-commit Gemfile.lock mise.toml -i "config"
Unknown providers in llm.providers.active: claude (ignored)
Failed to generate commit message: Unknown provider: glite. Supported providers: openrouter, zai
```

Issues:
- `claude` is listed as active but silently "ignored" — no explanation of why or what to do
- Falls through to `glite` which is not a valid provider — where does this come from?
- The only way to discover supported providers is from a crash stacktrace
- No docs listing: which providers exist, what env vars each needs, or example configurations

**Suggestion:** The error message should say something like:
```
Unknown provider: glite. Supported providers: openrouter (OPENROUTER_API_KEY), zai (ZAI_API_KEY).
Run 'ace-llm providers' to see available providers and their configuration.
```

## 5. Handbook skill sync only covers built-in ACE skills

`ace-handbook sync` syncs 27 built-in `as-*` workflow skills (task, assignment, e2e, bug) — which is great. But:

- There's no guidance on how to **add project-specific skills** to the handbook and have them sync
- Where should custom skills live? In `handbook/skills/`? In `.ace/handbook/`? No docs explain this
- The handbook config (`.ace/handbook/config.yml`, `.ace/handbook/providers/claude.yml`) is minimal and doesn't hint at extensibility
- Protocol URLs (`wfi://`, `guide://`, `tmpl://`, `skill://`) are mentioned but not explained for projects outside the ACE monorepo

A "creating your first handbook" guide covering both using the built-in skills and adding your own would go a long way.

## 6. `ace-framework init` generates broken presets

After running `ace-framework init`, `ace-bundle project` immediately fails:

```
$ ace-bundle project
Preset 'project' must define bundle sections
```

The generated `.ace/bundle/presets/project.md` uses a flat `files`/`commands` format in the frontmatter:

```yaml
bundle:
  files:
    - docs/vision.md
    - docs/architecture.md
  commands:
    - ace-taskflow recent --limit 3
```

But the actual required format uses `sections`:

```yaml
bundle:
  sections:
    overview:
      title: "Project Overview"
      files:
        - docs/vision.md
      commands:
        - ace-task list
```

So `ace-framework init` scaffolds a config that doesn't work out of the box. The preset also references `ace-taskflow` commands which don't exist as standalone CLIs — should be `ace-task list` etc.

## 7. Generated configs reference the ACE monorepo, not your project

Several files generated by `ace-framework init` contain ACE-monorepo-specific content:

- `project.md` preset says *"You are working on the Coding Agent Workflow Toolkit (Meta)"* — should be blank or a placeholder
- `project.md` lists `ace-taskflow recent`, `ace-taskflow next` — these aren't standalone commands
- Commit scopes default to `core, api, ui, docs, test` — generic but not harmful
- `core/settings.yml` has `name: "your-project"` — at least this one is an obvious placeholder

**Suggestion:** Either generate truly generic configs (with `TODO` markers) or make `ace-framework init` prompt for project name/type and customize the output.

## 8. `ace-handbook sync` only syncs 27 of 94 available skills

After running `ace-handbook sync`, we got 27 skills. But the actual gem suite ships 94 skills (visible after a Gemfile update triggered a re-sync). The first sync was incomplete because some gems weren't fully loaded yet.

This means the order matters:
1. `bundle install` (all gems)
2. `ace-framework init`
3. `ace-handbook sync` ← must happen AFTER all gems are installed

But there's no documentation about this dependency. If you run `ace-handbook sync` before all gems are installed, you get a partial set with no warning.

## 9. Suggested getting-started section for README

Something like this would eliminate most of the friction:

```markdown
## Getting Started (new project)

1. Add to your Gemfile:
   ```ruby
   gem "ace-framework"
   ```

2. Install and initialize:
   ```sh
   bundle install
   ace-framework init
   ace-handbook sync
   ```

3. Configure LLM providers in `~/.ace/.env`:
   ```sh
   ZAI_API_KEY=your_key_here
   OPENROUTER_API_KEY=your_key_here
   ```

4. Start using:
   ```sh
   ace-git-commit --help
   ace-task create "My first task"
   ```
```

## 9. `work-on-task` preset crashes on non-gem projects (`wfi://release/publish` missing)

The `work-on-task` assignment preset includes a `release-minor` step that references `wfi://release/publish` via `assign.source`. This workflow doesn't ship in any of the installed gems — it appears to be part of an unreleased `ace-release` package.

```
$ ace-assign create --preset work-on-task --task h3p
Error: Could not resolve assign.source 'wfi://release/publish' for skill 'release-minor'.
Searched: [...18 gem paths...] (none contain release/publish)
```

This makes the **main assignment preset unusable for any project** without a workaround.

**Additional issue:** `ace-assign create` resolves `assign.source` WFI references using its own hardcoded search paths (`<gem>/handbook/workflow-instructions/` and `<project>/ace-*/handbook/workflow-instructions/`), **not** the nav protocol system configured in `.ace/nav/protocols/wfi-sources/`. So registering a project-level wfi-source in nav config doesn't help — you must place files in `<project>/ace-assign/handbook/workflow-instructions/` for `ace-assign` to find them.

**Workaround applied:**
1. Created `ace-assign/handbook/workflow-instructions/release/publish.wf.md` (lightweight changelog-only workflow)
2. Also registered in `.ace/nav/protocols/wfi-sources/project.yml` for `ace-bundle wfi://` resolution

**Suggestions:**
- Ship a default `release/publish.wf.md` in `ace-assign` or `ace-bundle` that works generically (just changelog)
- Or make the `release-minor` step conditional/skippable when no release workflow exists
- Unify WFI resolution: `ace-assign` should use the same nav protocol system as `ace-bundle`
- Document how to add project-level workflow instructions

## TL;DR — the friction map

| # | Issue | Severity | Fix effort |
|---|-------|----------|-----------|
| 1 | Gem deps broken on fresh install | High | Investigate RubyGems API metadata |
| 2 | No Gemfile snippet in README | High | One paragraph in README |
| 3 | `ace-framework init` + `ace-handbook sync` undiscoverable | Critical | Document in README |
| 4 | LLM provider crashes instead of guiding | Medium | Better error messages |
| 5 | Handbook skill extensibility undocumented | Medium | Guide doc |
| 6 | Generated bundle preset is broken | High | Fix template in ace-framework |
| 7 | Generated configs reference ACE monorepo | Medium | Use generic placeholders |
| 8 | Partial skill sync with no warning | Medium | Warn or document order dependency |
| 9 | `work-on-task` preset crashes (missing `wfi://release/publish`) | Critical | Ship default release workflow or make step conditional |

---

Thanks for building ACE — the architecture is solid and the tooling is genuinely powerful once configured. Every one of these friction points is solvable and the fixes are mostly documentation + template fixes, not architectural changes. The gap is the last mile of discoverability for new users.
