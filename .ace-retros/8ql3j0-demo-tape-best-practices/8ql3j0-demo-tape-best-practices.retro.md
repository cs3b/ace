# How to Get the Most Out of Recorded Demos

Learnings from writing and iterating on VHS demo tapes for ace-assign, ace-overseer, ace-tmux, and ace-retro.

## What Went Well

- **Show real output, not help screens.** The first tape iteration used `--help` for most scenes ‚Äî technically correct but useless for a user deciding whether to adopt the tool. Switching to actual commands with real data (assignment trees, progress bars, retro content) made the demos immediately compelling.
- **Let the tool's visual design speak.** ace-assign's tree status with `‚úì/‚úó/‚óã` symbols, ace-overseer's color-coded progress bars, ace-retro's rich listing with tags and task refs ‚Äî these outputs are already designed to impress. The tape just needs to trigger them.
- **Dry-run flags are demo gold.** `ace-retro create --dry-run` shows exactly what would happen without side effects. This is the ideal demo pattern: real behavior, zero cleanup.
- **Live workflows beat static examples.** ace-tmux's tape now starts a session, adds a window, and verifies the result ‚Äî a complete workflow users can follow. Much more useful than showing `start --help`.

## What Could Be Improved

- **First iteration was all `--help` screens.** Tapes showed command surface but not command value. A user watching `--help` four times in a row learns nothing they couldn't get from the README. The demo should answer "what does this look like in practice?" not "what flags exist?"
- **Sleep times matter for readability.** The initial 3s sleep was too short for dense output like `ace-assign status` (60+ lines of tree). Bumped to 5-8s for rich output, kept 3-4s for simple commands. Rule of thumb: count the visible lines, give ~1s per 10 lines minimum.
- **Hardcoded IDs break reproducibility.** First-generation tapes used `8ql0mx@010.01` (a real assignment) ‚Äî works on this machine, fails everywhere else. Fixed by showing `list` and `status` which work with whatever data exists, or `--dry-run` which needs nothing.
- **Placeholder GIFs are worse than no GIF.** The 20-34K ffmpeg fallbacks were single-frame stubs. Users see them in the README and assume the tool is broken or trivial. Better to ship no GIF than a misleading one.

## Key Learnings

- **The best demo scene is a real command that produces visually rich output.** Rank commands by output quality, not by feature importance. `ace-overseer status` (progress bars, colors, PR status) is a better opener than `ace-overseer --help` even though help is more "complete."
- **One workflow > four features.** ace-tmux demo: `list` ‚Üí `start` ‚Üí `window` ‚Üí `verify` tells a story. Four disconnected `--help` screens are a reference card, not a demo.
- **Match sleep duration to output density.** Short output (2 lines): 3s. Medium output (10-20 lines): 4-5s. Dense output (tree/table): 6-8s. Users need time to scan and understand what they're seeing.
- **Tapes should be self-contained and reproducible.** Every command must work on a clean clone or with any repo state. Use `--dry-run`, `list`, `status`, and `doctor` ‚Äî commands that show features without requiring specific state.

## Action Items

- **Continue**: Using real data output over help screens in demo tapes
- **Continue**: Testing tapes are reproducible by checking each command works without specific state
- **Start**: Adding a "demo tape checklist" to the ace-demo docs:
  1. Does every scene show useful output (not just `--help`)?
  2. Are sleep times proportional to output density?
  3. Can every command run on a clean clone?
  4. Does the sequence tell a workflow story?
- **Start**: Considering `--dry-run` flags on more CLI tools specifically to enable safe demos
- **Stop**: Shipping placeholder GIFs generated by ffmpeg ‚Äî either record properly or omit the image
