# Improve Large Change Handling in `git-commit`

## Intention

Enhance the `git-commit` tool to better handle large code changes by intelligently splitting diffs or limiting their size for commit message generation, and to improve the inference of commit intention when it's not explicitly provided.

## Problem It Solves

**Observed Issues:**
- Large code changes result in overly long and potentially unfocused commit messages generated by the LLM.
- The LLM may struggle to extract a coherent commit intention or summary from extremely large diffs, leading to generic or inaccurate commit messages.
- Lack of intelligent diff processing can lead to inefficient use of LLM context windows and increased API costs for large changes.
- When commit intention is not explicitly passed, the inference mechanism may not always capture the most relevant aspects of a large diff.

**Impact:**
- Commit messages for large changes are less useful for understanding the scope and purpose of the commit.
- Developers may resort to manual editing of commit messages, negating the automation benefits.
- Inefficient processing of large diffs can lead to higher LLM API costs and slower commit generation times.
- The `git-commit` tool's effectiveness is diminished for complex or extensive code modifications.

## Key Patterns from Reflections

- **LLM Integration Architecture**: The project leverages LLMs for various tasks, including commit message generation. Improving the input quality for these LLMs is crucial. (ADR-014)
- **ATOM Architecture**: Components are classified into Atoms, Molecules, Organisms, and Ecosystems. This task likely involves enhancing a Molecule or Organism responsible for Git operations and LLM interaction. (ADR-011)
- **CLI Tool Patterns**: Existing CLI tools like `git-commit` follow specific patterns that need to be maintained. (docs/tools.md)
- **Security-First Development**: While not directly related to large changes, any new logic for diff processing must consider security implications, particularly around input sanitization. (docs/architecture.md)
- **Workflow Self-Containment**: Any changes should ideally not break the self-contained nature of workflows that might invoke `git-commit`. (ADR-001)

## Solution Direction

1. **Intelligent Diff Splitting**: Split large diffs into logical parts (e.g., by file or by hunk) and process them sequentially or in parallel for LLM summarization.
2. **Diff Size Limiting**: Implement a strategy to limit the size of the diff passed to the LLM (e.g., first 1000 lines, or diffs of changed files with the most lines), potentially summarizing smaller changes first.
3. **Enhanced Intention Inference**: Develop better heuristics or LLM prompts to infer the commit intention from the diff itself, especially when the `--intention` flag is not provided. This could involve identifying key file changes, types of modifications (e.g., bug fix, feature addition), or common patterns in commit history.

## Critical Questions

**Before proceeding, we need to answer:**
1. What is the optimal threshold for considering a diff "large" that warrants special handling?
2. What is the best strategy for splitting large diffs (by file, by hunk, by change type)?
3. How should the LLM be prompted to effectively summarize split or limited diffs, and how do we combine these summaries into a coherent commit message?

**Open Questions:**
- What are the performance implications of splitting large diffs or running multiple LLM calls for a single commit message?
- How do we ensure that the commit message remains focused and accurate when processing only a portion of a large diff?
- What are the best practices for inferring commit intention from diff content, particularly for complex or multi-faceted changes?

## Assumptions to Validate

**We assume that:**
- The LLM can effectively generate coherent summaries from smaller, segmented diffs. *Needs validation through experimentation.*
- Limiting diff size or splitting it will not obscure critical context necessary for an accurate commit message. *Needs validation through testing with representative large changes.*
- The `--intention` flag will continue to be the primary way to guide commit message generation, with fallback strategies only used when it's absent. *Needs validation based on user behavior and tool design.*

## Expected Benefits

- More focused, accurate, and useful commit messages for large code changes.
- Improved efficiency and reduced costs for LLM-based commit message generation.
- Enhanced developer experience when dealing with extensive modifications.
- Increased reliability of the `git-commit` tool across a wider range of code change sizes.
