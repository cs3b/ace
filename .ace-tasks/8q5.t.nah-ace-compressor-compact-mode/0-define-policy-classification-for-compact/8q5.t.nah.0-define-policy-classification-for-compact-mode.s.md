# Define policy classification for compact mode

## Behavioral Specification

### User Experience
- **Input**: A document corpus with mixed file types such as vision docs, guides, decisions, and workflows.
- **Process**: Compact mode classifies input before deciding what can be reduced aggressively.
- **Output**: A predictable policy choice per source or section that explains why content was compacted, preserved, or refused.

### Expected Behavior
Compact mode must not treat all files equally. Narrative-heavy documents are good compact candidates. Rule-heavy policy and decision documents must either preserve rule-bearing sections exactly or refuse compacting when safety would be compromised.

### Interface Contract
```bash
ace-compressor compress docs/vision.md docs/decisions.md --mode compact
```

**Error Handling:**
- Unknown file class: use conservative compact behavior and report the fallback class.
- Source classified as unsafe for compacting: exit non-zero with guidance to use `--mode exact`.

**Edge Cases:**
- Mixed files in one invocation may take different policy paths.
- A narrative doc containing a short imperative rule still preserves that rule exactly.

### Success Criteria
- [ ] File/section classes are behaviorally distinct.
- [ ] Narrative-heavy inputs are compacted more aggressively.
- [ ] Rule-bearing content stays exact or triggers refusal.
- [ ] Unknown class behavior is conservative and explicit.

### Validation Questions
- [x] Classification happens before aggressive compacting.
- [x] Unknown classes do not silently opt into the most destructive policy.
- [x] Rule-bearing sections keep exact semantics even in compact mode.
- [x] Refusal is preferred over unsafe compacting.
