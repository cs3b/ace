# Define documentation and README brand updates

## Behavioral Specification

### User Experience
- **Input**: Approved naming policy and GitHub metadata copy
- **Process**: Maintainer gets a file-by-file docs update plan with replacement rules
- **Output**: Public docs consistently present ACE as brand and ADE as category

### Expected Behavior
The subtask specifies required content updates for:
- Root `README.md` and repo-level docs (`docs/vision.md`, `docs/architecture.md`, `docs/blueprint.md`, `docs/tools.md`)
- Any core guidance files with top-level project naming statements (`CLAUDE.md`, `AGENTS.md`, `GEMINI.md` if applicable)
- Heading/tagline phrasing and glossary language

### Interface Contract
```text
Doc update artifact format:
1) File path
2) Current phrase
3) Replacement phrase
4) Rationale tag (brand/category/compat)
```

## Acceptance Criteria

- [x] All core public docs have a consistent name expansion and category sentence
- [x] Wording avoids ambiguity between brand name and category term
- [x] Update plan flags any intentionally unchanged historical references

### Defaults
- **Scope boundary**: Only files under `docs/`, root-level `*.md`, and `CLAUDE.md`/`AGENTS.md`/`GEMINI.md` — per-package READMEs are handled in subtask e
- **Historical references**: Git commit messages and CHANGELOG entries are intentionally unchanged (do-not-change)
- **Replacement granularity**: Phrase-level replacement (not whole-paragraph rewrites) to minimize diff noise

## Verification Plan

### Coverage Check
- Terminology matrix covers all files identified in subtask a's inventory under the docs/root-docs category
- Every replacement row has: file path, current phrase, replacement phrase, rationale tag
- Canonical one-liner is stated exactly once and referenced by all replacement rows that touch lead sections

### Consistency Check
- All replacement phrases use identical brand/category terminology (no variations like "ACE Framework" vs "ACE Toolkit")
- Intentionally unchanged references are explicitly listed with justification

### Negative Check
- No per-package README replacements appear (those belong to subtask e)
- No `sed` or script commands — this is a specification, not an implementation

## Deliverables

- [x] Documentation terminology matrix
- [x] File-by-file replacement checklist
- [x] Final canonical one-liner for README and docs lead sections

## Results

### 1. Canonical One-Liner

The following phrase is the single source of truth for lead sections across all root docs:

> **ACE (Agentic Coding Environment)** — a modular CLI toolkit for AI-assisted development.

Usage rules:
- The heading form is: `# ACE (Agentic Coding Environment)`
- The prose introduction form is: `ACE (Agentic Coding Environment) is a ...`
- The category phrase "Agentic Development Environment (ADE)" appears **only** in positioning/marketing contexts, never in code docs or lead sections
- Never use variations like "ACE Framework", "ACE Toolkit", or "ACE Platform" — use "ACE" alone after the first expanded mention
