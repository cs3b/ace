# Decide repository slug strategy

## Behavioral Specification

### User Experience
- **Input**: Surface inventory and product-positioning direction (ACE brand + ADE category)
- **Process**: Author sees a side-by-side decision matrix with weighted criteria and explicit tradeoffs
- **Output**: A final decision statement for canonical repository slug and naming language

### Expected Behavior
The subtask defines objective criteria (brand continuity, discoverability, uniqueness, ecosystem fit, migration risk)
and records a recommendation with rationale. The expected default is:
- Brand: `ACE`
- Expansion: `Agentic Coding Environment`
- Category language: "Agentic Development Environment (ADE)"
- Repository slug preference order: `ace` (if available) -> keep `ace-meta` -> `agentic-coding-environment` only if brand slug is unavailable/unacceptable

### Interface Contract
```text
Decision output format:
1) Canonical brand name
2) Canonical expanded phrase
3) Canonical category phrase
4) Repository slug decision and fallback order
5) One-paragraph public rationale
```

## Acceptance Criteria

- [ ] Decision matrix includes explicit pass/fail thresholds for each criterion
- [ ] Final recommendation includes both primary and fallback slug options
- [ ] Public-facing wording is consistent with prior decision: ACE brand, ADE category

## Deliverables

- [ ] Naming decision matrix
- [ ] Canonical naming policy statement
- [ ] Final repo slug recommendation with fallback logic

## Validation Questions

- [ ] Is GitHub org-level namespace availability for `ace` confirmed before execution?
- [ ] Are there external integrations that depend on current `ace-meta` slug and need transition notes?
