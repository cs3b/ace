# Standardized Review Runner Integration for ace-review

## Behavioral Specification

### User Experience
- **Input**: Users configure review presets with a `runners:` key listing active runners (e.g., `llm`, `lint`). Users can also enable runners ad-hoc via CLI flags like `--lint`.
- **Process**: ace-review orchestrates all configured runners in parallel or sequence, collects findings into a standardized `ReviewFinding` model, then synthesizes a unified report that clearly labels each finding's source.
- **Output**: A single review report combining deterministic findings (lint errors with exact file/line) and subjective LLM suggestions, with source attribution on each finding.

### Expected Behavior

When a user runs `ace-review --preset code-deep --lint --pr 123`, the system:
1. Loads the preset config and identifies active runners (LLM reviewers + LintRunner)
2. Executes each runner, collecting `ReviewFinding` instances
3. The LintRunner delegates to `ace-lint` via CLI, parses machine-readable output
4. The LLM runner(s) execute as today, but their findings are also normalized to `ReviewFinding`
5. All findings are passed to synthesis, which produces a unified report
6. Deterministic findings (lint) are prominently marked and distinguished from subjective (LLM) findings

When `--lint` is not passed and no `runners:` config exists, behavior is identical to today's LLM-only flow.

### Interface Contract

```bash
# Enable lint runner alongside LLM reviewers
ace-review --preset code-deep --lint --pr 123

# Preset config with runners
# .ace/review/presets/code-deep.yml
runners:
  - type: llm           # existing LLM reviewers (default, always active)
  - type: lint           # delegates to ace-lint
    config:
      format: json       # machine-readable output from ace-lint

# ReviewFinding model (internal interface)
# source: "llm:google:gemini-2.5-flash" | "lint:standardb" | "lint:markdownlint"
# file_path: "lib/ace/review/organisms/review_manager.rb"
# line_number: 42
# severity: "error" | "warning" | "info"
# message: "Unused variable assignment"
# category: "style" | "correctness" | "security" | "performance"
```

**Error Handling:**
- ace-lint not installed or not found: Warning logged, lint runner skipped, review continues with LLM only
- ace-lint returns non-zero exit: Warning logged with stderr, lint findings omitted, review continues
- No runners configured: Falls back to default LLM-only behavior

**Edge Cases:**
- Overlapping findings (lint and LLM flag same issue): Both appear in report, synthesis may note overlap
- Empty lint output (no issues found): Runner reports zero findings, synthesis notes "no lint issues"
- `--lint` with no files to lint: Runner skips gracefully

## Objective

Expand ace-review from an LLM-only review tool to a pluggable multi-runner review platform. The immediate use case is integrating deterministic lint findings (via ace-lint) alongside subjective LLM analysis, producing comprehensive unified reports. This establishes the architecture for future runners (test coverage, vulnerability scanning, etc.).

## Scope of Work

- **User Experience Scope**: Users can enable lint analysis in reviews via CLI flag or preset config, and receive unified reports with source-attributed findings
- **System Behavior Scope**: Runner abstraction layer, LintRunner implementation, ReviewFinding model, synthesis integration
- **Interface Scope**: `--lint` CLI flag, `runners:` preset config key, ReviewFinding internal model

### Deliverables

#### Behavioral Specifications
- Runner interface contract (what each runner must implement)
- ReviewFinding model specification
- Synthesis behavior for multi-source findings

#### Validation Artifacts
- Spike validates end-to-end runner flow with real ace-lint execution
- Unit tests for ReviewFinding model and LintRunner
- Integration test for multi-runner orchestration

## Out of Scope

- Implementation Details: File structures, code organization, technical architecture
- Technology Decisions: Tool selections, library choices, framework decisions
- Performance Optimization: Specific performance improvement strategies
- Future runners beyond LintRunner (test coverage, vulnerability scanning)
- Changes to ace-lint itself (assumed to provide adequate output)
- ace-assign integration (separate concern, see task 8pp5z7)
