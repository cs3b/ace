# Spike: Runner Abstraction with LintRunner End-to-End

## Behavioral Specification

### User Experience
- **Input**: Developer runs a time-boxed spike to validate the runner abstraction pattern
- **Process**: Build minimal ReviewFinding model, runner interface, and LintRunner that calls ace-lint, then wire through ReviewManager to produce a combined report
- **Output**: Working end-to-end proof that lint findings flow through the same pipeline as LLM findings; concept inventory of what survives

### Expected Behavior

The spike validates these architectural questions:
1. Can a runner interface (`#execute(subject) -> Array<ReviewFinding>`) work for both LLM and lint sources?
2. Can ace-lint produce machine-readable output that LintRunner can parse reliably?
3. Can the existing synthesis step handle mixed finding sources without major refactoring?
4. What existing concepts (FeedbackItem, Reviewer, ReviewManager) need adaptation vs replacement?

The spike produces a concept inventory showing:
- Which existing concepts survive (FeedbackItem, Reviewer model)
- What new concepts are needed (ReviewFinding, Runner interface, LintRunner)
- How the synthesis step must change to handle multi-source findings

```ruby
  line_number: 42,
  severity: "warning",

```

**Error Handling:**
- ace-lint binary not found: Spike documents this as a runtime dependency check needed
- Unexpected ace-lint output format: Spike documents required output format

**Edge Cases:**
- ace-lint produces no findings: Empty array returned, synthesis handles gracefully
- ace-lint output format varies by tool: Spike identifies which format(s) to support

### Success Criteria

- [ ] **Runner interface validated**: A single `#execute` method works for both LLM and lint runners
- [ ] **ace-lint output parsed**: LintRunner can invoke ace-lint and parse its output into ReviewFinding instances
- [ ] **End-to-end flow**: Lint findings appear in the final review output alongside LLM findings
- [ ] **Concept inventory produced**: Document which concepts are kept, added, or removed
- [ ] **Existing behavior preserved**: LLM-only path still works correctly after spike changes
