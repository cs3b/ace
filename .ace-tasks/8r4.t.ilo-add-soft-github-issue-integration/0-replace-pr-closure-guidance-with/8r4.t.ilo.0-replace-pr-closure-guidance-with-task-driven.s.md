# Replace PR closure guidance with task-driven issue sync

## Objective

Replace ACE guidance that relies on PR footers to close linked GitHub issues with guidance that treats `ace-task` as the owner of linked issue lifecycle.

## Behavioral Specification

### User Experience

- A user reading ACE task docs or drafting/reviewing linked tasks sees GitHub issue linkage described as task metadata plus automatic `ace-task` sync.
- A user no longer needs to remember that PR descriptions must contain `Closes #...` for ACE-linked tasks.
- Drafted tasks with GitHub-linked work can describe the issue relationship without binding correctness to PR body text.

### Expected Behavior

1. ACE docs and task/workflow guidance describe `ace-task` metadata and sync as the canonical GitHub issue integration path.
2. Guidance that previously used PR-body closure text for ACE-linked issue lifecycle is replaced or narrowed to non-ACE-linked/manual cases.
3. Existing task drafting guidance remains behavior-first, but now includes the machine-readable GitHub issue metadata pattern where issue-linked tracking is required.

### Interface Contract

- **Documentation contract**
  - ACE docs explain:
    - how to link a task to GitHub issue IDs
    - that `ace-task` syncs GitHub issue comments, labels, and issue state
    - that PR footers are optional and no longer required for ACE-linked issue closure
- **Task-spec contract**
  - Linked tasks may include GitHub issue metadata in frontmatter instead of relying on markdown-only issue references

### Success Criteria

- [ ] ACE docs no longer instruct users to rely on PR closure text for ACE-linked GitHub issue lifecycle.
- [ ] Task drafting/review guidance reflects machine-readable task linkage to GitHub issues.
- [ ] Existing examples and workflow guidance remain coherent with the new ACE-driven issue sync model.

### Validation Questions

- Should legacy PR-footer examples remain in docs as optional GitHub-native guidance for non-ACE-linked work?

## Vertical Slice Decomposition (Task/Subtask Model)

- **Slice type**: subtask
- **Slice outcome**: users and future task specs learn the GitHub issue lifecycle from `ace-task` docs/workflows rather than PR footer conventions
- **Advisory size**: small
- **Context dependencies**: task docs, task workflow instructions, existing linked task specs
