# Add soft GitHub issue integration for ace-task

## Objective

Make linked GitHub issues track ACE task state directly through `ace-task`, so issue lifecycle and traceability are no longer coupled to PR body footers.

## Behavioral Specification

### User Experience

- A user creates or updates an ACE task and links one or more GitHub issues in machine-readable frontmatter.
- If a task has linked GitHub issues, `ace-task` automatically keeps those GitHub issues in sync with the ACE task without requiring PR footer conventions.
- The GitHub issue stays visibly connected to ACE through a stable tracking label and a single updatable comment that shows the ACE task ref, title, directory path, and current ACE state.
- When the ACE task reaches a terminal state, the linked GitHub issue closes. If the task later becomes active again, the GitHub issue reopens automatically.

### Expected Behavior

1. `ace-task` supports explicit GitHub issue linkage as task metadata rather than relying on prose-only references or PR body conventions.
2. Linked issues are synced automatically after relevant task mutations, including create, title/status updates, reparenting, path changes, and archive moves.
3. A linked GitHub issue gets one stable label, `ace:tracked`, and one sticky comment managed by ACE.
4. The sticky comment is updated in place rather than duplicated, and it reflects the current task directory path even after rename/archive moves.
5. ACE closes linked GitHub issues when the task reaches `done`, `skipped`, or `cancelled`, and reopens them if the task later moves back to a non-terminal state.
6. PRs no longer need to be the primary mechanism for closing ACE-linked GitHub issues.

### Interface Contract

- **Task metadata contract**
  ```yml
  github:
    issues: [276, 278]
  ```
- **CLI contract**
  ```bash
  ace-task create "Add feature" --github-issue 276
  ace-task github-sync t.abc
  ace-task github-sync --all
  ```
- **GitHub issue contract**
  - Label: `ace:tracked`
  - Sticky comment template:
    ```md
    Tracked in ace-task: [8r4.t.ilo](https://github.com/cs3b/ace/blob/main/.ace-tasks/8r4.t.ilo-add-soft-github-issue-integration/8r4.t.ilo-add-soft-github-issue-integration-for-ace.s.md)
    ```
  - Comment format is intentionally minimal: one line with a clickable ACE task ref
- **Lifecycle contract**
  - GitHub issue remains `open` or `closed` natively.
  - ACE status remains the source of truth for work state.
  - GitHub issue open/close state follows ACE terminal vs non-terminal state only for linked tasks.

### Success Criteria

- [ ] Users can link GitHub issues to ACE tasks using machine-readable task metadata.
- [ ] Linked issues automatically receive one sticky ACE sync comment and the `ace:tracked` label.
- [ ] Sticky comments update when task ref or task file path changes, while staying one-line and clickable.
- [ ] Linked issues close on ACE terminal states and reopen when ACE tasks return to active states.
- [ ] PR footer-based issue closure is no longer required for ACE-linked work.
- [ ] Usage documentation explains the new ACE-driven issue lifecycle clearly.

### Validation Questions

- Should linked issues be required at task creation time, or remain optional metadata that can be added later?
- Should a task linked to multiple issues mirror identical status to all linked issues, or support per-issue overrides in a later phase?

## Review Questions (Resolved)

### [RESOLVED] Should linked issues be required at task creation time, or remain optional metadata that can be added later?
- **Decision**: Remain optional metadata. Creating a task does not require a GitHub issue link.
- **Rationale**: Many ACE workflows should not depend on GitHub presence at task creation time, and some tasks are not GitHub-centric.
- **Implications**: `ace-task` and docs should treat `github.issues` as optional metadata and only perform GitHub synchronization when the field is present.
- **Resolved by**: User
- **Date**: 2026-04-05

### [RESOLVED] Should a task linked to multiple issues mirror identical status to all linked issues, or support per-issue overrides in a later phase?
- **Decision**: Mirror the ACE task state identically to all linked issues in v1.
- **Rationale**: The current scope requires deterministic behavior and avoids introducing per-link policy complexity.
- **Implications**: `ace-task` sync behavior for v1 applies the same state updates and comments to each issue in `github.issues`.
- **Resolved by**: User
- **Date**: 2026-04-05
