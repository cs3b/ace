# Add CI Performance Gate to GitHub Actions

## Behavioral Specification

### User Experience
- **Input**: PR submitted with test changes
- **Process**: CI runs performance check on test suite
- **Output**: PR blocked if any unit test exceeds 100ms threshold

### Expected Behavior

When a PR is submitted:
1. CI runs `ace-test --profile 20` on affected packages
2. Parses output for tests exceeding thresholds
3. Fails the build if any unit/integration test >100ms
4. Reports specific slow tests in CI output

This enforces the "100ms Rule" from research synthesis.

### Interface Contract

```yaml
# .github/workflows/test.yml addition
- name: Check test performance
  run: |
    ace-test --profile 20 | tee profile.txt
    # Fail if any test > 100ms (0.1s)
    if grep -E "^\s+\d+\.\d+s" profile.txt | awk '$1 > 0.1 {exit 1}'; then
      echo "::error::Unit tests exceeding 100ms threshold"
      exit 1
    fi
```

**Thresholds (from research):**
| Layer | Budget | CI Action |
|-------|--------|-----------|
| Unit (Atoms) | <10ms | Warning |
| Integration (Molecules) | <50ms | Warning |
| Any Fast Loop | >100ms | **Fail** |
| Any Fast Loop | >200ms | **Block merge** |

### Success Criteria

- [ ] Performance gate added to test workflow
- [ ] Threshold set at 100ms for fast loop tests
- [ ] Clear error messages identifying slow tests
- [ ] Warning annotations for tests 50-100ms
- [ ] Documentation in workflow comments

## Objective

Automate enforcement of the 100ms Rule, preventing slow tests from being merged and catching hidden I/O leaks early.

## Technical Approach

### Current CI Architecture

The existing `.github/workflows/test.yml`:
- Matrix tests across Ruby 3.2, 3.3, 3.4
- Tests 11 packages in parallel
- Uses `bundle exec rake ci` per package
- Uploads artifacts on failure

### Performance Gate Strategy

**Option A: Add step to existing job**
- Pro: No additional job overhead
- Con: Must parse Minitest output format
- Con: Profile flag may need Rakefile changes

**Option B: Separate performance job**
- Pro: Clear separation of concerns
- Pro: Can run after tests pass
- Con: Additional CI time

**Recommended: Option A** - Add inline step using `ace-test --profile`

### Output Parsing

Minitest profile output format:
```
Finished in 3.456789s, 28.9583 runs/s, 73.1944 assertions/s.

Top 10 slowest tests:

  1) LongTest#test_slow_operation (0.523s)
  2) AnotherTest#test_another (0.102s)
```

Need to extract time values and compare to threshold.

### Execution Steps

- [ ] Add performance check step to test workflow
  ```yaml
  - name: Check test performance
    if: success()  # Only run if tests pass
    run: |
      cd ${{ matrix.package }}
      # Run with profiling and capture output
      bundle exec rake test TESTOPTS="--profile 20" 2>&1 | tee ../profile-output.txt

      # Parse output for slow tests (>100ms = 0.1s)
      # Minitest format: "  1) TestName#test_method (0.123s)"
      if grep -oP '\([\d.]+s\)' ../profile-output.txt | grep -oP '[\d.]+' | awk '$1 > 0.1 {found=1} END {exit !found}'; then
        echo "::error::Tests exceeding 100ms threshold detected"
        echo "## Slow Tests Detected" >> $GITHUB_STEP_SUMMARY
        grep -E '\([0-9]+\.[0-9]+s\)' ../profile-output.txt | head -20 >> $GITHUB_STEP_SUMMARY
        exit 1
      fi
  ```

## Risk Assessment

### Technical Risks
- **Risk:** Profile output format varies across packages
  - **Probability:** Medium
  - **Impact:** Medium (parsing may fail)
  - **Mitigation:** Test across multiple packages
  - **Rollback:** Make performance step optional initially

- **Risk:** Profile overhead slows CI
  - **Probability:** Low (profiling is lightweight)
  - **Impact:** Low (minor time increase)
  - **Mitigation:** Only profile top 20 tests
  - **Monitoring:** Compare CI times before/after

### Integration Risks
- **Risk:** Existing slow tests cause immediate failures
  - **Probability:** Medium (known slow tests exist)
  - **Impact:** High (blocks all PRs)
  - **Mitigation:** Start with warning-only mode
  - **Monitoring:** Review CI results after deployment
