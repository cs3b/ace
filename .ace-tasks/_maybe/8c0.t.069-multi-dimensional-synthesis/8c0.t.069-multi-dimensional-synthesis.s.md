# Explicit Review Presets with Provider Thinking Controls

## Goal

Reframe the future work from `8c0.t.069` around the architecture that proved highest-value in `ace-improve-review`: explicit single-file `ace-review` presets, provider-owned thinking/runtime controls in `ace-llm`, and optional narrow reviewer lanes only as a secondary extension.

## Current State

The original `8c0.t.069` direction assumed the highest-value future for `ace-review` was weighted multi-dimensional synthesis with reviewer metadata flowing through the execution chain. The work in `ace-improve-review` demonstrated a different result:

- **Explicit review presets** were easier to author, reason about, and debug than a pipeline/reviewer/partition stack
- **Provider thinking control** belonged in `ace-llm`, not `ace-review`
- **Repo-level `.ace/review` overrides** were part of the real runtime surface and could silently defeat package defaults if left unsynchronized
- **Simple model-output synthesis** was sufficient for the baseline product; weighted reviewer consensus was extra complexity without clear primary value
- **Optional narrow reviewer lanes** may still be valuable later, but only when they are explicit and subordinate to the preset definition

The next-value roadmap should therefore improve the explicit preset contract, strengthen provider-side thinking control, and reserve reviewer lanes for a clearly opt-in advanced mode.

## Concept Inventory

| Concept | Description | Introduced In |
|---------|-------------|---------------|
| Explicit preset contract | One self-contained preset file defines review intent, prompt composition, subject behavior, and model targets | 069.0 |
| Provider thinking controls | `ace-llm` maps explicit thinking levels to provider-native runtime settings while preserving `@ro`, `@rw`, and `@yolo` | 069.1 |
| Config cascade alignment | Package defaults and repo-level `.ace` overrides must resolve to the same runtime behavior | 069.0 |
| Optional narrow reviewer lanes | A preset may later opt into narrower internal lanes, but the baseline architecture remains preset-centric | 069.2 |
| Baseline synthesis simplicity | Default review execution stays model-output based and understandable without reviewer metadata | 069.2 |

## Subtasks

- **069.0**: Formalize the explicit preset contract and authoring model
  - Define the single-file preset contract as the primary review-definition unit
  - Clarify validation, authoring expectations, and config-cascade behavior
  - Foundation for the other subtasks
  - Advisory size: medium

- **069.1**: Extend provider-side thinking controls in `ace-llm`
  - Broaden provider coverage for explicit thinking levels
  - Keep effort/runtime control in `ace-llm`, not `ace-review`
  - Advisory size: medium

- **069.2**: Add optional narrow reviewer lanes as a secondary extension
  - Preserve explicit presets as the default path
  - Allow future lane-mode execution only when declared directly in the preset
  - Depends on 069.0
  - Advisory size: medium

## Acceptance Criteria

- [ ] The rewritten task clearly defines explicit presets as the primary `ace-review` architecture
- [ ] Provider effort/thinking control is explicitly owned by `ace-llm`
- [ ] Future preset growth is possible without reintroducing hidden reviewer/pipeline layers by default
- [ ] Package defaults and repo-level `.ace/review` overrides are treated as part of the same contract
- [ ] Optional narrow reviewer lanes are described as an opt-in extension, not the baseline model
- [ ] The three subtasks form a coherent roadmap under this direction
- [ ] The rewritten task is future-facing and no longer treats weighted reviewer synthesis as the primary goal

## Out of Scope

- Restoring weighted consensus, confidence scoring, or critical/focus synthesis as the default review model
- Reintroducing mandatory reviewer pipelines, partition builders, or tool-integrated review flows
- Designing provider-specific effort knobs inside `ace-review`
- Immediate implementation work; this task now defines future-value planning only
