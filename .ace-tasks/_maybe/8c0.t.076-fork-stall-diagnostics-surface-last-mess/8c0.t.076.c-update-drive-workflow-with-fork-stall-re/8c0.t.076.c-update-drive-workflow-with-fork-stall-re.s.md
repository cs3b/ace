# Update drive workflow with fork stall recovery patterns

## Behavioral Specification

### User Experience
- **Input**: Agent loads `wfi://assign/drive` workflow
- **Process**: Workflow now includes documented patterns for detecting and recovering from fork stalls
- **Output**: Agents following the drive workflow can handle stalled forks without manual debugging

### Expected Behavior

The drive workflow (`ace-assign/handbook/workflow-instructions/assign/drive.wf.md`) gains a new section "Recovering from Stalled Forks" that teaches agents:

1. How to read `stall_reason` from `ace-assign status` output
2. When to use `ace-assign fork-resume` vs fail + retry
3. Example messages for common stall causes (unexpected git state, auth failures, etc.)
4. The complete diagnostic-to-recovery flow

### Interface Contract

New section in `drive.wf.md`:

```markdown
#### Recovering from Stalled Forks

When fork-run fails with "did not complete within spawned session", the error includes the agent's last message.

Common stall causes and recovery:
1. Agent asked a question → `ace-assign fork-resume "answer" --assignment <id>@<root>`
2. Unexpected git state → `ace-assign fork-resume "Commit only your changes" --assignment ...`
3. Unrecoverable error → `ace-assign fail --message "reason"` then retry or skip
```

### Success Criteria

- [ ] Drive workflow has "Recovering from Stalled Forks" section
- [ ] Documents `ace-assign fork-resume` usage with examples
- [ ] Documents when to fail + retry vs resume
- [ ] Includes common stall patterns and suggested messages
- [ ] `ace-assign status` stall_reason reading documented
