# Human Approval Gates

## Objective

Pause workflow execution for explicit human approval at critical phases, persisting gate state for resume/decision handling.

## Scope of Work

### Included
- `gate: true` + optional `gate_prompt:` in phase frontmatter
- Assignment paused state persisted in `assignment.yaml`
- `ace-assign resume --approve` and `--reject --reason "..."` commands
- Status output shows gate prompt and resume instructions

### Excluded
- Multiple gate types (only human gate)
- Timeout/conditional gates
- Auto-rewind on rejection
- External approvals/webhooks

## Technical Approach

### Architecture Pattern
- [ ] Extend the queue state machine with an assignment-level paused state
- [ ] Treat gate phases as non-executable markers that block advancement
- [ ] Persist gate metadata in `assignment.yaml` for resume operations

### Technology Stack
- [ ] Use existing YAML serialization via `Assignment#to_h` and `AssignmentManager`
- [ ] Preserve current phase file schema with minimal new fields
- [ ] Ensure compatibility with queue scanning and status rendering

### Implementation Strategy
- [ ] Add gate metadata to `Phase` model and parsing pipeline
- [ ] Store assignment `status` and `gate` state with backward-compatible defaults
- [ ] Introduce a `resume` command that clears or rejects gate state

## Tool Selection

| Criteria | Option A | Option B | Option C | Selected |
|----------|----------|----------|----------|----------|
| Persistence | Extend assignment.yaml | New gate.yaml file | In-memory only | Option A |
| Integration | Centralized | Split state | Volatile | Option A |
| Maintenance | Low | Medium | High risk | Option A |
| Backward compatibility | High | Medium | Low | Option A |
| UX clarity | High | Medium | Low | Option A |

**Selection Rationale:** Extend `assignment.yaml` because assignment metadata already drives status selection and is the natural place to persist pause state.

### Dependencies

- [ ] Compatibility verified with existing YAML parsing

## Risk Assessment

### Technical Risks
- **Risk:** Assignment schema change breaks existing assignment loads
  - **Probability:** Medium
  - **Impact:** Medium
  - **Mitigation:** Provide defaults for new fields and tolerate unknown keys
  - **Rollback:** Revert schema additions and ignore gate fields

- **Risk:** Paused assignments still show an in_progress phase
  - **Probability:** Medium
  - **Impact:** Low
  - **Mitigation:** Gate logic clears in-progress phase before pausing
  - **Rollback:** Restore previous status rendering

### Integration Risks
- **Risk:** Status command lacks gate context after multi-assignment changes
  - **Mitigation:** Use assignment status and gate data over queue-only view
  - **Monitoring:** Add a status test that includes gate prompt
