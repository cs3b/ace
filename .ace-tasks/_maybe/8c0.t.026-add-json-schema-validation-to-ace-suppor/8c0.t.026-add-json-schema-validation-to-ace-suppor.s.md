# Add JSON Schema validation to ace-support-markdown (v0.2.0)

## Behavioral Specification

### User Experience
- **Input**: Developers define JSON Schema files for frontmatter validation in configuration directories
- **Process**: ace-support-markdown validates frontmatter against schemas before write operations, providing clear error messages
- **Output**: Validated markdown files with detailed, actionable error messages when validation fails

### Expected Behavior

Developers can define validation rules using JSON Schema instead of hardcoded validation logic. The system validates frontmatter against these schemas before writing, preventing invalid data from being persisted.

Instead of hardcoded validation rules in v0.1.0, v0.2.0 supports schema-based validation:

**Current Problem (v0.1.0):** Hardcoded validation rules scattered across atoms, difficult to maintain and extend.

**Solution (v0.2.0):** JSON Schema-based validation with configurable schema files.

### Interface Contract

```ruby
# Define schema for task frontmatter
schema = {
  "type" => "object",
  "required" => ["id", "status", "priority"],
  "properties" => {
    "id" => {"type" => "string", "pattern" => "^v\\.\\d+\\.\\d+\\.\\d+\\+task\\.\\d+$"},
    "status" => {"enum" => ["draft", "pending", "in-progress", "done", "blocked"]},
    "priority" => {"enum" => ["high", "medium", "low"]},
    "estimate" => {"type" => "string"},
    "dependencies" => {"type" => "array", "items" => {"type" => "string"}}
  }
}

# Validate with schema
editor = Ace::Support::Markdown::DocumentEditor.new("079.md")
editor.update_frontmatter({"status" => "invalid"})
result = editor.save!(validate: true, schema: schema)
# Returns: {success: false, errors: ["status: must be one of draft, pending, in-progress, done, blocked"]}

# Load schema from configuration
schema_path = ".ace.example/markdown/schemas/task.json"
editor.save!(validate: true, schema_file: schema_path)
```

**Error Handling:**
- Invalid frontmatter against schema: Clear error messages with field paths and expected formats
- Missing schema file: Fallback to hardcoded validation with warning
- Malformed schema file: Detailed JSON Schema parsing error
- Validation performance issues: <5ms overhead for typical schemas

**Edge Cases:**
- Multiple schema files: Support different schemas for different document types
- Schema versioning: Single schema version per document type (versioning deferred to later release)
- Backward compatibility: Hardcoded validation remains as fallback

## Objective

Add JSON Schema validation support to ace-support-markdown for flexible, maintainable frontmatter validation. This moves validation rules from hardcoded atoms to declarative schema files, making it easier to maintain and extend validation rules across different document types.

## Technical Approach

### Architecture Pattern
- **ATOM Architecture**: Follow ace-support-markdown pattern with Atoms (pure functions), Molecules (composed operations)
- **Integration Strategy**: Extend existing DocumentValidator atom with schema validation support
- **Dependency Management**: Add json-schema gem ~> 4.0 to gemspec

**Rationale**: Maintain consistency with v0.1.0 architecture, minimize disruption to existing validation flow.

### Technology Stack
- **JSON Schema Library**: json-schema gem ~> 4.0 (Ruby stdlib compatible, mature, actively maintained)
- **Schema Storage**: JSON files in .ace.example/markdown/schemas/ directory
- **Fallback**: Existing hardcoded validation from v0.1.0

**Version Requirements**:
- Ruby >= 2.7 (ACE standard)
- json-schema ~> 4.0 (latest stable)

**Performance Considerations**:
- JSON Schema validation: ~2-5ms for typical frontmatter
- Schema file loading: Cached after first load
- Target: <5ms overhead total

2. **Phase 2: Integration** - Extend DocumentValidator with schema support

4. **Phase 4: Error Formatting** - User-friendly error messages

## Risk Assessment

### Technical Risks

- **Risk:** json-schema gem performance issues with large schemas
  - **Probability:** Low
  - **Impact:** Medium (slower validation)
  - **Mitigation:** Schema caching, performance benchmarking
  - **Monitoring:** Add performance tests, set <5ms threshold

- **Risk:** Schema validation errors difficult to understand
  - **Probability:** Medium
  - **Impact:** Medium (poor UX)
  - **Mitigation:** Custom error formatting, clear field paths
  - **Rollback:** Improve error message formatting iteratively

### Integration Risks

- **Risk:** Breaking changes to DocumentValidator API
  - **Probability:** Low
  - **Impact:** High (breaks v0.1.0 users)
  - **Mitigation:** Optional schema parameter, backward compatible design
  - **Rollback:** Git revert if compatibility broken
