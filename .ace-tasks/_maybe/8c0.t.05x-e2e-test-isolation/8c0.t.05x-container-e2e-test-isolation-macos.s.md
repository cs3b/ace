# Container-Based E2E Test Isolation for macOS

## Behavioral Specification

### User Experience
- **Input**: Developer runs E2E tests via `ace-test-e2e` or agent runs via `/ace:run-e2e-test`
- **Process**: Tests execute inside a container strategy that matches platform:
  - macOS: lightweight Linux container via Lima/OrbStack
  - Linux/CI: lightweight LXC container with explicit host mount policy
- **Output**: Test results with full filesystem isolation, no host pollution

### Expected Behavior

When E2E tests require true filesystem isolation:
1. Container image with Ruby + project gems pre-installed is available
2. Test artifacts are mounted to a dedicated host artifact path for persistence
3. Git operations inside container don't affect host repositories
4. Tests run identically on CI and local development

## Runtime Profile

- **macOS primary**: Lima
- **macOS alternative**: OrbStack
- **Linux path**: LXC with project and artifact mount policy

## Interface Contract

```bash
# Option 1: Direct container execution
ace-test-e2e --container MT-WORKTREE-002

# Option 2: Container wrapper script
ace-container-test ace-git-worktree MT-WORKTREE-002

# Mounts
.cache/test-e2e/  -> /workspace/artifacts  (read-write)
project/          -> /workspace/src        (read-only, optional)
```

**Error Handling:**
- Container not running: Auto-start with helpful message
- Image missing: Pull or build with progress indicator
- Mount permissions: Clear error with resolution steps

## Linux runtime variant (LXC)

### Why this path exists
`8pkys1` established a Linux-focused isolation model where the container host surface is minimal and only artifacts are shared. This task now includes that path as the Linux-first option.

### LXC bootstrap and execution

1) Install tooling and validate kernel/container support

```bash
sudo pacman -Syu lxc lxc-templates
sudo pacman -S iptables dnsmasq bridge-utils
sudo lxc-checkconfig
```

2) Create a container (example)

```bash
sudo lxc-create -n ace-test-e2e -t download -- --dist archlinux --release latest --arch amd64
```

3) Bind-mount only artifact/report path

```text
mkdir -p /home/<user>/container_reports
lxc.mount.entry = /home/<user>/container_reports /reports none bind,create=dir,ro
```

4) Execute E2E workload in the container

- Full interactive:

```bash
sudo lxc-start -n ace-test-e2e
sudo lxc-attach -n ace-test-e2e
cd /reports
ace-test-e2e --container MT-WORKTREE-002
```

- One-shot command:

```bash
sudo lxc-execute -n ace-test-e2e -- /reports/run-script.sh
```

5) Optional: set `lxc.init.cmd = /reports/run-script.sh` for auto-start behavior

### LXC hardening rules

- Prefer unprivileged containers with user namespace mapping configured in `/etc/subuid` and `/etc/subgid`
- Keep host-visible mount surface to artifacts/reports-only
- Validate container bootstrap and command context before test execution

## Objective

E2E tests that manipulate git repositories need true filesystem isolation. The current approach creates temporary directories on the host, which:
- Can leak test artifacts if cleanup fails
- Cannot be safely run in parallel
- Behaves differently on macOS vs Linux (CI)

Container-based isolation provides:
- Hermetic test environments
- Parallel execution safety
- Cross-platform consistency
- Simple cleanup (remove container)

## Technical Notes

### Container Technology Options

**Lima (recommended)**
- Native macOS virtualization (Virtualization.framework)
- Light resource usage
- Supports containerd/nerdctl
- Free and open source

**OrbStack (alternative)**
- Drop-in Docker replacement for macOS
- Excellent performance
- Commercial product (free tier available)
- Simpler setup than Lima

### Volume Mounts

```yaml
# Lima config excerpt
mounts:
  - location: "~/.cache/test-e2e"
    writable: true
  - location: "~/Projects/ace-task"
    writable: false
```
