# ACE Context Ruby Code Parsing - Compact Code Structure Output

## Behavioral Specification

### User Experience
- **Input**: Users provide Ruby file paths or glob patterns to ace-context (e.g., `ace-context load --ruby-code 'lib/**/*.rb'`)
- **Process**: System parses Ruby files and extracts structural elements (classes, modules, methods, parameters, documentation) into a compact, deterministic representation
- **Output**: Users receive a token-efficient, structured "code skeleton" that shows the architecture and public interfaces of Ruby code without verbose implementation details

### Expected Behavior

When AI agents or developers need to understand the structure of existing Ruby code for planning features, refactoring, or debugging, they can use ace-context to load Ruby files in a compact format. Instead of receiving raw file content with all implementation details, they receive a structured overview showing:

- Class and module hierarchies
- Method signatures with parameters
- Public interfaces and APIs
- Associated documentation (YARD docs, comments)
- No implementation bodies or internal logic details

This compact representation significantly reduces token consumption while providing all necessary structural information for effective planning and code understanding. The output is consistent and deterministic across runs, ensuring reliable AI agent inputs.

### Interface Contract

```bash
# Load Ruby code structure into context
ace-context load --ruby-code 'lib/**/*.rb'
ace-context load --ruby-code 'lib/ace/core.rb'

# Configure parsing depth and inclusion rules
ace-context load --ruby-code 'lib/**/*.rb' --ruby-config 'ace.context.ruby_code_parser'

# Expected compact output format (example):
# Module: Ace::Context
#   Class: RubyCodeParser
#     - #initialize(file_path)
#     - #parse -> RubyCodeStructure
#     - #extract_methods(ast) -> Array<MethodSignature>
#   Class: MethodSignature
#     - attr_reader :name, :params, :doc
#     - #initialize(name, params, doc)
```

**Error Handling:**
- Malformed Ruby files: Report parsing errors with file path and line number, continue with other files
- Invalid glob patterns: Display helpful error message with pattern syntax examples
- Missing files: Clear error indicating which files were not found

**Edge Cases:**
- Empty Ruby files: Return minimal structure indicating empty file
- Ruby files with only comments: Extract documentation, indicate no code structures found
- Mixed valid/invalid files in glob: Process valid files, report errors for invalid ones
- Large codebases: Handle efficiently without significant performance degradation

### Success Criteria

- [ ] **Compact Representation**: Users receive Ruby code structures in 50-70% fewer tokens compared to raw file content
- [ ] **Structural Clarity**: Output clearly shows class/module hierarchies, method signatures, and parameters without implementation details
- [ ] **Documentation Integration**: Associated YARD docs and comments are included in the compact representation
- [ ] **Deterministic Output**: Same input files produce identical output across multiple runs
- [ ] **CLI Integration**: ace-context command accepts Ruby file paths and glob patterns seamlessly
- [ ] **Configuration Support**: Users can control parsing depth, inclusion/exclusion rules, and output format via configuration
- [ ] **Error Resilience**: System gracefully handles malformed files and provides actionable error messages
- [ ] **Performance**: Processing large Ruby codebases (100+ files) completes within acceptable time (< 5 seconds)

### Validation Questions

- [ ] **Output Format**: Should the compact representation use YAML, structured markdown, or a custom DSL format? What works best for LLM consumption?
- [ ] **Parsing Depth**: Should private methods be included by default, or only public interfaces? Should users control this?
- [ ] **Documentation Extraction**: Which documentation formats should be supported (YARD, RDoc, inline comments)? How detailed should doc extraction be?
- [ ] **Configuration Scope**: What specific parsing options do users need (e.g., exclude test files, include method bodies for specific patterns)?
- [ ] **Integration Points**: How should this integrate with existing ace-context caching mechanisms? Should cached structures be invalidated when source files change?
- [ ] **Multi-file Aggregation**: When processing multiple files, should output be per-file or aggregated into a unified structure? Both options?
- [ ] **Performance Targets**: What are acceptable performance thresholds for different codebase sizes?

## Objective

Enable AI agents and developers to efficiently understand Ruby codebase structure by providing compact, token-efficient representations of code architecture. This improves AI planning capabilities, reduces token consumption, and provides deterministic, structured context for better decision-making during feature development, refactoring, and debugging tasks.
