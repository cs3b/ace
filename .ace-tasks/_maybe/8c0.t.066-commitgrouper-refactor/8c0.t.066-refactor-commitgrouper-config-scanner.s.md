# Refactor CommitGrouper to use ProjectConfigScanner for ace-config grouping

## Behavioral Specification

### User Experience

- **Input**: List of changed files from git (includes `.ace/**` files)
- **Process**: Group files by scope, with all `.ace/**` files grouped as "ace-config"
- **Output**: Commit groups where `.ace/**` files are in single "ace-config" group

### Expected Behavior

When running `ace-git-commit` with changes to multiple `.ace/git/commit.yml` files across packages:

**Current (broken):**
```
15+ separate commits:
- chore(ace-bundle): Update commit config
- chore(ace-docs): Update commit config
- chore(ace-lint): Update commit config
...
```

**Expected (fixed):**
```
1 commit:
- chore(ace-config): Update git commit configurations
  Files: ace-bundle/.ace/git/commit.yml, ace-docs/.ace/git/commit.yml, ...
```

### Interface Contract

```ruby
# CommitGrouper already has this interface:
grouper = CommitGrouper.new
groups = grouper.group(files, project_root: project_root)

# Current behavior: each .ace file gets scope from its package
# New behavior: all .ace/** files get scope "ace-config"
```

**Integration with ProjectConfigScanner:**
```ruby
# Inside CommitGrouper#group:
scanner = Ace::Support::Config::Molecules::ProjectConfigScanner.new(project_root: project_root)
ace_locations = scanner.scan.keys  # All .ace folder locations

files.each do |file|
  if file_in_ace_folder?(file, ace_locations)
    groups["ace-config"].add_file(file)
  else
    # Normal resolution via FileConfigResolver
    resolved = @file_config_resolver.resolve(file)
    # ... existing logic
  end
end
```

**Error Handling:**
- Scanner fails: Fall back to current behavior (per-file resolution)
- Edge case .ace files: Include in ace-config group

## Technical Approach

### Architecture Pattern

- **Layer**: Molecule modification (CommitGrouper)
- **Pattern**: Pre-filter with scanner, then delegate to existing resolver
- **Integration**: Inject optional ProjectConfigScanner, graceful fallback

### Technology Stack

- **Dependencies**: ace-support-config (ProjectConfigScanner from 228.02)
- **No new gems required**

### Implementation Strategy

The refactoring will:
1. Add optional `config_scanner` parameter to CommitGrouper
2. In `group()` method, first check if file is in any `.ace/` folder
3. If yes, add to "ace-config" group directly (skip FileConfigResolver)
4. If no, use existing FileConfigResolver logic
5. Handle scanner failures gracefully (fallback to current behavior)

**Key Design Decisions:**
- "ace-config" is hardcoded scope name (per validation questions)
- Scanner is optional (constructor injection for testability)
- Fallback to current behavior on any scanner error
- Simple path matching: `file.include?("/.ace/")` or starts with `.ace/`

## Test Cases

### Happy Path

1. **Multiple .ace files from different packages group together**
   - Input: `["ace-bundle/.ace/git/commit.yml", "ace-docs/.ace/git/commit.yml", "ace-lint/.ace/git/commit.yml"]`
   - Expected: Single group with scope_name "ace-config", 3 files

2. **Mixed .ace and non-.ace files separated correctly**
   - Input: `["ace-bundle/.ace/git/commit.yml", "ace-bundle/lib/foo.rb", "docs/README.md"]`
   - Expected: Two groups: "ace-config" (1 file), normal groups for other files

### Edge Cases

3. **Root .ace/ files included**
   - Input: `[".ace/git/commit.yml"]`
   - Expected: Single group "ace-config" with 1 file

4. **Deeply nested .ace/ folders**
   - Input: `["pkg/sub/.ace/config.yml"]`
   - Expected: File in "ace-config" group

5. **No .ace files**
   - Input: `["foo.rb", "bar.md"]`
   - Expected: Normal grouping, no "ace-config" group created

6. **Only .ace files**
   - Input: `[".ace/a.yml", "pkg/.ace/b.yml"]`
   - Expected: Single "ace-config" group with 2 files, no other groups
