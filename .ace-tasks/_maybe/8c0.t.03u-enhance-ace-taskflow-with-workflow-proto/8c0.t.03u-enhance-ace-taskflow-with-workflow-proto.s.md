# Enhance ace-taskflow with Workflow Protocol and Prompt Overrides

## Behavioral Specification

### User Experience
- **Input**: Task definitions or preset configurations that specify protocol URIs (e.g., `wfi://load-project-context`, `prompt://git-commit.system`)
- **Process**: Agents and users interact with ace-taskflow using familiar commands, with protocols transparently resolved and executed
- **Output**: Dynamic workflow invocation, context loading, and prompt customization without manual intervention

### Expected Behavior

AI agents and users working with ace-taskflow should be able to declaratively specify workflow invocations and prompt overrides using standardized protocol URIs. When a task includes protocol definitions:

1. **Workflow Protocol (`wfi://`)**: Tasks can reference workflow instructions that should be executed
   - `wfi://load-project-context` - Dynamically loads project context via ace-context
   - `wfi://work-on-task` - Invokes task-specific navigation and resource discovery via ace-nav
   - Protocols are resolved through the configuration cascade
   - Execution is delegated to the appropriate tool (ace-context, ace-nav)

2. **Prompt Protocol (`prompt://`)**: Tasks can override default LLM prompts for specific tools
   - `prompt://git-commit.system` - Customizes the system prompt for ace-git-commit
   - Prompts are resolved from the configuration cascade (`.ace/prompts/`, `handbook/prompts/`)
   - Overrides follow clear precedence: task-specific > project-level > gem defaults
   - ace-llm consumes the resolved prompts for LLM interactions

The system transparently handles protocol resolution, validation, and execution, providing enhanced agent autonomy and workflow flexibility without requiring manual scripting or intervention.

### Interface Contract

```bash
# Protocol definitions in task frontmatter or presets
---
id: v.0.9.0+task.146
protocols:
  workflows:
    - wfi://load-project-context
    - wfi://work-on-task
  prompts:
    git-commit.system: prompt://git-commit.system
---

# ace-context resolves and loads workflow protocols
ace-context wfi://load-project-context
# Output: Loads project-specific context based on protocol definition

# ace-context resolves prompt overrides
ace-context prompt://git-commit.system
# Output: Returns prompt content from configuration cascade

# ace-nav executes workflow protocols
ace-nav wfi://work-on-task
# Output: Executes task-specific navigation workflow

# ace-git-commit uses overridden prompts
ace-git-commit
# Uses prompt://git-commit.system instead of default system prompt
```

**Error Handling:**
- Invalid protocol URI: Clear error message with valid protocol examples
- Protocol not found: Shows search path and available protocols
- Prompt injection attempt: Validates and sanitizes prompt content, rejects malicious content
- Circular protocol references: Detects and prevents infinite resolution loops

**Edge Cases:**
- Multiple prompt overrides for same tool: Last definition wins (task > project > gem)
- Protocol with no matching definition: Falls back to default behavior with warning
- Mixed protocol versions: Uses protocol version from active release context

### Success Criteria

- [ ] **Protocol Resolution**: ace-context and ace-nav can parse and resolve `wfi://` protocol URIs from task definitions
- [ ] **Prompt Override Resolution**: ace-context can locate and load prompt content via `prompt://` URIs following configuration cascade precedence
- [ ] **Workflow Invocation**: Tasks with `wfi://` protocols automatically trigger appropriate workflow execution
- [ ] **LLM Integration**: ace-llm and tools like ace-git-commit consume overridden prompts resolved by ace-context
- [ ] **Configuration Cascade**: Protocol and prompt resolution respects precedence (task > project > gem)
- [ ] **Validation and Security**: Robust validation prevents prompt injection and invalid protocol references
- [ ] **Error Reporting**: Clear, actionable error messages for invalid or missing protocols
- [ ] **Documentation**: Clear examples and guides for defining protocols in task presets

### Validation Questions

- [ ] **Protocol Syntax**: Should protocol URIs support parameters/query strings (e.g., `wfi://load-context?preset=project`)?
- [ ] **Scope of Protocols**: Should protocols be scoped per-task, per-release, or globally available?
- [ ] **Prompt Format**: What format should prompt overrides use (plain text, markdown, structured YAML)?
- [ ] **Conflict Resolution**: How should conflicts between multiple protocol definitions be resolved?
- [ ] **Backward Compatibility**: How will this affect existing tasks without protocol definitions?
- [ ] **CLI Discovery**: Should there be a command to list available protocols (e.g., `ace-nav --list-protocols`)?
- [ ] **Caching**: Should resolved protocols be cached for performance, and how to invalidate cache?

## Objective

Enable AI agents and users to dynamically customize ace-taskflow behavior through declarative protocol definitions, increasing workflow flexibility, agent autonomy, and prompt management capabilities while maintaining security and usability.
