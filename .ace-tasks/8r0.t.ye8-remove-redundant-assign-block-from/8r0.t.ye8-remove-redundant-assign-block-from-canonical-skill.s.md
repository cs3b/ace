# Remove Redundant assign Block from Canonical Skill Schema

## Objective

The `assign:` block in canonical SKILL.md files is redundant and should be removed. It duplicates data available through `skill.execution.workflow` and ace-assign catalog step files. Removing it simplifies the canonical skill schema, eliminates a lint/runtime mismatch, and makes the single-source-of-truth principle clearer.

**Why now**: The lint schema allows `assign.source` and `assign.phases` but not `assign.steps` — yet runtime reads `assign.steps`. Meanwhile, `assign.phases` is allowed by lint but never read by any code. This schema/runtime mismatch signals dead metadata that should be removed rather than fixed.

## Behavioral Specification

### User Experience

- **Input**: No user input required — this is an internal schema cleanup
- **Process**: Remove `assign:` from SKILL.md files, update resolver to use `skill.execution.workflow`, update lint to reject `assign:`, re-sync provider projections
- **Output**: Cleaner canonical skills, consistent schema, no behavioral change to ace-assign execution or provider skill projections

### Expected Behavior

After this change:
1. Canonical SKILL.md files no longer contain `assign:` blocks
2. `SkillAssignSourceResolver` reads workflow URI from `skill.execution.workflow` instead of `assign.source`
3. Assignment step catalog is built exclusively from catalog step files (`.ace-defaults/assign/catalog/steps/`)
4. `ace-lint` rejects `assign:` as an unknown field in SKILL.md
5. `ace-handbook sync` produces projections without `assign:` blocks
6. Assignment execution behavior is unchanged — presets and catalog steps already provide all necessary data

### Interface Contract

No external interfaces change. Internal changes:

**SkillAssignSourceResolver API** (internal, no BC guarantee pre-1.0):
- `resolve_assign_config(skill_name)` — reads `skill.execution.workflow` instead of `assign.source`
- `assign_step_catalog` — returns `[]` (catalog step files are authoritative)
- `assign_capable_skill_names` — checks `skill.execution.workflow` presence instead of `assign.source`

**Lint schema**:
- `assign:` becomes an unknown/rejected field in SKILL.md validation

Error Handling:
- Skills without `skill.execution.workflow` return nil from `resolve_assign_config` (same as current behavior for missing `assign.source`)
- Missing workflow files raise `Ace::Assign::Error` with clear message (unchanged)

Edge Cases:
- Skills that had `assign.source` but no `skill.execution.workflow` — verified all 17 affected skills have both with identical URIs, so no regression
