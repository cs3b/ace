# Remove Redundant assign Block from Canonical Skill Schema

## Objective

Canonical SKILL.md files currently duplicate assign metadata that `ace-assign` can derive from the bound workflow. In practice, the runtime child-step expansion already comes from workflow frontmatter (`assign.sub-steps` in `.wf.md` files), while skills still carry two extra fields: `assign.source` and `assign.steps`.

This task removes those two skill-level fields from canonical skills, updates runtime resolution to use `skill.execution.workflow`, and makes the workflow file the only authoritative source for assign metadata.

**Why now**: The current contract is split across two places:
- `assign.source` in skills duplicates `skill.execution.workflow`
- `assign.steps` in skills acts as side-car catalog metadata rather than the runtime sub-step list users actually observe

That split creates schema/runtime mismatch and weakens the single-source-of-truth model introduced by canonical skills.

## Behavioral Specification

### User Experience

- **Input**: No user input required — this is an internal schema cleanup
- **Process**: Remove `assign.source` and `assign.steps` from canonical SKILL.md files, update `ace-assign` to resolve assign metadata from the bound workflow, update lint to reject skill-level `assign:`, and re-sync provider projections
- **Output**: Cleaner canonical skills, consistent schema, no behavioral change to ace-assign execution or provider skill projections

### Expected Behavior

After this change:
1. Canonical SKILL.md files no longer contain skill-level `assign.source` or `assign.steps`
2. `SkillAssignSourceResolver` reads the workflow URI from `skill.execution.workflow`
3. Runtime sub-step expansion continues to come from workflow frontmatter (`assign.sub-steps` in `.wf.md`)
4. A skill is assign-capable only when its bound workflow file contains workflow-level `assign:` metadata
5. Assignment step catalog is built exclusively from YAML catalog step files, not from canonical skills
6. `ace-lint` rejects skill-level `assign:` in canonical SKILL.md files
7. `ace-handbook sync` produces provider projections without skill-level `assign:` blocks
8. The child steps users see for task-work remain unchanged because they already come from the workflow file

### Interface Contract

No external interfaces change. Internal changes:

**SkillAssignSourceResolver API** (internal, no BC guarantee pre-1.0):
- `resolve_assign_config(skill_name)` — resolves the workflow from `skill.execution.workflow`, then reads assign metadata from that workflow file
- `assign_step_catalog` — returns `[]` (catalog step files are authoritative)
- `assign_capable_skill_names` — includes only skills whose bound workflow file contains workflow-level `assign:` metadata

**Lint schema**:
- top-level `assign:` becomes an unknown/rejected field in canonical SKILL.md validation

Error Handling:
- Skills without `skill.execution.workflow` return nil from `resolve_assign_config`
- Skills whose workflow exists but has no workflow-level `assign:` metadata return nil from `resolve_assign_config`
- Missing workflow files raise `Ace::Assign::Error` with clear message updated to reference workflow resolution rather than `assign.source`

Edge Cases:
- All 17 affected canonical skills already declare matching `skill.execution.workflow`, so removing `assign.source` does not change workflow binding
- Workflow files without `assign:` remain valid workflows; they simply do not participate in assignment expansion
