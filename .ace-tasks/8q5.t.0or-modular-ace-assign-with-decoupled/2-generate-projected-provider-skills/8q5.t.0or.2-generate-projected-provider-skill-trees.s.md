# Generate projected provider skill trees

## Behavioral Specification

### User Experience
- **Input**: Package authors update canonical skills only in `handbook/skills`.
- **Process**: ACE generates `.agent/skills` and provider-specific skill trees from those canonical sources, applying any provider-safe filtering or reshaping required by each projection target.
- **Output**: Provider-facing skill trees remain available and current without manual wrapper maintenance or canonical authorship drift.

### Expected Behavior

This slice moves provider-facing skill trees to generated projections:
1. `.agent/skills` becomes a generated ACE-neutral projection instead of the canonical source.
2. `.claude/skills`, `.codex/skills`, `.gemini/skills`, and `.pi/skills` are generated from the same canonical package skills.
3. Projection generation may filter or reshape ACE-only fields when a provider cannot use them directly.
4. Projection generation preserves provider-safe baseline metadata and any required instruction body content.
5. Generated projections are reproducible and drift-detectable.
6. Manual edits to projected provider skills are treated as drift to repair, not as alternate sources of truth.

### Interface Contract

```text
Canonical source:
  ace-task/handbook/skills/as-task-plan/SKILL.md

Generated projections:
  .agent/skills/as-task-plan/SKILL.md
  .claude/skills/as-task-plan/SKILL.md
  .codex/skills/as-task-plan/SKILL.md
  .gemini/skills/as-task-plan/SKILL.md
  .pi/skills/as-task-plan/SKILL.md
```

```text
# Projection contract
- one canonical source drives all provider views
- projected files are reproducible
- provider-incompatible fields may be filtered or reshaped
- authors do not hand-maintain projected wrappers
```

**Error Handling:**
- Projection generation fails clearly when the canonical skill does not validate.
- A missing target provider tree is created or reported explicitly rather than silently skipped.
- Drift between projected files and canonical inputs is surfaced as actionable repair work.

**Edge Cases:**
- A canonical skill may project into a subset of providers if a provider cannot support that skill yet, but the decision must be explicit in the canonical metadata or generator rules.
- Provider projections may differ in small metadata details while preserving the same underlying capability.
- Canonical skills that are not user-invocable may still need provider projections when downstream tooling depends on them.

- [ ] **Generated projections adopted**: provider-facing skill trees are generated from canonical package skills.

- [ ] **Drift detection exists**: manual projection drift is detectable and repairable.
- [ ] **Provider compatibility preserved**: representative skills remain readable by provider-facing consumers after projection.

## Objective

Replace hand-maintained provider-facing skill wrappers with generated projections from canonical package skills while preserving provider compatibility.

## Scope of Work

- **User Experience Scope**: Provider-facing skills remain available in familiar locations.
- **System Behavior Scope**: Projection generation, provider-safe filtering, and drift detection.
- **Interface Scope**: Generated provider skill tree behavior and canonical ownership rules.

### Deliverables

#### Behavioral Specifications
- Generated projection contract for `.agent` and provider trees
- Provider-safe filtering/reshaping contract
- Projection drift-detection contract

#### Validation Artifacts
- Representative projected skill examples
- Drift detection and provider-compatibility checks

### Consumer Packages

- **ace-llm-providers-cli**: consumes provider-facing skill trees.
- **ace-integration-claude**: consumes Claude-facing projected skills.
- **Other provider integrations**: consume codex, gemini, and pi projections.
