# Migrate workflow-backed package skills and provider views

## Behavioral Specification

### User Experience
- **Input**: Package capabilities backed by workflows are authored once as canonical package-owned skills and remain available through provider-facing skill trees.
- **Process**: Each package with workflow-backed capabilities adds canonical `handbook/skills` entries, projects compatible provider views, and contributes those skills to `skill://` discovery and assign composition.
- **Output**: The repo has system-wide canonical skill coverage for the intended workflow-backed surfaces, and provider-facing skill trees continue to expose the same capabilities without duplicated authoring.

### Expected Behavior

This slice performs the broad migration once the schema, registry, and assign composition path are proven:
1. Packages with workflow-backed capabilities gain canonical `handbook/skills` entries.
2. Each canonical skill carries the enriched assign metadata needed by the migration scope.
3. Provider-facing skill trees are projected from canonical package skills instead of being independently maintained.
4. `skill://` can discover the migrated skills across packages.
5. Cross-gem assign composition works for representative flows once the migrated skills are present.
6. Internal or niche workflows may still be represented by non-user-invocable canonical skills, but they follow the same ownership model.

### Interface Contract

```bash
# Discover migrated skills across packages
ace-nav skill://*

# Representative cross-gem composition flow
/as-assign-compose "work on task 123, review the PR, and update docs"
```

```bash
# Provider-compatible projected views remain available
ls .agent/skills/as-task-work
ls .agent/skills/as-review-pr
```

**Error Handling:**
- A package that claims migration but lacks its canonical skill entries fails validation rather than silently relying on stale projected views.
- Projected provider views that do not match canonical skill content are treated as migration drift to repair, not as alternate sources of truth.
- Cross-gem composition failures identify the missing canonical skill or package instead of falling back silently to hidden assumptions.

**Edge Cases:**
- Packages may expose user-invocable and non-user-invocable canonical skills under the same ownership model.
- Workflow-backed packages with multiple related skills may emit multiple phases from a small set of canonical skills.
- Provider-view trees remain a compatibility layer even after migration; they are not removed in this slice.

### Success Criteria

- [ ] **Workflow-backed packages migrated**: Canonical package-owned skills exist for the intended workflow-backed migration scope across the repo.
- [ ] **Provider views projected**: `.agent/skills` and equivalent provider trees reflect canonical package skills.
- [ ] **Cross-package discovery works**: `skill://` lists migrated skills across packages.
- [ ] **Cross-gem composition works**: Representative multi-package assign flows use the migrated skills successfully.
- [ ] **No duplicated authorship**: Canonical package skills, not projected views, are the maintained source.

### Validation Questions

- [x] **Is this repo-wide or selective?** -> Repo-wide for workflow-backed package skills in the migration scope.
- [x] **Do projected provider views remain required?** -> Yes.
- [x] **Can non-user-invocable canonical skills exist?** -> Yes, if they still represent workflow-backed capabilities that need canonical ownership.

## Objective

Complete the system-wide migration from scattered provider-facing skill wrappers to canonical package-owned skills for workflow-backed capabilities, while preserving provider compatibility and shared discovery.

## Scope of Work

- **User Experience Scope**: Skills are discoverable and usable across packages through canonical ownership and projected provider views.
- **System Behavior Scope**: Repo-wide canonical skill coverage, provider-view projection, and cross-gem composition readiness.
- **Interface Scope**: Canonical package skill layout, projected provider skill trees, and cross-package discovery/composition behavior.

### Deliverables

#### Behavioral Specifications
- Repo-wide workflow-backed skill migration contract
- Projected provider-view behavior during and after migration
- Cross-gem composition readiness contract

#### Validation Artifacts
- Package coverage checks
- Cross-gem discovery/composition scenarios
- Provider-view drift checks

### Consumer Packages

- **Packages with workflows**: own and maintain canonical skills.
- **Provider-facing skill trees**: project compatible views.
- **ace-assign** and **ace-support-nav**: consume the migrated canonical skills.

## Out of Scope

- Final retirement of legacy phase-YAML ownership
- Broader documentation cleanup of canonical-path references
- Direct runtime execution of skills
