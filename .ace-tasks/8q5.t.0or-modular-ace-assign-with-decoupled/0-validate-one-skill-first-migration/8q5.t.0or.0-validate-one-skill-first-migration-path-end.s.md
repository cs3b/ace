# Validate one skill-first migration path end to end

## Behavioral Specification

### User Experience
- **Input**: A representative workflow-backed capability such as task work is migrated to a canonical package-owned skill in `handbook/skills`, while provider-facing views continue to expose the same skill to agents.
- **Process**: The migrated skill is discoverable through `skill://`, projected into provider trees, and used by `ace-assign` to compose one assignment path without relying on the old phase YAML as the authoritative definition.
- **Output**: The system proves the end-state with one real path before broader migration: canonical skill, provider view, `skill://` discovery, and assignment composition all agree on the same capability definition.

### Expected Behavior

This spike validates the architecture before the repo-wide migration:
1. One representative skill is moved or authored in a package-owned `handbook/skills` location.
2. The canonical skill includes enriched `assign` metadata sufficient for one composed phase path.
3. A projected provider view exists for the same skill without separately authored behavior text.
4. `ace-nav skill://...` resolves the canonical skill.
5. `ace-assign` can compose from the migrated metadata while leaving `ace-assign create FILE` unchanged.
6. The spike records which concepts survive into the final system and which legacy assumptions should be removed later.

The representative path for this spike should be workflow-backed and exercise subtree expansion. `as-task-work` / `wfi://task/work` is the default representative path.

### Interface Contract

```bash
# Resolve the canonical skill
ace-nav skill://as-task-work

# Compose from the migrated skill path
/as-assign-compose "work on task 123 and create a PR"
/as-assign-create work-on-task --taskref 123
```

```text
# Expected behavior contract
- One canonical package skill exists for the representative path
- One projected provider view exists for the same skill
- skill:// resolves the canonical file
- compose/create use the migrated skill metadata
- ace-assign create FILE remains the runtime boundary
```

**Error Handling:**
- If the canonical skill is missing required `assign` metadata, the spike fails with a concrete schema or validation error.
- If the projected provider view drifts from the canonical skill, the spike surfaces that mismatch rather than silently using stale content.
- If the canonical skill points at a missing workflow, composition fails before an assignment is partially created.

**Edge Cases:**
- The representative skill may emit one composed phase that expands into a subtree through workflow runtime metadata.
- Provider compatibility must survive even if the canonical skill carries ACE-specific additive fields.
- The spike proves one path only; it does not claim the repo-wide migration is complete.

### Success Criteria

- [ ] **Canonical skill exists**: One representative capability lives in `handbook/skills` as the authoritative source.
- [ ] **Provider view survives**: One provider-facing skill view is projected from the canonical skill and remains directly usable.
- [ ] **skill:// resolves the path**: The canonical skill is discoverable through `ace-nav skill://...`.
- [ ] **ace-assign composes from it**: One assignment path is composed from migrated skill metadata rather than phase YAML ownership.
- [ ] **Runtime boundary preserved**: `ace-assign create FILE` remains unchanged in the proven path.
- [ ] **Concept inventory recorded**: The spike produces clear guidance on what later slices must preserve or remove.

### Validation Questions

- [x] **Which path should prove the end-state?** -> A workflow-backed path with subtree expansion, defaulting to `as-task-work`.
- [x] **Must provider views be included in the spike?** -> Yes; they are part of the required end-state, not a follow-up.
- [x] **Should the spike use `skill://` immediately?** -> Yes; it proves the shared discovery path instead of a temporary scanner.

### Vertical Slice Decomposition (Task/Subtask Model)

- **Slice Type**: Subtask
- **Slice Outcome**: One canonical package skill proves the full skill-first path from discovery to provider projection to assign composition.
- **Advisory Size**: medium
- **Context Dependencies**: representative skill files, workflow runtime expansion, nav protocol registration, compose/create flows

## Objective

Validate the skill-first end-state with one real capability before drafting the repo-wide migration around assumptions that may later change.

## Scope of Work

- **User Experience Scope**: One representative capability is discoverable and composable from canonical skill metadata.
- **System Behavior Scope**: Canonical skill ownership, provider projection, `skill://` discovery, and one composed assign path.
- **Interface Scope**: `skill://` resolution for one skill plus one compose/create flow.

### Deliverables

#### Behavioral Specifications
- Representative canonical-skill contract
- Provider-view projection contract for one skill
- End-to-end discovery/composition proof

#### Validation Artifacts
- Spike verification commands
- Concept inventory notes for downstream slices

### Consumer Packages

- **ace-task**: likely owner of the representative workflow-backed skill.
- **ace-assign**: proves composition from migrated metadata.
- **ace-support-nav**: proves shared `skill://` discovery.

## Out of Scope

- Repo-wide migration of all package skills
- Final schema hardening for every edge case
- Legacy cleanup and authority retirement
