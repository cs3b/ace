# Prove typed canonical skills with three ACE examples

## Behavioral Specification

### User Experience
- **Input**: One representative ACE skill of each type is expressed as a canonical package-owned skill: one `capability`, one `workflow`, and one `orchestration`.
- **Process**: Each representative skill binds to a separate workflow body, projects into provider-facing trees, and resolves through canonical `skill://` discovery.
- **Output**: The architecture is proven with real ACE examples before the broader migration commits the ecosystem to a typed model.

### Expected Behavior

This tracer slice validates the taxonomy and core boundaries with representative examples:
1. A narrow deterministic utility such as `as-b36ts` proves the `capability` path.
2. A bounded process such as `as-task-plan` proves the `workflow` path.
3. A multi-step coordinator such as `as-assign-start` proves the `orchestration` path.
4. All three canonical skills live in package-owned `handbook/skills`.
5. All three bind to separate `wfi://...` workflows instead of embedding execution bodies directly.
6. Provider projections exist for all three and are clearly derived from the canonical files.
7. `skill://` resolves all three examples from canonical package paths.

The tracer proves the taxonomy itself. It does not yet require full repo-wide migration or final `ace-assign` integration.

### Interface Contract

```bash
# Canonical discovery for all three skill kinds
ace-nav skill://as-b36ts
ace-nav skill://as-task-plan
ace-nav skill://as-assign-start
```

```text
# Proven contract
- one canonical capability skill exists
- one canonical workflow skill exists
- one canonical orchestration skill exists
- each binds to a workflow body
- each has projected provider views
- each resolves via skill://
```

**Error Handling:**
- If a representative skill is missing `skill.kind`, the tracer fails schema validation.
- If a representative skill points at a missing workflow body, the tracer fails before projection or discovery claims success.
- If a projected provider skill drifts from its canonical source, the tracer surfaces drift instead of accepting stale output.

**Edge Cases:**
- The `capability` example remains discoverable even though it is not assignment-capable.
- The tracer may use non-user-invocable examples if needed, but the default examples should be stable public skills.
- The tracer proves projection and discovery for all three kinds without claiming repo-wide coverage.
