# Compose ace-assign from assign-capable skills

## Behavioral Specification

### User Experience
- **Input**: Users continue to compose or create assignments through `ace-assign catalog`, `/as-assign-compose`, and `/as-assign-create`.
- **Process**: `ace-assign` resolves canonical skills through `skill://`, filters to workflow/orchestration skills with explicit `assign:` metadata, builds an in-memory phase registry from that subset, and continues to hand off deterministic runtime work to `ace-assign create FILE`.
- **Output**: Public assignment surfaces stay familiar, but the source of truth becomes the assign-capable typed skill subset rather than provider-facing skills or phase YAML ownership.

### Expected Behavior

This slice integrates `ace-assign` with the typed canonical model:
1. `ace-assign` consumes canonical skills through `skill://`, not through provider-skill path scanning.
2. Only `workflow` and `orchestration` skills with explicit `assign:` metadata enter the phase registry.
3. `capability` skills remain discoverable through `skill://` but are excluded from assignment composition.
4. Workflow files remain the execution body and runtime subtree-expansion source.
5. `ace-assign create FILE` remains the deterministic runtime boundary.
6. A temporary compatibility bridge may still read legacy sources for unmigrated packages until the final cleanup slice, but the long-term source of truth is the assign-capable canonical subset.

### Interface Contract

```bash
# Inspect phases sourced from assign-capable canonical skills
ace-assign catalog

# Compose through existing public surfaces
/as-assign-compose "plan task 123 and then start the assignment"
/as-assign-create plan-task --taskref 123
```

```text
# Expected behavior contract
- catalog output comes from canonical workflow/orchestration skills
- capability skills do not appear as assignment phases
- hidden spec rendering still hands off to ace-assign create FILE
- workflow runtime expansion remains authoritative
```

**Error Handling:**
- Missing or invalid `assign:` metadata on an otherwise assign-capable skill fails before phase registry construction.
- Duplicate emitted phase names resolve deterministically and expose the shadowed canonical source.
- Capability skills that appear in canonical discovery never enter `ace-assign` catalogs or compose results.

**Edge Cases:**
- A workflow or orchestration skill may be discoverable through `skill://` yet excluded from `ace-assign` if it omits `assign:`.
- Partially migrated packages may continue through a temporary compatibility bridge until cleanup is complete.
- Provider-facing projected skills remain separate compatibility artifacts; `ace-assign` consumes canonical skill metadata, not provider projections.

### Success Criteria

- [ ] **Canonical subset used**: `ace-assign` builds its phase registry from canonical workflow/orchestration skills with `assign:`.
- [ ] **Capability skills excluded**: discoverable capability skills do not appear in assignment composition.
- [ ] **Shared discovery path used**: `ace-assign` consumes canonical `skill://` discovery rather than path-scanning provider trees.
- [ ] **Workflow boundaries preserved**: runtime execution remains workflow-backed and deterministic.
- [ ] **Compatibility bridge bounded**: any migration bridge is explicitly temporary and removed later.
