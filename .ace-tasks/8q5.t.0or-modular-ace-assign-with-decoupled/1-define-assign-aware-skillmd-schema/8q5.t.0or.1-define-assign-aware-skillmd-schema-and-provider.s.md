# Define assign-aware SKILL.md schema and provider projections

## Behavioral Specification

### User Experience
- **Input**: Package authors define skills once in `handbook/skills/<skill>/SKILL.md` with richer frontmatter that remains valid for provider consumers and sufficiently descriptive for `ace-assign`.
- **Process**: The canonical `SKILL.md` shape captures assign metadata, workflow references, phase emission rules, and provider-compatible baseline fields in one place. Projected provider views reuse that canonical content instead of maintaining duplicated manual wrappers.
- **Output**: A package author can add or update one canonical `SKILL.md`, and the system can validate it, discover it, project it, and compose from it consistently.

### Expected Behavior

This slice defines the schema contract for canonical package skills:
1. `SKILL.md` remains provider-compatible at the top level (`name`, `description`, `allowed-tools`, `source`, and similar existing fields stay intact).
2. Additional ACE-specific fields are additive, not provider-breaking.
3. `assign:` becomes the source of phase-definition metadata used by `ace-assign`.
4. `assign` supports one or more emitted phases per skill.
5. The canonical skill references its workflow execution body explicitly.
6. Projected provider views are derived from the canonical skill, not reauthored separately.

### Interface Contract

```yaml
---
name: as-task-work
description: Execute task implementation with context loading and change commits
user-invocable: true
source: ace-task
assign:
  workflows:
    primary: wfi://task/work
  phases:
    - name: work-on-task
      description: Implement task changes following project conventions
      emits:
        mode: workflow-subtree
      tags: [implementation, task]
      consumes: [project-base-context, task-spec, implementation-plan]
      produces: [code-changes, commits]
      context:
        default: fork
      prerequisites:
        - name: onboard-base
          strength: required
        - name: task-load
          strength: required
    - name: plan-task
      description: Refresh the implementation plan before work begins
      emits:
        mode: direct-phase
      tags: [planning, task]
      produces: [implementation-plan]
---
```

**Error Handling:**
- Unknown or malformed `assign` fields fail validation with field-level error messages.
- A canonical skill without `assign.workflows.primary` or emitted `assign.phases` is invalid for assign composition.
- Duplicate emitted phase names within one skill fail validation before registry indexing.

**Edge Cases:**
- A skill may emit one phase or several phases.
- A phase may be direct or may expand into a workflow-defined subtree.
- A provider consumer that ignores ACE-specific `assign` fields must still read the remaining skill successfully.

### Success Criteria

- [ ] **Canonical schema defined**: The enriched `SKILL.md` frontmatter contract is explicit and sufficient for assign composition.
- [ ] **Provider-safe additions**: ACE-specific fields are additive and do not invalidate provider consumers.
- [ ] **Multi-phase emission supported**: One canonical skill may emit multiple phases when needed.
- [ ] **Workflow references explicit**: The canonical skill declares the workflow execution body it relies on.
- [ ] **Projection behavior defined**: Provider-view projection rules are concrete and testable.

### Validation Questions

- [x] **Where does assign metadata live?** -> In canonical `SKILL.md` frontmatter.
- [x] **Can one skill emit multiple phases?** -> Yes.
- [x] **Does this replace workflows?** -> No; workflows remain the execution body and runtime subtree source.
- [x] **Can providers ignore ACE-specific fields?** -> Yes; compatibility requires additive fields only.

## Objective

Define one durable, package-owned `SKILL.md` contract that can support provider compatibility, skill discovery, and assign composition without duplicating behavior definitions across files and trees.

## Scope of Work

- **User Experience Scope**: Package authors edit one canonical skill file and get consistent discovery/projection behavior.
- **System Behavior Scope**: Canonical skill schema, emitted phase metadata, workflow references, and provider-view projection rules.
- **Interface Scope**: `SKILL.md` frontmatter fields and projected skill file layout.

### Deliverables

#### Behavioral Specifications
- Canonical `SKILL.md` assign schema
- Emitted-phase metadata contract
- Provider-view projection contract

#### Validation Artifacts
- Valid/invalid schema scenarios
- Provider-compatibility examples

### Consumer Packages

- **Packages with workflows**: author canonical package-owned skills.
- **ace-assign**: consumes the emitted phase metadata.
- **Provider-facing skill trees**: receive projected compatible views.

## Out of Scope

- `skill://` registry implementation
- `ace-assign` composition logic changes
- Repo-wide skill migration beyond schema examples
