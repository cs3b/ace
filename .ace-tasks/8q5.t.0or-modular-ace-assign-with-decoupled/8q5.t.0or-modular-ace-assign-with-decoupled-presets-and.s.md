# Modular ace-assign with Decoupled Presets and Sources

## Behavioral Specification

### User Experience
- **Input**: Users compose assignments from modular building blocks registered by individual gems. Instead of ace-assign knowing about ace-task or ace-review internals, each gem registers its own phases and presets into a discoverable catalog.
- **Process**: ace-assign discovers registered sources (similar to ace-nav's protocol sources), loads their phase definitions and presets, and composes them into assignments. The `assign:` frontmatter block in skill/workflow files becomes the declarative interface for registering building blocks.
- **Output**: Assignments that compose phases from multiple gems seamlessly. Adding a new assignable capability to any gem requires only registering a phase definition — no changes to ace-assign core.

### Expected Behavior

Today ace-assign has:
- A hardcoded phase catalog in `.ace-defaults/assign/catalog/phases/` (work-on-task, lint, create-pr, etc.)
- Preset composition via `composition_rules.rb` with knowledge of specific phases
- `skill_assign_source_resolver.rb` that resolves skill references

The modular redesign:
1. **Source registration**: Each gem declares its assignable phases in a standard location (e.g., `.ace-defaults/assign/catalog/phases/*.phase.yml`). ace-assign discovers these via the configuration cascade, similar to how ace-nav discovers protocol sources.
2. **Decoupled phase catalog**: The merged catalog is built from all registered sources at runtime. No gem needs to know about phases from other gems.
3. **Preset composition**: Presets reference phases by name. Since phases are registered by source gems, presets can compose cross-gem workflows without coupling.
4. **Frontmatter registration**: Gems can declare assignable behaviors in their `assign:` frontmatter blocks, making their capabilities discoverable.

The flow after redesign:
- `ace-assign compose` discovers all registered phases from all gems
- User selects phases or uses a preset that references them
- `ace-assign create` builds the job.yaml from selected phases
- Assignment execution uses existing mechanisms (phases, fork, skills)

### Interface Contract

```yaml
# Source registration: gem declares its phases
# ace-review/.ace-defaults/assign/catalog/phases/run-review.phase.yml
name: run-review
skill: ace-review-run
tags: [review, quality]
produces: [review-report]
prerequisites:
  - name: onboard
    strength: recommended
    reason: "Context loading improves review quality"
source: ace-review                # NEW: declares which gem owns this phase

# ace-task/.ace-defaults/assign/catalog/phases/work-on-task.phase.yml
name: work-on-task
skill: ace-task-work
tags: [implementation, task]
produces: [code-changes]
source: ace-task
```

```yaml
# Preset composition references phases by name (no gem coupling)
# .ace/assign/presets/full-delivery.yml
name: full-delivery
description: Complete delivery workflow
phases:
  - onboard
  - work-on-task
  - run-review            # from ace-review (discovered, not hardcoded)
  - lint                  # from ace-lint
  - create-pr
```

```bash
# Discovery: see all available phases from all gems
ace-assign catalog
# Output:
# Available phases:
#   onboard          (ace-assign)    [setup]
#   work-on-task     (ace-task)      [implementation, task]
#   run-review       (ace-review)    [review, quality]
#   lint             (ace-lint)      [quality, lint]
#   run-batch-e2e    (ace-test-e2e)  [testing, batch]    # from migrated batch (task 0m6)
#   create-pr        (ace-assign)    [delivery]

# Compose with discovered phases
ace-assign compose --phases onboard,work-on-task,run-review,create-pr
```

**Error Handling:**
- Phase name conflict across gems: Last-loaded wins with warning (config cascade order)
- Referenced phase not found: Clear error listing available phases and their sources
- Source gem not installed: Phases from that gem absent from catalog (no error unless referenced)
- Circular prerequisites: Detected and reported during compose

**Edge Cases:**
- No external gems installed: ace-assign's built-in phases still work (backward compatible)
- Multiple gems register same phase name: Warning logged, most-specific scope wins (project > user > gem defaults)
- Preset references phase from uninstalled gem: Error with suggestion to install the gem
- Empty catalog after filtering: Clear message "no phases match your criteria"

## Objective

Transition ace-assign from a monolithic package with hardcoded knowledge of ace-task and ace-review into a flexible orchestration engine. By adopting a source registration pattern (like ace-nav's protocol sources), individual gems register their assignable capabilities, and ace-assign discovers and composes them without coupling. This is the architectural capstone that leverages all building blocks: enhanced FORK delegation (task 0ki), batch-as-phases migration (task 0m6), review runner integration (task 0hm), and native review (task 0n9).

## Scope of Work

- **User Experience Scope**: Users compose assignments from a discoverable catalog of phases contributed by any gem; `ace-assign catalog` shows available phases with source attribution
- **System Behavior Scope**: Source registration convention, catalog discovery via config cascade, decoupled preset composition, removal of gem-specific coupling from ace-assign core
- **Interface Scope**: `ace-assign catalog` command, `source:` field in phase definitions, phase discovery via config cascade

### Deliverables

#### Behavioral Specifications
- Source registration convention for gems
- Catalog discovery and merging behavior
- Preset composition with dynamic phase resolution

#### Validation Artifacts
- Integration tests for cross-gem phase discovery
- Decoupling verification (no direct imports from ace-task/ace-review in ace-assign)
- Backward compatibility tests for existing presets

## Out of Scope
