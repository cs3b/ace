# Shift ace-assign to a package-owned skill registry

## Behavioral Specification

### User Experience
- **Input**: Developers and agents discover capabilities from package-owned `handbook/skills` entries instead of a separate phase catalog, while provider-facing skill views remain available under `.agent/skills`, `.claude/skills`, `.codex/skills`, and related trees.
- **Process**: `skill://` becomes the shared discovery surface for canonical skills. `ace-assign` reads the same skill metadata to build its phase database, while workflow files remain the execution bodies and subtree-expansion sources.
- **Output**: Users can list skills, inspect one canonical skill, compose assignments from migrated skill metadata, and continue using provider-compatible skill views without maintaining duplicate behavioral definitions.

### Expected Behavior

Today the system splits responsibility across three places:
- Phase definitions live in `ace-assign/.ace-defaults/assign/catalog/phases/*.phase.yml`
- Provider-facing skill wrappers live mostly in `.agent/skills/`
- Workflow runtime expansion lives in workflow `assign:` frontmatter

The redesigned system converges on one authoritative source of assign metadata:
1. **Canonical ownership**: Each package owns its skills in `handbook/skills/<skill>/SKILL.md`.
2. **Provider compatibility**: Provider-facing skill trees are generated or symlinked views of the canonical package skills, not separate authored copies.
3. **Shared discovery**: `ace-nav skill://...` lists and resolves canonical skills across packages.
4. **Assign metadata in skills**: `SKILL.md` frontmatter becomes the source of phase-definition metadata used by `ace-assign`.
5. **Workflow role preserved**: Workflows remain the execution body and may still supply runtime subtree-expansion semantics; `ace-assign` must not invoke skills directly as runtime work.
6. **Early shared registry**: `ace-assign` consumes the same `skill://`-backed registry that users and other tools inspect; no second long-lived skill scanner is introduced.
7. **System-wide migration**: Workflow-backed package skills are created across the repo, projected into provider views, and used by composition before the legacy phase catalog ownership is retired.

This parent task is an orchestrator. It coordinates six vertical slices:
- `8q5.t.0or.0` proves one representative skill-first path end to end
- `8q5.t.0or.1` defines the enriched `SKILL.md` schema and provider-view projection rules
- `8q5.t.0or.2` adds `skill://` protocol discovery and gem registrations
- `8q5.t.0or.3` makes `ace-assign` compose from the skill registry
- `8q5.t.0or.4` migrates workflow-backed package skills and provider views across the repo
- `8q5.t.0or.5` retires legacy catalog ownership and outdated canonical-path assumptions

### Interface Contract

```yaml
# Canonical package-owned skill
# ace-task/handbook/skills/as-task-work/SKILL.md
name: as-task-work
description: Execute task implementation with context loading and change commits
user-invocable: true
source: ace-task
assign:
  workflows:
    primary: wfi://task/work
  phases:
    - name: work-on-task
      description: Implement task changes following project conventions
      emits:
        mode: workflow-subtree
      tags: [implementation, task]
      consumes: [project-base-context, task-spec, implementation-plan]
      produces: [code-changes, commits]
      context:
        default: fork
      prerequisites:
        - name: onboard-base
          strength: required
        - name: task-load
          strength: required
```

```bash
# Discover canonical skills
ace-nav skill://*
ace-nav skill://as-task-work

# Compose from the shared skill registry
ace-assign catalog
/as-assign-compose "work on task 123 and create a PR"
/as-assign-create work-on-task --taskref 123
```

**Error Handling:**
- Duplicate canonical skill names resolve with deterministic nav precedence and surface enough source detail to diagnose shadowing.
- Missing or invalid `assign` metadata on a migrated canonical skill fails validation with a concrete field-level error.
- A canonical skill that references a missing workflow produces a clear discovery/composition error rather than a partial assignment.
- Provider-view projection drift is detectable and repairable without reauthoring the projected skill files manually.

**Edge Cases:**
- All package skills are discoverable through `skill://`, even while repo migration is incomplete.
- One skill may emit multiple assign phases, but the common case remains one primary phase that expands via workflow runtime metadata.
- `ace-assign create FILE` remains the deterministic runtime boundary even after composition metadata moves into skills.
- Packages with no `handbook/skills` entries contribute nothing to `skill://` until migrated.

### Success Criteria

- [ ] **Canonical package ownership**: Package-owned `handbook/skills` is the authoritative source for skill and assign metadata.
- [ ] **Provider compatibility preserved**: `.agent/skills` and provider-specific skill trees remain usable without separately authored behavior definitions.
- [ ] **Shared skill discovery**: `skill://` lists and resolves canonical skills across registered gems.
- [ ] **Skill-backed composition**: `ace-assign` builds its phase database from skill metadata rather than phase YAML ownership.
- [ ] **Workflow boundaries preserved**: Workflows remain execution bodies and runtime subtree sources; skills are used as configuration, not runtime dispatch targets.
- [ ] **Repo migration completed**: Workflow-backed package skills and provider projections exist system-wide for the intended migration scope.
- [ ] **Legacy ownership retired**: The phase catalog and old `.agent/skills` canonical assumption are no longer the authoritative model.

### Validation Questions

- [x] **Where is the canonical skill source?** -> `ace-*/handbook/skills/<skill>/SKILL.md`.
- [x] **How strict is provider compatibility?** -> Strict; extra ACE fields are additive and ignorable by provider consumers.
- [x] **Should ace-assign invoke skills directly?** -> No; skills provide composition/configuration data, while runtime execution continues through rendered assignment phases and workflows.
- [x] **Can a canonical skill emit more than one phase?** -> Yes; a skill may emit one or more phases, and workflows may still expand those phases into subtrees at runtime.
- [x] **When does `skill://` land?** -> Early; it is the shared discovery path used by both nav and `ace-assign`.
- [x] **Should all package skills be discoverable?** -> Yes; `skill://` lists all package skills, while migration ensures the needed assign metadata exists where composition depends on it.

## Objective

Replace the current split between phase YAML, provider-facing skill wrappers, and workflow runtime metadata with one package-owned skill registry that can drive both capability discovery and assignment composition across the repo.

## Scope of Work

- **User Experience Scope**: Users and agents discover capabilities via `skill://`, compose assignments from migrated skill metadata, and continue to use provider-compatible skill views.
- **System Behavior Scope**: Canonical package skills become the source of assign metadata; nav and `ace-assign` share the same skill discovery path; workflows remain runtime execution bodies.
- **Interface Scope**: `SKILL.md` frontmatter schema, `skill://` protocol, projected provider skill views, and `ace-assign` catalog/compose/create behavior.

### Deliverables

#### Behavioral Specifications
- Canonical package-owned skill model
- Enriched `SKILL.md` assign schema and provider-projection contract
- Shared `skill://` discovery behavior
- Skill-backed `ace-assign` composition contract
- Repo-wide migration and legacy-retirement contract

#### Validation Artifacts
- Spike proof for one representative end-to-end path
- Usage scenarios for discovery, provider views, and composition
- Verification commands covering discovery, composition, migration, and cleanup

### Consumer Packages

- **ace-assign**: consumes skill metadata for catalog/composition behavior.
- **ace-support-nav**: exposes `skill://` discovery and precedence behavior.
- **Packages with workflows**: own canonical `handbook/skills` entries and projected provider views.
- **Provider-facing skill trees**: continue to expose compatible `SKILL.md` files generated from canonical package skills.
