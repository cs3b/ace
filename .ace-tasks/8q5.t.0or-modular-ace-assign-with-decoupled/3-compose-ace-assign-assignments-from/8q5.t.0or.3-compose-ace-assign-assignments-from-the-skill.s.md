# Compose ace-assign assignments from the skill registry

## Behavioral Specification

### User Experience
- **Input**: Users compose or create assignments through the existing `ace-assign catalog`, `/as-assign-compose`, and `/as-assign-create` surfaces.
- **Process**: `ace-assign` resolves canonical skills through `skill://`, builds an in-memory phase registry from skill `assign` metadata, applies existing recipes and composition rules, and continues to hand off deterministic runtime creation through `ace-assign create FILE`.
- **Output**: Users see the same assignment-oriented surfaces, but the source of truth for available phases is now the skill registry instead of phase YAML ownership.

### Expected Behavior

This slice changes the source of composition truth without changing runtime boundaries:
1. `ace-assign` reads skill metadata from the shared `skill://` discovery path.
2. The in-memory phase catalog is derived from canonical skill `assign.phases`.
3. Recipes and composition rules remain guidance artifacts and continue to apply to the emitted phase names.
4. Workflow files remain the execution body and runtime subtree-expansion source.
5. `ace-assign create FILE` remains the deterministic runtime boundary after composition renders the hidden spec.
6. While repo migration is incomplete, a compatibility bridge may still read legacy phase YAML for unmigrated packages; this bridge is temporary and retired in `0or.5`.

### Interface Contract

```bash
# Inspect available phases from skill-backed metadata
ace-assign catalog

# Compose through existing public surfaces
/as-assign-compose "work on task 123 and create a PR"
/as-assign-create work-on-task --taskref 123
```

```text
# Expected behavior contract
- catalog output reflects phases emitted from canonical skills
- compose/create use the shared skill registry as the source of truth
- hidden spec rendering still hands off to ace-assign create FILE
- workflow-backed phases may still expand into runtime subtrees
```

**Error Handling:**
- If a canonical skill needed for composition is missing `assign.phases`, composition fails with a concrete metadata error.
- If two canonical skills emit the same phase name, the system uses deterministic precedence and surfaces the shadowed source.
- If a selected phase references a missing workflow execution body, assignment creation fails before a partial hidden spec is accepted.

**Edge Cases:**
- One skill may emit several phases used independently by recipes and explicit user intent.
- Partially migrated packages may still resolve through the compatibility bridge until the final cleanup slice lands.
- Direct provider invocation of projected skills must remain separate from assign composition; `ace-assign` consumes metadata, not skill execution.

### Success Criteria

- [ ] **Skill-backed catalog**: `ace-assign` derives its available phases from canonical skill metadata.
- [ ] **Shared discovery path**: `ace-assign` consumes `skill://` rather than creating a separate registry.
- [ ] **Workflow boundaries preserved**: Workflow files still define execution and subtree-expansion behavior.
- [ ] **Deterministic runtime preserved**: `ace-assign create FILE` remains the creation boundary.
- [ ] **Temporary compatibility bridge defined**: Unmigrated packages can still compose during migration without becoming the new long-term source of truth.

### Validation Questions

- [x] **Should `ace-assign` use a temporary scanner?** -> No; it should consume the shared `skill://` path.
- [x] **Do recipes and rules survive?** -> Yes; they remain composition guidance, but target phase names emitted from skills.
- [x] **Does skill metadata replace workflow execution?** -> No; workflows remain the execution body.

## Objective

Move `ace-assign` off phase-YAML ownership and onto the shared canonical skill registry without changing the public composition surfaces or the deterministic create boundary.

## Scope of Work

- **User Experience Scope**: Catalog, compose, and create continue to work with familiar entrypoints.
- **System Behavior Scope**: Skill-backed phase registry, recipe/rule application, workflow runtime expansion, and temporary migration bridge.
- **Interface Scope**: `ace-assign catalog`, `/as-assign-compose`, `/as-assign-create`, and emitted phase metadata consumption.

### Deliverables

#### Behavioral Specifications
- Skill-backed phase-registry contract
- Compatibility-bridge behavior for migration
- Workflow boundary preservation

#### Validation Artifacts
- Catalog output scenarios
- Compose/create scenarios and failure-path checks

### Consumer Packages

- **ace-assign**: owns catalog, compose, create, and runtime boundary behavior.
- **Packages with canonical skills**: emit the phase definitions consumed by composition.

## Out of Scope

- Repo-wide migration of every canonical skill
- Final retirement of the compatibility bridge
- Provider-view projection implementation details
