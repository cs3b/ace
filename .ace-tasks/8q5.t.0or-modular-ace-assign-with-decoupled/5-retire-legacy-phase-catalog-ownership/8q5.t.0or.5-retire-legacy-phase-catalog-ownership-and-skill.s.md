# Retire legacy phase catalog ownership and skill path assumptions

## Behavioral Specification

### User Experience
- **Input**: Users continue to discover skills and compose assignments through the already-migrated public surfaces.
- **Process**: Once canonical package skills, `skill://`, projected provider views, and skill-backed assign composition are all in place, the system removes the old ownership model and updates docs and code to reflect the new source of truth.
- **Output**: The public surfaces remain stable, while the implementation and documentation no longer imply that phase YAML or `.agent/skills` are the authoritative model.

### Expected Behavior

This cleanup slice removes transitional ambiguity:
1. Phase YAML is no longer treated as the authoritative source of assign phase definitions.
2. `.agent/skills` and provider trees remain compatibility views, but no longer claim canonical ownership.
3. Documentation, architecture notes, and task references describe `handbook/skills` as the canonical source.
4. Temporary compatibility bridges used during migration are removed once migration coverage is complete.
5. Recipes and composition rules remain, but they target phase names emitted from canonical skills rather than phase YAML ownership.

### Interface Contract

```bash
# Public behavior remains stable
ace-nav skill://*
ace-assign catalog
/as-assign-create work-on-task --taskref 123
```

```text
# End-state contract
- handbook/skills is canonical
- provider-facing skill trees remain compatibility views
- phase catalog YAML is no longer the authoritative phase-definition source
```

**Error Handling:**
- Cleanup does not remove migration bridges until migrated skill coverage is sufficient to keep public behavior stable.
- If a doc or code path still claims `.agent/skills` or phase YAML is canonical, that is treated as cleanup debt to remove before closure.

**Edge Cases:**
- Recipes and rules remain part of the system after cleanup.
- Provider-facing skill trees remain required compatibility outputs.
- Legacy phase YAML may remain as historical artifacts or fallback references only if clearly marked non-authoritative; the end-state should not compose from them.

### Success Criteria

- [ ] **Legacy authority removed**: Phase YAML is no longer the authoritative source of assign phase definitions.
- [ ] **Canonical path documented**: Repo docs and guidance consistently describe `handbook/skills` as canonical.
- [ ] **Compatibility layer clarified**: Provider-facing skill trees remain supported but are documented as projected views.
- [ ] **Migration bridges retired**: Temporary bridges introduced during migration are removed once they are no longer needed.
- [ ] **Public behavior stable**: Existing discovery and composition surfaces keep working through the new ownership model.
