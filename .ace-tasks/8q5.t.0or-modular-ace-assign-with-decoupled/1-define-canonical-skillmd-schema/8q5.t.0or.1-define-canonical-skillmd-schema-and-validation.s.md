# Define canonical SKILL.md schema and validation

## Behavioral Specification

### User Experience
- **Input**: Package authors define one canonical `SKILL.md` with provider-compatible baseline fields plus ACE-specific typed metadata.
- **Process**: Validation enforces the required taxonomy, workflow binding, and assign restrictions while provider-facing consumers continue to tolerate richer frontmatter.
- **Output**: Authors can create canonical skills confidently, and downstream projection, discovery, and assignment logic receive a stable contract.

### Expected Behavior

This slice defines the durable canonical skill schema:
1. Existing provider-safe top-level fields remain supported: `name`, `description`, `user-invocable`, `allowed-tools`, `source`, and related metadata.
2. Shared optional metadata such as `version`, `author`, `license`, `dependencies`, and `tags` is supported in canonical skills.
3. ACE-specific behavior is encoded explicitly under `skill:` and `assign:`.
4. `skill.kind` is required and only allows `capability`, `workflow`, or `orchestration`.
5. `skill.execution.workflow` is required for all canonical skills.
6. `assign:` is optional, but only valid for `workflow` and `orchestration` skills.
7. Provider readers that only care about baseline fields continue to operate without understanding ACE-only metadata.

### Interface Contract

```yaml
---
name: as-task-plan
description: Creates JIT implementation plan with ephemeral output
version: 1.0.0
author: ace-task
source: ace-task
user-invocable: true
allowed-tools:
  - Bash(ace-task:*)
  - Bash(ace-bundle:*)
  - Read
  - TodoWrite
tags: [planning, task]
skill:
  kind: workflow
  execution:
    workflow: wfi://task/plan
assign:
  phases:
    - name: plan-task
      description: Refresh the implementation plan before work begins
      emits:
        mode: direct-phase
---
```

```yaml
---
name: as-b36ts
description: Encode and decode timestamps to compact Base36 ids
source: ace-b36ts
user-invocable: true
allowed-tools:
  - Bash(ace-b36ts:*)
  - Bash(ace-bundle:*)
  - Read
skill:
  kind: capability
  execution:
    workflow: wfi://b36ts
---
```

**Error Handling:**
- Unknown `skill.kind` values fail validation with field-level errors.
- Missing `skill.execution.workflow` fails validation even if the skill is otherwise provider-compatible.
- `capability` skills with `assign:` fail validation instead of being silently ignored.
- Duplicate emitted phase names within a canonical skill fail validation before registry indexing.

**Edge Cases:**
- A canonical skill may be non-user-invocable and still be valid if it is a stable public capability for projections or composition.
- A `workflow` or `orchestration` skill may omit `assign:` and still be discoverable, but it does not enter the assignment registry.
- Provider-facing consumers may ignore `skill:` and `assign:` without rejecting the rest of the file.

### Success Criteria

- [ ] **Typed schema defined**: canonical `SKILL.md` requires `skill.kind` and `skill.execution.workflow`.
- [ ] **Provider-safe baseline preserved**: current provider-facing baseline fields remain valid.
- [ ] **Shared metadata supported**: optional shared fields are explicitly allowed in canonical skills.
- [ ] **Assign restriction enforced**: `assign:` is valid only on `workflow` and `orchestration` skills.
- [ ] **Validation path updated**: lint and other schema-aware consumers can validate the new canonical contract.

### Validation Questions

- [x] **Is `skill.kind` optional?** -> No; it is required on canonical skills.
- [x] **Does canonical execution live in the skill body?** -> No; execution stays in `skill.execution.workflow`.
- [x] **Can capability skills participate in assign?** -> No.
- [x] **Must provider consumers understand ACE-only fields?** -> No; ACE-only fields are additive.

## Objective

Define one durable canonical `SKILL.md` schema that supports typed skill ownership, workflow binding, provider projections, and assignment-aware behavior without breaking existing provider integrations.

## Scope of Work

- **User Experience Scope**: Package authors define one canonical skill contract instead of maintaining separate metadata dialects.
- **System Behavior Scope**: Typed schema enforcement, workflow binding, shared metadata support, and assign restrictions.
- **Interface Scope**: Canonical `SKILL.md` fields and validation rules.

### Deliverables

#### Behavioral Specifications
- Typed canonical frontmatter contract
- Assign restriction rules by skill kind
- Provider-compatible metadata contract

#### Validation Artifacts
- Valid/invalid schema examples
- Validation coverage for capability, workflow, and orchestration skills

### Consumer Packages

- **ace-lint**: primary schema validator for canonical skills.
- **ace-llm-providers-cli**: consumes provider projections and must tolerate enriched frontmatter.
- **ace-integration-claude**: documents provider-facing metadata expectations.

## Out of Scope

- Canonical discovery implementation
- Generated projection machinery
- Repo-wide migration inventory
