# Inventory and migrate the typed skill ecosystem

## Behavioral Specification

### User Experience
- **Input**: ACE maintainers classify current public skills and workflows across packages to decide which ones become canonical typed skills and which remain workflow-only internals.
- **Process**: The repo is inventoried, existing public capabilities are mapped to `capability`, `workflow`, or `orchestration`, canonical skills are created where appropriate, and generated provider projections are produced from the migrated set.
- **Output**: The repo ends with a curated typed canonical skill ecosystem instead of a blanket "every workflow becomes a skill" migration.

### Expected Behavior

This slice performs the broad repo migration after the platform pieces are proven:
1. Existing skills and public workflows are inventoried package by package.
2. Each candidate is classified as `capability`, `workflow`, `orchestration`, or workflow-only internal helper.
3. Stable public capabilities receive canonical package-owned skills.
4. Workflow-only internals are documented as intentionally not becoming canonical skills.
5. Generated provider projections are produced from the migrated canonical set.
6. Representative cross-package discovery and assignment flows keep working after migration.

### Interface Contract

```text
# Example migration inventory row
skill: as-b36ts
kind: capability
canonical: ace-b36ts/handbook/skills/as-b36ts/SKILL.md
workflow: wfi://b36ts
assign-capable: no
```

```text
# Example migration inventory row
skill: as-assign-start
kind: orchestration
canonical: ace-assign/handbook/skills/as-assign-start/SKILL.md
workflow: wfi://assign/start
assign-capable: yes
```

**Error Handling:**
- A workflow that is intentionally kept workflow-only must be recorded as such rather than silently skipped.
- A package that claims migration but still relies on hand-maintained provider wrappers is treated as incomplete.
- Cross-package gaps surface as missing canonical skills or incomplete projections, not as hidden fallback behavior.

**Edge Cases:**
- Some packages may already have canonical skills; those should be adopted into the typed model instead of duplicated.
- Non-user-invocable canonical skills may exist if they represent stable public capabilities needed by downstream systems.
- One package may expose multiple skills that bind to a small set of workflow bodies.

### Success Criteria

- [ ] **Repo inventory completed**: current public skills and workflows are classified by type and migration disposition.
- [ ] **Selective migration applied**: stable public capabilities gain canonical typed skills without forcing every workflow into a skill.
- [ ] **Workflow-only exceptions documented**: internal helpers that remain workflow-only are explicitly tracked.
- [ ] **Provider projections regenerated**: migrated canonical skills drive provider-facing projections.
- [ ] **Representative ecosystem flows preserved**: cross-package discovery and assignment behavior remains intact after migration.

### Validation Questions

- [x] **Does every workflow become a skill?** -> No.
- [x] **Should workflow-only internals be tracked explicitly?** -> Yes.
- [x] **Is repo-wide migration selective or blanket?** -> Selective by stable public capability.

### Vertical Slice Decomposition (Task/Subtask Model)

- **Slice Type**: Subtask
- **Slice Outcome**: ACE gains a curated repo-wide typed canonical skill ecosystem with explicit classification and selective migration boundaries.
- **Advisory Size**: large
- **Context Dependencies**: typed schema, generated projections, canonical discovery, assign-capable subset behavior, package workflow inventories

### Verification Plan

#### Unit / Component Validation
- [ ] Canonical skills exist for the intended migrated public capabilities.
- [ ] Workflow-only exceptions are recorded for public workflow inventories that stay internal.

#### Integration / E2E Validation (if cross-boundary behavior exists)
- [ ] `ace-nav skill://*` lists migrated canonical skills from multiple packages and multiple kinds.
- [ ] Representative assignment flows still compose from migrated workflow/orchestration skills.

#### Failure / Invalid-Path Validation
- [ ] Missing canonical skills or stale projections are surfaced as migration gaps.
- [ ] Inventory gaps do not silently fall back to hand-maintained provider wrappers.

#### Verification Commands
- [ ] `ace-nav skill://*`
- [ ] `find ace-*/handbook/skills -name SKILL.md`
- [ ] `find .agent/skills -name SKILL.md`

## Objective

Inventory ACE's current public skill and workflow surfaces, classify them by type, and migrate only the stable public capability set into the typed canonical model.

## Scope of Work

- **User Experience Scope**: Users keep using the same public capabilities through canonical skills and generated provider projections.
- **System Behavior Scope**: Repo-wide inventory, type classification, selective migration, explicit workflow-only exceptions, regenerated projections.
- **Interface Scope**: Canonical skill coverage and migration inventory outputs.

### Deliverables

#### Behavioral Specifications
- Repo-wide type-classification contract
- Selective migration contract
- Workflow-only exception contract

#### Validation Artifacts
- Inventory matrix for current public capabilities
- Cross-package discovery and composition checks

### Consumer Packages

- **Packages with public skills/workflows**: contribute inventory rows and canonical skills where appropriate.
- **ace-assign**: consumes migrated workflow/orchestration skills for composition.
- **Provider integrations**: consume regenerated projected skill trees.

## Out of Scope

- Redefining the canonical schema
- Adding new discovery protocols
- Final cleanup of old canonical-path assumptions
