# Build a typed canonical skill platform for ACE

## Behavioral Specification

### User Experience
- **Input**: Package authors define stable public capabilities once in `ace-*/handbook/skills/<skill>/SKILL.md` with richer metadata that can drive provider projections, canonical discovery, and assignment composition.
- **Process**: Each canonical skill declares a required skill type, binds to a separate execution workflow, and may optionally declare `assign:` metadata. Generated projections publish provider-facing skill views, `skill://` exposes canonical discovery, and `ace-assign` consumes only the assign-capable subset.
- **Output**: Users can discover canonical capability, workflow, and orchestration skills across packages, provider-facing skill trees remain usable, and `ace-assign` composes only from workflow/orchestration skills that explicitly opt into assignment behavior.

### Expected Behavior

Today ACE uses a split model:
- `.agent/skills` is treated as canonical in docs and some runtime paths
- `handbook/workflow-instructions` holds the actual executable bodies
- `ace-assign` resolves provider-facing skills and workflow `assign:` frontmatter to compose phases
- only a small subset of packages already use `handbook/skills`

The redesigned platform introduces a typed canonical skill model:
1. **Canonical ownership**: stable public skills live in `ace-*/handbook/skills/<skill>/SKILL.md`.
2. **Required taxonomy**: each canonical skill declares `skill.kind: capability | workflow | orchestration`.
3. **Separate execution body**: canonical skills always bind to `skill.execution.workflow: wfi://...`; workflows remain the executable bodies.
4. **Generated provider projections**: `.agent/skills`, `.claude/skills`, `.codex/skills`, `.gemini/skills`, and `.pi/skills` are generated views, not canonical authored copies.
5. **Canonical discovery**: `skill://` lists and resolves canonical package skills, independent of projected provider trees.
6. **Assign-capable subset**: only `workflow` and `orchestration` skills with explicit `assign:` metadata participate in `ace-assign` composition.
7. **Selective migration**: not every workflow becomes a skill. Internal helper workflows may remain workflow-only if they are not stable public capabilities.

This parent task is an orchestrator coordinating seven vertical slices:
- `8q5.t.0or.0` proves the taxonomy with real ACE examples across all three skill kinds
- `8q5.t.0or.1` defines canonical `SKILL.md` schema and validation rules
- `8q5.t.0or.2` generates projected provider skill trees from canonical package skills
- `8q5.t.0or.3` adds canonical `skill://` discovery
- `8q5.t.0or.4` makes `ace-assign` compose from assign-capable typed skills
- `8q5.t.0or.5` inventories and migrates the typed skill ecosystem
- `8q5.t.0or.6` retires old canonical-path and ownership assumptions

### Interface Contract

```yaml
# Canonical workflow skill
name: as-task-plan
description: Creates JIT implementation plan with ephemeral output
version: 1.0.0
author: ace-task
user-invocable: true
source: ace-task
allowed-tools:
  - Bash(ace-task:*)
  - Bash(ace-bundle:*)
  - Read
  - TodoWrite
tags: [planning, task]
skill:
  kind: workflow
  execution:
    workflow: wfi://task/plan
  projections:
    providers: [agent, claude, codex, gemini, pi]
assign:
  phases:
    - name: plan-task
      description: Refresh the implementation plan before work begins
      emits:
        mode: direct-phase
      context:
        default: no-fork
```

```yaml
# Canonical capability skill
name: as-b36ts
description: Encode and decode timestamps to compact Base36 ids
version: 1.0.0
source: ace-b36ts
user-invocable: true
allowed-tools:
  - Bash(ace-b36ts:*)
  - Bash(ace-bundle:*)
  - Read
skill:
  kind: capability
  execution:
    workflow: wfi://b36ts
```

```bash
# Canonical discovery
ace-nav skill://*
ace-nav skill://as-b36ts
ace-nav skill://as-task-plan
ace-nav skill://as-assign-start

# Assignment composition from the assign-capable subset only
ace-assign catalog
/as-assign-compose "plan task 123 and then start the assignment"
```

**Error Handling:**
- Missing or invalid `skill.kind` fails validation before projection or discovery.
- Missing `skill.execution.workflow` fails validation even if the rest of the skill is provider-compatible.
- `capability` skills that declare `assign:` are rejected rather than silently entering the assignment registry.
- Duplicate canonical skill names resolve through deterministic `skill://` precedence and expose enough source detail to diagnose shadowing.

**Edge Cases:**
- A skill may exist in canonical form without being user-invocable.
- A canonical skill may be projected differently per provider as long as authorship remains single-source.
- A workflow may remain workflow-only and never gain a canonical skill if it is an internal helper.
- `ace-assign` must ignore capability skills even when they are discoverable through `skill://`.

### Success Criteria

- [ ] **Typed canonical skills adopted**: stable public capabilities use package-owned canonical `SKILL.md` files with required `skill.kind`.
- [ ] **Execution remains separate**: canonical skills bind to workflows instead of replacing workflow bodies.
- [ ] **Generated projections adopted**: provider-facing skill trees are generated from canonical package skills.
- [ ] **Canonical discovery exists**: `skill://` lists and resolves canonical package skills across registered gems.
- [ ] **Assign subset enforced**: `ace-assign` composes only from `workflow` and `orchestration` skills with explicit `assign:` metadata.
- [ ] **Selective migration completed**: current stable public skills are inventoried and migrated by type without forcing every workflow into a skill.
- [ ] **Old assumptions removed**: docs and runtime paths no longer treat `.agent/skills` or provider trees as canonical.

### Validation Questions

- [x] **Should ACE encode skill types explicitly?** -> Yes; `skill.kind` is required and uses `capability`, `workflow`, or `orchestration`.
- [x] **Should execution stay in workflows?** -> Yes; canonical skills always bind to separate `wfi://...` workflow bodies.
- [x] **Are provider trees generated or hand-authored?** -> Generated projections from canonical package skills.
- [x] **Should `ace-assign` use all skills?** -> No; it consumes only the assign-capable subset of workflow/orchestration skills with explicit `assign:` metadata.
- [x] **Does every workflow become a skill?** -> No; only stable public capabilities become canonical skills.

## Objective

Replace ACE's current split between provider-facing skills, workflow bodies, and assignment metadata with a typed canonical skill platform that keeps execution in workflows and uses generated projections plus canonical discovery everywhere else.

## Scope of Work

- **User Experience Scope**: Users discover typed canonical skills, keep using provider-facing skills, and compose assignments from the assign-capable subset only.
- **System Behavior Scope**: Typed canonical ownership, workflow binding, generated projections, canonical `skill://` discovery, selective assignment integration, and repo-wide migration.
- **Interface Scope**: `SKILL.md` frontmatter contract, projection behavior, `skill://` protocol, and assignment composition source-of-truth rules.

### Deliverables

#### Behavioral Specifications
- Typed canonical `SKILL.md` contract
- Generated provider-projection contract
- Canonical `skill://` discovery contract
- Assign-capable subset contract for `ace-assign`
- Repo-wide migration and cleanup contract

#### Validation Artifacts
- Tracer proof using one skill of each type
- Usage scenarios for discovery, projections, and assign composition
- Verification commands covering schema, projections, discovery, migration, and cleanup

### Consumer Packages

- **ace-lint**: validates canonical skill schema and projection-safe metadata.
- **ace-llm-providers-cli**: continues provider skill discovery from projected trees.
- **ace-support-nav**: exposes canonical `skill://` discovery and precedence.
- **ace-assign**: composes from the assign-capable typed subset.
- **Packages with public workflows**: own canonical skills for stable public capabilities and keep workflows as execution bodies.
