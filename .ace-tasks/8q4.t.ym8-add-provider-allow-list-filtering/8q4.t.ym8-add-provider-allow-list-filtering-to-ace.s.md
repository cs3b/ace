# Add provider allow-list filtering to ace-llm configuration

## Behavioral Specification

### User Experience
- **Input**: Users declare which providers they want active via `llm.providers.active` in config YAML or `ACE_LLM_PROVIDERS_ACTIVE` env var
- **Process**: When ace-llm loads providers, it filters the full provider set to only those on the allow-list. Users see only active providers in CLI output and can only query active providers.
- **Output**: Filtered provider list in `--list-providers`, clear error messages when querying inactive providers, unchanged behavior when no filter is configured

### Expected Behavior

When a project or user configures `llm.providers.active`, only those providers are available through ace-llm. This controls which providers appear in `--list-providers` output, which providers can be queried, and which provider aliases are registered.

**When `active` list is present and non-empty:**
- Only listed providers are loaded and available
- Provider names are normalized the same way as `ClientRegistry` (case-insensitive, strip `-_`)
- Unlisted providers exist in config files but are not accessible
- Aliases from inactive providers are not registered (natural consequence of filtering before ClientRegistry sees them)

**When `active` is absent, null, or empty `[]`:**
- All discovered providers are available (backward compatible, no change from today)
- This is the forgiving default to avoid accidental lockout

**Env var override:**
- `ACE_LLM_PROVIDERS_ACTIVE=google,anthropic` sets the allow-list
- Env var takes precedence over YAML config (consistent with ADR-022 cascade: CLI/ENV > Project > User > Gem)
- Comma-separated, same normalization rules

### Interface Contract

```yaml
# .ace/llm/config.yml (project-level) or ~/.ace/llm/config.yml (user-level)
llm:
  providers:
    active:          # Optional allow-list
      - google
      - anthropic
      - openai
```

```bash
# Environment variable override
ACE_LLM_PROVIDERS_ACTIVE=google,anthropic ace-llm --list-providers

# List providers (when filter is active)
ace-llm --list-providers
# Output:
# Available LLM Providers (filtered — 3 of 26 active):
#
# ✓ anthropic (API key configured)
#   Models: claude-3-5-sonnet, ...
#
# ✓ google (API key configured)
#   Models: gemini-2.5-flash, ...
#
# ✓ openai (API key configured)
#   Models: gpt-4, ...
#
# Inactive providers (23):
#   deepseek, mistral, opencode, claude, codex, ...

# List providers (when NO filter is active — unchanged from today)
ace-llm --list-providers
# Output identical to current behavior

# Query active provider (works normally)
ace-llm google:gemini-2.5-flash "Hello"

# Query inactive provider (clear error)
ace-llm deepseek:deepseek-chat "Hello"
# Error: Provider 'deepseek' is inactive. It exists but is not in llm.providers.active.
# To enable it, add 'deepseek' to llm.providers.active in your config.
# Active providers: anthropic, google, openai

# Query unknown provider (different error — no config at all)
ace-llm nonexistent:model "Hello"
# Error: Unknown provider: nonexistent. Available providers: anthropic, google, openai
```

**Error Handling:**
- Inactive provider query: "Provider 'X' is inactive. It exists but is not in llm.providers.active. To enable it, add 'X' to llm.providers.active in your config. Active providers: a, b, c"
- Unknown provider query: Existing error message unchanged ("Unknown provider: X. Available providers: ...")
- Invalid provider name in `active` list (no matching config): Warning on stderr during load, silently ignored (does not error)
- Empty `active: []` or absent key: All providers available (no warning)

**Edge Cases:**
- `active: ["google"]` with no google provider config file: Warning logged, no providers crash — google simply won't appear since there's no config for it
- `active: ["Google", "ANTHROPIC"]`: Normalized to match `google`, `anthropic` — works correctly
- Env var with spaces: `ACE_LLM_PROVIDERS_ACTIVE="google, anthropic"` — trimmed, works
- Config cascade: project `.ace/llm/config.yml` sets `active: [google]`, user `~/.ace/llm/config.yml` sets `active: [anthropic, openai]` — project wins per cascade rules

## Objective

Provide a simple way for projects and users to control which LLM providers are active without manipulating individual provider config files. Currently ace-llm loads ALL provider configs found in the cascade (16+ project-level + 10 gem defaults), with no way to limit which are available. A config-driven allow-list (`llm.providers.active`) lets teams declare exactly which providers they use, reducing noise in `--list-providers` output and preventing accidental use of unapproved providers.

## Scope of Work

- **User Experience Scope**: Config YAML editing, env var setting, `--list-providers` CLI output, error messages on inactive provider query
- **System Behavior Scope**: Provider filtering in `Configuration.providers`, updated error messages in `ClientRegistry`, updated `--list-providers` formatting in CLI
- **Interface Scope**: New `llm.providers.active` config key, new `ACE_LLM_PROVIDERS_ACTIVE` env var, updated `--list-providers` output format, new "inactive provider" error message

### Deliverables

#### Behavioral Specifications
- Provider filtering logic in `Configuration.providers`
- Differentiated error messages (inactive vs unknown)
- Updated `--list-providers` output with filter header and inactive section

#### Validation Artifacts
- Unit tests for filtering logic and normalization
- Integration tests for CLI output and error messages
- Failure-path tests for edge cases

## Out of Scope

- **Blacklist/deny-list**: Only allow-list pattern implemented (no `inactive` or `blocked` config key)
- **Per-model filtering**: This filters at the provider level, not individual models
- **Provider groups/profiles**: No named groups like `active_profile: minimal` — just a flat list
- **ProviderConfigReader changes**: Reader stays pure (reads all files); filtering happens in Configuration
- **ClientRegistry refactoring**: Registry receives pre-filtered set, no internal changes needed
