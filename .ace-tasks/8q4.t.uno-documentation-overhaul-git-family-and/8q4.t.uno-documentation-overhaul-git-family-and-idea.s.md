# Documentation Overhaul - Git Family and Idea

## Objective

Rewrite documentation for ace-git, ace-git-worktree, ace-git-secrets, and ace-idea to create a user-first landing experience. Transform reference-heavy READMEs into landing pages with hero GIFs, taglines, and quick start guides. Create tutorial-style getting-started.md files. Create docs/usage.md (CLI reference), docs/handbook.md (skill/workflow catalog), and update gemspec metadata. Audit handbook for misplaced skills and dead content.

This orchestrator depends on 8q4.t.ums (Core User Tools), which established the documentation template. See retros 8qkop7, 8qku8y, 8qkvc4 for learnings applied here.

Research basis: Diataxis framework, clig.dev, top OSS README patterns (gum, lazygit, httpie, ripgrep).

## Behavioral Specification

### User Experience
- **Input**: Existing package READMEs, --help output, current docs/usage.md files for ace-git, ace-git-worktree, ace-git-secrets, ace-idea
- **Process**: Rewrite each package README as landing page (following template from 8q4.t.ums), create getting-started tutorial, record demo GIF, create usage.md + handbook.md, update gemspec, audit handbook, record demo GIF to docs/demo/
- **Output**: Per package: rewritten README.md, new docs/getting-started.md, docs/usage.md, docs/handbook.md, updated gemspec, docs/demo/*.gif committed to git

### Expected Behavior
When a first-time visitor opens any package README on GitHub, they should:
1. See a clear tagline explaining the tool's value proposition (not implementation)
2. See a hero GIF demo (<=30s) showing the tool's core value
3. Find a "Why" section with value-first framing (no commands)
4. Find feature bullets framed as benefits
5. Find "Works With" showing integration packages
6. Find "Agent Skills" listing package-owned skills only
7. Find documentation links to getting-started, usage, handbook
8. NOT find --help output, commands, full config schemas, or API docs in the README

### Interface Contract

Each package gets these files:

README.md structure:
- One-line tagline (value proposition)
- Demo GIF (committed to docs/demo/, referenced as relative path)
- Why section (2-3 sentences, value-first, no commands)
- Works With (integration packages with one-line descriptions)
- Agent Skills (package-owned skills only)
- Features (benefit-oriented bullets)
- Documentation links
- Part of ACE footer

docs/getting-started.md structure:
- Prerequisites
- Installation
- First real task walkthrough
- Common Commands table
- Documentation links at bottom
- What to try next
