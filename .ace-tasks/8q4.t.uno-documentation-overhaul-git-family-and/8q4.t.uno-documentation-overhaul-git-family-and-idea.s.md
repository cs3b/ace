# Documentation Overhaul - Git Family and Idea

## Objective

Rewrite documentation for ace-git, ace-git-worktree, ace-git-secrets, and ace-idea to create a user-first landing experience. Transform reference-heavy READMEs into landing pages with hero GIFs, taglines, and quick start guides. Create tutorial-style getting-started.md files. Delete usage.md files (CLI reference stays in --help only).

This orchestrator depends on 8q4.t.ums (Core User Tools), which establishes the landing page README template via the root README subtask. All packages here follow that template.

Research basis: Diataxis framework, clig.dev, top OSS README patterns (gum, lazygit, httpie, ripgrep).

## Behavioral Specification

### User Experience
- **Input**: Existing package READMEs, --help output, current docs/usage.md files for ace-git, ace-git-worktree, ace-git-secrets, ace-idea
- **Process**: Rewrite each package README as landing page (following template from 8q4.t.ums), create getting-started tutorial, record demo GIF, delete usage.md
- **Output**: Per package: rewritten README.md, new docs/getting-started.md, new docs/demo.gif, deleted docs/usage.md

### Expected Behavior
When a first-time visitor opens any package README on GitHub, they should:
1. See a clear tagline explaining the tool's value proposition (not implementation)
2. See a hero GIF demo (<=30s) showing the tool's core value
3. Find a Quick Start section to get running in <2 minutes
4. Find feature bullets framed as benefits
5. Find links to getting-started.md tutorial and --help for CLI reference
6. NOT find --help output, full config schemas, or API docs in the README

### Interface Contract

Each package gets these files:

README.md structure:
- One-line tagline (value proposition)
- Demo GIF
- Why section (2-3 sentences)
- Quick Start (install + primary usage)
- Features (benefit-oriented bullets)
- Documentation links
- Part of ACE footer

docs/getting-started.md structure:
- Prerequisites
- Installation
- First real task walkthrough
- What to try next
