# Native Client Review for ace-assign Subtree Pre-Commit

## Review Questions (Pending Human Input)

### [HIGH] Critical Design Questions

- [ ] **Dependency 0ki is obsolete — `fork_mode: workflow` doesn't exist**: The spec references `fork_mode: workflow` from task 0ki, but that task was closed as `resolution: obsolete` with unresolved HIGH questions about `skill` semantics and invocation protocol. The `ForkSessionLauncher` has no `fork_mode` concept — it always sends `/ace-assign-drive`. How should the pre-commit review phase be invoked without `fork_mode: workflow`?
  - **Research conducted**: Read `fork_session_launcher.rb` — hardcodes `/ace-assign-drive` prompt. Read 0ki spec — marked obsolete with unresolved semantic conflicts around `skill` meaning.
  - **Suggested default**: Drop `fork_mode: workflow` reference. Define the review phase as a regular inline phase (not forked) since review is fast and doesn't need fork isolation.
  - **Why needs human input**: Fundamental architectural dependency is invalid; spec needs restructuring.

- [ ] **ace-llm has no "native review" capability concept**: The spec says "queries ace-llm's provider abstraction to identify the current CLI provider" and checks for "native review capability." However, `VALID_CAPABILITIES` in `provider_config_validator.rb` only includes: `text_generation`, `streaming`, `function_calling`, `vision`, `embeddings`, `code_generation`, `chat_completion`. There is no `native_review` or similar capability. How should provider review capability be detected?
  - **Research conducted**: Read `provider_config_validator.rb` — capabilities are a closed set. No review-related capability exists. No provider config files reference review.
  - **Suggested default**: Add `native_review` to `VALID_CAPABILITIES` and declare it in Claude Code's provider config, OR detect review capability by checking if the provider is a CLI provider (not API) and has known review commands.
  - **Why needs human input**: Extending the capability model vs. hardcoding provider-specific knowledge is a design choice.

- [ ] **ace-llm is a consumer package but not listed**: The spec lists ace-assign and ace-review as affected packages but doesn't mention ace-llm, which would need `VALID_CAPABILITIES` extended or a new capability detection mechanism added. The `bundle.files` list also doesn't include the provider config validator.
  - **Research conducted**: Confirmed `provider_config_validator.rb` would need changes. Provider config YAML files for CLI providers would need `capabilities: [native_review]` added.
  - **Suggested default**: Add ace-llm to consumer packages list and `ace-llm/lib/ace/llm/atoms/provider_config_validator.rb` to `bundle.files`.
  - **Why needs human input**: Confirms scope expansion is acceptable.

- [ ] **Dependency 0hm is still draft — ReviewFinding model doesn't exist yet**: This task depends on `8q5.t.0hm` for the `ReviewFinding` model, but 0hm is still in `draft` status. The dependency chain means this task cannot be implemented until 0hm is promoted and implemented. Should this task be promoted to pending given its dependency is still draft?
  - **Research conducted**: Checked 0hm status — `status: draft`, no `ReviewFinding` model in codebase yet.
  - **Suggested default**: Promote to pending but note implementation is blocked until 0hm delivers ReviewFinding. Alternatively, define a minimal finding struct inline as a temporary measure.
  - **Why needs human input**: Sequencing decision — promote now or wait for dependency?

### [MEDIUM] Enhancement Questions

- [ ] **Empty review output handling**: What happens when the native review returns no findings? The spec covers "no changed files" (skip) but not "files reviewed, zero findings." Should this produce a positive confirmation message or be silent?
  - **Suggested default**: Brief confirmation: "Review complete: no findings (5 files checked)."

- [ ] **Review output format from native providers**: Claude Code's `/review` produces markdown text, not structured data. How should unstructured review text be parsed into `ReviewFinding` instances? The spec assumes direct mapping (file/line/message) but native review output may not be structured.
  - **Research conducted**: Claude Code `/review` outputs human-readable markdown with file references but not a machine-parseable format.
  - **Suggested default**: Use regex/heuristic parsing for known provider output formats, with raw text fallback when parsing fails.

## Behavioral Specification

### User Experience
- **Input**: User runs `ace-assign work-on subtree` (or equivalent subtree workflow). Before committing, the system automatically invokes a fast native review of the work-in-progress changes.
- **Process**: The workflow detects which CLI provider is active (Claude Code, Codex, etc.), checks if it has a native review capability, and runs it against the current diff. Findings are presented in a structured format before the commit step.
- **Output**: User sees review findings (if any) before committing. They can address findings or skip the review via config. The review output is compatible with ace-review's finding format (leveraging ReviewFinding from task 0hm).

### Expected Behavior

The subtree "work-on" workflow gains a pre-commit review gate:

1. After code changes are complete but before commit, the workflow checks for native review capability
2. Provider detection: queries ace-llm's provider abstraction to identify the current CLI provider
3. Capability check: determines if the provider has a native review command (e.g., Claude Code's `/review`, Codex's equivalent)
4. If available: executes the native review command against the current diff/staged changes
5. Findings are formatted as ReviewFinding instances (from task 0hm's model) for consistency
6. Findings are presented to the user/agent with severity levels
7. Commit proceeds (findings are informational, not blocking by default)

If no native review capability is detected, the step is skipped silently. The review can also use ace-review's runner integration (task 0hm) as a fallback — e.g., running the LintRunner for a quick deterministic check.

### Interface Contract

```yaml
# In ace-assign work-on subtree phase definition
# This is the pre-commit review phase added to the subtree workflow

---
name: pre-commit-review
status: pending
skill: ace-review-run          # uses ace-review with native runner
context: fork                  # optional: run in fork for isolation
fork_mode: workflow            # from task 0ki
---

Review work-in-progress changes using native client review before commit.
Use the fastest available review method (native CLI provider review or ace-lint).
```

```bash
# What the user/agent sees during subtree execution:

# [Phase: pre-commit-review]
# Detected provider: claude (native review available)
# Running native review on 5 changed files...
#
# Review findings:
#   WARNING lib/foo.rb:15 - Consider extracting method (complexity)
#   INFO    lib/bar.rb:42 - Unused import
#
# 2 findings (0 critical, 1 warning, 1 info)
# Proceeding to commit...
```

```yaml
# Configuration: .ace/assign/config.yml
subtree:
  pre_commit_review: true          # default: true
  pre_commit_review_provider: auto # "auto" | "native" | "ace-review" | "skip"
  pre_commit_review_block: false   # if true, block commit on critical findings
```

**Error Handling:**
- No native review capability detected: Step skipped silently, commit proceeds
- Native review command fails: Warning logged, commit proceeds (non-blocking by default)
- Provider detection fails: Falls back to ace-review lint runner (if available), then skips

**Edge Cases:**
- No changed files in subtree: Review step skipped ("nothing to review")
- Provider has review but it's slow (>30s): Configurable timeout, default 30s for pre-commit
- `pre_commit_review_block: true` with critical findings: Commit is blocked, user must address findings
- Multiple providers available: Uses the provider that launched the current session

## Objective

Integrate fast native client review capabilities into `ace-assign work-on subtree` to provide automated review before committing changes. This leverages CLI provider review commands (Claude Code's `/review`, Codex equivalent) for near-instant feedback, with ace-review's runner integration as fallback, ensuring consistent review quality without additional LLM API calls.
