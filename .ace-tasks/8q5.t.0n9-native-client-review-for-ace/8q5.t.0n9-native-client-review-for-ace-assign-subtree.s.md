# Native Client Review for ace-assign Subtree Pre-Commit

## Behavioral Specification

### User Experience
- **Input**: User runs `ace-assign work-on subtree` (or equivalent subtree workflow). Before committing, the system automatically invokes a fast native review of the work-in-progress changes.
- **Process**: The workflow detects which CLI provider is active (Claude Code, Codex, etc.), checks if it has a native review capability, and runs it against the current diff. Findings are presented in a structured format before the commit step.
- **Output**: User sees review findings (if any) before committing. They can address findings or skip the review via config. The review output is compatible with ace-review's finding format (leveraging ReviewFinding from task 0hm).

### Expected Behavior

The subtree "work-on" workflow gains a pre-commit review gate:

1. After code changes are complete but before commit, the workflow checks for native review capability
2. Provider detection: queries ace-llm's provider abstraction to identify the current CLI provider
3. Capability check: determines if the provider has a native review command (e.g., Claude Code's `/review`, Codex's equivalent)
4. If available: executes the native review command against the current diff/staged changes
5. Findings are formatted as ReviewFinding instances (from task 0hm's model) for consistency
6. Findings are presented to the user/agent with severity levels
7. Commit proceeds (findings are informational, not blocking by default)

If no native review capability is detected, the step is skipped silently. The review can also use ace-review's runner integration (task 0hm) as a fallback — e.g., running the LintRunner for a quick deterministic check.

### Interface Contract

```yaml
# In ace-assign work-on subtree phase definition
# This is the pre-commit review phase added to the subtree workflow

---
name: pre-commit-review
status: pending
skill: ace-review-run          # uses ace-review with native runner
context: fork                  # optional: run in fork for isolation
fork_mode: workflow            # from task 0ki
---

Review work-in-progress changes using native client review before commit.
Use the fastest available review method (native CLI provider review or ace-lint).
```

```bash
# What the user/agent sees during subtree execution:

# [Phase: pre-commit-review]
# Detected provider: claude (native review available)
# Running native review on 5 changed files...
#
# Review findings:
#   WARNING lib/foo.rb:15 - Consider extracting method (complexity)
#   INFO    lib/bar.rb:42 - Unused import
#
# 2 findings (0 critical, 1 warning, 1 info)
# Proceeding to commit...
```

```yaml
# Configuration: .ace/assign/config.yml
subtree:
  pre_commit_review: true          # default: true
  pre_commit_review_provider: auto # "auto" | "native" | "ace-review" | "skip"
  pre_commit_review_block: false   # if true, block commit on critical findings
```

**Error Handling:**
- No native review capability detected: Step skipped silently, commit proceeds
- Native review command fails: Warning logged, commit proceeds (non-blocking by default)
- Provider detection fails: Falls back to ace-review lint runner (if available), then skips

**Edge Cases:**
- No changed files in subtree: Review step skipped ("nothing to review")
- Provider has review but it's slow (>30s): Configurable timeout, default 30s for pre-commit
- `pre_commit_review_block: true` with critical findings: Commit is blocked, user must address findings
- Multiple providers available: Uses the provider that launched the current session

## Objective

Integrate fast native client review capabilities into `ace-assign work-on subtree` to provide automated review before committing changes. This leverages CLI provider review commands (Claude Code's `/review`, Codex equivalent) for near-instant feedback, with ace-review's runner integration as fallback, ensuring consistent review quality without additional LLM API calls.
