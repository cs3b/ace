## Behavioral Specification

### User Experience
- **Input**: User runs `ace-assign work-on subtree` (or equivalent subtree workflow). After work-on completes and before committing, the system optionally invokes a fast native review of the work-in-progress changes, optionally in an ace-assign forked phase.
- **Process**: The workflow uses ace-assign's existing phase/fork execution model to run a review step against the current diff. If the active client exposes its own `/review` command, ace-assign uses it directly.
- **Output**: User sees review findings before committing. They can address findings or continue. Raw native review output is acceptable when structured extraction is incomplete.

### Expected Behavior

The subtree "work-on" workflow gains a pre-commit review gate:

1. After code changes are complete but before commit, the workflow enters a pre-commit review phase
2. The phase may run inline or with `context: fork`, but only through existing ace-assign fork delegation
3. If the active client supports a native `/review` command, ace-assign invokes that native review path after work-on completes
4. If no native review path is available, the phase skips
5. Findings are shown to the user/agent before commit
6. Commit proceeds unless `pre_commit_review_block: true` is enabled and the implementation can confidently identify a blocking finding

If the configured native review path is unavailable, the step skips gracefully.

### Interface Contract

```yaml
# In ace-assign work-on subtree phase definition
# This is the pre-commit review phase added to the subtree workflow

---
name: pre-commit-review
status: pending
context: fork                  # optional: run in ace-assign fork isolation
---

Review work-in-progress changes before commit.
If the active client provides a native `/review` command, use it directly; otherwise skip this phase.
```

```bash
# What the user/agent sees during subtree execution:

# [Phase: pre-commit-review]
# Detected provider: claude (native review available)
# Running native review on 5 changed files...
#
# Review findings:
#   WARNING lib/foo.rb:15 - Consider extracting method (complexity)
#   INFO    lib/bar.rb:42 - Unused import
#
# 2 findings (0 critical, 1 warning, 1 info)
# Proceeding to commit...
```

```yaml
# Configuration: .ace/assign/config.yml
subtree:
  pre_commit_review: true          # default: true
  pre_commit_review_provider: auto # "auto" | "native" | "skip"
  pre_commit_review_block: false   # if true, block commit on critical findings
  native_review_clients: [claude]  # allowlist of clients with supported /review commands
```

**Error Handling:**
- Native review unavailable: Step skipped, commit proceeds
- Review command fails: Warning logged, commit proceeds (non-blocking by default)
- Fork execution unavailable: Runs inline if supported, otherwise skip review

**Edge Cases:**
- No changed files in subtree: Review step skipped ("nothing to review")
- Review takes too long (>30s): Configurable timeout, default 30s for pre-commit
- `pre_commit_review_block: true` with critical findings: Commit is blocked, user must address findings
- Forked review phase uses the same assignment/session context as the subtree execution

## Objective

Integrate fast native client review capabilities into `ace-assign work-on subtree` to provide automated review before committing changes. This leverages CLI provider review commands (Claude Code's `/review`, Codex equivalent) for near-instant feedback, with ace-review's runner integration as fallback, ensuring consistent review quality without additional LLM API calls.
