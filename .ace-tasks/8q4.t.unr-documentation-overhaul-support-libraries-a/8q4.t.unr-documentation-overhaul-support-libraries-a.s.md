# Documentation Overhaul - Support Libraries A (Light Refresh)

## Objective

Light refresh of support library READMEs for ace-support-core, ace-support-config, ace-support-fs, ace-support-items, ace-support-mac-clipboard, and ace-support-cli to apply consistent structure. No GIFs needed. Add clear tagline, consistent sections, keep existing docs. These are developer-facing libraries, not user-facing tools. Update gemspec summary/description to match README tagline (no internal jargon).

See retros 8qkop7, 8qku8y, 8qkvc4 for learnings applied here.

Research basis: Diataxis framework, top OSS library README patterns for internal/developer-facing packages.

## Behavioral Specification

### User Experience

- **Input**: Existing support library READMEs
- **Process**: Refresh README structure with consistent tagline, sections, purpose statement, update gemspec metadata
- **Output**: Per package: refreshed README.md with consistent structure, updated gemspec

### Expected Behavior

When a developer opens any support library README, they should:
1. See a clear tagline explaining the library's purpose
2. See a concise API overview
3. Find installation and basic usage
4. Find link to parent ACE project

### Interface Contract

Lightweight template applied to each support library:
- Tagline: one-line purpose statement
- Purpose: brief explanation of what the library provides and who uses it
- Installation: gem install or Gemfile entry
- Basic Usage: minimal code example showing primary API
- API Overview: concise list of key modules/classes and their roles
- Link to parent ACE project for broader context

No GIF, no getting-started.md, no usage.md deletion required. Keep existing docs intact.
