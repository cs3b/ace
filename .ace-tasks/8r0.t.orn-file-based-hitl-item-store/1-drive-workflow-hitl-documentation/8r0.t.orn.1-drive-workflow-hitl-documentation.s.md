# Implement ace-hitl Item Store and CLI

## Objective

Implement the actual `ace-hitl` item-store behavior and command surface inside the new package so both humans and agents can manage HITL items through a dedicated CLI instead of manual ad hoc file operations.

## Behavioral Specification

### User Experience

- **Input**: A human or agent needs to create, inspect, answer, or archive a HITL item.
- **Process**: `ace-hitl` manages the `.ace-hitl/` store with `list`, `show`, `create`, and `update` commands backed by a package-owned manager and item model.
- **Output**: HITL items can be worked end-to-end through the CLI while remaining simple markdown files in the repo.

### Expected Behavior

1. `ace-hitl create` creates a new `.hitl.s.md` item under `.ace-hitl/<id>-<slug>/`.
2. `ace-hitl list` shows items with filtering by status, kind, tags, and folder.
3. `ace-hitl show` resolves refs the same way ACE item tools do and can print path or raw content when requested.
4. `ace-hitl update` supports:
   - `--set key=value`
   - `--add key=value`
   - `--remove key=value`
   - `--move-to archive|next`
   - `--answer "..."` to write the `## Answer` body and mark the item answered
5. `ace-hitl` owns the HITL file contract, including frontmatter fields and markdown sections.
6. Each item represents one clarification/decision thread, not a multi-turn conversation log.
7. `ace-support-items` may be reused for generic scanning/frontmatter/folder operations, but no standalone HITL feature slice is introduced there.

### Interface Contract

CLI:

```bash
ace-hitl list --status pending
ace-hitl show t.abc
ace-hitl show t.abc --content
ace-hitl create "Which auth strategy?" --question "JWT or sessions?" --question "Refresh token storage?" --assignment 8qr5kx --step 020 --step-name implement-auth --resume "/as-assign-drive 8qr5kx"
ace-hitl update t.abc --answer "Use JWT with server-side refresh tokens."
ace-hitl update t.abc --move-to archive
```

Library:

```ruby
manager = Ace::Hitl::Organisms::HitlManager.new
item = manager.create("Which auth strategy?",
  questions: ["JWT or sessions?", "Refresh token storage?"],
  assignment: "8qr5kx",
  step: "020",
  step_name: "implement-auth",
  resume_instructions: "/as-assign-drive 8qr5kx"
)

answered = manager.update(item.id, answer: "Use JWT with server-side refresh tokens.")
```

### Success Criteria

- [ ] `ace-hitl list/show/create/update` all exist and are wired into the package CLI
- [ ] HITL items are created with the agreed `.ace-hitl/` file layout
- [ ] `update --answer` writes the answer body and marks the item answered
- [ ] Item loading/listing/filtering follow established ACE item-tool patterns
- [ ] Generic support code reuse stays internal to `ace-hitl`

## Vertical Slice Decomposition

Single feature slice:
- **Outcome**: `ace-hitl` can manage HITL items end-to-end
- **Advisory size**: medium
- **Context**: mirror manager/CLI patterns from `ace-retro` and `ace-task`

## Verification Plan

### Unit/Component Validation
- Create/list/show/update flows are covered by package tests
- Answer writing updates both frontmatter and body correctly
- Archive moves preserve item identity and reload behavior

### Failure/Invalid Path Validation
- Invalid refs return clear errors
- Empty answers do not falsely mark an item answered unless explicitly requested
- Missing `## Answer` section is handled by package-owned update logic instead of corrupting the document
