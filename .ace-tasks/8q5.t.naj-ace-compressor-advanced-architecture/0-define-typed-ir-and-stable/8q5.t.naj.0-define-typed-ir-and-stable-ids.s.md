# Define typed IR and stable IDs

## Behavioral Specification

### User Experience
- **Input**: Architectural requirements for deterministic compression and future patching.
- **Process**: The task defines the canonical intermediate objects and ID rules that downstream behaviors rely on.
- **Output**: A stable data model and ID strategy that future implementations can follow exactly.

### Expected Behavior
The architecture must define typed document, section, rule, fact, example, and patch objects. Stable IDs must be derived from normalized semantic identity, not line numbers, so formatting-only edits do not churn downstream patches.

### Interface Contract
```text
Document{id,path,type,sections[]}
Section{id,title,blocks[],summary}
Rule{id,subject,modality,action,provenance[]}
Fact{id,subject,predicate,object,qualifiers{},provenance[]}
```

**Error Handling:**
- ID scheme tied to line positions: reject.
- Untyped freeform intermediate representation: reject.

**Edge Cases:**
- Heading punctuation changes do not rotate section IDs.
- Frontmatter changes do not rotate semantic unit IDs.

### Success Criteria
- [ ] Typed IR is explicit and minimal.
- [ ] Stable-ID rules survive formatting-only edits.
- [ ] ID inputs are semantic, not positional.
- [ ] Canonical unit types are sufficient for later patching.

### Validation Questions
- [x] Typed objects are preferred over freeform strings.
- [x] Line numbers are not primary identity.
- [x] Section and unit IDs survive punctuation and frontmatter churn.
- [x] Future patching depends on this contract.
