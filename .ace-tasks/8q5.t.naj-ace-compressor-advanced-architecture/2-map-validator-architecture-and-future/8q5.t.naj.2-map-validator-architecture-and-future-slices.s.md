# Map validator architecture and future slices

## Behavioral Specification

### User Experience
- **Input**: Candidate architecture decisions and future scope beyond the first two modes.
- **Process**: The task defines how the system validates fidelity and how later implementation should be sliced.
- **Output**: Explicit validator requirements, rejection conditions, and future implementation slices that reduce churn.

### Expected Behavior
The validator must check rule preservation, numeric fidelity, section coverage, unresolved-item visibility, and token-budget compliance. The task must also produce a future slice map so later implementation can proceed without rediscovering the architecture.

### Interface Contract
```text
VALIDATE|rules_preserved=true
VALIDATE|numeric_fidelity=true
VALIDATE|section_coverage=acceptable
VALIDATE|token_budget=met
```

**Error Handling:**
- Validation dimension omitted from the architecture: reject.
- Future slice plan that reintroduces removed concepts: flag as churn risk.

**Edge Cases:**
- Compact mode may pass token budget and fail rule preservation; validator rejects it.
- Exact mode may pass fidelity and exceed a target budget; validator reports but does not reinterpret semantics.

### Success Criteria
- [ ] Validator dimensions are explicit.
- [ ] Failure semantics are explicit.
- [ ] Future slices are sequenced without concept churn.
- [ ] Acceptance criteria are directly testable later.

### Validation Questions
- [x] Fidelity dimensions outrank compression ratio.
- [x] Validator output must distinguish warning from rejection.
- [x] Future slices should avoid introducing concepts that later tasks remove.
- [x] This task closes architectural ambiguity for subsequent implementation work.

### Vertical Slice Decomposition (Task/Subtask Model)
- **Slice Type**: Subtask
- **Slice Outcome**: A full validator and future-slice contract for post-architecture work.
- **Advisory Size**: small
- **Context Dependencies**: deep research, advanced architecture note

### Verification Plan

#### Unit / Component Validation
- [ ] Rule, numeric, coverage, and budget checks are all represented.

#### Integration / E2E Validation (if cross-boundary behavior exists)
- [ ] A future implementation plan can be derived without reopening core architecture decisions.

#### Failure / Invalid-Path Validation
- [ ] Missing validator dimensions reject the architecture.

- [ ] `mise exec -- ace-task show 8q5.t.naj.2 --content`
