# Validate dedup provenance update flows

## Behavioral Specification

### User Experience
- **Input**: Multiple exact-mode sources with repeated facts/rules plus a change over time.
- **Process**: The CLI deduplicates equivalent units, preserves provenance, and validates update-path correctness.
- **Output**: A runnable exact update flow that remains traceable across multi-source inputs.

### Expected Behavior
This slice connects deduplication, provenance, and validation to the live update path. Repeated facts may collapse into canonical units, but each unit retains provenance. Validation must check rule preservation, section coverage, unresolved-item visibility, and patch/base compatibility.

### Interface Contract
```bash
ace-compressor compress docs/vision.md docs/architecture.md --mode exact --verbose
ace-compressor patch prev.contextpack docs/architecture.md --mode exact
```

**Error Handling:**
- Canonical unit without provenance is invalid.
- Patch generated from incompatible baseline is invalid.

**Edge Cases:**
- Near-duplicate facts from two files may merge while keeping both provenance references.
- Contradictory facts stay explicit instead of silently overwriting each other.

### Success Criteria
- [ ] Multi-file exact output can deduplicate repeated content.
- [ ] Canonical units retain provenance.
- [ ] Update-path validation is testable and explicit.
- [ ] Contradictory facts remain visible.

### Validation Questions
- [x] Dedup/provenance are part of runnable behavior here.
- [x] Contradictions must remain explicit.
- [x] Validator checks include more than patch-base compatibility alone.
- [x] This slice closes the minimal update-friendly phase.
