# Specify provenance patches and dedup

## Behavioral Specification

### User Experience
- **Input**: Multiple related documents and later updates to them.
- **Process**: The architecture keeps canonical units, tracks provenance, and emits patch operations by stable ID.
- **Output**: Compact, traceable, update-friendly contracts for deduped context.

### Expected Behavior
The architecture must preserve provenance for every rule/fact/table/chart surrogate. Near-duplicate content must consolidate into canonical units with provenance lists. Incremental updates must target stable IDs via `PatchPack/1`, not line diffs.

### Interface Contract
```text
P|id=fact:config_cascade|src=docs/architecture.md#configuration-cascade
PATCH|base=sha256:...
R|id=rule:cli_first|value=...
I|after=fact:config_cascade|id=fact:new_flag|value=...
D|id=example:legacy_cli
```

**Error Handling:**
- Patch without base compatibility metadata: reject.
- Canonical unit without provenance: reject.

**Edge Cases:**
- Two equivalent facts from different files merge into one canonical fact with both provenance refs.
- Contradictory facts stay explicit instead of silently overwriting each other.

### Success Criteria
- [ ] Every canonical unit has provenance.
- [ ] Patch operations are ID-addressed.
- [ ] Deduplication preserves traceability.
- [ ] Contradictions remain explicit.

### Validation Questions
- [x] Provenance is mandatory for every canonical unit.
- [x] Patch operations target stable IDs only.
- [x] Deduplication never erases source traceability.
- [x] Conflicts stay visible rather than silently merged away.
