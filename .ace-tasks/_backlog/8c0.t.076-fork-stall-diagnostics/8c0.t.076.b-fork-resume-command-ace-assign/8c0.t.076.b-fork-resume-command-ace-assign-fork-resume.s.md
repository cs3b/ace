# Fork resume command (ace-assign fork-resume)

## Behavioral Specification

### User Experience
- **Input**: `ace-assign fork-resume "message" --assignment <id>@<root>` — sends a follow-up message to a stalled forked session
- **Process**: Finds the stored session ID, invokes `codex exec resume <session_id> "message"` to continue the stalled agent
- **Output**: Agent resumes work; subtree state is re-checked and reported

### Expected Behavior

When a fork stalls (detected via 279.01), the user or orchestrator can send a follow-up message:

```bash
# Read what the agent was stuck on
ace-assign status --assignment 8plhhk
# > Stall Reason: I hit unexpected state change... unrelated files in git status

# Send instruction to resume
ace-assign fork-resume "Proceed. Only commit files you changed. Ignore unrelated bundle changes." \
  --assignment 8plhhk@010.03

# Agent resumes, completes remaining phases
# > Fork subtree 010.03 completed after resume.
```

### Interface Contract

```bash
# New CLI command
ace-assign fork-resume MESSAGE [OPTIONS]

Arguments:
  MESSAGE                  Follow-up message to send to the stalled agent (required)

Options:
  --assignment ID[@ROOT]   Target assignment (with optional fork root scope)
  --root PHASE             Fork subtree root phase number
  --session-id UUID        Explicit Codex session ID (auto-detected if omitted)
  --provider PROVIDER      LLM provider override
  --timeout SECONDS        Execution timeout (default: 1800)
  -q, --quiet              Suppress non-essential output

# Session ID storage
.cache/ace-assign/<id>/sessions/<fork_root>-session-id.txt
```

**Error Handling:**
- No stored session ID + no `--session-id` flag: raise clear error with instructions
- Provider doesn't support resume (non-codex): raise "Provider X does not support session resume"
- Session ID expired/invalid: raise codex error, suggest `fork-run` fresh start

### Success Criteria

- [ ] `ace-assign fork-resume "msg" --assignment <id>@<root>` resumes a stalled fork
- [ ] Session ID auto-detected from `.cache/ace-assign/<id>/sessions/` storage
- [ ] `--session-id` flag allows explicit session ID override
- [ ] CodexClient supports `resume_session_id` option (`codex exec resume <id>`)
- [ ] Session ID captured from codex stdout and persisted by ForkSessionLauncher
- [ ] Non-codex providers get clear "not supported" error
- [ ] Subtree state re-checked after resume completes
- [ ] Unit tests for ForkResume command, ForkSessionResumer, session ID capture
