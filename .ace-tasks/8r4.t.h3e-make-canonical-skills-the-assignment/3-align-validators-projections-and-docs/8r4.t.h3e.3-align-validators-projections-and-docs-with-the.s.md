# Align Validators, Projections, and Docs With the Skill-First Contract

## Objective

Align schema validation, provider projection behavior, and user-facing documentation with the final skill-first assignment contract after the public metadata, helper migration, and runtime `source:` changes are in place.

## Behavioral Specification

### User Experience

- **Input**: A maintainer edits canonical skills, runs lint/schema validation, syncs provider projections, or reads usage docs for the assignment discovery model.
- **Process**: Validation rules accept the final canonical contract, provider sync/status continues to project the canonical skills deterministically, and usage/docs describe skill-first discovery plus internal-only `wfi://...` execution accurately.
- **Output**: Canonical skills lint cleanly, projections stay in sync, and documentation matches the shipped contract instead of the retired duplicated model.

### Expected Behavior

1. `ace-lint` accepts workflowless direct skills where the final contract allows them.
2. `ace-lint` continues to reject invalid `assign:` usage outside the allowed skill kinds.
3. Provider sync/status behavior continues to project canonical skills without changing their underlying assignment contract.
4. Assignment docs and usage scenarios describe:
   - skill-first public discovery
   - fallback-only direct skill execution
   - internal-only explicit `wfi://...` execution
5. Obsolete references to duplicated public step ownership or public workflow discovery are removed from relevant docs/tests.

### Interface Contract

Validator expectations after alignment:

```yaml
skill:
  kind: capability
```

Valid when:
- the skill is directly executable
- the skill has no `assign:` block
- runtime fallback discovery rules are documented elsewhere

Projection/documentation expectations:

```yaml
assign:
  steps:
    - name: work-on-task
```

- remains canonical source metadata
- is projected consistently to provider-native trees
- is documented as canonical rather than provider-owned

Error Handling:

- If validation rules and projection assumptions disagree about the canonical schema, this task must resolve that mismatch explicitly.
- If docs still describe public `wfi://...` discovery or duplicated YAML ownership, they must be updated or called out as stale before completion.

Edge Cases:

- Capability skills may still have workflow bindings; this task should not conflate “capability” with “workflowless.”
- Provider projections may add provider-specific frontmatter, but must not change canonical assignment ownership semantics.

### Success Criteria

- [ ] `ace-lint` validation rules match the final canonical assignment contract
- [ ] Provider sync/status coverage still passes for the updated canonical skill frontmatter
- [ ] Usage/docs explain skill-first discovery, fallback direct skills, and internal-only explicit `wfi://...`
- [ ] Stale references to duplicated public ownership or public workflow discovery are removed or updated
- [ ] Validator and projection consumers no longer assume the retired duplicated model

## Vertical Slice Decomposition (Task/Subtask Model)

Single migration subtask.

- **Slice type**: subtask
- **Slice outcome**: validator, projection, and docs consumers align with the final contract
- **Advisory size**: medium
- **Context dependencies**: final public/helper/runtime contract from sibling subtasks, ADR-027, usage docs

## Verification Plan

### Unit/Component Validation

- Verify lint schema and validator tests cover allowed direct-skill and `assign:` combinations
- Verify provider sync/status tests still reflect canonical ownership and projection behavior
- Verify usage docs and assignment docs describe the same contract the code validates

### Integration/E2E Validation

- Trace one canonical skill update through validation and provider projection expectations
- Trace one assignment usage scenario that references public discovery and one that references internal-only explicit `wfi://...`

### Failure/Invalid Path Validation

- Verify invalid `assign:` on unsupported skill kinds still fails
- Verify docs do not advertise workflows as a public discovery inventory
- Verify projection logic does not reintroduce provider-owned assignment semantics
