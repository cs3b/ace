# Make Canonical Skills the Assignment Discovery Layer

## Objective

Replace the current split assignment metadata model with a skill-first design where canonical skills are the single discovery unit for both agents and `ace-assign`, workflow binding is declared in exactly one place, assignment steps use a single protocol-based execution target, and duplicated step-definition YAML entries are retired as their metadata is migrated into canonical skills.

## Behavioral Specification

### User Experience

- **Input**: A developer or agent wants to discover, compose, and execute assignment steps such as `work-on-task`, `create-pr`, or `verify-test-suite`.
- **Process**: `ace-assign` discovers canonical skills, reads composition metadata from those skills, applies recipes and composition rules, binds workflow-backed execution through `skill.execution.workflow`, allows fallback-only direct execution for skills with no workflow binding and no `assign:`, and performs runtime subtree expansion from workflow frontmatter when a bound workflow declares it. Explicit `wfi://...` step targets remain available for internal helpers and explicitly authored assignment steps, but they are not a public discovery surface.
- **Output**: A single canonical source of truth for public assignment steps, one non-duplicated workflow-binding field, a unified protocol-based step target contract, and no duplicated public step definitions across skills and YAML step files.

### Expected Behavior

1. Canonical skills become the authoritative discovery and composition layer for public assignment steps.
2. `skill.execution.workflow` is the only workflow-binding field on canonical skills.
3. Top-level skill `assign:` remains valid, but only as skill-side composition metadata rather than workflow lookup.
4. Workflow frontmatter `assign:` remains the authoritative source for runtime subtree expansion such as `assign.sub-steps` and runtime `context`.
5. Assignment steps move toward a single `source:` URI contract, with targets such as `skill://as-task-work` or `wfi://task/work`, instead of long-term separate `skill:` and `workflow:` fields.
6. Skills with no workflow binding and no `assign:` remain directly executable and are composable only through fallback name and description matching when compose needs to discover them.
7. `ace-assign` retains recipes and composition rules as orchestrator policy, but stops owning duplicated public step definitions in YAML.
8. Public step-definition YAML entries under `ace-assign/.ace-defaults/assign/catalog/steps/*.step.yml` are retired as their metadata migrates into skills.
9. Helper steps that still require composition metadata but have no canonical skill today are migrated to internal skills and workflows rather than remaining as permanent `skill: null` YAML entries.
10. Provider projections continue to mirror canonical skills after the canonical schema is updated.

### Interface Contract

Workflow-bound canonical skill contract after migration:

```yaml
skill:
  kind: workflow
  execution:
    workflow: wfi://task/work
assign:
  steps:
    - name: work-on-task
      description: Implement task changes following project conventions
      intent:
        phrases: ["work on task", "implement task"]
      prerequisites:
        - name: plan-task
          strength: recommended
      context:
        default: fork
```

Direct-execution canonical skill contract after migration:

```yaml
name: external-review
description: Run an external review directly from the skill body
skill:
  kind: capability
```

Step execution contract after migration:

```yaml
steps:
  - name: work-on-task
    source: skill://as-task-work
  - name: review-external
    source: skill://external-review
  - name: validate-workflow
    source: wfi://test/verify-suite
```

`wfi://...` targets are valid only for explicitly authored assignment steps and internal helper flows. They are not part of public discovery; public step discovery remains skill-first.

Workflow runtime contract:

```yaml
assign:
  sub-steps:
    - onboard-base
    - task-load
    - plan-task
    - work-on-task
  context: fork
```

Composition defaults for skills with no workflow and no `assign:`:

- compose may discover them from skill name + description matching
- execution uses the skill body directly
- they contribute no implicit prerequisites, context, or tags unless those are declared explicitly by future schema

`ace-assign` policy ownership:

- Recipes stay under `ace-assign/.ace-defaults/assign/catalog/recipes/`
- Composition rules stay under `ace-assign/.ace-defaults/assign/catalog/composition-rules.yml`
- Public step-definition YAML files are migration targets, not the desired long-term source of truth

Error Handling:

- Skills without `skill.execution.workflow` are valid direct-execution skills when the executor targets `skill://...`.
- Skills without `assign.steps` remain directly executable and discoverable only through name/description-based fallback composition metadata.
- Workflows without workflow-level `assign:` remain valid execution workflows when subtree expansion is not needed.
- Internal helper steps may remain temporarily in YAML only while they are being migrated to internal skills and workflows.
- A `source:` URI with an unsupported protocol must fail clearly rather than falling back silently.

Edge Cases:

- The final design must support both user-invocable public skills and internal non-user-invocable helper skills.
- The final design must support both workflow-bound skills and direct skills with no workflow.
- The migration must define how legacy `skill:` / `workflow:` step fields normalize into the new `source:` field.
- Migration must avoid a period where both skill and YAML metadata silently diverge for the same public step.
- Agent discovery and `ace-assign` composition must be able to enumerate the same canonical public step inventory.
- Explicit `wfi://...` steps must remain available for internal helpers and hand-authored assignments without becoming a second discoverable public inventory.

### Success Criteria

- [ ] The task locks a skill-first metadata ownership model instead of the earlier YAML-first cleanup direction

- [ ] The task states that workflow binding is declared only in `skill.execution.workflow`
- [ ] The intended long-term home for public step metadata is canonical skill `assign.steps`
- [ ] The intended long-term home for runtime subtree expansion is workflow frontmatter `assign:`
- [ ] The task states that assignment step execution targets move to a single protocol-based `source:` field
- [ ] The task states that workflowless skills remain directly executable and composable
- [ ] The task states that workflowless/no-assign skills compose through name + description matching
- [ ] The task states that `wfi://...` remains internal/explicit-only and not a public discovery source
- [ ] The task states that recipes and composition rules stay in `ace-assign`
- [ ] The task states that helper steps must migrate to internal skills/workflows rather than remain permanent `skill: null` catalog entries

### Validation Questions

- [x] Should skills be the main discovery layer for `ace-assign` and agents? -> Yes. Skills are the canonical discovery unit.
- [x] Should workflows remain the execution layer? -> Yes. Execution instructions and runtime subtree expansion stay on workflows.
- [x] Should workflow binding live anywhere other than `skill.execution.workflow`? -> No. Workflow binding should exist in one place only.
- [x] Should `ace-assign` support workflowless direct skills? -> Yes. Direct skills remain composable and executable.
- [x] How should direct skills enter composition when they have no `assign:`? -> Through skill name and description matching.
- [x] How should assignment steps identify execution targets long-term? -> Through one protocol-based `source:` field such as `skill://...` or `wfi://...`, with `wfi://...` limited to internal helpers and explicitly authored assignments rather than public discovery.
- [x] Should recipes and composition rules remain in `ace-assign`? -> Yes. They are orchestration policy rather than duplicated step definitions.
