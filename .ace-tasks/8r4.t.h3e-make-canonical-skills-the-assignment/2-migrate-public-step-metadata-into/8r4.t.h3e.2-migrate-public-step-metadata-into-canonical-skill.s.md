# Migrate Public Step Metadata Into Canonical Skill Frontmatter

## Objective

Move public step-definition ownership from duplicated `ace-assign` catalog YAML entries into canonical skill frontmatter so public discovery and composition are driven by `assign.steps`, while leaving recipes and composition rules in `ace-assign` as orchestration policy.

## Behavioral Specification

### User Experience

- **Input**: A maintainer composes assignments that reference public steps such as `onboard`, `plan-task`, `work-on-task`, `review-pr`, `create-pr`, or `verify-test-suite`.
- **Process**: Public step metadata that is currently duplicated in catalog YAML moves into canonical package skills, and `ace-assign` consumes that canonical metadata as the only long-term public source of truth.
- **Output**: Public step discovery stays stable for users and agents, but the owning metadata lives on canonical skills rather than duplicated YAML entries.

### Expected Behavior

1. Every public step classified by `8r4.t.h3e.0` gains a canonical owner in `assign.steps`.
2. Public workflow binding is no longer owned by `assign.source`; workflow binding comes from `skill.execution.workflow`.
3. Public step metadata such as `description`, `intent`, `prerequisites`, `produces`, `consumes`, `context`, `when_to_skip`, `effort`, and `tags` lives in canonical skill frontmatter.
4. Public catalog YAML files become migration targets only:
   - transitional compatibility shims where needed
   - removed once runtime compatibility no longer depends on them
5. Recipes and composition rules remain in `ace-assign` and are not pulled into canonical skills.

### Interface Contract

Canonical public skill contract:

```yaml
name: as-task-work
description: Execute task implementation with context loading and change commits
skill:
  kind: workflow
  execution:
    workflow: wfi://task/work
assign:
  steps:
    - name: work-on-task
      description: Implement task changes following project conventions
      intent:
        phrases:
          - work on task
          - implement task
      produces: [code-changes, commits]
      consumes: [project-base-context, task-spec]
      when_to_skip:
        - Task is documentation-only
      effort: medium
      tags: [implementation, core-workflow]
```

Policy ownership that remains in `ace-assign`:

```yaml
ordering:
  - rule: plan-before-implementation
    before: plan-task
    after: work-on-task
```

Error Handling:

- If public YAML contains metadata that does not belong in skills, this task must classify it as orchestration policy or compatibility-only data rather than duplicating it.
- If a public step lacks a canonical skill owner, this task must keep the gap visible and documented instead of silently inventing ownership.

Edge Cases:

- Public steps with current nested `steps:` or mixed helper behavior must be separated from true public metadata ownership.
- Public skills may preserve direct execution behavior while still owning public step metadata.

### Success Criteria

- [ ] Every public step from the spike keep/remove matrix is mapped to a canonical skill owner
- [ ] Public step metadata moves to canonical `assign.steps`
- [ ] `skill.execution.workflow` is the only workflow-binding owner for public skills
- [ ] No public step retains duplicated long-term ownership between catalog YAML and canonical skill frontmatter
- [ ] Recipes and composition rules remain under `ace-assign` and are explicitly out of migration scope

## Verification Plan

### Unit/Component Validation

- Compare each public YAML entry selected by the spike against its target canonical skill `assign.steps` entry
- Verify workflow binding is read from `skill.execution.workflow` rather than duplicated public metadata fields
- Verify policy-only data remains in composition rules or recipes

### Integration/E2E Validation

- Trace a public composed path such as `onboard -> plan-task -> work-on-task -> create-pr` using canonical skill metadata as the discovery source
- Verify the same public step names remain available to assignment composition after ownership moves

### Failure/Invalid Path Validation

- Verify no public step keeps two long-term metadata owners
- Verify missing canonical skill coverage is surfaced as a migration gap
- Verify helper-only metadata is not incorrectly imported into public `assign.steps`
