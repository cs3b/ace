# Validate Skill-First Assignment Metadata Model

## Objective

Validate the target skill-first assignment metadata model on workflow-bound, direct-skill, and explicit-workflow tracer paths and produce the decision-complete contract needed to draft the remaining migration subtasks without rework.

## Behavioral Specification

### User Experience

- **Input**: A maintainer reviews the current assignment metadata split across skills, workflows, and YAML step files.
- **Process**: The spike inventories the current duplicated owners, traces workflow-bound and direct-skill assignment paths, validates a protocol-based step target contract, and defines the final contract for canonical skills, workflows, internal helper skills, and retained `ace-assign` policy assets. It also identifies every consumer that must change once the contract is implemented.
- **Output**: A documented end-state and migration inventory that is precise enough to draft the remaining implementation subtasks without inventing ownership decisions later.

### Expected Behavior

1. The spike produces a contract table covering:
   - skill-side discovery/composition ownership
   - workflow-side runtime execution ownership
   - `ace-assign` orchestration-policy ownership
2. The spike validates one workflow-bound tracer path using `skill://as-task-work` / `task/work.wf.md`.
3. The spike validates one direct-skill tracer path using a skill with no workflow binding and no `assign:`, composed only through fallback name/description matching.
4. The spike validates one explicit-workflow tracer path using `source: wfi://...` for an internal helper or explicitly authored assignment step, not as a public discovery source.
5. The spike identifies which fields currently living in public step YAML must be absorbed into canonical skill `assign.steps`.
6. The spike inventories current `skill: null` helper steps and classifies each as:
   - migrate to internal skill/workflow
   - merge into another concept
   - retain temporarily during migration
7. The spike identifies which current `ace-assign/.ace-defaults/assign/catalog/*` assets stay:
   - recipes
   - composition rules
   - any temporary transitional files
8. The spike confirms the earlier `8r0.t.ye8` cleanup direction is invalid and should not be implemented.
9. The spike defines how legacy `skill:` / `workflow:` step fields migrate to a unified `source:` field.

### Interface Contract

Spike outputs must include:

- A keep/remove matrix for:
  - public step-definition YAMLs
  - helper-step YAMLs
  - recipe YAMLs
  - composition-rules YAML
- A proposed step target contract based on:
  - `source: skill://<skill-name>`
  - `source: wfi://<workflow-name>` for internal helpers and explicitly authored steps only
- A proposed canonical skill assign schema covering at minimum:
  - `name`
  - `description`
  - `intent.phrases`
  - `prerequisites`
  - `produces`
  - `consumes`
  - `context`
  - `instructions` where helper steps need agent-facing instructions
  - `when_to_skip`
  - `effort`
  - `tags`
- A tracer scenario showing:
  - skill discovery from canonical skills
  - composition from skill metadata
  - workflow binding from `skill.execution.workflow`
  - direct skill execution from the skill body when no workflow binding exists
  - runtime subtree expansion from workflow `assign.sub-steps`
  - explicit `wfi://...` execution without treating workflows as a discoverable public inventory

Error Handling:

- If the tracer path reveals a field that cannot be cleanly owned by either skills, workflows, or policy files, the spike must call that out explicitly rather than silently inventing a fourth owner.
- If a helper step cannot reasonably become a skill/workflow, the spike must document why and propose a bounded exception instead of assuming YAML remains forever.
- If direct skills need richer composition metadata than name/description matching can provide, the spike must define the minimum explicit schema rather than relying on undocumented heuristics.
- If no current canonical skill cleanly fits the direct-skill tracer, the spike may use a synthetic contract-only tracer rather than forcing speculative implementation work.

Edge Cases:

- Internal helper steps may need `user-invocable: false`.
- Some helper steps carry inline `instructions` or nested `steps`; the spike must define where those move.
- Some skills may be directly executable but intentionally not composable; the spike must define whether the schema needs an explicit opt-out.
- The final contract must preserve deterministic composition and execution ordering.
- The final contract must identify each implementation consumer that must change: `ace-assign`, `ace-lint`, and provider projection sync.

### Success Criteria

- [ ] A workflow-bound tracer path is documented for `skill://as-task-work` -> `task/work.wf.md`
- [ ] A direct-skill tracer path is documented for a skill with no workflow binding
- [ ] An explicit-workflow tracer path is documented for `source: wfi://...`
- [ ] A final ownership table is produced for skill, workflow, and policy responsibilities
- [ ] A final step-target contract is produced for protocol-based `source:` URIs
- [ ] The required canonical skill schema delta is enumerated from current YAML fields
- [ ] Every current `skill: null` helper step is classified for migration or consolidation
- [ ] The spike states which catalog assets remain as `ace-assign` policy
- [ ] The spike documents why `8r0.t.ye8` is superseded
- [ ] The spike identifies the consumer updates required in `ace-assign`, `ace-lint`, and provider projection sync
- [ ] The spike leaves no open ownership decision for the follow-on migration subtasks
