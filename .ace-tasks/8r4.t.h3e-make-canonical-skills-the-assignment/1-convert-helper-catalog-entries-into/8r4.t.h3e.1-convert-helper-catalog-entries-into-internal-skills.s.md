# Convert Helper Catalog Entries Into Internal Skills and Workflows

## Objective

Convert helper and orchestration-oriented catalog entries away from permanent `skill: null` ownership into internal canonical skills and workflow-backed explicit steps, using the `8r4.t.h3e.0` spike artifacts as the authoritative migration inventory and exception list.

## Behavioral Specification

### User Experience

- **Input**: A maintainer composes or executes assignments that currently rely on helper catalog entries such as `task-load`, `mark-task-done`, `pre-commit-review`, `verify-test`, `update-docs`, or subtree orchestration helpers.
- **Process**: Helper entries are re-homed into internal non-user-invocable skills and workflows where that gives them canonical ownership, while orchestration-only or explicit-workflow cases remain executable through internal or authored `wfi://...` paths without becoming discoverable public steps.
- **Output**: Helper behavior stays available to `ace-assign`, but the long-term source of truth is no longer permanent `skill: null` YAML entries for cases that should live as canonical internal skills/workflows.

### Expected Behavior

1. Every helper entry classified by `8r4.t.h3e.0` is assigned one target state:
   - internal canonical skill with optional workflow binding
   - explicit internal `wfi://...` execution path
   - bounded transitional YAML exception documented in the spike artifacts
2. Internal helper skills remain non-user-invocable and are not added to the public discovery inventory.
3. Instruction-heavy helpers preserve their observable behavior, including dependency loading, status update verification, nested review/checkpoint steps, and subtree orchestration guidance.
4. Explicit `wfi://...` remains valid only for internal helpers and hand-authored assignment steps, never for public discovery.
5. Any helper that cannot yet move out of YAML remains documented as a temporary exception with an explicit reason and follow-up path.

### Interface Contract

Internal helper ownership contract:

```yaml
name: as-task-load-internal
description: Internal helper for loading task context into assignment execution
user-invocable: false
skill:
  kind: workflow
  execution:
    workflow: wfi://task/load-internal
assign:
  steps:
    - name: task-load
      description: Load task behavioral spec into assignment context
      prerequisites:
        - name: onboard-base
          strength: recommended
```

Explicit internal workflow step contract:

```yaml
steps:
  - name: split-subtree-root
    source: wfi://assign/split-subtree-root
```

Error Handling:

- If a helper cannot be expressed cleanly as a canonical internal skill or explicit internal workflow, this task must preserve the YAML entry only as a documented transitional exception.
- If a helper currently carries nested `steps:` or large `instructions:` blocks, the migrated ownership must preserve those user-visible execution expectations rather than silently dropping them.

Edge Cases:

- Helpers may need `user-invocable: false`.
- Helpers may be orchestration-only and intentionally not direct user-facing workflows.
- Transition exceptions must not be presented as permanent design choices.

### Success Criteria

- [ ] Every helper entry from the spike inventory is classified and migrated or explicitly deferred with a bounded exception
- [ ] Internal helper ownership no longer relies on permanent `skill: null` YAML for cases that have canonical internal skill/workflow homes
- [ ] Instruction-heavy helpers preserve their observable execution behavior
- [ ] Explicit `wfi://...` helper execution remains internal-only and non-discoverable
- [ ] The task records which helper YAML entries remain transitional and why

## Vertical Slice Decomposition (Task/Subtask Model)

Single migration subtask.

- **Slice type**: subtask
- **Slice outcome**: helper and internal execution paths have canonical ownership
- **Advisory size**: medium
- **Context dependencies**: `8r4.t.h3e.0` artifact set, helper catalog YAMLs, internal workflow conventions

## Verification Plan

### Unit/Component Validation

- Compare each helper YAML selected by the spike inventory against its target internal skill/workflow home
- Verify helper instruction fields, nested `steps:`, and context defaults remain represented in the target ownership model
- Verify transitional exceptions are explicitly enumerated rather than inferred

### Integration/E2E Validation

- Trace one migrated helper flow such as `task-load` or `mark-task-done` through its internal skill/workflow ownership path
- Trace one orchestration-only helper such as `split-subtree-root` through its explicit internal `wfi://...` execution path

### Failure/Invalid Path Validation

- Verify no migrated helper becomes part of the public discoverable step inventory
- Verify any helper left in YAML is marked as temporary with a migration reason
- Verify removal of helper YAML ownership does not erase nested review/checkpoint behavior

### Verification Commands

- `rg -n "^skill: null|^workflow: " ace-assign/.ace-defaults/assign/catalog/steps/*.yml`
- `rg -n "user-invocable: false|assign:" ace-*/handbook/skills/*/SKILL.md`
- `rg -n "split-subtree-root|task-load|mark-task-done|pre-commit-review" ace-assign/lib ace-assign/test ace-assign/handbook`

## Scope of Work

- Migrate helper catalog ownership into internal skills/workflows where the spike says they belong
- Preserve helper behavior that is currently encoded in YAML `instructions:` or nested `steps:`
- Document bounded transitional exceptions for helpers that cannot move yet

## Deliverables

### Behavioral Specifications

- Internal helper ownership contract
- Mapping of each helper inventory entry to its target home
- Internal-only `wfi://...` contract for explicit helper execution

### Validation Artifacts

- Helper migration inventory aligned with `8r4.t.h3e.0`
- Transitional exception list with reasons and follow-up path

## Out of Scope

- Public step metadata migration for skill-backed public steps
- Runtime parser/executor implementation details for the unified `source:` contract
- Validator, projection, and docs alignment outside helper ownership
