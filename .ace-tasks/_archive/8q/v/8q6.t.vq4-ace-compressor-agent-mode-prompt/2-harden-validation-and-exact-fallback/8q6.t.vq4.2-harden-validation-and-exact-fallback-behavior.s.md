# harden validation and exact fallback behavior

## Behavioral Specification

### User Experience
- **Input**: An `--mode agent` run where the provider is unavailable or the minified result does not meet fidelity gates.
- **Process**: The compressor detects the degraded condition, emits exact-mode output instead, adds explicit fallback markers, and explains that a degraded path was used.
- **Output**: A successful-but-degraded exact artifact that downstream tooling can detect without ambiguity.

### Expected Behavior
Fallback to exact is a supported degraded success path, not a silent substitution. The CLI must keep the run usable for automation while making it obvious that agent compression did not complete successfully. Validation should protect instructions, rules, numbers, commands, tables, and provenance from silent loss.

### Interface Contract
```bash
ace-compressor docs/decisions.md --mode agent --verbose
```

**Error Handling:**
- Provider-unavailable and validator-failed paths both produce exact output plus fallback metadata.
- The degraded path exits `0`, not non-zero.
- Hidden fallback is not allowed.

**Edge Cases:**
- Exact fallback still needs provenance and deterministic output.
- Fallback messaging must be visible in both human-readable and machine-readable paths.

### Success Criteria
- [x] Provider-unavailable agent runs degrade to exact with explicit `FALLBACK|...`.
- [x] Validator-failed agent runs degrade to exact with explicit `FALLBACK|...`.
- [x] Degraded runs exit `0`.
- [x] Fallback remains machine-detectable and user-visible.

### Validation Questions
- [x] Fallback to exact is a supported degraded success path.
- [x] Exit code remains `0`.
- [x] Hidden fallback is unacceptable.
- [x] Validator coverage must be broader than compact-mode refusal checks.
