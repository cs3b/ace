# Update task/work workflow to be plan-driven with fallback planning

## Behavioral Specification

### User Experience
- **Input**: Agent starts work on a task ref.
- **Process**: Workflow resolves task, ensures a fresh plan exists (generates when needed), then executes against plan checklist.
- **Output**: Predictable execution progress tied to plan items and task status updates.

### Expected Behavior
Work execution must be explicitly plan-first. If no plan is available or freshness checks fail, the workflow must invoke planning before implementation steps. Work still updates task state and checklist evidence, but cannot proceed with task-only interpretation.

### Interface Contract
```bash
# resolve task and path
mise exec -- ace-task show <ref> --path

# ensure plan exists/fresh (new CLI)
mise exec -- ace-task plan <ref>

# task state updates during work
mise exec -- ace-task update <ref> --set status=in-progress
mise exec -- ace-task update <ref> --set status=done
```

**Error Handling:**
- If plan cannot be resolved/generated, workflow stops and reports actionable error.
- If task is `draft`, workflow blocks and requires review promotion.

**Edge Cases:**
- Stale plan found after task/context change -> regenerate and continue.
- Orchestrator tasks with subtasks must run plan-first per subtask path.

### Success Criteria
- [ ] `work.wf.md` defines plan as mandatory input, not optional context.
- [ ] Workflow includes explicit fallback: run planning when plan absent/stale.
- [ ] Workflow keeps task status updates in work phase.
