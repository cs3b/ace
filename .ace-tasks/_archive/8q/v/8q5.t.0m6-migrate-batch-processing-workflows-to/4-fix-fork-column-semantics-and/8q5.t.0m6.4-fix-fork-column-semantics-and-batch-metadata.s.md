# Fix fork column semantics and batch metadata visibility

## Behavioral Context

**Issue**: Assignment status output used `FORK: yes` to mean "has children", which could mislead drivers into delegating non-fork batch parents.

**Key Behavioral Requirements**:
- `FORK` must indicate `context: fork` only.
- Batch containers without fork context must not appear as fork targets.
- Drive guidance should align with the corrected interpretation.

## Objective

Correct fork-signal semantics so delegation decisions are based on actual fork context rather than child presence.

## Scope of Work

- Updated status rendering to compute `FORK` from phase `context: fork`.
- Updated drive workflow examples/rules to use context-based fork detection.
- Added regression coverage for the corrected status-column behavior.

### Deliverables

#### Modify
- `ace-assign/lib/ace/assign/cli/commands/status.rb`
- `ace-assign/handbook/workflow-instructions/assign/drive.wf.md`
- `ace-assign/test/commands/status_command_test.rb`

## Implementation Summary

### What Was Done

- **Problem Identification**: Batch parent phases showed `FORK: yes` even when not fork-enabled.
- **Investigation**: Status table logic tied `FORK` to `children.any?` instead of phase context.
- **Solution**: Switched signal to `phase.fork?` and updated drive workflow logic/examples accordingly.
- **Validation**: Added assertions that parent-with-children/no-context does not show `FORK: yes`, while leaf fork phases do.
