# Generalize per-item fanout and retire e2e runner

## Behavioral Specification

### User Experience
- **Input**: Users run `/as-assign-run-in-batches` with one instruction template and an explicit item list instead of relying on a tool-specific batch workflow.
- **Process**: The workflow builds one parent assignment plus one child phase per item, forks children by default, and optionally continues directly into drive.
- **Output**: Users get one reusable repeated-item orchestration interface, and the old E2E-specific batch wrapper is no longer needed.

### Expected Behavior

This slice turns repeated-item orchestration into a reusable assign behavior:

1. `/as-assign-run-in-batches` accepts one instruction template and one explicit `--items` list.
2. The workflow creates a parent plus one child phase per item.
3. Child phases fork by default to support independent repeated work; `--sequential` disables that when ordering matters.
4. The instruction template substitutes `{{item}}`; if omitted, the workflow prepends a deterministic `Target item:` line.
5. E2E batch usage is migrated to this generic flow, allowing the bespoke `e2e/run-batch` and `as-e2e-run-batch` surfaces to be removed.

### Interface Contract

```bash
/as-assign-run-in-batches "Run E2E scenario {{item}}" --items TS-001,TS-002 --run
/as-assign-run-in-batches "Update docs for {{item}}" --items ace-assign,ace-task --sequential
```

**Error Handling:**
- Missing `--items` value: refuse creation with an actionable message.
- Empty or duplicate item list after normalization: refuse creation instead of generating degenerate phases.

**Edge Cases:**
- If the template omits `{{item}}`, generated instructions still identify the target item explicitly.
- `--sequential` disables fork context on item children.
- A single-item list still creates a valid assignment; fan-out becomes a one-child parent/child tree.
