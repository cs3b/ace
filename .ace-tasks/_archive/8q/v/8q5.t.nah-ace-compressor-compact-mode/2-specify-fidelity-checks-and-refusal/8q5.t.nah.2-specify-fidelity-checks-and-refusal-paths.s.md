# Compress tables and examples with loss metadata

## Behavioral Specification

### User Experience
- **Input**: Table-heavy and example-heavy docs passed to compact mode.
- **Process**: The CLI chooses an explicit table strategy, shrinks rows/examples/code references where safe, preserves sensitive structures exactly when needed, and emits explicit metadata describing what was lost or retained.
- **Output**: Compact output with visible `TABLE|...`, `LOSS|...`, `EXAMPLE_REF|...`, and `FIDELITY|...` style records.

### Expected Behavior
This slice turns compact mode into something trustworthy for structured docs. Tables may be `preserve`, `schema_plus_key_rows`, or `summarize_with_loss`; examples may collapse to refs with provenance; code blocks may compact to essential lines or refs; rule-bearing structured sections may stay exact. Every reduction must be explicitly marked.

### Interface Contract
```bash
ace-compressor compress docs/ace-gems.g.md --mode compact --verbose
ace-compressor compress README.md docs/vision.md --mode compact
ace-compressor compress docs/blueprint.md --mode compact --verbose
```

**Error Handling:**
- Table too sensitive to summarize safely: preserve more structure or refuse.
- Missing loss metadata for a summarized table/example is a test failure.

**Edge Cases:**
- Repeated examples across sources collapse to one ref with provenance.
- Header rows are not counted as retained data rows in loss metadata.
- Mimicry-required examples stay explicit instead of collapsing to refs.
- Rule-bearing structured sections may emit exact-preserve records alongside compact narrative/table output.

### Success Criteria
- [x] Tables compact with explicit `LOSS|...` metadata when rows are elided.
- [x] Repeated examples compact to refs.

- [x] Sensitive structured content stays explicit or refuses.
