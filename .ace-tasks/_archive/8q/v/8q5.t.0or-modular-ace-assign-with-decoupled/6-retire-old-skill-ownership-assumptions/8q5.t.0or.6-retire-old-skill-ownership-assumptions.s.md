# Retire old skill ownership assumptions

## Behavioral Specification

### User Experience
- **Input**: Users continue to discover skills and compose assignments through the already-migrated public surfaces.
- **Process**: Once typed canonical skills, generated projections, canonical discovery, and selective assignment integration are all in place, ACE removes the old ownership assumptions from code, docs, and compatibility bridges.
- **Output**: Public behavior stays stable while the repo no longer implies that provider skill trees or phase YAML are the canonical model.

### Expected Behavior

This cleanup slice removes the old model completely:
1. `.agent/skills` and other provider trees are documented and implemented as generated projections, not canonical sources.
2. Hardcoded provider-skill scanning in `ace-assign` is removed in favor of canonical discovery.
3. Legacy phase YAML is no longer treated as the authoritative source of assignment phase definitions.
4. Temporary migration bridges are removed once repo-wide migrated coverage is sufficient.
5. Docs and guidance consistently describe package-owned canonical skills plus workflow execution bodies as the end-state.

### Interface Contract

```bash
# Public behavior remains stable
ace-nav skill://*
ace-assign catalog
/as-assign-create plan-task --taskref 123
```

```text
# Final contract
- handbook/skills is canonical
- provider trees are generated projections
- workflows remain execution bodies
- phase YAML is not the long-term assign authority
```

**Error Handling:**
- Cleanup must not remove compatibility bridges until migrated canonical coverage keeps public behavior stable.
- Residual docs or code paths that still claim `.agent/skills` is canonical are treated as cleanup debt to remove before closure.
- If final cleanup exposes a missing migrated capability, the missing capability blocks cleanup instead of silently restoring the old model.

**Edge Cases:**
- Provider-facing skill trees remain required outputs after cleanup.
- Recipes and composition rules remain part of the system after cleanup.
- Legacy phase YAML may remain as historical artifacts, but not as the authoritative composition source.
