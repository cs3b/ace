# Minimal runnable narrow-review engine

## Behavioral Specification

### User Experience
- **Input**: A maintainer defines reviewers, providers, preset-qualified model targets, and a pipeline instead of embedding mixed review intent directly inside one large preset.
- **Process**: `ace-review` resolves one narrow preset through the new reviewer/provider/pipeline path and executes reviewers independently.
- **Output**: One runnable narrow review flow produces ordinary feedback items with provenance, proving the new engine can replace preset-heavy source logic.

### Expected Behavior

This slice must create the first working end-to-end path through the new architecture. It is not enough to add config files or internal models in isolation. At the end of the slice, one narrow preset must resolve through reviewer definitions, provider definitions, preset-qualified model targets, and one pipeline definition, then emit findings through the existing feedback-item workflow.

The slice must include at least one evidence lane from the start so the engine proves it can merge more than plain LLM outputs. The evidence lane can be deterministic tool output or imported human review evidence, but it must go through the same normalized finding path.

### Interface Contract

```bash
# New config surfaces
.ace/review/reviewers/<name>.yml
.ace/review/providers/<name>.yml
.ace/review/pipelines/<name>.yml
.ace/llm/presets/<name>.yml

# One runnable narrow preset
ace-review --preset pr-risk-based --auto-execute

# Provider targets may use generic ace-llm execution presets
model: "gflash@review-fast"
model: "claude:sonnet@review-deep"
```

**Error Handling:**
- Missing reviewer reference: fail with a message naming the missing reviewer.
- Missing provider reference: fail with a message naming the missing provider.
- No optional lanes selected by pipeline logic: fall back to the safe minimal reviewer set.

**Edge Cases:**
- Reviewer metadata must affect runtime behavior, not remain passive metadata.
- The new engine must preserve feedback items as the primary review artifact.

### Success Criteria

- [x] Reviewer, provider, and pipeline definitions exist as first-class config surfaces.
- [x] Provider definitions can target preset-qualified `ace-llm` models without ad hoc runtime fields.
- [x] One narrow preset executes end-to-end through the new engine.
- [x] One evidence lane participates in the same finding path.
- [x] Findings normalize into feedback items without introducing a second output system.

### Validation Questions

- [x] **Minimal Preset**: The first runnable preset is PR-focused only; broader subject-agnostic coverage is deferred.
- [x] **Evidence Lane Choice**: `lint` is the first evidence lane because it proves normalized non-LLM merging without adding PR-ingestion scope.
- [x] **Always-On Set**: The always-on reviewer set for this slice is `correctness + contracts`.

## Objective

Build the minimum viable narrow-review engine that proves reviewer decomposition, provider reuse, pipeline composition, and feedback-item continuity in one runnable path.

## Scope of Work

- **User Experience Scope**: One new narrow preset can run through the new architecture.
- **System Behavior Scope**: Reviewer/provider/pipeline resolution, normalized findings, and one evidence lane.
- **Interface Scope**: New review config surfaces and one executable preset path.

### Resolved Defaults

- The first runnable narrow preset is PR-focused and diff-based.
- The minimum runnable reviewer set is:
  - `correctness`
  - `contracts`
- The first evidence lane is:
  - `lint`
- The first engine slice must prove:
  - reviewer resolution
  - provider resolution
  - preset-qualified model target passthrough
  - pipeline resolution
  - finding normalization
  - merge of LLM and deterministic evidence into feedback items
- Deferred from this slice:
  - PR comment ingestion as human evidence
  - subject-agnostic preset support
  - skeptic/referee stages
  - phased preset cutover

### Deliverables

#### Behavioral Specifications
- Reviewer/provider/pipeline configuration contracts
- One runnable narrow preset behavior contract

#### Validation Artifacts
- Resolver verification scenarios
- Feedback continuity checks
