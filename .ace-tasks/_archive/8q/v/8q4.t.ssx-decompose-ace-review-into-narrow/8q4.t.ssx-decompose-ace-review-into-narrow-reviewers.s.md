# Decompose ace-review into narrow reviewers

## Behavioral Specification

### User Experience
- **Input**: Maintainers define narrow review perspectives, provider configs, and orchestration rules instead of encoding broad mixed review logic in large presets.
- **Process**: A review run selects the right narrow reviewers for the change surface, executes them independently, optionally challenges disputed findings, and produces actionable feedback with provenance.
- **Output**: Users receive feedback items that are easier to trust, narrower in scope, cheaper to run when risk is low, and composable into broader review modes when needed.

### Expected Behavior

`ace-review` must shift from preset-centric mixed review definitions to a reviewer-centric architecture where narrow reviewers are the canonical source of review intent. The system must support reusable reviewer definitions, reusable provider definitions, preset-qualified model targets resolved by `ace-llm`, and orchestration pipelines that compose them into review runs.

The first milestone is not to preserve any specific old preset name. The first milestone is to preserve the value that old mixed reviews provided by decomposing that value into narrow, reusable reviewers. Broad mixed presets such as `code-deep` may be recreated later from those narrow reviewers, but broad presets must no longer be the source of truth.

`ace-assign` must become the first consumer of this architecture. Instead of always running a fixed sequence of broad review cycles, it must be able to select review lanes based on risk and changed surface so simple tasks get narrower reviews and risky tasks get broader coverage.

### Interface Contract

```bash
# Reviewer definitions become a first-class config surface
.ace/review/reviewers/<name>.yml

# Provider definitions become a first-class config surface
.ace/review/providers/<name>.yml

# Pipelines orchestrate narrow reviewers
.ace/review/pipelines/<name>.yml

# Presets stay user-facing but become thin compositions
.ace/review/presets/<name>.yml

# Example future usage
ace-review --preset pr-risk-based --auto-execute
ace-review --pipeline narrow-risk-based --reviewer correctness --reviewer tests --auto-execute
ace-llm query claude:sonnet@review-deep "Summarize this diff"
```

**Error Handling:**
- Missing reviewer definition: review setup fails with an actionable message naming the missing reviewer reference.
- Missing provider definition: review setup fails with an actionable message naming the missing provider reference.
- No matching risk-based lanes: system falls back to a minimal safe default lane set rather than silently doing nothing.

**Edge Cases:**
- Broad preset reconstruction: recreated wide presets must behave as compositions over narrow reviewers, not bespoke prompt bundles.
- Mixed reviewer kinds: LLM, tool, and human evidence sources must normalize into one merge path without losing provenance.

## Objective

Turn the current ace-review redesign research into a decision-complete task that replaces preset-heavy mixed review definitions with narrow reusable reviewers, reusable providers, and orchestration pipelines, while making risk-based lane selection in `ace-assign` the first concrete consumer.

## Scope of Work

- **User Experience Scope**: Review authors define and run narrower, more explainable review perspectives without losing the ability to assemble broader mixed reviews later.
- **System Behavior Scope**: The review system resolves reviewers, providers, and pipelines as separate concepts and supports risk-based lane selection for assignment-driven review workflows.
- **Interface Scope**: Review config surfaces, review preset semantics, future review CLI composition, and assignment review-policy behavior.

## Resolved Defaults

### v1 review defaults

- Always-on lanes:
  - `correctness`
  - `contracts`
- Human evidence in scope:
  - PR review comments
  - PR review threads
- Deferred from v1:
  - saved offline review-session imports
  - broad mixed preset reconstruction such as `code-deep`
  - skeptic/referee trigger policy as part of the initial cutover

### Deterministic v1 lane-selection matrix

- Add `tests` when:
  - Ruby source changes
  - public behavior changes
  - bug-fix logic changes
  - CLI/config/workflow branching changes
- Add `security` when:
  - auth, tokens, secrets, shell execution, filesystem writes, network/API/LLM/provider boundaries, GitHub boundaries, or untrusted input handling change
- Add `architecture-fit` when:
  - new modules, resolvers, workflows, gems, or config layers are introduced
- Add `simplicity` when:
  - the change adds multiple abstractions, new indirection layers, or architecture-oriented refactors
- Add `performance` when:
  - loops, batch processing, parsing, filesystem walking, network calls, or synthesis hot paths change
- Add `docs-dx` when:
  - CLI flags, user-visible workflows, presets, handbook usage, README docs, or public output formats change
- Add `lint` when:
  - touched file types match supported lint surfaces
