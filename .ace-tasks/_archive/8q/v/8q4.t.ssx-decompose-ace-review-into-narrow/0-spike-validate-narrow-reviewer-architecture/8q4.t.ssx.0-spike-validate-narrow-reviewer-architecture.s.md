# Spike validate narrow reviewer architecture

## Behavioral Specification

### User Experience
- **Input**: A maintainer selects one representative review scenario from the current ace-review/ace-assign workflow.
- **Process**: The spike traces how that scenario would work in the narrow-reviewer architecture, identifies which existing concepts survive, and records the minimum viable reviewer/provider/pipeline model.
- **Output**: A validated concept inventory and one end-to-end scenario proving the target design is coherent before more subtasks are drafted.

### Expected Behavior

This spike must validate the redesign with one concrete end-to-end scenario, not a broad speculative architecture. The scenario must show:

- a review preset resolving into a pipeline
- the pipeline selecting narrow reviewers
- at least one deterministic or human evidence lane
- findings converging into feedback items with provenance
- risk-based lane selection rules that `ace-assign` could use

The spike must explicitly classify existing concepts as `KEPT`, `REMOVE`, or `DEFER`. If the spike cannot explain one coherent end-state with fewer concepts than the current preset-heavy model, no additional implementation subtasks should be drafted.

### Interface Contract

```bash
# Example spike scenario
ace-review --preset pr-risk-based --auto-execute

# Example assignment-driven scenario
ace-assign ... -> review policy selects:
  correctness
  contracts
  tests
  simplicity
```

**Error Handling:**
- If the spike cannot demonstrate a coherent minimal reviewer/provider/pipeline split, the output must say which concept boundary failed and why.
- If risk-based selection is under-specified, the spike must enumerate the missing signals rather than hand-waving them.

**Edge Cases:**
- Wide presets remain reconstructable even if not implemented in the spike.
- Mixed evidence from tool and human sources must fit the same finding contract.

### Success Criteria

- [x] One end-to-end scenario is fully described using narrow reviewers, providers, and a pipeline.
- [x] The spike produces a concept inventory with `KEPT`, `REMOVE`, and `DEFER` labels.
- [x] The spike identifies the minimal always-on reviewer set.
- [x] The spike defines when `ace-assign` adds optional lanes such as security, performance, and docs-dx.

### Validation Questions

- [x] **Always-On Lanes**: The safe default is `correctness + contracts`; `tests` remains optional and signal-driven.
- [x] **Evidence Merge**: One normalized finding contract is sufficient for LLM, tool, and PR-comment evidence in v1.
- [x] **Preset UX**: The first reconstructed user-facing surface is `code-valid`, `code-fit`, and `code-shine`; `code-deep` remains deferred.
- [x] **Challenge Trigger**: Skeptic/referee triggering is deferred from the first implementation and is not required for this spike to validate the core concept set.

## Objective

Validate the target architecture end-to-end with a single scenario so the final implementation can be decomposed from proven concepts instead of speculative phase slicing.

## Scope of Work

- **User Experience Scope**: One coherent future review flow that is narrower and more explainable than the current broad mixed-review preset model.
- **System Behavior Scope**: Reviewer decomposition, pipeline execution, risk-based lane selection, and finding normalization.
- **Interface Scope**: Reviewer/provider/pipeline config surfaces and the assignment review-policy behavior they imply.

### Resolved Defaults

- The spike scenario must assume `correctness + contracts` as the always-on lane set.
- Human evidence in v1 means PR comments and review threads only.
- The normalized finding contract must support LLM, tool, and PR-comment evidence without separate merge paths.
- The spike does not need to validate skeptic/referee behavior; it only needs to confirm those stages can stay deferred without blocking the core engine design.

### Deliverables

#### Behavioral Specifications
- One scenario narrative covering reviewer resolution, pipeline flow, and output behavior
  (`docs/scenario-validation.md`)
- One concept inventory showing what survives the redesign (`docs/concept-inventory.md`)

#### Validation Artifacts
- Explicit keep/remove/defer table (`docs/concept-inventory.md`)
- Lane selection rule summary for `ace-assign` (`docs/scenario-validation.md`)

## Out of Scope

- ❌ Drafting the full implementation breakdown
- ❌ Creating additional subtasks before the spike validates the concept set
- ❌ Runtime code changes
