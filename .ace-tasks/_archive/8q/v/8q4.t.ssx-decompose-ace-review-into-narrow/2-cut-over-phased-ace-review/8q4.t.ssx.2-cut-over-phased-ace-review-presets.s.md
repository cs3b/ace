# Cut over phased ace-review presets

## Behavioral Specification

### User Experience
- **Input**: Users continue invoking the phased review presets they already trust.
- **Process**: Those phased presets resolve through the new reviewer/provider/pipeline engine instead of carrying review intent as duplicated preset-specific boilerplate.
- **Output**: Users keep the familiar phased review surface while the architecture underneath now uses narrow reviewers as the source of truth.

### Expected Behavior

This slice moves the current useful ace-review surface onto the new engine. `code-valid`, `code-fit`, and `code-shine` must remain usable and semantically recognizable, but their logic must now come from composed reviewer definitions and pipeline rules rather than large preset-local instruction bundles.

This slice is the actual ace-review cutover. After it lands, the old preset-heavy architecture should no longer remain as a supported runtime path. Temporary migration glue inside the branch is acceptable, but the delivered system should have one runtime path.

### Interface Contract

```bash
# Existing phased review surface remains user-facing
ace-review --preset code-valid --auto-execute
ace-review --preset code-fit --auto-execute
ace-review --preset code-shine --auto-execute
```

**Error Handling:**
- If a phased preset cannot be expressed through the new engine, the failure must identify which reviewer or pipeline capability is missing.
- If a preset resolves but produces no reviewer set, fail explicitly rather than silently skipping the review.

**Edge Cases:**
- Feedback item semantics must remain stable enough for the existing feedback workflow.
- Broad mixed preset reconstruction remains optional and can stay deferred.

### Success Criteria

- [ ] `code-valid`, `code-fit`, and `code-shine` run through the new engine.
- [ ] Narrow reviewers become the source of truth for phased review behavior.
- [ ] Feedback items remain the primary review artifact.
- [ ] No long-lived dual engine remains after the cutover.

### Validation Questions

- [x] **Compatibility Surface**: Preserve behavioral focus and output shape, not exact prompt wording.
- [x] **Preset Coverage**: Only `code-valid`, `code-fit`, and `code-shine` are required in this cutover slice.
- [x] **Feedback Stability**: Core feedback lifecycle and actionable fields remain stable; provenance-oriented metadata may expand.

## Objective

Replace the preset-heavy ace-review runtime with the new narrow-review engine while preserving the current phased user-facing review experience.

## Scope of Work

- **User Experience Scope**: Preserve the familiar phased review commands.
- **System Behavior Scope**: Route the phased review surface through reviewer/provider/pipeline composition and normalized findings.
- **Interface Scope**: Phased preset semantics and feedback workflow compatibility.

### Resolved Defaults

- Required phased presets in this slice:
  - `code-valid`
  - `code-fit`
  - `code-shine`
- Not required in this slice:
  - `code-deep`
  - any experimental or example presets beyond the phased trio
- Compatibility contract for phased presets:
  - preserve review focus
  - preserve user-facing preset names
  - preserve actionable feedback-item output shape
  - do not preserve exact legacy prompt wording
- Allowed feedback metadata expansion in this slice:
  - reviewer provenance
  - provider provenance
  - confidence
  - stage trace
  - adjudication-ready metadata fields that remain unused until later slices
- Disallowed breakage in this slice:
  - replacing feedback items with a new primary artifact
  - introducing a second supported legacy runtime path after cutover

### Deliverables

#### Behavioral Specifications
- Behavior contract for each phased preset on the new engine
- Cutover rules for phasing out the old preset-heavy runtime path

#### Validation Artifacts
- Phased preset compatibility checks
- Feedback output continuity checks

## Out of Scope

- ❌ `ace-assign` risk-based selection
- ❌ Skeptic/referee meta-review
- ❌ Broad mixed preset reconstruction unless compatibility proves it necessary
