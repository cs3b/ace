# Add risk-based review lane selection

## Behavioral Specification

### User Experience
- **Input**: A maintainer runs an assignment workflow for a task or task batch.
- **Process**: The assignment workflow selects review lanes from task risk and changed-surface signals instead of assuming a fixed sequence of review phases for every task.
- **Output**: Simpler tasks get narrower review coverage; riskier tasks get broader reviewer coverage; the chosen lanes and rationale are visible.

### Expected Behavior

`ace-assign` must become the first consumer of the new reviewer architecture. Instead of always instructing follow-up review work as a hardcoded `valid -> fit -> shine` sequence, assignment presets must be able to derive the right lane set from changed-surface and task-risk signals.

This selection must be deterministic enough to reason about in specs and tests. When no optional lanes are triggered, the workflow must still run the safe minimal set rather than silently skipping review.

### Interface Contract

```yaml
review_policy:
  mode: risk-based
  always:
    - correctness
    - contracts
  optional:
    - tests
    - security
    - simplicity
    - architecture-fit
    - performance
    - docs-dx
    - lint
```

```bash
# Assignment workflow behavior
ace-assign ...  # review policy selects a deterministic reviewer lane set
```

**Error Handling:**
- Invalid reviewer name in review policy: fail with a message naming the invalid lane.
- No optional lanes selected: run the always-on minimal set and record that no optional lanes matched.

**Edge Cases:**
- Batch tasks may expand the reviewer set when any member task triggers a higher-risk lane.
- Documentation-only tasks may skip code-focused optional lanes while preserving the minimal safe set if review remains required.

### Success Criteria

- [ ] Assignment presets can express risk-based review policy.
- [ ] Reviewer lane selection is deterministic and explainable.
- [ ] The workflow still has a safe default when no optional lanes are selected.
- [ ] `ace-assign` is the first real consumer of the new reviewer library.

### Validation Questions

- [x] **Signal Source**: v1 uses the deterministic rule matrix from the orchestrator defaults; advisory heuristics are out of scope.

## Objective

Make assignment workflows choose narrow review lanes from risk and change signals so the new reviewer architecture delivers immediate workflow value instead of remaining only a lower-level ace-review refactor.

## Scope of Work

- **User Experience Scope**: Assignment presets can choose review lanes instead of hardcoded review phases.
- **System Behavior Scope**: Deterministic lane selection and explicit reporting of chosen reviewers.
- **Interface Scope**: Assignment preset `review_policy` behavior and review-lane selection semantics.

### Resolved Defaults

- Always-on lanes in v1:
  - `correctness`
  - `contracts`
- Deterministic optional-lane triggers in v1:
  - `tests`
  - `security`
  - `simplicity`
  - `architecture-fit`
  - `performance`
  - `docs-dx`
  - `lint`
- Rule source for those triggers:
  - the explicit changed-surface matrix defined in `8q4.t.ssx`
- Batch policy in v1:
  - union all triggered lanes across the batch
  - do not partition one batch into multiple reviewer plans
- Docs-only policy in v1:
  - skip code-focused optional lanes unless separately triggered by public workflow or config changes
  - retain the always-on lane set
- Out of scope for this slice:
  - advisory-only heuristics
  - dynamic per-task partitioning inside one batch assignment
  - full review fast-exit that bypasses the always-on lane set

### Deliverables

#### Behavioral Specifications
- Risk-based review policy contract for assignment presets
- Lane-selection behavior contract for single-task and batch workflows

#### Validation Artifacts
- Reviewer-selection scenario matrix
- Safe-default and invalid-policy checks

## Out of Scope

- ❌ Additional ace-review meta-review stages
- ❌ Broad preset reconstruction
- ❌ Automatic partitioning of one batch into multiple independent reviewer plans unless later required
