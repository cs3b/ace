# Align Validators, Projections, and Docs With the Skill-First Contract

## Review Questions (Resolved During Review)

### [HIGH] What is the final validator matrix for skill kinds and workflow bindings?

- [x] Which skill kinds require `skill.execution.workflow`, and when is `assign:` allowed?
- **Repository evidence**: Current lint tests require `skill.execution.workflow` for all skills and reject `assign:` on capability skills.
- **Decision**:
  - `workflow` and `orchestration` skills require `skill.execution.workflow`
  - `capability` skills may omit `skill.execution.workflow`
  - `capability` skills may still optionally declare `skill.execution.workflow`
  - `assign:` remains valid only for `workflow` and `orchestration` skills
- **Why resolved without human input**: This matches the target contract already implied by the parent task, tracer scenarios, and current capability-with-workflow examples.

## Objective

Align schema validation, provider projection behavior, and user-facing documentation with the final skill-first assignment contract after the public metadata, helper migration, and runtime `source:` changes are in place.

## Behavioral Specification

### User Experience

- **Input**: A maintainer edits canonical skills, runs lint/schema validation, syncs provider projections, or reads usage docs for the assignment discovery model.
- **Process**: Validation rules accept the final canonical contract, provider sync/status continues to project the canonical skills deterministically, and usage/docs describe skill-first discovery plus internal-only `wfi://...` execution accurately.
- **Output**: Canonical skills lint cleanly, projections stay in sync, and documentation matches the shipped contract instead of the retired duplicated model.

### Expected Behavior

1. `ace-lint` accepts workflowless direct skills where the final contract allows them.
2. `ace-lint` continues to reject invalid `assign:` usage outside the allowed skill kinds.
3. `ace-lint` requires `skill.execution.workflow` for `workflow` and `orchestration` skills, but not for `capability` skills.
4. Provider sync/status behavior continues to project canonical skills without changing their underlying assignment contract.
5. Assignment docs and usage scenarios describe:
   - skill-first public discovery
   - fallback-only direct skill execution
   - internal-only explicit `wfi://...` execution
6. Obsolete references to duplicated public step ownership or public workflow discovery are removed from relevant docs/tests.

### Interface Contract

Validator expectations after alignment:

```yaml
skill:
  kind: capability
```

Valid when:
- the skill is directly executable
- the skill has no `assign:` block
- runtime fallback discovery rules are documented elsewhere

Validator matrix:

```yaml
skill:
  kind: workflow | orchestration
  execution:
    workflow: wfi://...
assign:
  steps: [...]
```

- `workflow` and `orchestration` require `skill.execution.workflow`
- `assign:` is allowed only on `workflow` and `orchestration`
- `capability` may omit `skill.execution.workflow`
- `capability` may still declare `skill.execution.workflow` when direct capability execution intentionally binds to a workflow body

Projection/documentation expectations:

```yaml
assign:
  steps:
    - name: work-on-task
```

- remains canonical source metadata
- is projected consistently to provider-native trees
- is documented as canonical rather than provider-owned

Error Handling:

- If validation rules and projection assumptions disagree about the canonical schema, this task must resolve that mismatch explicitly.
- If docs still describe public `wfi://...` discovery or duplicated YAML ownership, they must be updated or called out as stale before completion.

Edge Cases:

- Capability skills may still have workflow bindings; this task should not conflate “capability” with “workflowless.”
- Provider projections may add provider-specific frontmatter, but must not change canonical assignment ownership semantics.
