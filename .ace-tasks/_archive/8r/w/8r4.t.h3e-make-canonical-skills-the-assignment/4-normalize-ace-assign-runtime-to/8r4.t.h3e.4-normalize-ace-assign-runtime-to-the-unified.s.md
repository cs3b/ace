# Normalize ace-assign Runtime to the Unified Source Contract

## Review Questions (Resolved During Review)

### [HIGH] Where should invalid public `wfi://...` usage be rejected?

- [x] Should public `wfi://...` misuse fail at runtime only, or earlier?
- **Decision**: Public canonical discovery must reject `wfi://...` during lint/compose/catalog-generation time. Runtime still supports explicit `wfi://...` execution for internal helpers and hand-authored assignments.
- **Why resolved without human input**: This preserves the parent task’s internal-only rule and prevents workflows from becoming a second discoverable public inventory.

## Objective

Normalize `ace-assign` parsing, resolution, rendering, and runtime expansion around a single `source:` contract while preserving compatibility for legacy `skill:` and `workflow:` fields during the migration window.

## Behavioral Specification

### User Experience

- **Input**: A maintainer composes or executes assignment steps that may currently reference legacy `skill:` or `workflow:` fields, canonical `skill://...` sources, or explicit internal `wfi://...` sources.
- **Process**: `ace-assign` normalizes step definitions to `source:`, resolves workflow-bound skills through `skill.execution.workflow`, executes fallback direct skills through the skill body when no workflow binding exists, and continues to use workflow frontmatter for runtime subtree expansion.
- **Output**: Assignment runtime follows one explicit source contract and no longer depends on duplicated long-term metadata ownership.

### Expected Behavior

1. Assignment steps use `source:` as the canonical runtime contract.
2. `source: skill://...` resolves:
   - workflow-bound skills through `skill.execution.workflow`
   - fallback-only direct skills through the skill body when no workflow binding exists
3. `source: wfi://...` remains valid only for internal helpers and explicitly authored steps.
   Public canonical discovery must reject `wfi://...` before runtime; runtime support remains for explicit authored/internal paths.
4. Legacy `skill:` and `workflow:` fields are normalized into `source:` compatibility behavior during migration.
5. Runtime subtree expansion continues to read workflow `assign:` frontmatter and not public skill metadata.
6. Public step rendering and composed assignment behavior remain stable while ownership moves underneath.

### Interface Contract

Canonical step contract:

```yaml
steps:
  - name: work-on-task
    source: skill://as-task-work
  - name: task-load
    source: wfi://task/load-internal
```

Compatibility normalization contract:

```yaml
steps:
  - name: work-on-task
    skill: as-task-work
```

Normalizes to:

```yaml
steps:
  - name: work-on-task
    source: skill://as-task-work
```

Error Handling:

- Unsupported `source:` protocols fail clearly.
- Steps that cannot be normalized from legacy fields fail clearly instead of silently dropping execution data.
- Explicit `wfi://...` sources used in discoverable public metadata are rejected or flagged according to the final contract.
- Rejection of discoverable public `wfi://...` must happen in lint/compose/catalog-generation paths, not only during runtime execution.

Edge Cases:

- Direct skills may have no workflow binding and no `assign:`.
- Capability skills may still carry workflow bindings.
- Runtime rendering must preserve helper/internal behavior that relies on explicit workflow rendering.

### Success Criteria

- [x] `source:` becomes the canonical runtime step contract
- [x] Workflow-bound `skill://...` execution resolves through `skill.execution.workflow`
- [x] Fallback direct skill execution works without fake workflow duplication
- [x] Explicit internal `wfi://...` execution works without becoming public discovery
- [x] Legacy `skill:` and `workflow:` compatibility normalization is explicit and temporary
- [x] Runtime no longer depends on duplicated public metadata ownership
