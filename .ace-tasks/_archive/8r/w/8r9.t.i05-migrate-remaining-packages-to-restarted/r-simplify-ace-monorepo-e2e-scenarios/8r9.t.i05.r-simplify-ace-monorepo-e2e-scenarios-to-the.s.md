# Simplify ace-monorepo-e2e to the fast/feat/e2e testing model

## Objective

Rebaseline ace-monorepo-e2e on the new public testing taxonomy by running the canonical E2E lifecycle first, then simplifying or retaining scenario coverage based on the approved review plan instead of assuming the package should remain scenario-only.

## User Experience

- `as-e2e-review` on `ace-monorepo-e2e` starts from the current suite of 2 scenarios and 8 test cases and produces a coverage matrix before any rewrite decisions are made.
- Operators continue to use `ace-test-e2e ace-monorepo-e2e` unless the approved rewrite promotes deterministic coverage out of the current scenarios.
- Reviewers can see whether any workflow assertions should move to `test/fast/` or `test/feat` instead of staying in scenario coverage.

## Expected Behavior

1. Run `as-e2e-review` via `ace-bundle wfi://e2e/review` for `ace-monorepo-e2e` and capture a coverage matrix for the existing suite.
2. Run `as-e2e-plan-changes` via `ace-bundle wfi://e2e/plan-changes` and classify every current scenario and TC as `KEEP`, `MODIFY`, `REMOVE`, `CONSOLIDATE`, or `ADD`.
3. Run `as-e2e-rewrite` via `ace-bundle wfi://e2e/rewrite` and apply the approved scenario changes.
4. Keep the package scenario-only only if the review explicitly concludes that no deterministic coverage should move out of E2E; otherwise promote review-identified deterministic behavior into `test/fast/` or `test/feat`.
5. Update docs, workflow text, and task references so the public contract is expressed in terms of `fast`, `feat`, and `e2e`, with no stale `ace-test <pkg> e2e` or `test-e2e/scenarios` guidance.
6. Do not create placeholder `test/fast/` or `test/feat/` directories unless the approved rewrite introduces real deterministic coverage.

## Package Migration Shape

- The current suite starts at 2 scenarios and 8 test cases and must be reviewed before deciding what survives.
- Default expectation is scenario simplification or consolidation where the review identifies low-value overlap.
- `test/e2e/` remains only for retained scenarios that still justify real workflow cost.
- Deterministic layers are introduced only when the review and change plan justify them.

## Interface Contract

- `ace-test-e2e ace-monorepo-e2e`
- `ace-test ace-monorepo-e2e` when the approved rewrite introduces explicit deterministic coverage
- `ace-test ace-monorepo-e2e feat` when the approved rewrite introduces explicit `test/feat/` coverage

## Validation Questions

- None. The package starts scenario-only, so the remaining decisions are the review-driven `KEEP` / `MODIFY` / `REMOVE` / `CONSOLIDATE` / `ADD` outcomes and whether any explicit deterministic layer is justified.

## Vertical Slice Decomposition

- This task is one standalone scenario-focused migration slice for `ace-monorepo-e2e`. It ships independently once the package completes `review -> plan-changes -> rewrite` and the resulting public contract is explicit.
