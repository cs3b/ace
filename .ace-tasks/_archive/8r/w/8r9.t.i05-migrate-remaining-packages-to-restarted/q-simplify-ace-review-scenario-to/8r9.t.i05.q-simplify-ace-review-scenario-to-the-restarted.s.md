# Migrate ace-review to the fast/feat/e2e testing model

## Objective

Move ace-review into the new `fast`/`feat`/`e2e` model by running the canonical E2E lifecycle first, then applying the approved rewrite so low-value scenario coverage moves into deterministic layers and legacy `test/integration/` coverage lands in `test/feat/`.

## User Experience

- `as-e2e-review` on `ace-review` starts from the current suite of 1 scenario and 7 test cases and produces a coverage matrix before any rewrite decisions are made.
- `ace-test ace-review` becomes the default fast feedback loop for deterministic package tests.
- `ace-test ace-review feat` holds deterministic feature coverage promoted out of legacy `test/integration/` or removed from low-value E2E overlap.
- `ace-test-e2e ace-review` remains only for retained real workflow scenarios.

## Expected Behavior

1. Run `as-e2e-review` via `ace-bundle wfi://e2e/review` for `ace-review` and capture a coverage matrix for the existing suite.
2. Run `as-e2e-plan-changes` via `ace-bundle wfi://e2e/plan-changes` and classify every current scenario and TC as `KEEP`, `MODIFY`, `REMOVE`, `CONSOLIDATE`, or `ADD`.
3. Run `as-e2e-rewrite` via `ace-bundle wfi://e2e/rewrite` and apply the approved scenario changes.
4. Move legacy `test/integration/` coverage into `test/feat/`, and move any deterministic behavior that fails the E2E value gate into `test/fast/` or `test/feat`.
5. Keep `test/e2e/` only for scenarios that still justify real workflow cost, and update package-local docs, config, demo assets, and references to teach only `fast`, `feat`, and `e2e`.
6. Do not reintroduce deterministic `*_test.rb` files under `test/e2e/` or `ace-test ace-review e2e` guidance.

## Package Migration Shape

- The current suite starts at 1 scenario and 7 test cases and must be reviewed before deciding what survives.
- Existing deterministic `test/integration/` coverage is a candidate source for `test/feat/`.
- Existing ATOM or flat package tests should move under `test/fast/`.
- Retained scenario assets stay under `test/e2e/` only when the review and change plan justify them.

## Interface Contract

- `ace-test ace-review`
- `ace-test ace-review feat`
- `ace-test ace-review all`
- `ace-test-e2e ace-review`

## Validation Questions

- None. The package already has legacy deterministic `test/integration/` coverage, so the remaining decisions are the review-driven `KEEP` / `MODIFY` / `REMOVE` / `CONSOLIDATE` / `ADD` outcomes.

## Vertical Slice Decomposition

- This task is one standalone package migration slice for `ace-review`. It ships independently once the package completes `review -> plan-changes -> rewrite` and exposes the required `fast` / `feat` / `e2e` contract.
