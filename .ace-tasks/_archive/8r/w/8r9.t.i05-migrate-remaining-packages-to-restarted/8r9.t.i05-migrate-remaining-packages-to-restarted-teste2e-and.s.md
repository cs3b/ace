# Migrate batch 1 packages to the fast/feat/e2e testing model

## Objective

Apply the validated `ace-b36ts` pilot taxonomy to the first migration batch through one package-specific review-first migration subtask per package.

### User Experience

- Batch 1 operators can pick up any child task and see the exact `as-e2e-review -> as-e2e-plan-changes -> as-e2e-rewrite` path it must execute before rewriting package tests.
- Reviewers can confirm that the batch no longer teaches legacy `unit` / `integration` / `e2e` command usage before implementation starts.
- The batch stays blocked until every child spec is complete enough to execute without inventing missing migration behavior or E2E rewrite criteria.

## Behavioral Specification

### User Experience

- Every package assigned to batch 1 gets an explicit migration task.
- Each pending package task starts from the canonical E2E lifecycle and states how deterministic tests move into `test/fast/` and, when justified by review findings, `test/feat/`.
- No batch 1 migration starts from stale `unit` / `integration` / `e2e` assumptions or from an assumption that current scenarios survive unchanged.

### Expected Behavior

1. Run subtask `8r9.t.i05.0` first to refresh all batch 1 child specs from the `ace-b36ts` pilot outputs.
2. Make every pending child task start with `as-e2e-review` via `ace-bundle wfi://e2e/review`, followed by `as-e2e-plan-changes` and `as-e2e-rewrite`.
3. Use the resulting change plan to move deterministic overlap out of `test/e2e/` and into `test/fast/` or `test/feat`.
4. Make `ace-test <package>` default to `fast`.
5. Use `ace-test <package> feat` when the approved rewrite retains or creates deterministic feature-layer coverage.
6. Keep `ace-test-e2e <package>` only for retained real workflow scenarios.
7. Hand remaining tested packages to sibling batch task `8r9.t.i06`.
8. Run sibling cleanup task `8r9.t.i07` only after both migration batches are complete.

### Interface Contract

- `ace-test <package>`
- `ace-test <package> feat`
- `ace-test <package> all`
- `ace-test-e2e <package>`

### Success Criteria

- [ ] Every package assigned to batch 1 has one explicit migration subtask.
- [ ] Every pending batch 1 package subtask requires `as-e2e-review`, `as-e2e-plan-changes`, and `as-e2e-rewrite` before scenario changes are applied.
- [ ] Every batch 1 package subtask lists deterministic migration destinations under `test/fast/` and, when justified by review findings, `test/feat/`.
- [ ] The refresh subtask runs before batch 1 migration implementation begins.
- [ ] No batch 1 child task teaches `ace-test <package> e2e`.
- [ ] No batch 1 child task assumes current E2E coverage survives without review-driven `KEEP` / `MODIFY` / `REMOVE` / `CONSOLIDATE` / `ADD` decisions.
- [ ] Batch 1 cleanly hands off the remaining tested packages to `8r9.t.i06`.

## Vertical Slice Decomposition

- `8r9.t.i05.0` refresh all batch 1 package subtasks from pilot outputs
- `8r9.t.i05.1` through `8r9.t.i05.r` migrate or simplify batch 1 package-local coverage under `fast` / `feat` / `e2e`
- sibling `8r9.t.i06` owns batch 2 packages that do not fit under the child-slot limit of `8r9.t.i05`
- sibling `8r9.t.i07` removes legacy `unit` / `integration` naming and bridge code after both batches land
