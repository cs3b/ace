# Migrate ace-overseer to the fast/feat/e2e testing model

## Objective

Move ace-overseer into the new `fast`/`feat`/`e2e` model by running the canonical E2E lifecycle first, then applying the approved rewrite so low-value scenario coverage moves into deterministic layers and any real feature contracts become explicit outside E2E.

## User Experience

- `as-e2e-review` on `ace-overseer` starts from the current suite of 1 scenario and 5 test cases and produces a coverage matrix before any rewrite decisions are made.
- `ace-test ace-overseer` becomes the default fast feedback loop for deterministic package tests.
- `ace-test ace-overseer feat` is added only when review findings identify deterministic feature coverage that should not remain inside E2E.
- `ace-test-e2e ace-overseer` remains only for retained real workflow scenarios.

## Expected Behavior

1. Run `as-e2e-review` via `ace-bundle wfi://e2e/review` for `ace-overseer` and capture a coverage matrix for the existing suite.
2. Run `as-e2e-plan-changes` via `ace-bundle wfi://e2e/plan-changes` and classify every current scenario and TC as `KEEP`, `MODIFY`, `REMOVE`, `CONSOLIDATE`, or `ADD`.
3. Run `as-e2e-rewrite` via `ace-bundle wfi://e2e/rewrite` and apply the approved scenario changes.
4. Create or expand `test/feat/` only when the review identifies deterministic feature contracts that should leave E2E; otherwise keep deterministic coverage in `test/fast/` and narrow `test/e2e/` to retained workflow value.
5. Keep `test/e2e/` only for scenarios that still justify real workflow cost, and update package-local docs, config, demo assets, and references to teach only `fast`, `feat`, and `e2e`.
6. Do not reintroduce deterministic `*_test.rb` files under `test/e2e/` or `ace-test ace-overseer e2e` guidance.

## Package Migration Shape

- The current suite starts at 1 scenario and 5 test cases and must be reviewed before deciding what survives.
- No legacy `test/integration/` layer exists, so `test/feat/` is created only if the review identifies deterministic feature contracts that should not stay in `test/e2e/`.
- Existing ATOM or flat package tests move under `test/fast/`.
- Retained scenario assets stay under `test/e2e/` only when the review and change plan justify them.

## Interface Contract

- `ace-test ace-overseer`
- `ace-test ace-overseer feat` when the approved rewrite introduces explicit `test/feat/` coverage
- `ace-test ace-overseer all`
- `ace-test-e2e ace-overseer`

## Validation Questions

- None. The package has no legacy `test/integration/` layer, so the remaining decisions are the review-driven `KEEP` / `MODIFY` / `REMOVE` / `CONSOLIDATE` / `ADD` outcomes and whether any explicit `feat` layer is justified.

## Vertical Slice Decomposition

- This task is one standalone package migration slice for `ace-overseer`. It ships independently once the package completes `review -> plan-changes -> rewrite` and exposes the required `fast` / `feat` / `e2e` contract.
