# Fix ace-taskflow idea file organization to enforce subfolder structure

## Behavioral Specification

### User Experience
- **Input**: Users run `ace-taskflow idea create "description"` command
- **Process**: System validates and ensures proper directory structure, saves idea in correct location with clear feedback
- **Output**: Idea file saved in `.ace-taskflow/<release>/ideas/` subfolder, confirmation message shows correct path

### Expected Behavior
When users create ideas using ace-taskflow, the system must enforce that all idea files are saved within the `ideas/` subfolder structure. The old format where ideas could be saved at the root level or in incorrect locations must be completely prevented. Users should receive clear feedback about where their ideas are being saved, and any attempts to save in incorrect locations should fail with helpful error messages.

The system should also detect any existing misplaced idea files and provide guidance on how to correct the structure.

### Interface Contract

```bash
# CLI Interface
ace-taskflow idea create "idea description"
# Success output: "Idea created: .ace-taskflow/v.0.9.0/ideas/20251113-1234-idea-description.md"

# Error conditions
ace-taskflow idea create "description"  # When structure is misconfigured
# Error output: "Error: Ideas must be saved in ideas/ subfolder. Check configuration."

# Detection of misplaced ideas
ace-taskflow idea list
# Warning: "Warning: Found 3 idea files in incorrect locations. Run 'ace-taskflow idea fix-structure' for details."

# Structure validation
ace-taskflow idea validate-structure
# Output: "✓ All ideas properly organized in ideas/ subfolders"
# Or: "✗ Found misplaced ideas: [list of files]"
```

**Error Handling:**
- Configuration missing ideas subfolder: Error with setup instructions
- Write permission issues: Clear permission error with fix suggestions
- Misplaced existing ideas: Warning with migration guidance

**Edge Cases:**
- Empty idea description: Still saves in correct location with generic filename
- Very long descriptions: Truncates filename but maintains correct path
- Concurrent idea creation: Each gets unique timestamp, all in correct location

### Success Criteria
- [ ] **Behavioral Outcome 1**: All new ideas are saved exclusively in `ideas/` subfolder structure
- [ ] **User Experience Goal 2**: Clear error messages prevent and explain any incorrect file placement
- [ ] **System Performance 3**: Existing misplaced ideas are detected and reported to users

### Validation Questions
- [ ] **Requirement Clarity**: Should the system automatically migrate existing misplaced ideas or just report them?
- [ ] **Edge Case Handling**: How should the system handle ideas created while in a git worktree?

## Objective

Fix the ace-taskflow idea creation to enforce proper file organization, ensuring all ideas are saved in the `ideas/` subfolder structure. This prevents confusion, maintains consistency, and ensures ideas can be properly tracked and managed.

## Technical Approach

### Current Implementation Analysis

Research reveals the current implementation already enforces the ideas/ subdirectory structure through four redundant layers:

1. **IdeaCommand** (commands/idea_command.rb) - Always sets directory to include "ideas" subdirectory
2. **IdeaWriter** (organisms/idea_writer.rb) - Always creates directories via FileNamer
3. **FileNamer** (molecules/file_namer.rb) - Always generates directory paths
4. **IdeaLoader** (molecules/idea_loader.rb) - Always searches within ideas/ directory

The issue appears to be legacy ideas created before this enforcement was implemented.

### Architecture Pattern
- Maintain existing ATOM architecture with clear separation of concerns
- Add validation layer to detect and report misplaced ideas
- Enhance error messaging for structure violations

### Technology Stack
- Ruby (existing ace-taskflow infrastructure)
- FileUtils for directory operations
- Thor for CLI command extensions
- No new dependencies required

## Risk Assessment

### Technical Risks
- **Risk:** False positives detecting valid ideas as misplaced
  - **Probability:** Low
  - **Impact:** Medium
  - **Mitigation:** Comprehensive pattern matching with multiple validation rules
  - **Rollback:** Validation can be disabled via configuration

### Integration Risks
- **Risk:** Breaking existing workflows that expect specific file locations
  - **Probability:** Low
  - **Impact:** High
  - **Mitigation:** Only add warnings, don't break existing functionality
  - **Monitoring:** Log validation results without enforcement initially
