## Objective

Enable ace-review to execute a single review against multiple LLM models concurrently, allowing users to compare insights from different AI providers (e.g., GPT-4 vs Gemini vs Claude) in a single command. This provides diverse perspectives on code quality while saving time through parallel execution.

## Scope of Work

- Extend CLI to accept multiple model specifications (comma-separated or repeated flags)
- Add `models:` array support to preset YAML schema (backward compatible with `model:` scalar)
- Implement concurrent execution with Thread-based parallelism (Ruby stdlib)
- Add real-time progress display showing per-model status
- Implement fault isolation so one model failure doesn't abort others
- Generate per-model output files with model-identifying filenames
- Enhance session metadata to track multi-model execution results

## Technical Approach

### Architecture Pattern

**Concurrent Execution with Thread Pool**:
- Use Ruby's built-in `Thread` class for concurrent model execution
- Simple thread pool pattern (no external dependencies like `concurrent-ruby`)
- Max concurrent limit (default 3, configurable via ENV)
- Progress tracking with thread-safe mutex for status updates

**Backward Compatibility**:
- Single model execution path remains unchanged
- Preset `model:` scalar still supported (existing behavior)
- New `models:` array triggers multi-model execution
- CLI `--model` flag parsing extended but maintains existing single-model behavior

### Technology Stack

**Concurrency**: Ruby stdlib `Thread` class
- No additional dependencies required
- Simple thread pool implementation
- Proven pattern from ace-support-core CommandExecutor

**Progress Display**: stderr output with emoji indicators
- `⏳` for querying, `✓` for success, `✗` for failure
- Real-time updates (not buffered)
- Compatible with CI/CD environments

**Model Parsing**: String splitting with deduplication
- Comma-separated: `"model1,model2,model3"` → `["model1", "model2", "model3"]`
- Repeated flags: collected into array by OptionParser
- Deduplicated via `Array#uniq`

    class MultiModelExecutor
      def execute(models, system_prompt:, user_prompt:, session_dir:)
        results = {}
        mutex = Mutex.new

        threads = models.map do |model|
          Thread.new do
            result = execute_single_model(model, system_prompt, user_prompt, session_dir)
            mutex.synchronize { results[model] = result }
          end
        end

        threads.each(&:join)
        results
      end
    end

## Risk Assessment

### Technical Risks

- **Risk**: Thread concurrency bugs (race conditions, deadlocks)
  - **Probability**: Medium
  - **Impact**: High
  - **Mitigation**: Use Mutex for shared state, thorough testing, follow patterns from ace-support-core
  - **Rollback**: Feature flag to disable multi-model execution

- **Risk**: Progress display buffering in CI/CD environments
  - **Probability**: Low
  - **Impact**: Medium
  - **Mitigation**: Use `$stderr.flush` after each progress update, test in CI environment
  - **Rollback**: N/A (progress is informational only)

- **Risk**: Memory usage with many concurrent models
  - **Probability**: Low
  - **Impact**: Medium
  - **Mitigation**: Default max concurrent to 3, configurable via ENV
  - **Rollback**: Reduce default concurrent limit

### Integration Risks

- **Risk**: Breaking existing single-model workflows
  - **Probability**: Low
  - **Impact**: High
  - **Mitigation**: Extensive backward compatibility testing, keep single-model path unchanged
  - **Monitoring**: Full test suite must pass before merging

- **Risk**: Preset schema conflicts with existing presets
  - **Probability**: Low
  - **Impact**: Medium
  - **Mitigation**: `models:` array is optional, `model:` scalar still supported
  - **Monitoring**: Validate all existing presets load correctly

### Performance Risks

- **Risk**: Concurrent execution doesn't actually run in parallel (GIL)
  - **Mitigation**: LLM queries are I/O bound (HTTP requests), not CPU bound - GIL releases during I/O
  - **Monitoring**: Measure total duration vs sum of individual durations
  - **Thresholds**: Total duration should be < 2x slowest individual model
