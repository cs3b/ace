# Declarative Assignment Frontmatter

## Objective

Allow task files (`.s.md`) and workflow files (`.wf.md`) to declare assignment intent via an `assign:` frontmatter block. Compose reads this declaration and resolves it against the phase catalog — the frontmatter says *what*, compose decides *how*. Add parent-child assignment modeling and `ace-assign list --tree`.

## Design Context

### Why Declarative (Not Prescriptive)

The compose workflow (240.06) already handles phase selection, ordering, prerequisite validation, and review cycle expansion via the phase catalog and composition rules. Frontmatter should not duplicate this — it should feed into compose as a structured input, replacing or supplementing the natural-language description.

### The Flow

```
task.s.md / workflow.wf.md    (declares assign: intent)
         │
         ▼
    compose workflow           (resolves against catalog + rules)
         │
         ▼
    assignment.yaml            (includes parent-child links)
         │
         ▼
    runner / executor           (enforces fork for sub-phase trees)
```

### Where Frontmatter Lives

- **Task files** (`.s.md`): "This task needs implementation with review"
- **Workflow files** (`.wf.md`): "This workflow expands to these sub-phases"
- **NOT skills**: Skills stay lightweight (~10 lines)

## Scope of Work

### Included
- Parse `assign:` frontmatter block from `.s.md` and `.wf.md` files
- Declarative intent fields: `goal`, `phases` (hints not prescriptions), `variables`
- Compose integration: compose reads frontmatter as structured input alongside description
- Parent-child assignment linking via `parent` field in `assignment.yaml`
- Runner enforcement: phases with sub-phases execute in fork context
- `ace-assign list --tree` hierarchy view
- Update usage docs with frontmatter examples

### Excluded
- Skills frontmatter (skills stay minimal)
- Frontmatter as standalone assignment creation (always goes through compose)
- Full 234.08 schema (context, delegation, gates, report blocks)
- Cross-directory hierarchy discovery

## Frontmatter Schema

### Task File Example (`.s.md`)

```yaml
---
id: v.0.9.0+task.148
status: in-progress
assign:
  goal: implement-with-pr        # maps to recipe or intent keyword
  variables:
    taskref: "148"
    review_cycles: 2
  hints:                          # soft guidance for compose, not hard phase list
    - include: security-audit
    - skip: lint
---
```

### Workflow File Example (`.wf.md`)

```yaml
---
name: work-on-task
assign:
  sub-phases:                     # this workflow expands to sub-phases
    - onboard
    - implement
    - verify-tests
  context: fork                   # entire sub-tree runs in fork
---
```

### Schema Fields

| Field | Type | Purpose |
|-------|------|---------|
| `assign.goal` | String | Intent keyword or recipe name — compose resolves to phases |
| `assign.variables` | Hash | Parameters passed through to compose (taskref, review_cycles, etc.) |
| `assign.hints` | Array | Soft include/skip guidance for compose phase selection |
| `assign.sub-phases` | Array | Sub-phase expansion for workflow files (runner creates child phases) |
| `assign.context` | String | Fork context for the sub-phase tree |
| `assign.parent` | String | Parent assignment ID for hierarchy linking |

## Technical Approach

### 1. AssignFrontmatterParser (Atom)

Pure function: takes raw YAML frontmatter hash, extracts `assign:` block, validates fields, returns structured result.

```ruby
# Input:  full frontmatter hash from any .s.md or .wf.md
# Output: { config: { goal:, variables:, hints:, sub_phases:, context:, parent: },
#           valid: true/false, errors: [] }
AssignFrontmatterParser.parse(frontmatter_hash)
```

No file I/O — reuses existing frontmatter extraction from `PhaseFileParser` or `ace-support-markdown`.

### 2. Compose Integration

The compose workflow gains a new optional input: a source file path. When provided:
1. Extract frontmatter with `AssignFrontmatterParser`
2. Use `goal` as recipe/intent (replaces natural-language description)
3. Apply `variables` as compose parameters
4. Apply `hints` as phase selection guidance
5. Continue with normal compose flow (catalog, rules, validation)

This is an **additive input** — compose still works with pure descriptions. Frontmatter just gives it structured data instead of parsing natural language.

### 3. Parent-Child Assignment Model

Add `parent` field to `Assignment` model. `AssignmentManager` persists it in `assignment.yaml`. `AssignmentDiscoverer` returns it in `AssignmentInfo`.

### 4. Sub-Phase Fork Enforcement

When `AssignmentExecutor.start` creates phases and a phase references `sub-phases` (from workflow frontmatter), the runner:
- Creates child phases under a batch parent
- Sets `context: fork` on the parent
- Existing `auto_complete_parents` handles completion cascade

This reuses the batch-parent pattern already in compose (see `expansion.batch-parent` in compose workflow).

### 5. Tree Formatter (Atom)

Pure function: takes flat list of `AssignmentInfo` with parent fields, returns indented tree string.

```ruby
TreeFormatter.format(assignments)
# =>  task-148-implement
#     ├── onboard (completed)
#     ├── work-on-task (running)
#     │   ├── onboard (completed)
#     │   ├── implement (in_progress)
#     │   └── verify-tests (pending)
#     └── review-pr (pending)
```

## Risk Assessment

### Technical Risks
- **Risk:** Compose workflow changes are manual (LLM-executed) — harder to test
  - **Probability:** Medium
  - **Impact:** Low
  - **Mitigation:** Keep compose changes minimal (add one input path, not restructure)

- **Risk:** Sub-phase expansion at executor level adds complexity to start flow
  - **Probability:** Medium
  - **Impact:** Medium
  - **Mitigation:** Reuse existing batch-parent pattern; sub-phases are just pre-resolved children

- **Risk:** Parent metadata not present for older assignments
  - **Probability:** High
  - **Impact:** Low
  - **Mitigation:** Treat missing parent as root in tree rendering
