# Migrate ACE CLI gems to exception-based exit code pattern

## Behavioral Specification

### User Experience
- **Input**: User runs any ACE CLI command that encounters an error (e.g., `ace-lint nonexistent.md`)
- **Process**: Command detects error condition, raises exception with appropriate exit code
- **Output**: Error message printed to stderr, process exits with non-zero status code

### Expected Behavior

Currently, ACE CLI gems return integers from command `call()` methods expecting them to become exit codes. However, dry-cli's `call()` method returns `nil`, not the command's return value. This means **all ACE CLI commands currently exit with code 0 even on failure**.

After this migration:
1. Commands that fail will exit with non-zero codes (typically 1)
2. Error messages will be printed to stderr (not stdout)
3. Shell scripts and CI/CD pipelines can detect command failures via `$?`
4. Commands remain testable by asserting on raised exceptions

### Interface Contract

```bash
# Before (broken - always exits 0)
$ ace-lint nonexistent-file.md
Error: File not found
$ echo $?
0  # WRONG - should be 1

# After (correct - exits 1 on failure)
$ ace-lint nonexistent-file.md
Error: File not found
$ echo $?
1  # CORRECT - non-zero indicates failure

# Success case (unchanged)
$ ace-lint valid-file.md
✓ No issues found
$ echo $?
0  # Success
```

**Error Handling:**
- Validation errors: Exit code 1
- Runtime errors: Exit code 1
- Warnings (success with issues): Exit code 2 (optional)

**Edge Cases:**
- Help/version flags: Exit code 0 (not an error)
- Keyboard interrupt (Ctrl+C): Preserve signal handling (exit 130)

## Objective

Fix broken exit code handling in ACE CLI gems. Currently, dry-cli does not propagate command return values as exit codes, causing all commands to exit 0 even on failure. This breaks CI/CD pipelines, shell scripts, and automation that depends on exit codes.

## Scope of Work

- **Foundation**: Create `Ace::Core::CLI::Error` class in ace-support-core
- **Migration**: Update all CLI gems to use exception-based pattern
- **Testing**: Update test patterns to use `assert_raises`
- **Documentation**: ADR-023 already updated with correct pattern

### Deliverables

#### Behavioral Specifications
- Exception class with configurable exit code
- Exe wrapper pattern that catches and exits
- Command pattern that raises on failure

#### Validation Artifacts
- Exit code verification tests for each gem
- CI/CD integration test (shell script checking `$?`)

## Out of Scope

- ❌ **Changing command behavior**: Only exit code handling changes
- ❌ **New CLI features**: Focus on fixing existing exit codes
- ❌ **Performance optimization**: Functional fix only

## Technical Approach

### Architecture Pattern
- Exception-based error signaling (matches Hanami CLI pattern)
- Single error class with configurable exit code
- Exe wrapper handles exception → exit code translation

### Technology Stack
- Ruby exceptions (no external dependencies)
- Existing dry-cli infrastructure unchanged
- Compatible with all Ruby 3.x versions
