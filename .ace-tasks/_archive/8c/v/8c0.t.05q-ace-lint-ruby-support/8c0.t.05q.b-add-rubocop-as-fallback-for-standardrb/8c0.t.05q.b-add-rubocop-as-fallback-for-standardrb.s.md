# Add RuboCop as Fallback for StandardRB

## Objective

Improve user experience by adding RuboCop as a fallback validator when StandardRB is not available. This provides better compatibility for users who already have RuboCop installed, while maintaining StandardRB as the preferred zero-config option.

## Scope of Work

- **User Experience Scope**: Ruby file linting with automatic tool detection and fallback
- **System Behavior Scope**: Tool availability checking, command execution, JSON parsing, result display
- **Interface Scope**: Existing `ace-lint` CLI command interface (no changes to command syntax)

### Deliverables

#### Create

- `ace-lint/lib/ace/lint/atoms/rubocop_runner.rb` - RuboCop runner atom
- `ace-lint/.ace-defaults/lint/.rubocop.yml` - Minimal RuboCop configuration
- `ace-lint/test/atoms/rubocop_runner_test.rb` - Tests for RuboCop runner

#### Modify

- `ace-lint/lib/ace/lint/molecules/ruby_linter.rb` - Add fallback logic
- `ace-lint/.ace-defaults/lint/ruby.yml` - Add fallback_linter config
- `ace-lint/README.md` - Document fallback behavior
- `ace-lint/test/molecules/ruby_linter_test.rb` - Add fallback tests

## Behavioral Specification

### User Experience
- **Input**: User runs `ace-lint` on a Ruby file (e.g., `ace-lint lib/test.rb`)
- **Process**:
  - If StandardRB is installed: Uses StandardRB (zero-config, preferred)
  - If StandardRB is NOT installed but RuboCop is: Uses RuboCop as fallback
  - If neither is installed: Shows helpful error message with installation instructions
- **Output**: Linting results displayed with color coding (✓ success, ✗ error, ⚠ warnings)

### Expected Behavior

1. **StandardRB Available (Preferred Path)**:
   - System uses StandardRB for linting
   - No configuration required (zero-config)
   - StandardRB opinionated rules applied

2. **StandardRB Unavailable, RuboCop Available (Fallback Path)**:
   - System automatically falls back to RuboCop
   - Uses minimal default `.rubocop.yml` configuration
   - User informed via output that fallback is in use

3. **Neither Tool Available**:
   - System displays clear error message with installation instructions

### Fallback Notification

When RuboCop is used as fallback, the user is notified:

- **Message**: `[info] Using RuboCop (StandardRB not available)`
- **Output**: stderr
- **Suppressed by**: `--quiet` flag

### Success Criteria

- [ ] **Behavioral Outcome 1**: When StandardRB is not installed but RuboCop is, ace-lint automatically uses RuboCop and displays linting results
- [ ] **User Experience Goal 2**: Users see clear indication when fallback is in use vs when StandardRB is being used
- [ ] **System Performance 3**: Batch processing of multiple Ruby files works with both StandardRB and RuboCop fallback
- [ ] **Error Handling 4**: Clear, helpful error message when neither tool is installed

## Out of Scope

- ❌ User-facing config to choose between tools
- ❌ Ensuring identical linting results (different rules are expected)
- ❌ Performance benchmarking

## Technical Approach

### Architecture Pattern

- **ATOM pattern**: Create `RuboCopRunner` atom mirroring `StandardrbRunner`
- **Fallback logic in molecule**: `RubyLinter` tries StandardRB first, then RuboCop
- **Shared JSON parsing**: Both tools output identical RuboCop JSON format

### Technology Stack

- **RuboCop**: Ruby static code analyzer (validator)
- **StandardRB**: Opinionated RuboCop wrapper (zero-config validator)
- **JSON format**: Identical output format (StandardRB built on RuboCop)

### Implementation Strategy

1. **Copy-paste approach**: `RuboCopRunner` based on `StandardrbRunner` template
2. **Minimal changes**: Command name, availability check, flag mapping
3. **Fallback in molecule**: Try StandardRB first, fall back to RuboCop
4. **Reuse existing parsing**: JSON parsing works for both tools

## Risk Assessment

### Technical Risks

- **Risk**: RuboCop configuration differs significantly from StandardRB defaults
  - **Probability**: Medium
  - **Impact**: Low (different linting results expected)
  - **Mitigation**: Provide minimal `.rubocop.yml` with reasonable defaults

- **Risk**: Flag mapping incomplete or incompatible
  - **Probability**: Low
  - **Impact**: Medium (some features may not work with RuboCop)
  - **Mitigation**: Test all flag mappings, document limitations

### Integration Risks

- **Risk**: Existing tests may assume StandardRB is always available
  - **Mitigation**: Update tests to handle both tools and fallback scenarios
