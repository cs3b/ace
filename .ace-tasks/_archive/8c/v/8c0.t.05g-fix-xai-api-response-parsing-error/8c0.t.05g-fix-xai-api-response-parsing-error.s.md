# Fix xAI API Response Parsing Error

## Behavioral Specification

### User Experience
- **Input**: User invokes xAI/Grok API via ace-llm (e.g., `ace-llm-query "Hello" -m xai:grok-4`)
- **Process**: API returns error response (502/503 gateway error with HTML body)
- **Output**: User receives clear error message instead of Ruby exception

### Expected Behavior

When the xAI API returns a non-JSON error response (HTML error page, plain text, gateway errors), the system should:
1. Gracefully handle the non-Hash response body
2. Extract whatever error information is available
3. Return a clear, actionable error message to the user
4. Not crash with "String does not have #dig method"

Current behavior crashes when `response.body` is a String instead of Hash.

### Interface Contract

```bash
# When API returns non-JSON error (e.g., 502 Bad Gateway with HTML)
$ ace-llm-query "Hello" -m xai:grok-4

# Current (broken):
LLM error: xai API error: String does not have #dig method

# Expected (fixed):
LLM error: xAI API error (502): unknown - Unknown error: 502
```

**Error Handling:**
- Non-JSON response body: Treat as empty error details, include HTTP status code
- HTML error pages: Extract status code, provide generic error message
- Gateway errors (502, 503, 504): Include status in error message

**Edge Cases:**
- Response body is a String (HTML page): Handle gracefully
- Response body is nil: Handle gracefully
- Response body is empty: Handle gracefully
