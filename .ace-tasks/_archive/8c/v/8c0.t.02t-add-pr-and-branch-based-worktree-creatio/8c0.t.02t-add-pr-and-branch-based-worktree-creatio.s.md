## Behavioral Specification

### User Experience
- **Input**: PR number via `--pr` flag OR branch name (local/remote) via `-b/--branch` flag
- **Process**: Tool fetches branch information (from GitHub PR or git remote), creates isolated worktree, sets up remote tracking when source is remote branch
- **Output**: Ready-to-use worktree with code checked out, confirmation message showing worktree location, branch name, and tracking status

### Expected Behavior

Developers can create git worktrees from two new sources:

1. **GitHub Pull Requests**: Using `--pr <number>`, the tool queries GitHub CLI to get the PR's branch information, fetches the remote branch, and creates a worktree with proper remote tracking.

2. **Explicit Branches**: Using `-b <branch>` or `--branch <branch>`, the tool creates a worktree from any local or remote branch:
   - **Remote branches** (containing `/` like `origin/feature`): Automatically fetches if needed and sets up tracking
   - **Local branches**: Creates worktree without tracking setup

Both approaches enable quick context switching for PR review, testing remote branches, or collaboration without disrupting the current work environment.

### Interface Contract

```bash
# PR-based creation
ace-git-worktree create --pr 26
ace-git-worktree create --pull-request 26

# Branch-based creation (local or remote)
ace-git-worktree create -b feature/auth                # Local branch
ace-git-worktree create --branch origin/feature/auth   # Remote branch
ace-git-worktree create --branch upstream/main         # Fork remote

# Expected output for PR
✓ Fetched PR #26: "Add authentication feature"
✓ Remote branch: origin/claude/task-09-work-011CV2wtrwECwjaB8UPQPwjr
✓ Created worktree: ace-pr-26
✓ Branch: pr-26 tracking origin/claude/task-09-work-011CV2wtrwECwjaB8UPQPwjr
✓ Location: /Users/mc/Ps/ace-meta/.ace-wt/ace-pr-26

# Expected output for remote branch
✓ Fetched remote branch: origin/feature/auth
✓ Created worktree: feature-auth
✓ Branch: feature/auth tracking origin/feature/auth
✓ Location: /Users/mc/Ps/ace-meta/.ace-wt/feature-auth

# Expected output for local branch
✓ Created worktree: feature-auth
✓ Branch: feature/auth (local, no tracking)
✓ Location: /Users/mc/Ps/ace-meta/.ace-wt/feature-auth

# Dry run support
ace-git-worktree create --pr 26 --dry-run
ace-git-worktree create -b origin/feature --dry-run

# Integration with existing commands
ace-git-worktree list                    # Shows PR and branch worktrees
ace-git-worktree switch ace-pr-26        # Switch to PR worktree
ace-git-worktree remove ace-pr-26        # Remove PR worktree
```

**Error Handling:**
- **gh CLI not available**: `Error: gh CLI is required for PR worktrees. Install with: brew install gh`
- **PR not found**: `Error: PR #26 not found in this repository`
- **Branch not found**: `Error: Branch 'origin/nonexistent' not found`
- **Remote not fetched**: `Error: Remote branch 'upstream/main' not found. Run: git fetch upstream`
- **Conflicting options**: `Error: Cannot use --pr with --branch, --task, or positional branch name`
- **Network issues**: `Error: Failed to fetch PR information. Check your network connection`
- **Already exists**: `Error: Worktree already exists for branch 'origin/feature/auth' at: <path>`
- **Fork PR warning**: `⚠️  This PR is from a fork (user/repo). Branch owner: external-contributor`

**Edge Cases:**
- **Fork-based PRs**: Detect and add fork remote if needed, then fetch branch
- **Remote branch not fetched yet**: Automatically run `git fetch <remote>` before creating worktree
- **Branch name conflicts**: Local branch `pr-26` exists - use unique name or fail with clear error
- **Multiple worktrees for same branch**: Detect existing worktree and inform user of location
- **Deleted PR branches**: Handle gracefully with message: "PR branch has been deleted on remote"
- **Custom worktree paths**: Support `--path` option for both PR and branch worktrees

## Objective

Enable developers to quickly create isolated git worktrees from GitHub pull requests and any git branch (local or remote) with a single command. This reduces context switching friction for code review, testing remote branches, and collaboration workflows.

**User Value:**
- **Fast PR Review**: Create review environment in seconds: `ace-git-worktree create --pr 26`
- **Remote Branch Testing**: Test colleague's branch without affecting current work: `ace-git-worktree create -b origin/feature`
- **Parallel Work**: Review multiple PRs or test multiple branches simultaneously
- **Clean Workflow**: Automatic remote tracking ensures git operations work correctly
- **Consistency**: Same tool for task-based, PR-based, and branch-based worktrees

### Git Command Sequences

**PR-Based Worktree Creation:**
```bash
# 1. Check gh availability (fail fast if not installed)
which gh || exit 1

# 2. Fetch PR metadata (extended fields for fork detection)
gh pr view <pr_number> --json number,headRefName,baseRefName,title,headRepositoryOwner,isCrossRepository

# 3. Parse response
# {
#   "number": 26,
#   "headRefName": "claude/task-09-work-011CV2wtrwECwjaB8UPQPwjr",
#   "baseRefName": "main",
#   "title": "Add authentication feature",
#   "headRepositoryOwner": { "login": "external-contributor" },
#   "isCrossRepository": true
# }

# 4. Check if fork-based PR and display warning
if isCrossRepository:
  echo "⚠️  This PR is from a fork (${headRepositoryOwner.login}/repo)"
  echo "   Branch owner: ${headRepositoryOwner.login}"

# 3. Fetch remote branch
git fetch origin <headRefName>

# 4. Create worktree with remote tracking
git worktree add <worktree_path> -b <local_branch> origin/<headRefName>

# Example:
# git worktree add .ace-wt/ace-pr-26 -b pr-26 origin/claude/task-09-work-011CV2wtrwECwjaB8UPQPwjr
```

**Remote Branch Worktree Creation:**
```bash
# 1. Detect remote branch (contains '/')
# branch_name = "origin/feature/auth"

# 2. Parse remote and branch
# remote = "origin"
# branch = "feature/auth"

# 3. Fetch if needed
git fetch <remote> <branch>

# 4. Create worktree with tracking
git worktree add <worktree_path> -b <local_branch> <remote>/<branch>

# Example:
# git worktree add .ace-wt/feature-auth -b feature/auth origin/feature/auth
```

**Local Branch Worktree Creation:**
```bash
# 1. Verify branch exists locally
git show-ref --verify refs/heads/<branch>

# 2. Create worktree without tracking
git worktree add <worktree_path> <branch>

# Example:
# git worktree add .ace-wt/feature-auth feature/auth
```

## Risk Assessment

### Technical Risks

**Risk: GitHub CLI Not Available**
- **Probability:** Medium (not installed by default on all systems)
- **Impact:** High (PR worktree creation fails completely)
- **Mitigation:**
  - Early availability check in `PrFetcher.gh_available?` - fail fast
  - Clear error message: "Error: gh CLI is required for PR worktrees. Install with: brew install gh"
  - No fallback: gh is a hard requirement for `--pr` feature
  - Document requirement clearly in README and help text
- **Rollback:** Feature is opt-in via `--pr` flag, existing workflows unaffected

**Risk: GitHub CLI Authentication Failure**
- **Probability:** Medium (first-time users, expired tokens)
- **Impact:** High (cannot fetch PR information)
- **Mitigation:**
  - Detect auth errors from `gh` output
  - Clear message: "GitHub CLI not authenticated. Run: gh auth login"
  - Document authentication setup in troubleshooting guide
- **Monitoring:** Parse gh error output for auth-related messages

**Risk: Network Connectivity Issues**
- **Probability:** Low (usually have network for git operations)
- **Impact:** Medium (PR fetch fails, but branch creation may still work offline)
- **Mitigation:**
  - Set reasonable timeout (30 seconds) for gh and git fetch commands
  - Catch timeout errors and provide clear message
  - Document offline limitations
- **Rollback:** User can retry when network available

**Risk: Remote Branch Detection Ambiguity**
- **Probability:** Low (branch naming conventions are generally clear)
- **Impact:** Low (might treat remote as local or vice versa)
- **Mitigation:**
  - Use simple heuristic: contains `/` = remote branch
  - Validate branch existence before creation
  - Provide `--force-remote` or `--force-local` flags if needed (future enhancement)
- **Monitoring:** Unit tests cover various branch naming patterns

**Risk: Worktree Path Conflicts**
- **Probability:** Medium (user creates multiple worktrees for same PR/branch)
- **Impact:** Low (creation fails, but with clear error)
- **Mitigation:**
  - Check for existing worktrees on same branch before creation
  - Provide existing worktree location in error message
  - Option: Support `-2`, `-3` suffixes for duplicate paths (future enhancement)
- **Rollback:** User removes existing worktree or uses different path

### Integration Risks

- **Impact:** Medium (users need to update configs)
- **Mitigation:**
  - Add new config sections without changing existing structure
  - Provide sensible defaults for all new options
  - Make all new configuration optional
  - Document configuration migration if needed
- **Monitoring:** Config validation tests with various formats

**Risk: Git Worktree State Inconsistencies**
- **Probability:** Low (git worktree is mature feature)
- **Impact:** High (corrupted git state)
- **Mitigation:**
  - Follow git worktree best practices
  - Test with various git versions (CI matrix)
  - Document minimum git version requirements
  - Add rollback procedures to documentation
- **Monitoring:** Integration tests verify worktree integrity

### Performance Risks

**Risk: Network Operations Slow Down User Experience**
- **Mitigation:**
  - Set appropriate timeouts (gh: 30s, git fetch: 60s)
  - Show progress indication during network operations
  - Consider async/background fetching (future enhancement)
- **Monitoring:** Measure and log operation timings
- **Thresholds:**
  - PR fetch: < 5 seconds acceptable
  - Git fetch: < 10 seconds acceptable
  - Total operation: < 15 seconds acceptable

**Risk: Multiple PR Fetches Causing Rate Limiting**
- **Mitigation:**
  - GitHub CLI handles rate limiting internally
  - Document rate limits in troubleshooting guide
  - Consider caching PR metadata locally (future enhancement)
- **Monitoring:** Log gh API response codes
- **Thresholds:** GitHub API allows 5,000 requests/hour (authenticated)
