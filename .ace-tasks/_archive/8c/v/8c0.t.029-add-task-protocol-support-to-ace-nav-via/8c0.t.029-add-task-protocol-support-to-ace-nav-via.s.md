# Add task:// Protocol Support to ace-nav via Command Delegation

## Behavioral Specification

### User Experience
- **Input**: `ace-nav task://[task-ref]` with any valid task reference format (018, 018, v.0.9.0+task.018)
- **Process**: Seamless navigation to task resources through unified ace-nav interface that delegates to ace-taskflow
- **Output**: Task information displayed (path, content, tree) matching ace-taskflow task command output

### Expected Behavior

Users should be able to navigate to tasks using the task:// protocol through ace-nav, providing a unified interface for all resource navigation. The protocol delegates to the underlying ace-taskflow task command, preserving all functionality while maintaining consistency with other ace-nav protocols (wfi://, guide://, tmpl://).

This creates a single entry point for developers: ace-nav handles all resource types including tasks, workflows, guides, and templates. The command delegation pattern ensures ace-nav doesn't duplicate ace-taskflow's task management logic, instead acting as a smart router.

### Interface Contract

```bash
# Core Task Navigation (v1 - Single Task Lookup)
ace-nav task://018                     # Delegates to: ace-taskflow task 018
ace-nav task://018                # Delegates to: ace-taskflow task 018
ace-nav task://v.0.9.0+task.018        # Delegates to: ace-taskflow task v.0.9.0+task.018
ace-nav task://backlog+task.025             # Delegates to: ace-taskflow task backlog+task.025

# Display Options (pass-through to ace-taskflow)
ace-nav task://018 --path              # Delegates to: ace-taskflow task 018 --path
ace-nav task://018 --content           # Delegates to: ace-taskflow task 018 --content
ace-nav task://018 --tree              # Delegates to: ace-taskflow task 018 --tree

# Expected Output
# Returns whatever ace-taskflow task would return
# Exit codes match ace-taskflow behavior (0 for success, 1 for errors)
```

**Error Handling:**
- Task not found: Delegates to ace-taskflow which returns appropriate error message
- Invalid reference format: Delegates to ace-taskflow for validation and error reporting
- ace-taskflow not available: "Error: ace-taskflow command not found. Install ace-taskflow gem."
- Invalid options: Pass through to ace-taskflow for validation

**Edge Cases:**
- Next task (no ref): `ace-nav task://` → delegates to `ace-taskflow task` (shows next task)
- Ambiguous references: Let ace-taskflow handle disambiguation
- Multiple matching tasks: ace-taskflow's lookup logic handles this
- Cross-release references: Fully supported via delegation (v.0.9.0+task.018, backlog+task.025)

## Objective

Provide a unified navigation interface across all ACE resources by adding task:// protocol support to ace-nav. This maintains consistency with existing protocols (wfi://, guide://, tmpl://) while delegating actual task management to ace-taskflow's specialized functionality.

**User Value**: Single command for all resource navigation - users don't need to remember whether to use ace-nav or ace-taskflow for different resource types.

## Technical Approach

### Architecture Pattern

The implementation follows ace-nav's existing protocol architecture but adds a new "cmd" delegation type. Instead of resolving to file paths like existing protocols (wfi://, guide://, tmpl://), cmd protocols delegate execution to external commands.

**Pattern Selection:**
- **Existing**: File-based protocols (wfi, guide, tmpl) resolve URIs to file paths
- **New**: Command-based protocols delegate to CLI commands with argument pass-through
- **Integration**: Minimal changes to existing code; new organism handles delegation

**Key Design Decisions:**
1. **Type-based dispatch**: Protocol config includes `type: cmd` to differentiate from file-based
2. **Template-based delegation**: Config defines command template with placeholder substitution
3. **Argument preservation**: All ace-nav flags/args passed through to delegated command
4. **Exit code forwarding**: Delegate command's exit code becomes ace-nav's exit code

### Technology Stack

**No New Dependencies:**
- Uses Ruby's built-in `Kernel.system` for command delegation
- Leverages existing ace-nav config cascade (`.ace/nav/protocols/`)
- Reuses existing `ResourceUri` parsing and validation

**Integration Points:**
- `NavigationEngine` (organism): Add delegation path detection
- `CLI`: Detect cmd protocols and delegate before normal resolution
- Protocol config (`.ace/nav/protocols/task.yml`): Define delegation template

### Implementation Strategy

**Phase 1: Protocol Configuration**
1. Create task.yml protocol config with `type: cmd`
2. Define command template: `ace-taskflow task %{ref}`
3. Specify which options are pass-through vs filtered

**Phase 2: Command Delegation**
1. Create `CommandDelegator` organism to handle execution
2. Parse task:// URI to extract reference
3. Build command from template + ref + pass-through args
4. Execute and forward exit code

**Phase 3: Integration**
1. Modify `CLI` to detect cmd-type protocols early
2. Short-circuit normal file resolution for cmd protocols
3. Delegate to `CommandDelegator` organism

**Phase 4: Testing**
1. Unit tests for command template building
2. Integration tests for full delegation flow
3. Test all reference formats and options

## Risk Assessment

### Technical Risks

- **Risk:** Command not found (ace-taskflow not installed)
  - **Probability:** Medium
  - **Impact:** High (feature doesn't work)
  - **Mitigation:** Check command exists before delegation, provide clear error message
  - **Rollback:** Feature gracefully fails; existing protocols unaffected

- **Risk:** Argument escaping issues with special characters
  - **Probability:** Low
  - **Impact:** Medium (incorrect command execution)
  - **Mitigation:** Use shell-safe argument passing, test with special chars
  - **Rollback:** Fix argument building logic

- **Risk:** Exit code handling differences across Ruby versions
  - **Probability:** Low
  - **Impact:** Low (inconsistent error reporting)
  - **Mitigation:** Test on multiple Ruby versions, use `$?.exitstatus`
  - **Monitoring:** CI tests across Ruby versions

### Integration Risks

- **Risk:** Conflicts with future file-based task:// implementation
  - **Probability:** Low
  - **Impact:** Medium (would need refactoring)
  - **Mitigation:** Design allows type switching via config change
  - **Monitoring:** Track usage patterns and requirements

- **Risk:** Performance overhead from process spawning
  - **Probability:** High (inherent to delegation)
  - **Impact:** Low (acceptable for task lookup operations)
  - **Mitigation:** Document that delegation has subprocess overhead
  - **Monitoring:** Not needed; acceptable trade-off for simplicity

### Performance Risks

- **Risk:** Subprocess spawn adds ~150ms overhead
  - **Mitigation:** Acceptable for task lookup; delegation is explicit design choice
  - **Monitoring:** Not needed; performance acceptable for use case
  - **Thresholds:** < 500ms total for task lookup considered acceptable
