# Fix ace-git-commit to respect path restrictions when committing

## Behavioral Specification

### User Experience
- **Input**: User runs `ace-git-commit .ace-taskflow/` to commit only files within that directory
- **Process**: Tool stages only files within specified path, unstages files outside the path, generates commit message for the restricted set
- **Output**: Commit created containing ONLY files within the specified path(s)

### Expected Behavior

When a user specifies a path to `ace-git-commit`, they expect the tool to commit ONLY the files within that path, similar to how `git add <path>` works. Currently, the tool commits ALL staged files regardless of the path restriction, which violates the principle of least surprise.

**Current Broken Behavior:**
```bash
$ ace-git-commit .ace-taskflow/
# Commits: .ace-taskflow/ files + Gemfile.lock + ace-git-worktree/lib/...
# WRONG: Files outside .ace-taskflow/ should not be included
```

**Expected Correct Behavior:**
```bash
$ ace-git-commit .ace-taskflow/
# Commits: ONLY files within .ace-taskflow/
# Files outside this path remain in their previous state (staged or unstaged)
```

The system should behave like a focused commit tool - when given a path, it creates a commit scoped to just that path, allowing users to make precise, targeted commits even when they have multiple changes staged.

### Interface Contract

```bash
# CLI Interface
ace-git-commit [options] [path...]

# Basic path restriction
ace-git-commit .ace-taskflow/
# Expected: Commits only files in .ace-taskflow/

# Multiple paths
ace-git-commit lib/ test/
# Expected: Commits only files in lib/ and test/

# Single file
ace-git-commit README.md
# Expected: Commits only README.md

# With other options
ace-git-commit --message "Custom message" .ace-taskflow/
# Expected: Uses custom message, commits only .ace-taskflow/ files

# No path (current behavior preserved)
ace-git-commit
# Expected: Commits all staged files (backward compatible)
```

**Error Handling:**
- Non-existent path: "Error: Path 'xyz' does not exist"
- No changes in path: "No changes to commit in specified path(s)"
- Path with no tracked files: "No tracked files found in 'path/'"
- Mixed success: Clear report of what was/wasn't committed

**Edge Cases:**
- Symlinks: Follow git's standard symlink behavior
- Submodules: Respect submodule boundaries
- Binary files: Handle same as text files
- Deeply nested paths: Work correctly at any depth
- Glob patterns: Support if feasible (e.g., `**/*.rb`)

## Technical Approach

### Architecture Pattern
The solution follows the existing ATOM architecture pattern, adding a new PathResolver molecule to handle path filtering logic. This integrates cleanly with the existing CommitOrchestrator workflow without disrupting current functionality.

**Key Design Decisions:**
- Create PathResolver molecule for path expansion and filtering
- Enhance FileStager with path-aware staging methods
- Maintain backward compatibility by preserving existing behavior when no paths specified
- Leverage git's own path filtering capabilities rather than reimplementing

### Technology Stack
- Ruby (existing gem infrastructure)
- Git plumbing commands for precise control:
  - `git ls-files` for finding tracked files in paths
  - `git diff --name-only` for finding changed files in paths
  - `git reset` for unstaging files outside paths
- No new dependencies required - all functionality uses git and Ruby stdlib

### Implementation Strategy

**Approach: Enhanced Staging with Path Filtering**

When paths are specified:
1. Save current staging state
2. Reset (unstage) all files
3. Stage only files within specified paths
4. Proceed with normal commit flow

This ensures a clean, predictable state where only files in the specified paths are committed.

  def stage_paths(paths)
    return false if paths.nil? || paths.empty?

    # Reset staging area
    @git.execute("reset", "--quiet")

    # Stage only files in specified paths
    paths.each do |path|
      @git.execute("add", path) if File.exist?(path)
    end

    true
  end
