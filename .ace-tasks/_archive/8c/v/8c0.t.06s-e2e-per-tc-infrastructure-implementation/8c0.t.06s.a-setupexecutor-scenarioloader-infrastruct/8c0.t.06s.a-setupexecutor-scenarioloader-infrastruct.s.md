# SetupExecutor + ScenarioLoader Infrastructure

## Behavioral Specification

### User Experience

**Input:**
- Developer creates a scenario directory under `{package}/test-e2e/scenarios/TS-{AREA}-{NNN}-{slug}/` containing:
  - `scenario.yml` — metadata and setup steps
  - `fixtures/` — real files to copy into sandbox
  - `TC-*.tc.md` — independent test case files
- Developer runs `ace-test-e2e setup ace-lint TS-LINT-001` to create a populated sandbox for debugging

**Process:**
- `ScenarioLoader` reads `scenario.yml`, validates metadata, discovers `TC-*.tc.md` files, returns `TestScenario` + `TestCase` models
- `SetupExecutor` creates a fresh sandbox directory and executes setup steps deterministically via Ruby (no LLM involved):
  - `git-init`: initializes git repo with test user config
  - `copy-fixtures`: copies `fixtures/` tree into sandbox
  - `write-file`: writes inline content to a path
  - `run`: executes shell commands in sandbox
  - `env`: sets environment variables
- Setup completes in ~200ms (vs current LLM-driven setup taking 30-60s)

**Output:**
- A populated sandbox directory at `.cache/ace-test-e2e/{timestamp}-{pkg}-{scenario}-{tc}/`
- For `setup` subcommand: prints sandbox path to stdout for developer inspection
- Models loaded and ready for the execution pipeline (Subtask 2)

### Expected Behavior

**Current Pain Point:**
Test sandbox setup is embedded as bash heredocs inside `.mt.md` files, executed by the LLM agent. This is:
- Slow (30-60s per scenario via agent)
- Non-deterministic (agent may interpret heredocs differently)
- Opaque (test files are strings inside strings, not real files with IDE support)
- Repeated when re-running individual TCs (full scenario context sent every time)

**Desired Behavior:**
Sandbox setup becomes a deterministic Ruby operation that completes in milliseconds. Test fixture files are real files on disk with proper syntax highlighting and IDE integration. Each test case gets its own fresh sandbox created from the same setup steps.

**Key Behaviors:**
1. **Deterministic setup**: Same `scenario.yml` always produces identical sandbox state
2. **Fast execution**: Setup steps run via Ruby system calls, not LLM interpretation
3. **Real fixtures**: Test data files live as actual files in `fixtures/`, not heredocs
4. **Independent sandboxes**: Every TC gets its own sandbox directory (no shared state)
5. **Debuggable**: `ace-test-e2e setup` creates sandbox without running tests, so developer can inspect

### Interface Contract

```bash
# CLI: Setup-only subcommand (new)
ace-test-e2e setup <package> <scenario-id>
# Creates sandbox, runs setup steps, prints path
# Exit code: 0 on success, 1 on failure
# Output: sandbox path to stdout

# Example
ace-test-e2e setup ace-lint TS-LINT-001
# → .cache/ace-test-e2e/8xyz12-lint-ts001/
```

```yaml
# scenario.yml — Declarative setup specification
test-id: TS-LINT-001
title: Ruby Validator Fallback Behavior
area: lint
package: ace-lint
priority: high
requires:
  tools: [standardrb, rubocop]
  ruby: ">= 3.0"

setup:
  - git-init
  - copy-fixtures
  - run: git add -A && git commit -m "initial" --quiet
  - write-file:
      path: .config/settings.yml
      content: |
        verbose: true
  - env:
      PROJECT_ROOT_PATH: .
```

**Error Handling:**
- Missing `scenario.yml`: Clear error message with expected path
- Invalid YAML in `scenario.yml`: Parse error with line number
- Missing required fields (`test-id`, `title`, `area`): Validation error listing missing fields
- Missing `fixtures/` directory when `copy-fixtures` step present: Error with hint
- `run` step failure: Non-zero exit code reported with stderr output
- No `TC-*.tc.md` files found: Warning (setup can still run, but no test cases to execute)

**Edge Cases:**
- Scenario with no setup steps: Valid — sandbox created but empty
- Scenario with no fixtures: Valid — `copy-fixtures` step not required
- `run` step with multi-line command: YAML block scalar (`|`) supported
- Nested fixture directories: Entire tree preserved (e.g., `fixtures/lib/helper.rb`)
- `env` step: Variables available to subsequent `run` steps in setup

## Implementation Plan

### 1. Technical Approach

**ATOM Architecture Placement:**

| Component | Layer | Rationale |
|-----------|-------|-----------|
| `TestCase` model | Models | Pure data object — no I/O, no side effects |
| `TestScenario` model (updates) | Models | Adding new fields to existing pure data model |
| `FixtureCopier` atom | Atoms | Pure function: given source + dest paths, copies tree. Could be made pure by accepting file list, but the copy operation itself is the atom's purpose. Actually this performs filesystem I/O (FileUtils.cp_r), so it is a **Molecule** by the existing codebase convention (see ScenarioParser comment: "Molecule because it performs filesystem I/O"). Reclassify to **Molecules** layer. |
| `ScenarioLoader` molecule | Molecules | Performs filesystem I/O: reads `scenario.yml`, discovers `TC-*.tc.md` files via Dir.glob |
| `SetupExecutor` molecule | Molecules | Performs filesystem I/O and system calls: creates dirs, runs commands, writes files |
| `Setup` CLI command | CLI/Commands | Follows existing `RunTest`/`RunSuite` pattern with `Dry::CLI::Command` |

**Correction on FixtureCopier**: Following the codebase convention where any component performing filesystem I/O is a Molecule (not Atom), `FixtureCopier` should be placed in `molecules/` since it calls `FileUtils.cp_r`. The behavioral spec says "atom" but the ATOM architecture rules in this codebase classify it as a molecule. We will place it in `molecules/` for consistency.

**Integration with existing TestDiscoverer:**
- `TestDiscoverer` currently only finds `*.mt.md` files. In this subtask, we do NOT modify `TestDiscoverer` (that's Subtask 3).
- `ScenarioLoader` is a new, independent molecule that handles `TS-*/scenario.yml` discovery.
- Both can coexist: `TestDiscoverer` for legacy `.mt.md`, `ScenarioLoader` for new `TS-*/scenario.yml`.

**TestScenario model backward compatibility (MT- and TS- formats):**
- The `TestScenario` model currently has a `short_id` method that only handles `MT-{AREA}-{NNN}` format.
- Add support for `TS-{AREA}-{NNN}` format: `short_id` returns `"ts001"` for `"TS-LINT-001"`.
- New optional fields: `setup_steps` (Array, default `[]`), `dir_path` (String, nil), `fixture_path` (String, nil), `test_cases` (Array<TestCase>, default `[]`).
- All new fields are optional with defaults, so existing code constructing `TestScenario` for MT- files continues to work unchanged.
- `test_case_ids` method: for TS- scenarios, derive from `test_cases` array instead of parsing markdown content.

#### Step 5: SetupExecutor Molecule

Create the deterministic setup executor.

**Files to create:**
- `lib/ace/test/end_to_end_runner/molecules/setup_executor.rb`
- `test/molecules/setup_executor_test.rb`

**Implementation details:**
- `execute(setup_steps:, sandbox_dir:, fixture_source: nil)`:
  - `FileUtils.mkdir_p(sandbox_dir)`
  - Maintains `env` hash (starts from `ENV.to_h` or subset)
  - Iterates steps, dispatching by type:
    - String `"git-init"` → `handle_git_init`
    - String `"copy-fixtures"` → `handle_copy_fixtures`
    - Hash `{ "run" => cmd }` → `handle_run`
    - Hash `{ "write-file" => { "path" => ..., "content" => ... } }` → `handle_write_file`
    - Hash `{ "env" => { ... } }` → `handle_env`
  - Step parsing: `scenario.yml` setup is an array where each element is either a string (shorthand actions) or a hash (parameterized actions)
- `handle_git_init(sandbox_dir, env)`:
  - `system("git", "init", chdir: sandbox_dir)` or `Open3.capture3`
  - `system("git", "config", "user.name", "Test User", chdir: sandbox_dir)`
  - `system("git", "config", "user.email", "test@example.com", chdir: sandbox_dir)`
- `handle_copy_fixtures(sandbox_dir, fixture_source)`:
  - Raises if `fixture_source` is nil
  - Delegates to `FixtureCopier.new.copy(source_dir: fixture_source, target_dir: sandbox_dir)`
- `handle_run(command, sandbox_dir, env)`:
  - `Open3.capture3(env, command, chdir: sandbox_dir)` — shell execution
  - Raises on non-zero exit with stderr
- `handle_write_file(path, content, sandbox_dir)`:
  - Resolves path relative to sandbox: `File.join(sandbox_dir, path)`
  - `FileUtils.mkdir_p(File.dirname(full_path))`
  - `File.write(full_path, content)`
- `handle_env(vars, env)`:
  - Merges string key-value pairs into env hash
  - Does not modify process ENV (only the local env hash passed to subsequent `run` steps)
