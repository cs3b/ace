# Orchestration + Suite Support

## Behavioral Specification

### User Experience

**Input:**
- Developer runs `ace-test-e2e run ace-lint` (package-level) or `ace-test-e2e suite` (multi-package)
- Optional filters: `--test-id TS-LINT-001`, `--test-cases TC-001,TC-003`, `--only-failures`

**Process:**
- `TestDiscoverer` scans for `TS-*/scenario.yml` directories (new format)
- `TestOrchestrator` loops over discovered scenarios, calling SetupExecutor per-TC then TestExecutor
- `SuiteOrchestrator` manages TC-level execution queue with parallel subprocess isolation
- `FailureFinder` tracks failures at `{scenario}/{tc-id}` granularity
- Display managers show `TS-LINT-001/TC-001` format in progress output

**Output:**
- Per-TC results aggregated into scenario-level and suite-level reports
- Failure tracking at TC level for `--only-failures` re-runs
- Progress display showing individual TC status

### Expected Behavior

**Current Pain Point:**
TestOrchestrator and SuiteOrchestrator operate at scenario (`.mt.md`) level. Parallelism is per-scenario, not per-TC. Failure re-runs send the entire scenario to the LLM even if only one TC failed.

**Desired Behavior:**
Orchestration operates at TC level. Each TC gets its own sandbox, execution, and report. Parallelism is per-TC (finer granularity). Failure re-runs target specific TCs within a scenario.

**Key Behaviors:**
1. **TC-level orchestration**: TestOrchestrator creates sandbox per-TC, executes per-TC
2. **TC-level discovery**: TestDiscoverer finds `TS-*/scenario.yml` dirs, loads TC files
3. **TC-level parallelism**: SuiteOrchestrator queues individual TCs for parallel execution
4. **TC-level failure tracking**: FailureFinder stores `{scenario_id}/{tc_id}` in metadata
5. **TC-level display**: Progress shows `TS-LINT-001/TC-001: pass` format

### Interface Contract

```ruby
# TestDiscoverer — updated for directory-based format
discoverer.discover(package: "ace-lint")
# Returns: [TestScenario, ...] each with .test_cases => [TestCase, ...]
# Scans: {package}/test-e2e/scenarios/TS-*/scenario.yml

discoverer.discover(package: "ace-lint", test_id: "TS-LINT-001")
# Returns: [TestScenario] filtered by test_id
```

```ruby
# TestOrchestrator — TC-level execution
orchestrator.run(
  package: "ace-lint",
  test_id: "TS-LINT-001",      # Optional: specific scenario
  test_cases: ["TC-001"],       # Optional: specific TCs
  cli_args: {},
  run_id: "8xyz12"
)
# For each TC:
#   1. SetupExecutor.setup(scenario, sandbox_path)
#   2. TestExecutor.execute(test_case, sandbox_path, scenario)
#   3. ReportWriter.write(result, report_dir)
```

```ruby
# SuiteOrchestrator — TC-level queue
suite.run(
  packages: ["ace-lint", "ace-review"],
  parallel: 3,                   # Max concurrent TCs
  only_failures: false
)
# Discovers all TCs across packages, queues for parallel execution
```

```ruby
# FailureFinder — TC-level tracking
finder.find_failures(package: "ace-lint")
# Returns: { "TS-LINT-001" => ["TC-001", "TC-003"], "TS-LINT-002" => ["TC-005"] }

finder.find_failures_by_scenario
# Returns: { "ace-lint" => { "TS-LINT-001" => ["TC-001"] }, ... }
```

## Implementation Plan

### 1. Technical Approach

**Dual-Mode Discovery Strategy**

`TestDiscoverer` must support both legacy `.mt.md` files and new `TS-*/scenario.yml` directories during the transition period. The strategy is:

1. `find_tests` gains a second glob pattern: `{package}/test-e2e/scenarios/TS-*/scenario.yml`
2. For each matching `scenario.yml` directory, `ScenarioLoader` (from 261.01) loads the full `TestScenario` with `TestCase` models
3. A new method `find_scenarios` returns loaded `TestScenario` objects (not just file paths) for the TS- format
4. Existing `find_tests` continues returning `.mt.md` file paths for backward compatibility
5. Both result sets are merged when callers need the full list (e.g., `list_packages`)

The caller (TestOrchestrator/SuiteOrchestrator) dispatches based on the scenario format:
- `scenario.scenario_format == :mt` -- legacy path: `executor.execute(scenario)`
- `scenario.scenario_format == :ts` -- new path: per-TC loop with `SetupExecutor` + `executor.execute_tc`

**Per-TC Orchestration Flow (TestOrchestrator)**

For a TS-format scenario with N test cases:

```
TestOrchestrator.run(package, test_id, test_cases)
  |
  +--> TestDiscoverer.find_scenarios(package, test_id)
  |      returns [TestScenario] with .test_cases => [TestCase, ...]
  |
  +--> For each TestScenario:
  |      |
  |      +--> Filter TCs by test_cases arg (if provided)
  |      |
  |      +--> For each TestCase (parallel via thread pool):
  |      |      |
  |      |      +--> Generate sandbox path: {ts}-{pkg}-{scenario}-{tc}/
  |      |      +--> SetupExecutor.execute(scenario.setup_steps, sandbox_dir, fixture_source)
  |      |      +--> TestExecutor.execute_tc(test_case, sandbox_path, scenario)
  |      |      +--> ReportWriter.write(result, scenario, report_dir, test_case)
  |      |      +--> Return TestResult for this TC
  |      |
  |      +--> Aggregate TC results into scenario-level TestResult
  |
  +--> Display.show_summary(aggregated results)
```

**Parallel Execution Model (TC-Level Queue)**

The current `SuiteOrchestrator` builds a flat queue of `{package, test_file}` items and spawns subprocesses for parallel execution. For TS-format scenarios, the queue granularity changes to TC-level:

- Queue items become `{package, scenario_dir, tc_id}` (or the full `TestCase` + `TestScenario` reference)
- Each subprocess runs: `ace-test-e2e run {package} {scenario_id} --test-cases {tc_id} --run-id {id}`
- The `--test-cases` flag limits execution to a single TC within the scenario
- This naturally maps each TC to its own sandbox (created by the subprocess's TestOrchestrator)

For the sequential (in-process) path in `SuiteOrchestrator.run_sequential`, the approach is similar: instead of calling `run_single_test(package, test_file)` once per `.mt.md` file, it calls it once per TC within each TS-format scenario.

**Important design decision**: `SuiteOrchestrator` continues to use subprocess isolation (spawning `ace-test-e2e` subprocesses) for parallel execution. The TC-level granularity is achieved by passing `--test-cases {tc_id}` to each subprocess, not by having the suite orchestrator directly call `SetupExecutor` + `TestExecutor.execute_tc`. This preserves the existing isolation model and avoids mixing in-process and subprocess execution patterns.

**Failure Tracking Schema Change**

The per-TC report metadata (written by 261.02's updated `ReportWriter`) now includes `scenario-id` and `tc-id` fields in `metadata.yml`. `FailureFinder` already reads these files and groups by `test-id`. The changes needed are:

1. `FailureFinder` already supports composite `scenario_id/tc_id` tracking via the `find_failures_by_scenario` method -- it reads `test-id` and `failed_test_cases` from metadata
2. For TS-format per-TC reports, each `metadata.yml` has exactly one TC result. The `failed_test_cases` array will contain at most one entry (the TC itself), or be empty on pass
3. `FailureFinder` needs to also read the `scenario-id` field from per-TC metadata to correctly group results. Currently it groups by `test-id`, which for per-TC reports is the scenario ID (since `ReportWriter` in 261.02 still writes `test-id` as the scenario ID)
4. No schema change is strictly required -- the existing `find_failures_by_scenario` already produces `{ package => { test-id => [tc-ids] } }`, which works for both legacy and new formats

**Display Format Updates**

Display managers need to handle TS-format test cases:
- `SimpleDisplayManager` / `ProgressDisplayManager`: Currently keyed on `scenario.test_id`. For TS-format scenarios with multiple TCs, each TC becomes a separate display row: `TS-LINT-001/TC-001: running`, `TS-LINT-001/TC-002: pass`
- `SuiteSimpleDisplayManager` / `SuiteProgressDisplayManager`: Currently use `File.basename(test_file, ".mt.md")` for test names. For TS-format, the test name should be `{scenario_id}/{tc_id}` (e.g., `TS-LINT-001/TC-001`)
- The `extract_test_name` methods need to be updated to handle both `*.mt.md` files and `TS-*/scenario.yml` paths

  - MT-format files: existing `run_single_test` / `run_package_tests` path (unchanged)
  - TS-format scenarios: new `run_ts_scenario` / `run_ts_package` path

**New private method: `run_ts_scenario(scenario, timestamp, cli_args, output, test_cases:)`**
- For a single TS-format scenario:
  1. Filter TCs by `test_cases` arg (if provided)
  2. For each TC:
     a. Generate sandbox path: `{cache_dir}/{ts}-{pkg}-{scenario_short_id}-{tc_short_id}/`
     b. Call `@setup_executor.execute(setup_steps: scenario.setup_steps, sandbox_dir: sandbox_path, fixture_source: scenario.fixture_path)`
     c. Call `@executor.execute_tc(test_case: tc, sandbox_path: sandbox_path, scenario: scenario, cli_args: cli_args, run_id: run_id)`
     d. Call `@report_writer.write(result, scenario, report_dir: report_dir, test_case: tc)` (for API providers)
  3. Aggregate per-TC results into scenario-level `TestResult`
  4. Return `[scenario_result]`

**New private method: `run_ts_package(scenarios, package, timestamp, cli_args, output, test_cases:)`**
- Like `run_package_tests` but operates on `TestScenario` objects with per-TC execution
- Builds a flat queue of `[scenario, test_case]` pairs
- Uses thread pool (same pattern as `run_package_tests`) for parallel TC execution
- Each thread: setup sandbox, execute TC, write reports
- Display uses per-TC rows: `{scenario.test_id}/{tc.tc_id}`

**New private method: `aggregate_tc_results(scenario, tc_results)`**
- Combines per-TC `TestResult` objects into a single scenario-level `TestResult`
- Status: "pass" if all TCs pass, "fail" if any TC fails, "error" if any TC errors
- `test_cases` array: each TC mapped to `{ id: tc_id, status: tc_status, ... }`
- Summary: "N/M passed"

**New private method: `sandbox_dir_for(scenario, test_case, timestamp)`**
- Returns: `{cache_dir}/{timestamp}-{short_pkg}-{scenario_short_id}-{tc_short_id}/`
- Uses `TestScenario#short_id` and `TestCase#short_id` from 261.01

**New private method: `tc_report_dir_for(scenario, test_case, timestamp)`**
- Returns: `{cache_dir}/{timestamp}-{short_pkg}-{scenario_short_id}-{tc_short_id}-reports/`
