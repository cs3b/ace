# TC-Level Execution Pipeline

## Behavioral Specification

### User Experience

**Input:**
- A `TestScenario` model (from ScenarioLoader, subtask 01) with associated `TestCase` models
- A populated sandbox directory (from SetupExecutor, subtask 01)

**Process:**
- `TestExecutor` receives a single `TestCase` and its sandbox path
- `PromptBuilder` / `SkillPromptBuilder` constructs TC-level prompts (not full scenario)
- LLM executes only the steps in the individual TC file
- `ReportWriter` generates per-TC reports with `{scenario-id}/{tc-id}` metadata

**Output:**
- Per-TC execution results (`TestResult` model with TC-level granularity)
- Per-TC report files: `summary.r.md`, `experience.r.md`, `metadata.yml`
- Report directory: `.cache/ace-test-e2e/{timestamp}-{pkg}-{scenario}-{tc}-reports/`

### Expected Behavior

**Current Pain Point:**
TestExecutor sends the entire `.mt.md` content (including all TCs and setup) to the LLM. The LLM runs all TCs sequentially in a single session, making it impossible to re-run individual TCs or parallelize execution.

**Desired Behavior:**
TestExecutor sends only a single TC's content to the LLM. The sandbox is already set up (by SetupExecutor). The LLM only needs to execute the test steps and report results. This enables per-TC independence, parallelism, and targeted re-runs.

**Key Behaviors:**
1. **Single TC prompts**: PromptBuilder includes only the TC content, not the full scenario
2. **Sandbox pre-populated**: TC execution assumes sandbox already exists (setup done by SetupExecutor)
3. **Per-TC reports**: Each TC gets its own report directory with `{scenario-id}/{tc-id}` in metadata
4. **Simplified workflow**: `run-e2e-test.wf.md` becomes simpler — no setup heredocs, just TC steps

### Interface Contract

```ruby
# TestExecutor — updated signature
executor.execute(
  test_case: test_case,          # TestCase model (single TC)
  sandbox_path: "/path/to/sandbox",
  scenario: test_scenario,       # For context metadata
  provider: "claude:sonnet",
  cli_args: {}
)
# Returns: TestResult with single TC result
```

```ruby
# PromptBuilder — TC-level prompt
PromptBuilder.build(
  test_case: test_case,          # Single TC content
  scenario: test_scenario,       # For metadata context
  sandbox_path: sandbox_path
)
# Returns: { system: "...", user: "..." }
```

```ruby
# ReportWriter — per-TC reports
ReportWriter.write(
  result: test_result,
  report_dir: ".cache/ace-test-e2e/{ts}-{pkg}-{scenario}-{tc}-reports/"
)
# Generates: summary.r.md, experience.r.md, metadata.yml
# metadata.yml includes: scenario_id, tc_id, sandbox_path
```

## Implementation Plan

### 1. Technical Approach

**Core Change: From Scenario-Level to TC-Level LLM Invocations**

Currently, `TestExecutor.execute(scenario)` sends the entire `.mt.md` content (including environment setup, test data, and all TCs) to the LLM as a single prompt. The LLM performs setup, data creation, and all TC execution in one session.

In the new architecture (post-261.01), `SetupExecutor` handles sandbox creation deterministically in Ruby. Each `TestCase` is then sent individually to the LLM with only its specific content plus sandbox context. This means:

1. **Prompt content shrinks dramatically** -- from full scenario (setup + data + all TCs) to a single TC's objective/steps/expected results
2. **Sandbox path becomes a prompt parameter** -- the LLM must execute in the pre-populated sandbox, not create one
3. **Results are per-TC** -- each LLM invocation returns results for exactly one TC, not an array of TCs
4. **Workflow simplifies** -- `run-e2e-test.wf.md` for TC-level execution has no setup/data creation steps

**How TC-Level Prompts Differ from Scenario-Level Prompts:**

| Aspect | Scenario-Level (current) | TC-Level (new) |
|--------|--------------------------|----------------|
| Content sent | Full `.mt.md` (setup + data + all TCs) | Single TC markdown body only |
| Setup instructions | Included in prompt (heredoc scripts) | Omitted (pre-done by SetupExecutor) |
| Sandbox context | LLM creates sandbox from instructions | LLM receives `sandbox_path` parameter |
| Expected results format | JSON with `test_cases[]` array (multiple) | JSON with single `test_case` object (or `test_cases[]` with 1 entry) |
| Return contract (CLI) | Markdown with Passed/Failed/Total counts | Markdown with single TC status + details |
| Report directory | `{ts}-{pkg}-{scenario}-reports/` | `{ts}-{pkg}-{scenario}-{tc}-reports/` |

**Strategy for Maintaining Backward Compatibility:**

Both old (scenario-level, `.mt.md`) and new (TC-level, `.tc.md`) execution paths must coexist during the transition period (until 261.04 removes legacy support). The approach is:

1. **Add new methods alongside existing ones** -- `execute_tc` alongside `execute`, `build_tc` alongside `build`
2. **Do not modify existing method signatures** -- `execute(scenario)` continues to work identically
3. **Let the caller choose** -- `TestOrchestrator` (261.03) will call `execute_tc` for TS-format scenarios and `execute` for MT-format scenarios
4. **Result model is unchanged** -- `TestResult` works for both; for TC-level, it simply has a single-entry `test_cases` array
5. **ReportWriter gains an optional `tc_id` parameter** -- when provided, includes TC metadata; when omitted, behaves as before

**How Per-TC Reports Integrate with Existing Report Structure:**

Current report path: `.cache/ace-test-e2e/{ts}-{pkg}-{scenario}-reports/`
New TC report path: `.cache/ace-test-e2e/{ts}-{pkg}-{scenario}-{tc}-reports/`

Example directory tree after executing TS-LINT-001 with TC-001, TC-002:
```
.cache/ace-test-e2e/
  8xyz12-lint-ts001-tc001/              # TC-001 sandbox
  8xyz12-lint-ts001-tc001-reports/      # TC-001 reports
    summary.r.md
    experience.r.md
    metadata.yml                         # includes scenario_id + tc_id
  8xyz13-lint-ts001-tc002/              # TC-002 sandbox
  8xyz13-lint-ts001-tc002-reports/      # TC-002 reports
    summary.r.md
    experience.r.md
    metadata.yml
```

The `metadata.yml` for TC-level reports adds two new fields:
- `scenario-id`: The parent scenario ID (e.g., `TS-LINT-001`)
- `tc-id`: The specific test case ID (e.g., `TC-001`)

The `FailureFinder` (261.03 scope) will use `scenario-id/tc-id` composite keys for per-TC failure tracking.

**How the Workflow File Simplifies for TC Execution:**

The current `run-e2e-test.wf.md` is 763 lines covering everything from test discovery to sandbox setup to report writing. For TC-level execution, a new workflow section (or a separate lightweight workflow) will:

1. Skip steps 1-5 entirely (locate, read, verify, setup, test data) -- all pre-done
2. Receive: sandbox path, TC content, scenario metadata
3. Execute: only the TC steps in the pre-populated sandbox
4. Report: single TC result in the return contract format

The approach is to add a TC-level execution mode section within `run-e2e-test.wf.md` (gated by new arguments `--tc-mode`, `--sandbox-path`, `--tc-content`) rather than creating a separate workflow file, preserving a single source of truth.

### 2. File Modifications

#### Modify: `atoms/prompt_builder.rb`

**New method: `build_tc(test_case:, scenario:, sandbox_path:)`**

Builds a TC-level user prompt containing:
- Scenario metadata (test_id, package, title) for context
- The sandbox path where the test environment is pre-populated
- Only the single TC's markdown content (objective, steps, expected)
- Instructions to execute in the sandbox and return single-TC JSON results

Keeps existing `build(scenario, test_cases:)` method unchanged.

**New constant: `TC_SYSTEM_PROMPT`**

A TC-level system prompt that:
- Instructs the LLM it is executing a single test case (not a full scenario)
- Specifies the sandbox is pre-populated (no setup needed)
- Defines the JSON output format for a single TC result:
  ```json
  {
    "test_id": "TS-LINT-001",
    "tc_id": "TC-001",
    "status": "pass|fail",
    "actual": "What actually happened",
    "notes": "Observations",
    "summary": "Brief result"
  }
  ```
- Rules: execute in sandbox, do not modify sandbox setup, record actual results

#### Modify: `atoms/skill_prompt_builder.rb`

**New method: `build_tc_skill_prompt(test_case:, scenario:, sandbox_path:, run_id: nil)`**

For skill-aware providers (claude): generates a skill invocation command like:
```
/ace:run-e2e-test-tc ace-lint TS-LINT-001 TC-001 --sandbox /path/to/sandbox --run-id abc123
```

Note: This uses a new skill command `/ace:run-e2e-test-tc` (or the existing `/ace:run-e2e-test` with `--tc-mode` flag). The exact invocation depends on whether we add a new skill or extend the existing one.

**New method: `build_tc_workflow_prompt(test_case:, scenario:, sandbox_path:, workflow_content:, run_id: nil)`**

For non-skill-aware CLI providers (gemini, codex): embeds the TC-level workflow content plus TC content and sandbox path into a single prompt. Similar to `build_workflow_prompt` but with:
- TC content instead of full scenario content
- Sandbox path parameter
- Simplified workflow (no setup/data steps)
- Single-TC return contract

Keeps existing `build_skill_prompt` and `build_workflow_prompt` methods unchanged.

#### Modify: `atoms/result_parser.rb`

**New class method: `parse_tc(text)`**

Parses the TC-level JSON format which has:
- `tc_id` field (new, in addition to `test_id`)
- Single result (not array) -- though we normalize it to a single-entry `test_cases` array for compatibility with `TestResult`
- Same error handling as `parse` (ParseError on invalid/missing JSON)

Returns a hash in the same shape as `parse()` but always with exactly one entry in `test_cases`:
```ruby
{
  test_id: "TS-LINT-001",
  status: "pass",
  test_cases: [{ id: "TC-001", description: "", status: "pass", actual: "...", notes: "..." }],
  summary: "TC-001 passed",
  observations: ""
}
```

Falls back to `parse()` if the response contains the multi-TC format (backward compat for edge cases).

Keeps existing `parse(text)` method unchanged.

#### Modify: `atoms/skill_result_parser.rb`

**New class method: `parse_tc(text)`**

Parses TC-level markdown return contract:
```markdown
- **Test ID**: TS-LINT-001
- **TC ID**: TC-001
- **Status**: pass
- **Report Paths**: 8xyz12-lint-ts001-tc001-reports/*
- **Issues**: None
```

Falls back to `parse()` if the format matches the multi-TC contract instead.

Returns a hash compatible with `TestResult` (single-entry `test_cases` array).

Keeps existing `parse(text)` method unchanged.

#### Modify: `molecules/test_executor.rb`

**New public method: `execute_tc(test_case:, sandbox_path:, scenario:, cli_args: nil, run_id: nil)`**

Accepts:
- `test_case` -- a `Models::TestCase` instance (from 261.01)
- `sandbox_path` -- path to pre-populated sandbox directory
- `scenario` -- parent `Models::TestScenario` for metadata context
- `cli_args` -- optional CLI args
- `run_id` -- optional pre-generated run ID

Routes to `execute_tc_via_skill` or `execute_tc_via_prompt` based on provider type (same routing logic as `execute`).

Returns a `Models::TestResult` with:
- `test_id` set to `scenario.test_id`
- `test_cases` array with a single entry for this TC
- `status` reflecting this single TC's outcome

**New private method: `execute_tc_via_skill(test_case, sandbox_path, scenario, cli_args:, run_id:)`**

For CLI providers:
1. If skill-aware: `@skill_prompt_builder.build_tc_skill_prompt(test_case:, scenario:, sandbox_path:, run_id:)`
2. Otherwise: loads TC-level workflow content, calls `@skill_prompt_builder.build_tc_workflow_prompt(...)`
3. Calls `Ace::LLM::QueryInterface.query` with the prompt
4. Parses with `Atoms::SkillResultParser.parse_tc(response[:text])`
5. Returns `Models::TestResult`

**New private method: `execute_tc_via_prompt(test_case, sandbox_path, scenario, cli_args:)`**

For API providers:
1. Calls `@prompt_builder.build_tc(test_case:, scenario:, sandbox_path:)`
2. Calls `Ace::LLM::QueryInterface.query` with `system: Atoms::PromptBuilder::TC_SYSTEM_PROMPT`
3. Parses with `Atoms::ResultParser.parse_tc(response[:text])`
4. Returns `Models::TestResult`

Error handling mirrors `execute_via_skill`/`execute_via_prompt` patterns: catches `ParseError`, `Ace::LLM::Error`, and `StandardError`.

Keeps existing `execute(scenario)` method and all private methods unchanged.

#### Modify: `molecules/report_writer.rb`

**Updated `write` signature (backward-compatible):**

```ruby
def write(result, scenario, report_dir:, test_case: nil)
```

When `test_case` is provided (a `Models::TestCase` instance):

- Experience report references the specific TC
- Metadata YAML includes `scenario-id` and `tc-id` fields
- Metadata YAML includes `sandbox-path` field

When `test_case` is nil (default): behaves exactly as current implementation.

**Changes to `write_metadata`:**

```ruby
"tc-id" => test_case.tc_id,
"sandbox-path" => sandbox_path  # if available via result or parameter
```

**Changes to `write_summary`:**

When `test_case` is present:
- Frontmatter adds `tc-id:` and `scenario-id:` fields
- Test Information table adds TC ID and TC Title rows
- Results Summary shows single row for the one TC

#### Modify: `handbook/workflow-instructions/run-e2e-test.wf.md`

**Add new section: TC-Level Execution Mode**

Add a new gated section at the top of the workflow (before step 1) that activates when TC-mode arguments are provided:

```markdown
## TC-Level Execution Mode

When invoked with `--tc-mode`, `--sandbox-path`, and test case content, this workflow operates in TC-level mode. In this mode, steps 1-5 are skipped entirely and execution begins directly at TC execution.

### TC Arguments

- `SANDBOX_PATH` (required in TC mode) - Pre-populated sandbox path
- `TC_CONTENT` (required in TC mode) - The test case markdown content
- `SCENARIO_ID` (required in TC mode) - Parent scenario ID for metadata
- `TC_ID` (required in TC mode) - Test case ID
- `PACKAGE` (required in TC mode) - Package name for metadata
- `RUN_ID` (optional) - Pre-generated run ID

### TC Execution Steps

1. Verify sandbox exists at `SANDBOX_PATH`
2. Change to sandbox directory
3. Execute TC steps from `TC_CONTENT`
4. Write per-TC reports to `{SANDBOX_PATH}-reports/` (or `{RUN_ID}-{pkg}-{scenario}-{tc}-reports/`)
5. Return TC-level return contract:
   - **Test ID**: {scenario-id}
   - **TC ID**: {tc-id}
   - **Status**: pass | fail
   - **Report Paths**: {ts}-{pkg}-{scenario}-{tc}-reports/*
   - **Issues**: Brief description or "None"
```

The existing workflow steps (1-10) remain unchanged for legacy `.mt.md` execution.
