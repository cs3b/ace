# Simplified Feedback System - File-Based Output Architecture

## Overview

Transform review synthesis from a single monolithic document to individual feedback files. Each unique finding becomes a trackable `.s.md` file with its own lifecycle.

## Architecture

```
Step 1: ace-review --auto-execute
        ↓ produces review-report-*.md files (one per model)

Step 2: ace-review feedback create
        ↓ LLM synthesizes reviews → creates feedback/*.s.md files (one per finding)

Step 3: ace-review feedback verify <id> --valid|--invalid
        ↓ human/agent marks validity

Step 4: /ace:apply-feedback
        ↓ applies verified feedback, marks resolved
```

## Key Design Decisions

1. **LLM for extraction** - Reviewers have different formats; LLM needed to understand, correlate, and deduplicate findings across multiple review reports
2. **File-based feedback** - Each finding is a separate `.s.md` file (like tasks in ace-taskflow)
3. **Session-scoped** - Feedback lives in session directory: `.cache/ace-review/sessions/{session}/feedback/`
4. **Latest session default** - `feedback create` uses most recent session when no args provided

## Feedback File Format

**Filename**: `{6-char-id}-{slug}.s.md`

```yaml
---
id: 8p3abc
title: SQL Injection in User Handler
status: draft          # draft → pending → done | invalid | skip
priority: critical     # critical | high | medium | low
files:
  - src/user.rb:42
reviewers:             # Multiple reviewers if they found same issue
  - google:gemini-2.5-flash
  - anthropic:claude-sonnet
consensus: true        # true if 3+ reviewers agree
created: 2026-02-03T18:00:00Z
updated: 2026-02-03T18:00:00Z
---

## Finding

User input is passed directly to SQL query without sanitization.

## Context

This was identified by 2 of 4 reviewers. The `execute_query` method
at line 42 concatenates user input directly into the SQL string.

## Research

{Populated during verification - validation notes}

## Resolution

{Populated when resolved - how it was fixed}
```

## Status Workflow

```
draft (initial extraction from LLM)
  ↓
  ├─→ pending (verified valid, ready to fix)
  │     ↓
  │     └─→ done (fixed) → _archived/
  │
  ├─→ invalid (false positive) → _archived/
  │
  └─→ skip (out of scope) → _archived/
```

## Components to Implement

### 1. CLI Command: `feedback create`

**File**: `ace-review/lib/ace/review/cli/commands/feedback/create.rb`

```ruby
# Uses most recent session by default
# Finds review-report-*.md files
# Calls FeedbackManager.extract_and_save()
# Which uses FeedbackSynthesizer (LLM) + FeedbackFileWriter

option :session, type: :string   # Explicit session path
option :task, type: :string      # Task-based resolution
option :model, type: :string     # Model for synthesis
```

**Behavior**:
- No args → find latest session in `.cache/ace-review/sessions/`
- Find `review-report-*.md` files (exclude `review-dev-feedback.md`)
- Call existing `FeedbackManager.extract_and_save()`
- Display created files count

### 2. Workflow: `/ace:apply-feedback`

**Skill**: `.claude/skills/ace_apply-feedback/SKILL.md`
**Workflow**: `ace-review/handbook/workflow-instructions/apply-feedback.wf.md`

**Steps**:
1. List pending items: `ace-review feedback list --status pending`
2. For each item by priority (Critical → High → Medium):
   - Show details: `ace-review feedback show <id>`
   - Verify issue exists in code
   - Implement fix
   - Mark resolved: `ace-review feedback resolve <id> --resolution "..."`
3. Skip items with reason if not applicable
