# 150.37: Refactor ace-taskflow to Accept Options Hashes Directly

## Context (Technical Debt)

The current ace-taskflow implementation uses a hybrid approach where Thor options are converted back to an ARGV array via `options_to_args` to support legacy command classes. This translation layer should eventually be removed.

## Current State

- Thor CLI accepts options as a Hash via `options`
- Command classes convert this back to ARGV-style strings
- Underlying Organisms expect string arrays from legacy parsing

## Problem

```ruby
# Current: CLI (Thor options Hash) -> Command (ARGV strings) -> Organism (ARGV strings)
# Ideal:   CLI (Thor options Hash) -> Command (options Hash) -> Organism (options Hash)
```

The double conversion adds complexity and obscures the actual option flow.

## Proposed Solution

### Phase 1: Update Organisms to Accept Options Hash

1. Refactor Organisms to accept keyword arguments instead of ARGV arrays
2. Maintain backward compatibility during transition

```ruby
# Before
Organisms::TaskLoader.load(["--status", "pending"])

# After (with backward compat)
Organisms::TaskLoader.load(status: "pending")  # New way
Organisms::TaskLoader.load(["--status", "pending"])  # Still works
```

### Phase 2: Update Commands to Pass Options Directly

```ruby
# Before
def execute(args, options)
  argv = @options_to_args.call(options)
  Organisms::TaskLoader.load(argv)
end

# After
def execute(args, options)
  Organisms::TaskLoader.load(**options.slice(:status, :limit, :format))
end
```

### Phase 3: Remove Translation Layer

Remove `options_to_args` helpers once all Organisms accept options Hashes.

## Deliverables

1. Identify all Organisms that need refactoring
2. Create options mapping for each Command -> Organism boundary
3. Implement keyword argument acceptance in Organisms
4. Remove `options_to_args` translation layer
5. Update tests

## Impact

- Reduced code complexity
- Clearer option flow
- Better type safety (Hash vs string parsing)
- Easier to add new options
