# Fix: Thor default_task and method_missing pattern for CLI backward compatibility

## Description

**Unplanned fix task**: After Thor migration in Task 150, discovered that packages using `default_task` didn't support calling the default command without explicitly naming it.

This is a backward compatibility issue that affects user experience - users expect to call commands like `ace-nav wfi://workflow` without specifying the subcommand name.

## Issue

After Thor migration in Task 150, packages using `default_task` didn't support calling the default command without explicitly naming it.

**Example failures**:
- `ace-nav wfi://update-pr-description` → "Could not find command"
- `ace-git-commit ace-review/` → "Could not find command"
- `ace-taskflow retro` → "Action required" error (exit 1)

## Root Cause

Thor's `default_task` only handles the case of NO arguments provided to the CLI. When a positional argument is provided (e.g., `ace-nav wfi://...`), Thor treats it as a command name and tries to find a matching command, which fails.

Example:
```ruby
class CLI < Thor
  default_task :resolve  # Only handles: ace-nav (no args)
                       # Does NOT handle: ace-nav wfi://...
end
```

## Solution

Added `method_missing` handlers to 9 CLI packages that delegate unknown "commands" to their default task using Thor's `invoke` method.

**Pattern**:
```ruby
def method_missing(command, *args)
  invoke :default_command, [command.to_s] + args
end
```

Also fixed ace-taskflow retro command to show active retros when called without args (consistency with task/idea/release commands).
