# Parent Task as Subtask Context

## Behavioral Specification

### User Experience
- **Input**: Orchestrator agent delegates a subtask via the work-subtasks workflow (no change to invocation)
- **Process**: The delegation prompt instructs the subagent to read the parent (orchestrator) task for context *in addition to* its own subtask spec
- **Output**: Subagent has access to the parent's "why" and overall scope, enabling it to make better judgment calls during implementation

### Expected Behavior

When the work-subtasks workflow delegates to a subagent (step 3.2), the delegation prompt should include an instruction to read the orchestrator task for context:

**Current behavior** (step 3.2 delegation prompt):
```
2. Read task: `ace-task 122.01`
3. Implement changes per task file
```

**Target behavior**:
```
2. Read parent task for context: `ace-task 122`
3. Read your subtask: `ace-task 122.01`
4. Implement changes per subtask file
```

The parent task provides:
- **Why** the overall work exists (orchestrator Objective/Overview)
- **Scope boundaries** (what other subtasks handle, so this agent doesn't duplicate or conflict)
- **Concept inventory** (what concepts are being introduced/removed across subtasks)

The subtask provides:
- **What specific work** to do (behavioral spec, interface contract, success criteria)

**Reference-based, not inline embedding**: The subagent reads the parent task via `ace-task <id>` at runtime. The delegation prompt does NOT paste the parent's content inline — this keeps the prompt lightweight and ensures the subagent always gets the current version of the parent spec.

### Interface Contract

The workflow instruction file (`work-subtasks.wf.md`) is the interface. No CLI or API changes.

**Step 3.2 delegation prompt change** — add one line to the Steps section:

```markdown
## Steps

1. `cd <WORKTREE_PATH>`
2. Read parent task for context: `ace-task <orchestrator-id>`
3. Read your subtask: `ace-task <subtask-id>`
4. Implement changes per subtask file
5. Run tests: `ace-test`
6. Commit on subtask branch
7. Return: changes made, files modified, test results
```

And add a brief explanation after the Steps block:

```markdown
The parent task provides the "why" and overall scope.
Your subtask provides the specific work to do.
If your subtask's requirements seem to conflict with the parent's
intent, flag this in your return summary rather than guessing.
```

**Error Handling:**
- Parent task doesn't exist or `ace-task` fails: Subagent proceeds with subtask spec only (graceful degradation, same as current behavior)
- Parent task has no useful overview: Subagent proceeds normally — reading an empty overview is harmless

**Edge Cases:**
- Deeply nested subtasks (e.g., 122.01.01): Read the immediate parent (122.01), not the grandparent — one level of context is sufficient
- Orchestrator is a single flat task with no subtasks: This workflow doesn't apply (work-subtasks is only for orchestrators)

### Success Criteria

- [ ] Step 3.2 delegation prompt in `work-subtasks.wf.md` includes `ace-task <orchestrator-id>` as a context-reading step
- [ ] Parent task is read *before* the subtask (context-first ordering)
- [ ] Prompt includes brief guidance on how to use parent context vs. subtask spec
- [ ] No inline embedding of parent content — reference-based only
- [ ] Existing subtask delegation still works if parent read fails (graceful degradation)

### Validation Questions

- [x] Should the subagent read the parent first or the subtask first? → **Parent first — establishes context before diving into specifics**
- [x] Should deeply nested tasks read the full chain? → **No — one level of parent context is sufficient**
- [x] Should the parent context be a hard requirement or optional? → **Optional — if `ace-task` fails, proceed with subtask only**

## Objective

Give subtask agents awareness of the parent task's intent and scope. Currently, subagents see only their own spec and generic project context — they have no visibility into *why* the orchestrator exists or what sibling subtasks handle. Adding a single `ace-task <parent-id>` read to the delegation prompt closes this gap with minimal complexity.

## Scope of Work

- **User Experience Scope**: Subagents make better-informed decisions by understanding the parent's intent
- **System Behavior Scope**: Delegation prompt text changes only — no code changes
- **Interface Scope**: `work-subtasks.wf.md` step 3.2 delegation prompt

- Changes to worktree creation or PR workflows
- Multi-level parent chain traversal
