# Spec-Aware Planning at the Spec-to-Agent Handoff

## Behavioral Specification

### User Experience
- **Input**: No CLI or user-facing input changes. This subtask modifies workflow instructions read by agents (spec reviewers and planning agents)
- **Process**: Review workflow gains an "Operating Modes Covered" checklist item; planning workflow is updated to plan *against* the behavioral spec format (not just read it); ace-assign planning prompt teaches agents to use the spec structure for higher-quality plans
- **Output**: Planning agents produce plans that systematically address each behavioral spec section (interface contract, error handling, edge cases, operating modes) rather than cherry-picking what seems relevant. Gaps are surfaced as a natural byproduct of thorough spec-driven planning

### Expected Behavior

#### Review Gate (`review.wf.md`)

The spec review readiness checklist should include a check for operating modes:

**Current behavior**: The readiness checklist covers structural completeness (sections present, criteria verifiable, etc.) but does not prompt reviewers to check whether the spec addresses operating modes like `--dry-run`, `--force`, `--verbose`, `--quiet`.

**Target behavior**: Add an "Operating Modes Covered" item to the readiness checklist:

```markdown
- [ ] Operating modes covered — spec addresses relevant modes (dry-run, force, verbose, quiet) or explicitly marks them out-of-scope
- [ ] Degenerate inputs covered — spec considers identity operations (X=Y), empty inputs, and self-referential calls where the same entity appears in both argument positions
- [ ] Per-path variations covered — if the spec says "same behavior for X and Y", edge cases unique to each path are enumerated (guard logic, error handling, parameter differences)
```

Reviewers seeing these items will either:
1. Confirm the spec covers the concern, or
2. Flag the gap back to the spec author before the spec is approved

#### Planning Workflow (`plan.wf.md`)

The planning workflow's step 1 (read and understand the spec) should teach the agent to plan *against* the behavioral spec structure:

**Current behavior**: The planning agent reads the spec and proceeds to plan. It treats the spec as background context — cherry-picking details that seem relevant and silently skipping sections it doesn't know how to address.

**Target behavior**: Update step 1 to instruct the agent to use the spec's structure as a planning checklist:

```markdown
1.x Plan against the behavioral spec — for each section in the spec (Interface
    Contract, Error Handling, Edge Cases, Success Criteria), ensure your plan
    addresses it. Check that your plan covers all operating modes the spec
    defines (dry-run, force, verbose, quiet). If the spec omits a section or
    mode that your plan needs, note it in a "Behavioral Gaps" section at the
    end of your plan rather than silently working around it.

1.y When the spec covers multiple code paths with "same behavior" (e.g.,
    "same behavior for ace-task move"), enumerate each path explicitly in the
    plan and note any per-path divergences: guard logic differences, dry-run
    interaction, error handling variations, parameter differences. Do not
    assume one path's implementation generalizes — plan each path separately.
```

This makes the behavioral spec the *skeleton* of the plan rather than just input to it. Gap reporting falls out naturally — if the agent is systematically walking the spec sections and something is missing, it reports that. Per-path enumeration prevents the asymmetric implementation bug where one code path gets correct guard logic and the other silently omits it.

#### Planning Prompt in ace-assign (`assignment_executor.rb`)

The `plan-task` phase instruction (around line 651) should teach spec-driven planning:

**Current behavior**: The planning prompt tells the agent to read the spec and produce a plan. It doesn't tell the agent *how* to use the spec structure.

**Target behavior**: Add to the planning prompt:

```
Plan against the behavioral spec structure: walk each section (Interface
Contract, Error Handling, Edge Cases, operating modes) and ensure your plan
addresses it. If the spec is missing something your plan needs, list it in a
"Behavioral Gaps" section rather than silently working around the omission.
```

### Interface Contract

No CLI or API changes. The interfaces are the workflow instruction files and the planning prompt text.

**Review workflow** (`review.wf.md`): One new checklist item in the readiness section.

**Planning workflow** (`plan.wf.md`): One new sub-step in step 1.

**Planning prompt** (`assignment_executor.rb`): One new paragraph in the `plan-task` phase instruction.

**Error Handling:**
- Planning agent doesn't find any gaps: No "Behavioral Gaps" section is produced — this is expected for complete specs
- Reviewer finds modes aren't relevant (e.g., pure doc change): Checks the item as "out-of-scope" — the checklist item is a prompt, not a hard gate

**Edge Cases:**
- Specs for non-CLI features (e.g., workflow instruction changes): "Operating modes" doesn't apply — reviewer marks as N/A
- Planning agent over-reports gaps: Better than under-reporting — spec author can dismiss false positives

## Objective

Close the 4th intent loss point in the ACE pipeline: **spec → planning agent**. The task-282 experiment showed that planning agents treat behavioral specs as background context — they cherry-pick what seems useful and silently skip the rest. The fix is to make the spec the *skeleton* of the plan: agents walk the spec section-by-section (interface contract, error handling, edge cases, operating modes) and plan against each one. Gap reporting is a natural side effect — when the agent systematically covers the spec and finds something missing, it says so instead of silently assuming it doesn't matter.

## Scope of Work

- **User Experience Scope**: No user-facing changes — agents and reviewers see updated instructions
- **System Behavior Scope**: Workflow instruction text and planning prompt text changes only
- **Interface Scope**: `review.wf.md`, `plan.wf.md`, `assignment_executor.rb`

- Automated gap detection tooling — this is a prompt-level change only
- Changes to the behavioral spec template itself (that could be a follow-up)
- Enforcement (blocking plans that don't include a gaps section)
