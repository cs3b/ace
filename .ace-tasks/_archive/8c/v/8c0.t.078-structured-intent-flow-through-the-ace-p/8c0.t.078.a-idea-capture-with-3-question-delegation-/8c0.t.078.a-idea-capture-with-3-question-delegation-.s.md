# Idea Capture with 3-Question Delegation Brief

## Behavioral Specification

### User Experience
- **Input**: User runs `ace-idea capture "my idea" --llm-enhance` (no change to CLI interface)
- **Process**: LLM enhancement produces 3 structured intent sections instead of the current Problem/Solution/Benefits format
- **Output**: Enhanced idea file with `## What I Hope to Accomplish`, `## What "Complete" Looks Like`, `## Success Criteria` sections

### Expected Behavior

When a user enhances an idea with `--llm-enhance`, the LLM output should contain exactly 3 sections that map to the 3-Question Delegation Brief:

- **`## What I Hope to Accomplish`** — the impact, the *why*. Captures the change in the world the idea is after, not the steps to get there.
- **`## What "Complete" Looks Like`** — concrete end state. The reader should be able to compare an implementation against this and know whether it's done.
- **`## Success Criteria`** — verifiable pass/fail checks. Each criterion should be something an agent or reviewer can mechanically verify.

These 3 sections **replace** the current 5-section format (`Problem / Solution / Implementation Approach / Considerations / Benefits`) entirely.

**Full question wording as section headings**: The headings are self-documenting — they match the research phrasing so anyone reading the idea understands what each section answers without needing to know the framework.

**Gap handling**: If the user's raw input doesn't provide enough information for all 3 sections, the LLM should mark gaps with italicized clarifying prompts (e.g., `_What specific outcome would tell you this idea succeeded?_`) rather than inventing content.

**Non-enhanced ideas**: Ideas captured without `--llm-enhance` stay as-is — no forced structure is applied.

**Fallback stub**: When the LLM is unavailable, the `generate_enhanced_stub` method produces the same 3-section skeleton with placeholder content.

### Interface Contract

The LLM system prompt instructs the model to produce JSON with `enhanced_description` containing the 3-question sections:

```json
{
  "filename": "feat-taskflow-queue-system",
  "title": "File-Based Queue System for ACE",
  "enhanced_description": "## What I Hope to Accomplish\n[impact and why]\n\n## What \"Complete\" Looks Like\n[concrete end state]\n\n## Success Criteria\n- [verifiable criterion 1]\n- [verifiable criterion 2]"
}
```

**Stub fallback output** (when LLM unavailable):

```markdown
# My Idea Title

## Description

[original user input]

## What I Hope to Accomplish

_[What impact should this have? Why does it matter?]_

## What "Complete" Looks Like

_[What concrete end state would indicate this idea is fully realized?]_

## Success Criteria

_[What verifiable checks would confirm success?]_
```

**Error Handling:**
- LLM returns old-format sections: Falls through to stub (JSON parse succeeds but content is not validated for section headings — acceptable; the prompt instructs the new format)
- LLM unavailable: Stub uses 3-question skeleton with italicized clarifying prompts

**Edge Cases:**
- Very short input ("add caching"): LLM should still produce all 3 sections, using gap markers where intent is unclear
- Very long input (multi-paragraph): LLM synthesizes into the 3 sections, preserving key details

### Success Criteria

- [ ] LLM system prompt (`idea_enhancement.system.md`) instructs the 3-question format with the exact section headings
- [ ] JSON example in system prompt shows `enhanced_description` with the 3 new sections (not the old 5)
- [ ] Fallback stub (`generate_enhanced_stub` in `idea_enhancer.rb`) produces the 3-question skeleton
- [ ] Inline fallback (`default_system_prompt` in `idea_enhancer.rb`) mentions the 3-question format
- [ ] Non-enhanced ideas (no `--llm-enhance`) are unaffected
- [ ] Existing tests pass (no behavioral change to CLI interface or file structure)

## Objective

Replace the generic idea enhancement format with the 3-Question Delegation Brief so that enhanced ideas carry structured, inheritable intent from the moment of capture. This enables downstream workflows (task drafting, subtask delegation) to map idea intent directly into their own structures rather than restating from scratch.

## Scope of Work

- **User Experience Scope**: Only affects LLM-enhanced idea output format; CLI interface unchanged
- **System Behavior Scope**: LLM system prompt content, stub fallback method, inline fallback prompt
- **Interface Scope**: No CLI/API changes; only the *content* of enhanced idea files changes
