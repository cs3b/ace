# Review Agents Get Spec Context

## Behavioral Specification

### User Experience
- **Input**: No CLI or user-facing input changes. This subtask modifies how review sessions are initialized when reviewing a task's PR
- **Process**: When `ace-review` runs against a task PR, the task's behavioral spec (`.s.md` file) is automatically included in the review context alongside existing project context (README, architecture, vision)
- **Output**: Review agents can reference spec decisions when evaluating code changes, preventing false positives from contradicting explicit spec decisions

### Expected Behavior

#### Review Session Context Loading

**Current behavior**: Review sessions load project-level context (README, architecture docs, vision docs) but do not include the task's behavioral spec. Reviewers evaluate code against general project conventions and their own assumptions about what the code should do.

**Target behavior**: When a review session is created for a task PR, the review context bundle includes the task's `.s.md` spec file. The spec provides reviewers with:
1. The intended behavior (what the code *should* do)
2. Explicit edge case decisions (what was intentionally included or excluded)
3. Success criteria (what "correct" looks like)

This prevents the failure mode observed in task 282's review session 3, where a reviewer flagged archived-task conversion as a bug when the spec explicitly allowed it (line 86: "status is orthogonal to structure").

#### Spec Discovery

The review setup needs to locate the relevant spec for a given PR:

- **Task PR convention**: PRs created by the ACE pipeline reference their task ID in the branch name or PR description
- **Spec location convention**: Task specs live at `.ace-taskflow/*/tasks/<task-dir>/task.<id>.s.md` or `<task-dir>/<id>.s.md`
- **Fallback**: If no spec is found, the review proceeds without it (no hard dependency)

### Interface Contract

**Review context bundle**: The spec file is added to the review context as an additional document, alongside existing context sources.

**No CLI changes**: `ace-review` commands remain the same. Spec inclusion is automatic when a task spec is discoverable.

**Error Handling:**
- Spec file not found for the task: Review proceeds without spec context (graceful degradation)
- Multiple spec files found: Include the primary task spec (not subtask specs)
- Spec file is very large: Include as-is — specs are concise by convention

**Edge Cases:**
- PR not associated with a task: No spec to load — review uses project context only
- Subtask PR: Load the subtask's own spec, not the parent orchestrator
- Spec references other specs: Only include the directly relevant spec, not transitive references

### Success Criteria

- [ ] When `ace-review` creates a session for a task PR, the task's `.s.md` spec is included in the review context
- [ ] Review agents can reference spec decisions in their findings (e.g., "spec line 86 allows this")
- [ ] If no spec is found, review proceeds normally without error
- [ ] False positives from contradicting explicit spec decisions are eliminated

### Validation Questions

- [x] Should spec inclusion be automatic or opt-in? -> **Automatic with graceful fallback — spec is always valuable context when available**
- [x] Should subtask specs include the parent orchestrator? -> **No — only the directly relevant spec to keep context focused**
- [x] Which review package component handles context loading? -> **Session initialization in `ace-review` — where `context.md` or review preset is assembled**

## Objective

Close the intent loss point at the **spec → review agent** handoff. Task 282's experiment showed that review agents without spec context contradict explicit spec decisions (3 invalid findings, 1 directly contradicting spec line 86). Including the behavioral spec as review context lets reviewers validate code against intended behavior rather than inventing constraints.

## Scope of Work

- **User Experience Scope**: No user-facing changes — review agents see the spec as additional context
- **System Behavior Scope**: Review session context loading gains spec-awareness
- **Interface Scope**: `ace-review` session initialization
