# Evidence-Based PR Description Workflow

## Behavioral Specification

### User Experience
- **Input**: No new CLI commands. The existing PR description workflow (`update-pr-desc.wf.md`) and PR creation workflow (`create-pr.wf.md`) are updated
- **Process**: Agent derives every PR description section from code evidence — diff, commits, test output, CHANGELOGs — not from task specs
- **Output**: PR descriptions in the PR #215 style: summary leads with user intent, file changes from `ace-git diff --format grouped-stats`, test evidence maps test names to behaviors

### Expected Behavior

#### PR Description Section Structure

**Current structure** (from `update-pr-desc.wf.md` Section 8):
```
Summary → Changes → Breaking Changes → Test Results → Related Tasks
```

**Target structure**:
```
Summary → Changes → File Changes → Test Evidence → Releases
```

Each section has explicit sourcing rules:

| Section | Data Source | What NOT to Do |
|---------|-----------|----------------|
| **Summary** | Feature commits + error messages that changed in the diff | Copy from task spec objective |
| **Changes** | `git diff` grouped by concern, each traced to commit SHA | List commits verbatim |
| **File Changes** | `ace-git diff --format grouped-stats` output | Manual file listing |
| **Test Evidence** | Read test files, map test names → behaviors proved + suite totals | Just paste `ace-test-suite` output |
| **Releases** | CHANGELOG entries in the diff | Repeat what's already in Changes |

#### Summary Writing Rules

The workflow instructs the agent to:
1. Read the diff for **what changed** (old behavior vs new behavior)
2. Identify the **user-facing pain** that was removed (often visible in deleted error messages or removed manual steps)
3. Write summary as: "What's easier now" → "What you used to have to do"
4. Never start with method names or class names — start with what the user experiences

**Example from PR #215**:
- Bad: "`create_subtask` and `demote_to_subtask` now auto-convert non-orchestrator parents..."
- Good: "Organizing tasks into parent-child hierarchies now works in one step..."

#### Sections Removed

- **Commit History** — GitHub provides this in a separate tab
- **Related Tasks** — replaced by deriving everything from code evidence
- **Breaking Changes placeholder** — only include if breaking changes actually exist in the diff

### Interface Contract

**Workflow files updated**: `update-pr-desc.wf.md`, `create-pr.wf.md`

**PR templates updated**: `feature.template.md`, `default.template.md`

**No CLI changes**: `ace-git` commands remain the same. The workflow consumes `ace-git diff --format grouped-stats` output.

**Error Handling:**
- No CHANGELOG entries in diff: Omit the Releases section entirely
- No test files in diff: Test Evidence shows only suite pass/fail totals
- `ace-git diff --format grouped-stats` unavailable: Fall back to flat file list (graceful degradation)

**Edge Cases:**
- Single-file PR: All sections still apply, just shorter
- Documentation-only PR: No Test Evidence section needed, Changes section focuses on what docs clarify
- Hotfix PR: Summary focuses on what broke and what's fixed, not intent/value framing

### Success Criteria

- [ ] Updated workflow produces descriptions in the PR #215 style
- [ ] Summary section leads with intent/value, not implementation details
- [ ] File Changes section uses `ace-git diff --format grouped-stats` output
- [ ] Test Evidence maps specific tests to behaviors (not just pass/fail counts)
- [ ] No task spec content appears in the generated PR description
- [ ] Releases section derived from CHANGELOG entries in the diff
- [ ] Sections with no evidence are omitted rather than left empty

### Validation Questions

- [x] Should the summary ever reference task specs? -> **No — derive entirely from code evidence. The spec informed the code; the PR describes the code**
- [x] Should File Changes always use grouped-stats or allow flat format? -> **Default to grouped-stats with flat fallback if 281.06 isn't available**
- [x] Should Test Evidence include test output or just test names? -> **Test names mapped to behaviors, plus suite totals. Not raw output**

## Objective

Make PR descriptions evidence-based: every section derived from what the code actually does (diff, tests, CHANGELOGs) rather than what the task spec said to do. This closes the last mile of the structured intent flow — intent flows from spec to code to PR description, but the PR description proves intent through evidence rather than restating it.

## Scope of Work

- **User Experience Scope**: PR descriptions are higher quality with no extra effort from the user
- **System Behavior Scope**: Workflow instruction changes, template updates
- **Interface Scope**: Workflow files and PR templates

### What to Change

#### 1. Rewrite `update-pr-desc.wf.md`

Replace the description generation sections with the new evidence-based section structure and sourcing rules described above.

#### 2. Update `create-pr.wf.md`

Align the PR creation workflow with the same evidence-based approach so new PRs start with the right structure.

#### 3. Update PR templates

Update `feature.template.md` and `default.template.md` to match the new section structure:
```
## Summary
## Changes
## File Changes
## Test Evidence
## Releases
```

- Automated PR description generation without agent involvement
- Retroactively updating existing PR descriptions
