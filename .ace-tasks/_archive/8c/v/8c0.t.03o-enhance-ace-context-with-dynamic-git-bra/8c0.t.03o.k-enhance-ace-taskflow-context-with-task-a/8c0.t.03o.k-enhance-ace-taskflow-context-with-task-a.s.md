# Enhance ace-taskflow context with task activity awareness

## Problem

The merge bug in task 140 revealed that developers/agents lack visibility into what's happening around them in the task workflow. The current `ace-taskflow context` shows current task info but doesn't show:
- What was recently completed (work context)
- What else is in progress (parallel work awareness)
- What's coming next (planning context)

## Objective

Enhance `ace-taskflow context` to show task activity, providing awareness of:
- Recently completed tasks (what just finished)
- Tasks in progress except current (parallel work)
- Next pending tasks (what's coming)

## Behavioral Specification

### User Experience

- **Input**: `ace-taskflow context` (existing command)
- **Process**: Queries task status from .ace-taskflow directory
- **Output**: Enhanced context with task activity section

### Expected Behavior

When running `ace-taskflow context`, in addition to current output, show:
1. **Recently Done** (last 3): One line per task with ID, title, completion time
2. **In Progress** (excluding current): One line per task with ID, title
3. **Up Next** (next 3 pending): One line per task with ID, title

### Interface Contract

```bash
# Current output (unchanged)
ace-taskflow context
# Current Task: 140 - Enhance ace-context...
# Release: v.0.9.0 (92% complete)

# NEW: Task Activity section added
## Task Activity
### Recently Done
- 140.02: Update ace-taskflow to use ace-git (done 2h ago)
- 140.01: Create ace-git package (done 1d ago)
- 139: Fix documentation (done 2d ago)

### In Progress
- 143: Unified configuration loading (@worktree)

### Up Next
- 140.03: Update ace-review to use ace-git
- 140.04: Update ace-prompt to use ace-git
- 140.05: Update ace-context to use ace-git
```

**Flags:**
- `--json`: Includes task activity in JSON output
- Existing flags unchanged

**Error Handling:**
- No recent done tasks: Show "No recently completed tasks"
- No other in-progress: Show "No other tasks in progress"
- No pending tasks: Show "No pending tasks"

## Technical Approach

### Architecture Pattern
- Follow existing ATOM pattern in ace-taskflow
- Atoms: Pure formatting functions (no new atoms needed - use existing)
- Molecules: Task activity analysis (TaskActivityAnalyzer - new), extend TaskDisplayFormatter
- Organisms: Orchestration (TaskflowContextLoader updates)
- Models: Data carriers (extend context hash with task_activity attribute)

### Technology Stack
- Existing `TaskFilter.filter_by_status()` for status filtering
- Existing `TaskFilter.filter_recent()` for time-based filtering (uses File.mtime)
- Existing `TaskSelector.select_next()` for "up next" prioritization
- Existing `TaskDisplayFormatter` for status icons and formatting
- Ruby Time for relative time calculations

### Key Design Decisions
- **Reuse existing molecules**: TaskFilter already has `filter_recent()` and `filter_by_status()`
- **New analyzer molecule**: Create `TaskActivityAnalyzer` to orchestrate activity categorization
- **Relative time formatting**: Add method to TaskDisplayFormatter for "2h ago" style output
- **Graceful degradation**: Empty sections show helpful messages, don't fail

## Test Case Planning

### Happy Path Scenarios
1. Release with done, in-progress, and pending tasks - shows all three sections
2. Multiple in-progress tasks excluding current - shows all except current
3. JSON output includes task_activity object with all three arrays
4. Current task excluded from in_progress list

### Edge Case Scenarios
1. No done tasks - "Recently Done" shows empty message
2. No other in-progress tasks - "In Progress" shows empty message
3. No pending tasks - "Up Next" shows empty message
4. Very old done tasks (> 3 days) - not included in recently_done
5. No current task detected - shows all activity without exclusion

### Error Condition Scenarios
1. Task files without mtime - graceful handling
2. Invalid task status values - skip invalid tasks
3. Missing task path - skip tasks without valid path

## Risk Assessment

### Technical Risks
- **Risk:** File.mtime dependency for time filtering may be slow on large repos
  - **Probability:** Low
  - **Impact:** Low
  - **Mitigation:** TaskFilter.filter_recent already handles this; limit to current release

### Performance Risks
- **Risk:** Loading all tasks for activity analysis adds overhead
  - **Probability:** Medium
  - **Impact:** Low
  - **Mitigation:** Only load tasks from current release, limit result counts

## Out of Scope

- Changing existing context output format
- Task details beyond one-line summary
- Cross-release task visibility
