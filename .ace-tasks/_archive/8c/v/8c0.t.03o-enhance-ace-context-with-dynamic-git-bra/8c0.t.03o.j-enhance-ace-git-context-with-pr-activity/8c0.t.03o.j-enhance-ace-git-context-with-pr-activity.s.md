# Enhance ace-git context with PR activity awareness

## Problem

The merge bug in task 140 revealed a gap: `ace-git context` doesn't show enough about what's happening in the repository. Developers/agents can't see recent PR activity, leading to incorrect assumptions (e.g., thinking PRs aren't merged when they are).

## Objective

Enhance `ace-git context` to show PR activity around the current branch, providing awareness of:
- What was recently merged (context of completed work)
- What else is open (context of parallel work)

## Behavioral Specification

### User Experience

- **Input**: `ace-git context` (existing command)
- **Process**: Fetches PR data via `gh` CLI, formats concisely
- **Output**: Enhanced context with PR activity section

### Expected Behavior

When running `ace-git context`, in addition to current output, show:
1. **Recent Merged PRs** (last 3): One line per PR with number, title, merge date
2. **Open PRs** (excluding current branch): One line per PR with number, title, author

### Interface Contract

```bash
# Current output (unchanged)
ace-git context
# Branch: 140-enhance-ace-context...
# Task Pattern: 140
# PR: #85 (if current branch has PR)

# NEW: PR Activity section added
## PR Activity
### Recently Merged
- #84: Update ace-taskflow to use ace-git (merged 2h ago)
- #82: Create ace-git package (merged 1d ago)
- #80: Fix lint issues (merged 2d ago)

### Open PRs
- #83: Add documentation updates (@username)
- #81: Refactor search module (@other)
```

**Flags:**
- `--no-pr`: Skips ALL PR-related output (current + activity)
- `--json`: Includes PR activity in JSON output

**Error Handling:**
- gh CLI not installed: Skip PR activity section gracefully
- No recent merged PRs: Show "No recently merged PRs"
- No open PRs: Show "No other open PRs"

## Technical Approach

### Architecture Pattern
- Follow existing ATOM pattern in ace-git
- Atoms: Pure formatting functions (TimeFormatter, ContextFormatter updates)
- Molecules: gh CLI integration (PrMetadataFetcher new methods)
- Organisms: Orchestration (RepoContextLoader updates)
- Models: Data carriers (RepoContext pr_activity attribute)

### Technology Stack
- gh CLI for PR data fetching (`gh pr list --state merged/open --json`)
- Existing CommandExecutor for safe command execution
- Ruby Time.parse for ISO8601 timestamp handling

### gh Commands Needed
```bash
# Recently merged PRs
gh pr list --state merged --limit 3 --json number,title,mergedAt

# Open PRs (filter current branch client-side)
gh pr list --state open --json number,title,author,headRefName
```

## Test Case Planning

### Happy Path Scenarios
1. Repository with merged and open PRs - shows both sections
2. Repository with only merged PRs - shows only merged section
3. Repository with only open PRs - shows only open section
4. JSON output includes pr_activity object

### Edge Case Scenarios
1. No merged PRs - section omitted or shows empty message
2. No open PRs - section omitted or shows empty message
3. Current branch's PR filtered from open list
4. Very old merge dates (weeks ago formatting)

### Error Condition Scenarios
1. gh CLI not installed - skip PR activity gracefully
2. gh CLI not authenticated - skip PR activity gracefully
3. Network timeout - skip PR activity gracefully
4. JSON parse error - skip PR activity gracefully

## Risk Assessment

### Technical Risks
- **Risk:** gh pr list command format changes
  - **Probability:** Low
  - **Impact:** Medium
  - **Mitigation:** Use explicit --json fields, test with current gh version

### Performance Risks
- **Risk:** Additional gh CLI calls slow down context command
  - **Probability:** Medium
  - **Impact:** Low
  - **Mitigation:** Fetch merged and open PRs in sequence (not parallel to avoid rate limits), skip on --no-pr flag
