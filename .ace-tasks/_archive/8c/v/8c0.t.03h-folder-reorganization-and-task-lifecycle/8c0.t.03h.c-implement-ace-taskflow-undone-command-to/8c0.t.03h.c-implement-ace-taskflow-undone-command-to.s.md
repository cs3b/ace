# Implement ace-taskflow undone command to reopen tasks

## Description

Implement a new `ace-taskflow undone <task_identifier>` command that allows reopening of completed tasks. This command will locate a task in the `.ace-taskflow/<release>/tasks/_archive` directory, move it back to the active tasks folder, and update its status to `in-progress` with an updated timestamp. This addresses a critical gap in task lifecycle management where tasks need to be reactivated due to new information, incomplete work, or shifting priorities.

## Problem Statement

Currently, tasks moved to the `.ace-taskflow/<release>/tasks/_archive` directory are considered permanently closed. There is no standardized CLI command to revert a completed task back to an active state, requiring manual file system operations and metadata editing. This limitation hinders agile development practices and creates friction for both human developers and AI agents managing task workflows.

## Acceptance Criteria

- [ ] **CLI Command**: `ace-taskflow undone <task_identifier>` command is implemented and functional
- [ ] **Task Discovery**: Locates task file in `.ace-taskflow/<release>/tasks/_archive` directory by task ID, partial title, or filename
- [ ] **File Movement**: Moves task markdown file from `_archive/` directory back to active `.ace-taskflow/<release>/tasks` folder
- [ ] **Status Update**: Updates frontmatter `status` field to `in-progress` (or configurable default active status)
- [ ] **Timestamp Update**: Updates `last-updated` timestamp in task frontmatter

**Molecules**:

**Atoms**:

### Configuration

- Paths to `_archive` directory retrieved via ace-core's configuration cascade
- Current release directory path resolution
- Configurable default "reopened" status (default: `in-progress`)

**Config cascade example** (`.ace.example/taskflow/config.yml`):
```yaml
reopened_status: 'in-progress'  # Default status when task is reopened
```

### Cross-Release Reopening

When reopening a task from a closed/archived release (e.g., reopening v.0.8.0 task while working on v.0.9.0):

**Behavior**: Fail with clear error message
- **Rationale**: Cross-release reopening is ambiguous - should it go to the old release or current?
- **Error format**: `Cannot reopen task from release v.0.8.0. Task belongs to inactive release. Use --to-release v.0.9.0 to move to current release, or --force to reopen in original release.`
- **Options**:
  - `--to-release <version>`: Explicitly move to specified release when reopening
  - `--force`: Reopen in original release even if inactive

**Algorithm**:
1. Find task in _archive/ (any release)
2. Determine task's original release
3. Check if that release is active (current release or explicitly marked active)
4. If not active: fail with actionable error message
5. If active: proceed with normal reopening

### Task Identification Strategy

Support multiple identifier formats:

- Task ID (e.g., `v.0.9.0+task.123` or `task.123` or `T-123`)
- Partial title match (e.g., "undone command")
- Filename pattern
- Consider fuzzy matching for improved UX

### Considerations

- **Conflict Resolution**: If task already exists in target release, prompt user or fail with clear error
- **Status Options**: Allow `--status` flag to override default reopened status
- **Integration**: Consider interaction with `ace-nav` for task discovery
- **Git Integration**: Consider `ace-git-commit` integration if changes should be auto-committed
- **Subdirectory Structure**: Handle tasks organized in subdirectories within _archive/

### Example Usage

```bash
# Reopen by task ID
ace-taskflow undone task.131

# Reopen with custom status
ace-taskflow undone task.131 --status backlog

# Reopen by partial title
ace-taskflow undone "implement undone"
```
