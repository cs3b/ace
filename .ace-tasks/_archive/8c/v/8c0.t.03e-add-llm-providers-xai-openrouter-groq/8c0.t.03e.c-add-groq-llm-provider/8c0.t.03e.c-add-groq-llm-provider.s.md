# Add Groq LLM Provider

## Objective

Integrate Groq as a new LLM provider in ace-llm gem to enable ultra-fast inference for Llama and other open-source models. Groq provides sub-second response times through their LPU (Language Processing Unit) infrastructure using an OpenAI-compatible API.

## Scope

Integrate Groq as a new LLM provider in ace-llm gem. Groq provides ultra-fast inference for Llama and other models through an OpenAI-compatible API.

## API Details

| Property | Value |
|----------|-------|
| Base URL | `https://api.groq.com/openai/v1` |
| Endpoint | `/openai/v1/chat/completions` |
| Auth | `Authorization: Bearer $GROQ_API_KEY` |
| Compatibility | OpenAI-compatible |

## Models (Current, Non-deprecated)

- `llama-3.3-70b-versatile` (default, high quality)
- `llama-3.1-8b-instant` (fast)
- `meta-llama/llama-4-maverick-17b-128e-instruct`

**Note**: Several models deprecated in 2025 (gemma2-9b-it, mixtral-8x7b-32768, llama3-70b-8192).

## Technical Approach

### Architecture Pattern

Groq uses the same OpenAI-compatible chat completions API, making it nearly identical to the existing OpenAI and TogetherAI client implementations. The implementation follows the established BaseClient pattern:

- **Pattern**: Inherit from `BaseClient` and override provider-specific methods
- **Integration**: Seamlessly integrates with existing `QueryInterface` and `ClientRegistry`
- **Impact**: Minimal - adds new provider without affecting existing ones

### Technology Stack

- **Base class**: `Ace::LLM::Organisms::BaseClient` (provides common functionality)
- **HTTP client**: Uses existing `Ace::LLM::Atoms::HttpClient` (no new dependencies)
- **API compatibility**: OpenAI chat completions format (same as OpenAI, TogetherAI)
- **Configuration**: YAML-based provider config (same pattern as other providers)

**Dependencies**: None - uses existing ace-llm infrastructure

### Implementation Strategy

1. **Copy-and-modify approach**: Start with `openai_client.rb` as template
2. **Key differences from OpenAI**:
   - Base URL: `https://api.groq.com/openai/v1` (instead of `https://api.openai.com`)
   - Default model: `llama-3.3-70b-versatile` (instead of `gpt-4o`)
   - Provider name: `groq` (for routing and error messages)
3. **Testing strategy**: Unit tests for client initialization and error handling (API tests optional due to external dependency)
4. **Rollback**: Simple - remove files and revert changes to `llm.rb` and `env_reader.rb`

## Risk Assessment

### Technical Risks

- **Risk**: Groq API changes or deprecates models
  - **Probability**: Low (recent deprecations already handled)
  - **Impact**: Medium (would require model list update)
  - **Mitigation**: Document current models, add deprecation notes to groq.yml
  - **Rollback**: Update model list in groq.yml, no code changes needed

- **Risk**: OpenAI-compatible API has subtle differences
  - **Probability**: Low (Groq explicitly markets OpenAI compatibility)
  - **Impact**: Medium (response parsing might fail)
  - **Mitigation**: Test with actual API calls during implementation, handle edge cases in parse_response
  - **Rollback**: Fix parsing logic or fall back to simpler response handling

### Integration Risks

- **Risk**: ClientRegistry doesn't auto-discover GroqClient
  - **Probability**: Very Low (pattern proven with 6 existing providers)
  - **Impact**: High (provider completely non-functional)
  - **Mitigation**: Follow exact pattern from other providers (require in llm.rb, proper class structure)
  - **Monitoring**: Test with `ace-llm-query groq "test"` immediately after implementation

- **Risk**: Alias conflicts with existing aliases
  - **Probability**: Very Low (groq/groq-fast are unique)
  - **Impact**: Low (would just override or fail to load)
  - **Mitigation**: Verify no existing aliases match in other provider YAML files
  - **Monitoring**: Test alias resolution after configuration load

### Performance Risks

- **Risk**: N/A - Groq is designed for ultra-fast inference
  - **Mitigation**: N/A
  - **Monitoring**: Response times should be <1s for most queries
  - **Thresholds**: If >2s consistently, investigate network or API issues

## Out of Scope

- Real API integration testing (requires valid API key and external service availability)
- Streaming support (defer to future enhancement)
- Function calling support (Groq supports it but not in MVP)
- Additional model aliases beyond groq/groq-fast/groq70b
- Rate limiting or retry logic (handled by base infrastructure)
- Deprecated model support (explicitly excluded)
