# Add models.dev Integration (ace-llm-models-dev)

## Objective

Create a new standalone Ruby gem `ace-llm-models-dev` that integrates with models.dev API to provide model validation, cost tracking, and change monitoring capabilities. This gem will enable developers to validate model names, calculate query costs, and monitor pricing changes across 40+ LLM providers.

## Scope

Create a new gem `ace-llm-models-dev` that integrates with [models.dev](https://models.dev/) - an open-source database of AI models. This enables model validation, cost tracking, and change monitoring across providers.

## Data Source

| Property | Value |
|----------|-------|
| API | `https://models.dev/api.json` |
| Source | [github.com/sst/models.dev](https://github.com/sst/models.dev) |
| Format | JSON (generated from TOML) |
| Logos | `https://models.dev/logos/{provider}.svg` |

## Capabilities

### 1. Model Validation
- Validate model names exist for a provider
- Check if model is deprecated/alpha/beta
- Local cache avoids repeated API calls

### 2. Cost Tracking
- Query cost per million tokens (input/output/reasoning/cache)
- Calculate query costs based on token usage
- Support audio pricing where available

### 3. Change Monitoring
- Download and cache API snapshots
- Generate diff between updates
- Track new models, deprecations, pricing changes

## Model Data Structure

From models.dev API, each model includes:

```yaml
# Core
name: "Model Display Name"
model_id: "provider:model-name"  # AI SDK format

# Capabilities
attachment: true/false
reasoning: true/false
tool_call: true/false
structured_output: true/false
temperature: true/false

# Dates
knowledge: "2024-04"
release_date: "2025-02-19"
updated_date: "2025-03-01"

# Cost (per million tokens)
cost:
  input: 3.00
  output: 15.00
  reasoning: 12.00  # optional
  cache_write: 3.75  # optional
  cache_read: 0.30   # optional

# Limits
limit:
  context: 400000
  output: 8192

# Modalities
modalities:
  input: [text, image, audio, video, pdf]
  output: [text, audio]

# Status
status: deprecated/alpha/beta  # optional
open_weights: true/false
```

## CLI Examples

```bash
# Sync local cache
ace-llm-models sync
# Synced 847 models from 42 providers

# Validate model name
ace-llm-models validate xai:grok-3
# ✓ xai:grok-3 is valid

ace-llm-models validate xai:grok-99
# ✗ xai:grok-99 not found. Did you mean: grok-3, grok-4?

# Calculate cost
ace-llm-models cost openai:gpt-4o --input 10000 --output 2000
# Input: 10K tokens × $2.50/M = $0.025
# Output: 2K tokens × $10.00/M = $0.020
# Total: $0.045

# Show diff since last sync
ace-llm-models diff
# + anthropic:claude-4-opus (new)
# ~ openai:gpt-4o: output cost $10.00 → $7.50
# - google:gemini-1.0-pro (deprecated)
```

## Integration with ace-llm

Optional integration for query validation:

```ruby
# In ace-llm configuration (.ace/llm/config.yml)
validate_models: true  # Enable model validation before queries
cache_path: ~/.cache/ace-llm-models-dev  # Local cache location
```

## Technical Approach

### Architecture Pattern

Following ACE ATOM architecture with clear separation of concerns:

- **Atoms**: Pure functions for data parsing, HTTP requests, file I/O
- **Molecules**: Focused operations for validation, cost calculation, diff generation
- **Organisms**: Business orchestration for sync workflow
- **Models**: Data structures for API response, model info, pricing

### Technology Stack

**Core Dependencies:**
- `ace-support-core ~> 0.10` - Configuration cascade and project utilities
- `faraday ~> 2.0` - HTTP client for API calls (follows ADR-010)
- `addressable ~> 2.8` - URL handling and validation
- `thor ~> 1.3` - CLI framework (follows ADR-018)

**Development Dependencies:**
- `ace-support-test-helpers ~> 0.9` - Testing infrastructure
- `webmock ~> 3.0` - HTTP request mocking
- `vcr ~> 6.0` - HTTP interaction recording

### Implementation Strategy

1. **Gem Structure**: Follow standard ACE gem pattern with ATOM architecture
2. **HTTP Client**: Use Faraday with retry middleware per ADR-010
3. **Cache Management**: Store in `~/.cache/ace-llm-models-dev/` using XDG conventions
4. **Error Handling**: Custom exceptions for network, validation, and cache errors
5. **CLI Design**: Thor-based commands following ADR-018 patterns
6. **Testing**: Flat test structure per ADR-017, use VCR for API interactions

## Risk Assessment

### Technical Risks

- **Risk:** models.dev API structure changes
  - **Probability:** Medium
  - **Impact:** High
  - **Mitigation:** Version cache format, handle unknown fields gracefully
  - **Rollback:** Use cached data, warn about outdated schema

- **Risk:** Large API response size (performance)
  - **Probability:** Low
  - **Impact:** Medium
  - **Mitigation:** Stream JSON parsing, compress cache files
  - **Rollback:** Add compression option

### Integration Risks

- **Risk:** Cache directory permissions
  - **Probability:** Low
  - **Impact:** Medium
  - **Mitigation:** Use XDG directories, handle permission errors with helpful messages
  - **Monitoring:** Test on different OS/permission setups

### Performance Risks

- **Risk:** Slow diff generation on large datasets
  - **Mitigation:** Optimize diff algorithm, add progress indicators
  - **Monitoring:** Profile with realistic data
  - **Thresholds:** Diff should complete in < 1 second for typical datasets
