## Objective

Integrate OpenRouter as a new LLM provider in ace-llm gem. OpenRouter provides unified access to 400+ models from multiple providers through a single OpenAI-compatible API, enabling cost-effective access to models from OpenAI, Anthropic, Google, Meta, and others via one API key.

## Scope of Work

- Create OpenRouter client implementation following OpenAI-compatible pattern
- Add optional attribution headers (HTTP-Referer, X-Title) for OpenRouter's app ranking
- Configure provider metadata with model aliases
- Add OPENROUTER_API_KEY environment variable support
- Preserve native_finish_reason in metadata (OpenRouter normalizes this field)
- Create comprehensive tests for the new provider

## Technical Approach

### Architecture Pattern

OpenRouter is OpenAI-compatible, so we follow the OpenAIClient pattern with these additions:
- Store `@options` instance variable to access referer/title options
- Override `make_api_request` to conditionally add HTTP-Referer and X-Title headers
- Preserve `native_finish_reason` in metadata alongside normalized `finish_reason`

**Integration with existing architecture:**
- Extends `BaseClient` (inherits message building, error handling, system_append)
- Registered via `ace-llm/lib/ace/llm.rb` require
- Auto-discovered by `ClientRegistry` via provider YAML
- Accessible via `QueryInterface` and `ace-llm-query` CLI

**Design decisions:**
- Use OpenAI chat completions endpoint pattern
- Default model: `openai/gpt-4o` (widely available, balanced performance)
- Optional headers only sent if provided (maintains OpenAI compatibility)

### Technology Stack

**Libraries/Frameworks:**
- Uses existing `Atoms::HTTPClient` (Faraday-based, with retries)
- No new dependencies required

**Version Compatibility:**
- Ruby 2.7+ (same as ace-llm)
- Compatible with existing ace-llm test suite

**Performance Implications:**
- OpenRouter adds ~50-100ms latency vs direct API
- Same request/response format as OpenAI (no extra parsing overhead)

**Security Considerations:**
- API key stored in environment variable (OPENROUTER_API_KEY)
- Supports .ace/.env cascade via ace-core integration
- HTTPS only (base_url: https://openrouter.ai/api/v1)

### Implementation Strategy

**Step-by-step approach:**
1. Create OpenRouterClient by copying OpenAIClient structure
2. Modify initialize to store @options for header access
3. Override make_api_request to add conditional headers
4. Update parse_response to preserve native_finish_reason
5. Add OPENROUTER_API_KEY support in EnvReader
6. Create provider YAML with models and aliases
7. Add require to main lib file
8. Write comprehensive tests

**Rollback considerations:**
- New files can be removed without affecting existing providers
- No changes to base classes (only additions)
- If broken, revert 5 files: client, config, test, env_reader case, require line

**Testing strategy:**
- Unit tests for helper methods (similar to base_client_helpers_test.rb)
- Mock API responses for generate() method
- Test error handling (missing key, network errors)
- Test optional header inclusion
- Integration test via ace-llm-query CLI (manual verification)

**Performance monitoring:**
- No specific monitoring needed (uses existing HTTPClient instrumentation)
- Token usage tracked in metadata (same as other providers)

## Risk Assessment

### Technical Risks

- **Risk:** OpenRouter API changes breaking compatibility
  - **Probability:** Low
  - **Impact:** Medium
  - **Mitigation:** OpenRouter maintains OpenAI compatibility; monitor API changelog
  - **Rollback:** Revert 5 files (client, config, test, env_reader, require)

- **Risk:** Optional headers (HTTP-Referer, X-Title) causing issues
  - **Probability:** Low
  - **Impact:** Low
  - **Mitigation:** Only send headers when explicitly provided in options
  - **Rollback:** Remove header lines from make_api_request

- **Risk:** Model naming conflicts (provider/model format)
  - **Probability:** Low
  - **Impact:** Low
  - **Mitigation:** Model names are namespaced (e.g., openai/gpt-4o)
  - **Monitoring:** Test alias resolution in integration tests

### Integration Risks

- **Risk:** Breaking existing provider functionality
  - **Probability:** Very Low
  - **Impact:** High
  - **Mitigation:** No changes to base classes; only additions
  - **Monitoring:** Run full test suite (ace-test-suite)

- **Risk:** EnvReader changes affecting other providers
  - **Probability:** Very Low
  - **Impact:** Medium
  - **Mitigation:** Only adding new case, not modifying existing logic
  - **Monitoring:** Test all providers with ace-test

### Performance Risks

- **Risk:** Additional latency from OpenRouter proxy
  - **Mitigation:** Document expected ~50-100ms overhead
