# 242.01: FeedbackVerifier Molecule

## Objective

Create the core verification logic that implements Chain of Verification (CoV) pattern for validating review findings independently.

## Behavioral Specification

### User Experience

- **Input**: Feedback item + task context (goal, constraints)
- **Process**: Generate verification questions, run independent LLM verification, score confidence
- **Output**: Verification result with status (`verified`, `uncertain`, `invalid`) and answers

### Interface Contract

```ruby
verifier = Ace::Review::Molecules::FeedbackVerifier.new(llm_client:)

# Verify single feedback item
result = verifier.verify(feedback_item, task_context:)
# => { status: :verified, questions: [...], answers: [...], verified_at: Time }

# Result structure
result.status      # :verified | :uncertain | :invalid
result.questions   # Array of verification questions asked
result.answers     # Array of {question:, answer:, verdict:} hashes
result.verified_at # Timestamp of verification
```

### Expected Behavior

1. **Question Generation**: Generate 3 verification questions per finding:
   - "Does the code at {location} actually have this issue: {title}?"
   - "Would the suggested fix introduce new bugs or break existing functionality?"
   - "Is this change aligned with the task goal: {task_goal}?"

2. **Independent Verification**: For each question:
   - Use a verification prompt that does NOT include the original finding's reasoning
   - Show only the code location and the specific question
   - Request YES/NO/UNCERTAIN answer with brief explanation

3. **Confidence Scoring**:
   - `verified`: All questions answered YES (or YES/YES/NO for question 2 which is inverted)
   - `uncertain`: Mixed answers or any UNCERTAIN
   - `invalid`: Question 1 = NO (issue doesn't exist) OR Question 3 = NO (misaligned with goal)

## Implementation Notes

### Verification Prompt Design

Key insight from CoV research: **factored verification** - check claims with limited context so the model doesn't simply justify initial errors.

```markdown
# Verification Prompt

You are verifying a code review finding. Answer based ONLY on the code shown.

## Code Location
{code_snippet}

## Question
{question}

## Task Context
Goal: {task_goal}

Answer with exactly one of: YES, NO, or UNCERTAIN
Then provide a brief (1-2 sentence) explanation.

Format:
VERDICT: [YES|NO|UNCERTAIN]
REASON: [explanation]
```

### Answer Parsing

Parse LLM response to extract:
- `verdict`: :yes, :no, :uncertain
- `reason`: String explanation

Handle edge cases:
- Missing verdict → :uncertain
- Ambiguous response → :uncertain
- Timeout/error → :uncertain with error note
