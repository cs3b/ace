# Improve ace-git-worktree error message for dependency-blocked tasks

## Objective

When `ace-git-worktree create --task <id>` fails because the task has unmet dependencies, the error message should clearly explain WHY the task cannot be started, not just that it failed.

## Behavioral Specification

### User Experience

- **Input**: `ace-git-worktree create --task 157.14` (where task 157.14 has unmet dependencies)
- **Process**: System attempts to mark task as in-progress, TaskManager rejects due to dependencies
- **Output**: Clear error message explaining the specific blocker

### Current Behavior (Problem)

```bash
$ ace-git-worktree create --task 157.14
Creating worktree for task: 157.14
Failed to create worktree: Failed to update task status
```

The message "Failed to update task status" is too generic - user has no idea WHY it failed.

### Expected Behavior (Solution)

```bash
$ ace-git-worktree create --task 157.14
Creating worktree for task: 157.14
Failed to create worktree: Cannot start task - blocked by dependencies: 157.09 (pending)

Hint: Use --no-status-update to create worktree without changing task status
```

### Interface Contract

**CLI Output Changes:**
- Error message should include the actual reason from TaskManager
- Should list which dependencies are blocking (with their status)
- Should suggest workaround (`--no-status-update` flag)

**Error Message Format:**
```
Failed to create worktree: Cannot start task - blocked by dependencies: <dep1> (<status>), <dep2> (<status>)

Hint: Use --no-status-update to create worktree without changing task status
```

### Success Criteria

- [x] Error message shows actual blocker reason from TaskManager
- [x] Blocking dependency task IDs are listed
- [x] Blocking dependency statuses are shown (pending, draft, etc.)
- [x] Hint suggests `--no-status-update` workaround
- [x] User can understand exactly why task creation failed

### Validation Questions

- [x] Should the error list ALL blocking dependencies or just the first? → All
- [x] Should it show dependency status? → Yes, helps user understand what needs to be done

## Technical Context

**Call Chain:**
1. `TaskWorktreeOrchestrator.create_for_task` calls `update_task_status(task_id, "in-progress")`
2. `TaskStatusUpdater.update_status` calls `TaskManager.update_task_status`
3. `TaskManager` returns `{success: false, message: "Cannot start task: blocked by dependencies 157"}`
4. Currently: error message is discarded, generic "Failed to update task status" returned

**Key Files:**
- `ace-git-worktree/lib/ace/git/worktree/organisms/task_worktree_orchestrator.rb:84-88`
- `ace-git-worktree/lib/ace/git/worktree/molecules/task_status_updater.rb:264-283`

## Implementation Plan

   - Update `update_task_status` call to handle result hash
   - Use the message from result in `error_workflow_result`
   - Add hint about `--no-status-update` flag
