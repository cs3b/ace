# Robust LLM Response Parsing

## Overview

Handle varied LLM response formats in JSON parsing. This ~30 line fix was the actual root cause of the issues that prompted the entire task 227.

## The Real Problem

The original JSON parser only handled responses where JSON appeared at the start:

```ruby
# Original (broken with Claude Opus)
def parse_json(response)
  JSON.parse(response)
end
```

But Claude Opus returns responses like:
```
Based on my analysis of the code review reports, here are the findings:

```json
{"findings": [...]}
```
```

The parser failed because it expected pure JSON or code fence at the very start.

## Behavioral Specification

### User Experience
- **Input**: LLM response containing JSON (various formats)
- **Process**: Extract JSON regardless of surrounding text
- **Output**: Parsed JSON object

### Expected Behavior

The `extract_json_from_response` helper handles all formats:

1. **Pure JSON**: `{"findings": [...]}`
2. **Code-fenced JSON**: ` ```json\n{"findings": [...]}\n``` `
3. **Text + code fence**: `Based on analysis...\n```json\n{...}\n```\nSome trailing text`
4. **JSON embedded in text**: `Here is the result: {"findings": [...]} as requested`

### Interface Contract

```ruby
module Ace::Review::Support
  class JsonExtractor
    def self.extract(response)
      # Try code fence extraction first
      if (match = response.match(/```(?:json)?\s*\n?(.*?)\n?```/m))
        return JSON.parse(match[1].strip)
      end

      # Try direct JSON parse
      JSON.parse(response)
    rescue JSON::ParserError
      # Try to find JSON object/array in response
      if (match = response.match(/(\{.*\}|\[.*\])/m))
        JSON.parse(match[1])
      else
        raise
      end
    end
  end
end
```

### Success Criteria

- [x] All LLM providers (claude, gemini, openai, codex) parse successfully
- [x] Test coverage for each response format variant
- [x] Graceful error handling for truly invalid responses

## Implementation

The fix was ~30 lines in `json_extractor.rb`:

```ruby
def self.extract(response)
  return nil if response.nil? || response.empty?

  # Strategy 1: Code fence with json marker
  if (match = response.match(/```json\s*\n?(.*?)\n?```/m))
    return JSON.parse(match[1].strip)
  end

  # Strategy 2: Code fence without marker
  if (match = response.match(/```\s*\n?(.*?)\n?```/m))
    begin
      return JSON.parse(match[1].strip)
    rescue JSON::ParserError
      # Not valid JSON in fence, continue
    end
  end

  # Strategy 3: Direct parse (pure JSON)
  begin
    return JSON.parse(response.strip)
  rescue JSON::ParserError
    # Continue to next strategy
  end

  # Strategy 4: Find JSON object/array in text
  if (match = response.match(/(\{[\s\S]*\})/))
    JSON.parse(match[1])
  elsif (match = response.match(/(\[[\s\S]*\])/))
    JSON.parse(match[1])
  else
    raise JSON::ParserError, "No valid JSON found in response"
  end
end
```

## Lessons Learned

This 30-line fix was what actually solved the problem. The extensive FeedbackExtractor, FeedbackDeduplicator, and FeedbackContextResolver (500+ lines total) were unnecessary complexity built on top of a simple parsing bug.
