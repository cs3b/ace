## Technical Approach

### Architecture Pattern

This task follows the existing ace-assign configuration-driven pattern. Review behavior is communicated via instruction text in preset YAML files. The reviewing agent reads the instruction (e.g., "Use preset code-correctness") and passes it as `--preset code-correctness` to ace-review. No structured metadata or Ruby code changes are required -- the existing instruction-text pattern handles preset selection.

**Key pattern:** Composition rules define the progression mapping. Presets (work-on-task.yml) hardcode the default 2-cycle configuration. The compose-assignment workflow uses the composition rules to dynamically generate review steps for composed assignments. Both paths must be updated.

### Technology Stack

- No new dependencies
- No Ruby code changes
- YAML configuration files only (parsed by existing ace-assign tooling)
- Markdown workflow instructions (read by LLM agents during compose)

### Implementation Strategy

1. Update composition-rules.yml first (source of truth for preset progression)
2. Update the two preset files (work-on-task.yml, work-on-tasks.yml) in parallel
3. Update the recipe file (implement-with-pr.recipe.yml)
4. Update the compose-assignment workflow (compose-assignment.wf.md)
5. Validate by reading all modified files and checking consistency
