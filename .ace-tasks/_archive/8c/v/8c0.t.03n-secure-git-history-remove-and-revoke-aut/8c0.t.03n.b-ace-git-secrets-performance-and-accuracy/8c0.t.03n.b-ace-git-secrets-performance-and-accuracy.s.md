# ace-git-secrets Performance and Accuracy Improvements

## Objective

Improve ace-git-secrets to reduce false positives by 99%+ (38,000 → ~100), speed up scanning by 10x (11 min → ~1 min), and improve output UX with summary-to-stdout and full-report-to-file.

## Behavioral Specification

### User Experience

- **Input**: User runs `ace-git-secrets scan` on a repository
- **Process**:
  - Parallel scanning with 16 threads
  - Automatic exclusion of lock files, minified assets, test fixtures
  - Progress indication during scan
- **Output**:
  - Brief summary printed to stdout (token counts, scan time, report path)
  - Full detailed report saved to `.cache/ace-git-secrets/YYYYMMDD-HHMMSS-report.json`

### Expected Behavior

1. **Lock files are ignored by default** - `package-lock.json`, `yarn.lock`, `Gemfile.lock`, etc. should never trigger false positives
2. **Azure storage keys require context** - Only match when preceded by `AccountKey=` or similar environment variable patterns, not bare base64 strings
3. **Scanning is parallelized** - Use 16 threads to split git history and scan concurrently
4. **Reports saved to cache** - Full JSON report with all details saved to `.cache/ace-git-secrets/` directory
5. **Summary to stdout** - Only show counts and report path, not full token list

### Interface Contract

```bash
# Scan with default settings (JSON report, 16 threads)
ace-git-secrets scan
# Output:
# Scan completed in 45s (16 threads)
# Tokens found: 92 (high: 45, medium: 47)
# Report saved: .cache/ace-git-secrets/20251220-235500-report.json

# Scan with markdown report
ace-git-secrets scan --format markdown
# Report saved: .cache/ace-git-secrets/20251220-235500-report.md

# View full report
cat .cache/ace-git-secrets/20251220-235500-report.json | jq
```

## Technical Approach

### 1. Default File Exclusions

```ruby
# lib/ace/git/secrets.rb
DEFAULT_EXCLUSIONS = [
  # Lock files (contain integrity hashes, not secrets)
  "**/package-lock.json", "**/yarn.lock", "**/pnpm-lock.yaml",
  "**/Gemfile.lock", "**/composer.lock", "**/Cargo.lock", "**/poetry.lock",
  # Minified files
  "**/*.min.js", "**/*.min.css",
  # Test fixtures & cassettes
  "**/fixtures/**", "**/cassettes/**", "**/__snapshots__/**",
  # Build outputs
  "**/dist/**", "**/build/**", "**/node_modules/**", "**/vendor/**"
].freeze
```

### 2. Azure Storage Key Pattern Fix

Change from bare base64 matching to context-aware:
```ruby
# Before (matches any 86-char base64)
regex: /[A-Za-z0-9+\/]{86}==/

# After (requires AccountKey= or similar prefix)
regex: /(?:AccountKey|AZURE[_-]?STORAGE[_-]?(?:ACCOUNT[_-]?)?KEY|azure[_-]storage[_-]key)\s*[=:]\s*["']?([A-Za-z0-9+\/]{86}==)/i
```

### 3. Parallel Scanning

```ruby
require "parallel"

THREAD_COUNT = 16

def scan_commits_parallel(commits)
  Parallel.flat_map(commits, in_threads: THREAD_COUNT) do |commit|
    scan_commit(commit)
  end
end
```

### 4. Report File Output

```ruby
def save_report(format: :json)
  cache_dir = File.join(repository_path, ".cache", "ace-git-secrets")
  FileUtils.mkdir_p(cache_dir)

  timestamp = Time.now.strftime("%Y%m%d-%H%M%S")
  ext = format == :json ? "json" : "md"
  path = File.join(cache_dir, "#{timestamp}-report.#{ext}")

  File.write(path, format == :json ? to_json : to_markdown)
  path
end
```

## Risk Assessment

### Technical Risks

- **Risk:** Thread-safety issues with shared blob cache
  - **Probability:** Medium
  - **Impact:** High (data corruption, incorrect results)
  - **Mitigation:** Use Mutex for all cache access, add concurrent test cases
  - **Rollback:** Revert to sequential scanning if issues detected

- **Risk:** Azure pattern too strict, missing real tokens
  - **Probability:** Low
  - **Impact:** Medium (security gap)
  - **Mitigation:** Test against known Azure key formats, document edge cases
  - **Rollback:** Add additional context patterns if needed

- **Risk:** Parallel gem not available in all environments
  - **Probability:** Low
  - **Impact:** Low (fallback to sequential)
  - **Mitigation:** Add graceful fallback when parallel not loaded
  - **Monitoring:** Log warning if falling back to sequential

### Performance Risks

- **Risk:** Thread count too high for some machines
  - **Probability:** Medium
  - **Impact:** Low (slower performance)
  - **Mitigation:** Make thread_count configurable, default to sensible value
  - **Thresholds:** Log if scan takes >5 minutes

- **Risk:** Large blob cache memory usage in parallel mode
  - **Probability:** Low
  - **Impact:** Medium (memory exhaustion)
  - **Mitigation:** Blob cache already exists, parallel doesn't increase total size
  - **Monitoring:** Profile memory usage during testing
