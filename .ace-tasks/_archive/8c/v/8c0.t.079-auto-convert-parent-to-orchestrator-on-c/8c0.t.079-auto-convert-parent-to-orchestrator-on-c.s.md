# Auto-Convert Parent to Orchestrator on Child Operations

## Behavioral Specification

### User Experience
- **Input**: `ace-task create "Subtask" --child-of 100` or `ace-task move 101 --child-of 100` where task 100 is a regular (non-orchestrator) task
- **Process**: The system auto-converts task 100 to an orchestrator (its content becomes subtask .01), then adds the new child as the next subtask
- **Output**: Success — the user gets an orchestrator with subtasks, without needing a manual intermediate step

### Expected Behavior

**Current behavior (broken)**:
```
$ ace-task create "Phase 1" --child-of 100
Error: Task 100 is not an orchestrator. Convert it to an orchestrator first
with `ace-taskflow task move 100 --child-of self`.
```

The user must run two commands: first `ace-task move 100 --child-of self`, then retry. This is a usability trap — the CLI knows exactly what needs to happen but refuses to do it.

**Target behavior**:
```
$ ace-task create "Phase 1" --child-of 100
Converted task 100 to orchestrator with subtask .01
Created subtask v.0.9.0+task.100.02
Path: .ace-taskflow/v.0.9.0/tasks/100-my-task/100.02-phase-1.s.md
```

The system:
1. Detects that task 100 is not an orchestrator
2. Auto-converts it (original content → subtask .01, new orchestrator file created)
3. Proceeds to create the requested child as subtask .02
4. Reports both actions in the output

**Same behavior for `ace-task move`**:
```
$ ace-task move 101 --child-of 100
Converted task 100 to orchestrator with subtask .01
Moved task 101 as subtask 100.02
```

**Already-orchestrator case**: No change — if the parent is already an orchestrator, skip conversion and proceed directly (current working behavior).

**Conversion fails**: If auto-conversion fails (e.g., task is already a subtask, or task not found), return the conversion error. Don't fall through to the "not an orchestrator" error.

### Interface Contract

```bash
# ace-task create --child-of (non-orchestrator parent)
$ ace-task create "My Subtask" --child-of 100
Converted task 100 to orchestrator with subtask .01
Created subtask v.0.9.0+task.100.02
Path: .ace-taskflow/.../100.02-my-subtask.s.md

# ace-task move --child-of (non-orchestrator parent)
$ ace-task move 101 --child-of 100
Converted task 100 to orchestrator with subtask .01
Moved task 101 as subtask 100.02

# Already an orchestrator — no conversion message
$ ace-task create "Another" --child-of 100
Created subtask v.0.9.0+task.100.03
Path: .ace-taskflow/.../100.03-another.s.md
```

**Error Handling:**
- Parent task not found: `"Parent task 100 not found. Create the parent task first."` (unchanged)
- Parent is already a subtask (can't become orchestrator): `"Failed to auto-convert task 100 to orchestrator: Task is already a subtask of 99"` (conversion error passed through)
- Subtask limit reached: `"Maximum subtask limit (99) reached for task 100"` (unchanged)

**Edge Cases:**
- Parent task already has status `done` or `archived`: Conversion should still work (status is orthogonal to structure)
- `ace-task create --child-of` where parent was just created moments ago: Works — the create command finds the task file, converts, adds child

### Success Criteria

- [ ] `ace-task create "X" --child-of <flat-task>` succeeds without prior manual conversion
- [ ] `ace-task move <task> --child-of <flat-task>` succeeds without prior manual conversion
- [ ] Auto-conversion creates orchestrator file and .01 subtask (same as `--child-of self`)
- [ ] Output reports both the conversion and the child operation
- [ ] Already-orchestrator parents work unchanged (no regression)
- [ ] Conversion errors propagate clearly (not swallowed)
- [ ] Existing test expectations updated to match new behavior
- [ ] Handbook docs (`draft.wf.md`, `draft-batch.wf.md`) updated — Pattern B simplified

### Validation Questions

- [x] Should auto-conversion be silent or reported? → **Reported in output** so the user knows their task structure changed
- [x] Should this apply to both `create --child-of` and `move --child-of`? → **Yes, both**
- [x] What if parent is a subtask itself (can't convert)? → **Pass through the conversion error**

## Objective

Eliminate the manual `ace-task move NNN --child-of self` intermediate step. When a user says "make this a child of that", the system should make it happen — the orchestrator conversion is an implementation detail, not a user concern. This was the most common failure mode during the task-281 drafting session and will trip every agent that follows the (currently incorrect) workflow docs.

## Scope of Work

- **User Experience Scope**: `ace-task create --child-of` and `ace-task move --child-of` accept non-orchestrator parents
- **System Behavior Scope**: `TaskManager.create_subtask` and `TaskManager.demote_to_subtask` auto-convert before proceeding
- **Interface Scope**: CLI output adds conversion message; error messages change for conversion failures

### Deliverables

#### Code Changes
- `TaskManager.create_subtask` — replace orchestrator check with auto-convert
- `TaskManager.demote_to_subtask` — same pattern

#### Test Updates
- Update `test_demote_to_subtask_parent_not_orchestrator` to expect success
- Add test for `create_subtask` with non-orchestrator parent

#### Documentation Updates
- `draft.wf.md` Pattern B — remove manual conversion step
- `draft-batch.wf.md` — same update

## Out of Scope

- Changing `convert_to_orchestrator` itself (it works correctly)
- Adding `--child-of self` as a sub-option of `create` (separate concern)
- Changing how orchestrator detection works (`is_orchestrator_directory?`)
