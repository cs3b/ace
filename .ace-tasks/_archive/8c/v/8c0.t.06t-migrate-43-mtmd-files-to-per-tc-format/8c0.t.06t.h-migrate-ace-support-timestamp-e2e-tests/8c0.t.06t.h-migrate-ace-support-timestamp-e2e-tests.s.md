# Migrate ace-support-timestamp E2E Tests

## Scope

Convert `.mt.md` files in `ace-support-timestamp/test/e2e/` to directory-based per-TC format, **filtered through the E2E Value Gate**. The migration inventory below shows the source material; actual TC count will be determined by Phase 0's coverage review. Highest source TC count of all packages but NO fixture files — pure CLI testing makes this a high candidate for Value Gate reduction.

**Source material:** 4 `.mt.md` files, 46 TCs (pre-review)
**Expected output:** Significantly fewer TCs after Value Gate filtering and consolidation

## Phase 0: Coverage Review & Change Plan

Before migrating, run the 3-stage pipeline to determine which TCs have genuine E2E value:

1. **Review:** `/ace:review-e2e-tests ace-support-timestamp` — produces coverage matrix (functionality × unit tests × E2E)
2. **Plan:** `/ace:plan-e2e-changes ace-support-timestamp` — classifies each existing MT TC as REMOVE / KEEP / MODIFY / CONSOLIDATE / ADD
3. **Approve plan** before proceeding to migration

The ace-lint migration showed ~60% of TCs duplicate unit test coverage. With 46 pure CLI TCs and no fixtures, expect significant reduction — many encode/decode roundtrip tests likely overlap with unit tests.

### Migration scope is determined by the plan:
- **KEEP TCs** → migrate to TS format as-is
- **MODIFY TCs** → migrate to TS format with changes applied
- **CONSOLIDATE TCs** → merge during migration (single TS TC from multiple MT TCs)
- **REMOVE TCs** → skip (don't create TS files, delete MT files)
- **ADD TCs** → create new TS TCs for uncovered features (if any)

## Technical Approach

### Minimal Setup (all 4 scenarios)

```yaml
setup: []
```

No `git-init`, no `copy-fixtures`, no `env` needed. Pure CLI tests that invoke `ace-timestamp encode` and `ace-timestamp decode` with various flags and inputs.

### Key Characteristics

- **No fixtures directory** needed for any scenario
- All test data is generated via bash commands (date, echo) or hardcoded strings in TC steps
- Config files (if any) are created dynamically within TCs
- Highest volume of TC extraction (46 individual files) but simplest per-file work
