# Migrate ace-review E2E Tests

## Scope

Convert `.mt.md` files in `ace-review/test-e2e/scenarios/` to directory-based per-TC format, **filtered through the E2E Value Gate**. The migration inventory below shows the source material; actual TC count will be determined by Phase 0's coverage review.

**Source material:** 6 `.mt.md` files, 37 TCs (pre-review)
**Expected output:** Fewer scenarios/TCs after Value Gate filtering and consolidation

## Phase 0: Coverage Review & Change Plan

Before migrating, run the 3-stage pipeline to determine which TCs have genuine E2E value:

1. **Review:** `/ace:review-e2e-tests ace-review` — produces coverage matrix (functionality × unit tests × E2E)
2. **Plan:** `/ace:plan-e2e-changes ace-review` — classifies each existing MT TC as REMOVE / KEEP / MODIFY / CONSOLIDATE / ADD
3. **Approve plan** before proceeding to migration

The ace-lint migration showed ~60% of TCs duplicate unit test coverage. Expect significant TC reduction.

### Migration scope is determined by the plan:
- **KEEP TCs** → migrate to TS format as-is
- **MODIFY TCs** → migrate to TS format with changes applied
- **CONSOLIDATE TCs** → merge during migration (single TS TC from multiple MT TCs)
- **REMOVE TCs** → skip (don't create TS files, delete MT files)
- **ADD TCs** → create new TS TCs for uncovered features (if any)

## Technical Approach

### Fixture Extraction

Most fixtures are YAML presets and Ruby source files. Extract all to `fixtures/` preserving directory structure:

```
fixtures/
├── .ace/review/
│   ├── config.yml
│   └── presets/
│       ├── code.yml, code-pr.yml, ...
├── lib/
│   ├── example.rb, helper.rb
├── test.rb
└── README.md
```

### Setup Patterns

All scenarios need: `git-init`, `copy-fixtures`, `env: { PROJECT_ROOT_PATH: "." }`

Additional per-scenario:
- **002:** Initial commit + staged changes (multi `run:` steps)
- **003:** Nested `.ace-taskflow/` directory structure in fixtures
- **004:** `requires: { tools: [gh] }` — external auth dependency
- **005, 006:** `requires: { env: [GOOGLE_API_KEY] }` — API key required

### TC Extraction

- Strip `ace-test-e2e-sh "$TEST_DIR"` and nested `bash << 'SANDBOX'` wrappers
- Preserve multi-command sequences in Steps code blocks
- Slug from title: e.g., `TC-004: Error - Circular Dependency Detection` → `TC-004-error-circular-dependency-detection.tc.md`

## Implementation Plan

### Phase 0: Review & Plan
- [ ] Run `/ace:review-e2e-tests ace-review` — capture coverage matrix
- [ ] Run `/ace:plan-e2e-changes ace-review` — get TC classifications
- [ ] Present plan for approval (expected: N REMOVE, N KEEP, N MODIFY, N CONSOLIDATE, N ADD)

### Phase 1: Informed Migration
- [ ] For each KEEP/MODIFY TC: extract to TS format following migration-pattern.md
- [ ] For each CONSOLIDATE group: merge into single TC during extraction
- [ ] For each ADD: create new TS TC based on plan
- [ ] Validate: `ace-test-e2e setup ace-review TS-REVIEW-{NNN}` for each scenario
- [ ] Remove all original `.mt.md` files
- [ ] Final verification: scenario count, TC count, no stale references

## Acceptance Criteria

- [ ] Final scenario count: 2-5 TCs per scenario (cost-aware guideline)
