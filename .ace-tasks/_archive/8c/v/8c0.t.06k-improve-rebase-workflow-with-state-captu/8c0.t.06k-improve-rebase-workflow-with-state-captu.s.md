# Improve rebase workflow with state capture and verification

## Behavioral Specification

### User Experience
- **Input**: Target branch to rebase against, optionally `--no-verify` to skip post-verification
- **Process**:
  1. Agent captures pre-rebase state to cache directory
  2. Attempts simple `git rebase` first
  3. If conflicts detected, aborts and switches to cherry-pick fallback
  4. After completion, verifies diff stats match expectations
- **Output**: Rebased branch with verification report showing pre/post comparison

### Expected Behavior

When a user invokes the rebase workflow:
1. **State capture phase**: Before any rebase operation, the workflow captures:
   - Full diff against merge-base (for recovery)
   - List of commit SHAs with messages (for cherry-pick fallback)
   - Diff statistics (files changed, insertions, deletions) for verification

2. **Strategy selection**:
   - Default: Simple `git rebase $target` - preserves exact commit history
   - Fallback: Cherry-pick each commit from saved list when conflicts detected

3. **Conflict handling**:
   - Simple rebase conflict: Abort immediately, switch to cherry-pick
   - Cherry-pick conflict: Pause for manual resolution per-commit

4. **Verification**: Compare pre/post diff stats to ensure rebase preserved intended changes

### Interface Contract

```bash
# Workflow invocation (in Claude Code)
/ace:rebase [target-branch]
/ace:rebase origin/main --no-verify

# Cache directory created at:
.cache/ace-git/{compact_id}-rebase/
├── metadata.yml        # Session info
├── commits.txt         # SHA + message list
├── pre-rebase.diff     # Full diff for recovery
├── pre-rebase.stats    # Stats for verification
└── post-rebase.stats   # Stats after completion (if verified)
```

**Error Handling:**
- Dirty working directory: Abort with message to commit or stash changes
- Invalid target branch: Report error with available branches
- Verification mismatch: Warn but don't fail (stats may differ slightly due to CHANGELOG)

**Edge Cases:**
- No commits to rebase: Report "Already up to date"
- Single commit: Use simple rebase (no cherry-pick benefit)
- Rebase already in progress: Detect and offer to continue/abort

## Objective

Replace the current reset-split default strategy with a simpler, more predictable approach:
1. Try simple rebase first (most common case works)
2. Fall back to cherry-pick for per-commit conflict resolution
3. Always capture state for recovery and verification

This improves reliability by catching unexpected changes (verification) and simplifies the common case (no conflicts = simple rebase).
