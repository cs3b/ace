## Behavioral Specification

### User Experience
- **Input**: User runs `ace-sim run` with either `--scenario <name>` or `--steps <csv>`, plus `--source` and at least one `--provider`.
- **Process**: Tool resolves scenario order from `.ace/sim/scenarios/*.yml` (or defaults), resolves step contracts from `.ace/sim/steps/*.md`, renders step prompts through `ace-bundle`, executes each step in sequence, validates YAML output schema, and chains step output to the next step.
- **Output**: Deterministic run artifacts under `.cache/ace-sim/simulations/<run-id>/` including `session.yml`, `stages/*.yml`, `synthesis.yml`, and `result.yml`, with CLI summary including run id, run dir, status, and result path.

### Expected Behavior
- Run ID uses default `ace-b36ts` token style.
- Scenario order is authoritative when `--scenario` is provided.
- `--scenario` and `--steps` can both be passed, but scenario wins and tool prints an explicit notice.
- Output must be strict YAML matching step contract requirements.
- Invalid YAML or missing required keys marks provider run failed (`implementation failure`) and influences step/session status accordingly.
- Default `next-phase` scenario uses ordered steps: `draft`, `plan`, `work`.

### Interface Contract
```bash
# Scenario-driven run (authoritative order from scenario file)
ace-sim run \
  --scenario next-phase \
  --source .ace-taskflow/v.0.9.0/ideas/8pp5z7-llm-enhance/idea.idea.s.md \
  --provider codex:mini

# Steps-only run
ace-sim run \
  --steps draft,plan,work \
  --source .ace-taskflow/v.0.9.0/ideas/8pp5z7-llm-enhance/idea.idea.s.md \
  --provider codex:mini

# Both provided: scenario wins and emits notice
ace-sim run \
  --scenario next-phase \
  --steps work,plan,draft \
  --source .ace-taskflow/v.0.9.0/ideas/8pp5z7-llm-enhance/idea.idea.s.md \
  --provider codex:mini
```

**Error Handling**:
- Unknown scenario: non-zero exit with known scenario list.
- Missing step contract file: step fails with `implementation failure`.
- Invalid provider output YAML/schema: run fails or becomes partial depending on successful providers.

**Edge Cases**:
- Existing run-id directory collision: regenerate run-id and retry.
- Provider returns markdown/text: rejected by schema validation.

### Success Criteria
- [ ] Run IDs match `ace-b36ts` style and are collision-safe.
- [ ] Scenario and step config are loaded from `.ace/sim/scenarios/*.yml` and `.ace/sim/steps/*.md` with cascade defaults.
- [ ] `next-phase` default order is `draft -> plan -> work`.
- [ ] Strict schema validation prevents false `ok` states for malformed outputs.
- [ ] `result.yml` is written and surfaced in CLI output on success.
- [ ] Tests cover CLI precedence, ordering, validation failures, and happy-path artifacts.

### Validation Questions
- [ ] Should `work` eventually become the only terminal step for `next-phase`, or remain extensible for future scenarios?
- [ ] Should result summary in CLI be full YAML or a compact subset of keys by default?
