# Rename Timestamp Format Options to Precision-Based Names

## Description

Rename timestamp format options to clearly communicate the precision they provide. The current names mix technical concerns (character length) with semantic meaning. New names should immediately tell users what time precision they get.

## Behavioral Specification

### Current vs New Naming

| Current | New | Length | Precision | Change Type |
|---------|-----|--------|-----------|-------------|
| `compact` | `2sec` | 6 chars | ~1.85s (default) | Rename |
| `month` | `month` | 2 chars | month | No change |
| `week` | `week` | 3 chars | ISO week-in-month | No change |
| `day` | `day` | 3 chars | day-in-month | No change |
| `hour` | `40min` | 4 chars | 40-min block | **Fix + Rename** |
| `high_7` | `50ms` | 7 chars | ~50ms | Rename |
| `high_8` | `ms` | 8 chars | ~1.4ms | Rename |

### Critical Fix: 4-char Format

The current 4-char "hour" format incorrectly encodes **hour (0-23)**. The 4-char format is designed for 40-minute block precision to be consistent with position 4 of the compact format. This bug must be fixed:

**Current (bug):**
```ruby
# encode_hour incorrectly uses hour (0-23)
hour = time.hour.clamp(0, 23)
result << encode_value(hour, 1, alphabet)
```

**Correct implementation:**
```ruby
# Use 40-min blocks like position 4 of compact format
minutes_of_day = (time.hour * 60) + time.min
block = minutes_of_day / BLOCK_MINUTES  # 0-35
result << encode_value(block, 1, alphabet)
```

### Interface Contract

```bash
# New format names
ace-timestamp encode now --format 2sec    # default, ~1.85s precision
ace-timestamp encode now --format month   # month precision
ace-timestamp encode now --format week    # ISO week precision
ace-timestamp encode now --format day     # day precision
ace-timestamp encode now --format 40min   # 40-minute block precision
ace-timestamp encode now --format 50ms    # ~50ms precision
ace-timestamp encode now --format ms      # ~1.4ms precision

# Old names no longer work (breaking change)
ace-timestamp encode now --format compact  # ERROR: unknown format
ace-timestamp encode now --format hour     # ERROR: unknown format
ace-timestamp encode now --format high_7   # ERROR: unknown format
```

**Error Handling:**
- `--format compact` → Error: "Unknown format 'compact'. Did you mean '2sec'?"
- `--format hour` → Error: "Unknown format 'hour'. Did you mean '40min'?"
- `--format high_7` → Error: "Unknown format 'high_7'. Did you mean '50ms'?"

## Technical Approach

### Architecture Pattern

The changes follow the existing ATOM architecture:
- **Atoms** (format_specs.rb, compact_id_encoder.rb): Core encoding/decoding logic
- **CLI** (encode.rb, decode.rb): User interface layer
- No new patterns introduced; this is a refactoring task

### Symbol Naming Convention

New format symbols use precision-based names:
- `:2sec` - Ruby symbols can start with digits when quoted (:"2sec")
- `:40min`, `:50ms`, `:ms` - Clear precision indicators
- `:month`, `:week`, `:day` - Unchanged (already precision-based)

### Method Naming Convention

Internal methods renamed to match precision:

| Current Method | New Method |
|----------------|------------|
| `encode_compact` | `encode_2sec` |
| `decode_compact` | `decode_2sec` |
| `encode_hour` | `encode_40min` |
| `decode_hour` | `decode_40min` |
| `encode_high_7` | `encode_50ms` |
| `decode_high_7` | `decode_50ms` |
| `encode_high_8` | `encode_ms` |
| `decode_high_8` | `decode_ms` |

Public API methods (`encode`, `decode`) remain unchanged for backward compatibility.

## Risk Assessment

### Technical Risks

- **Risk:** Quoted symbols (:"2sec", :"40min") may cause issues in some contexts
  - **Probability:** Low
  - **Impact:** Medium
  - **Mitigation:** Ruby supports quoted symbols; test thoroughly
  - **Rollback:** Revert to descriptive names without special chars if issues arise

- **Risk:** Breaking change affects downstream consumers
  - **Probability:** Low (pre-1.0, mono-repo)
  - **Impact:** Low
  - **Mitigation:** Per ADR-024, no backward compatibility needed pre-1.0
  - **Rollback:** N/A - intentional breaking change

### Integration Risks

- **Risk:** Config files using old format names will break
  - **Probability:** Medium
  - **Impact:** Low
  - **Mitigation:** Update default config; users will see clear error messages
  - **Monitoring:** Check CI for config-related failures

## Out of Scope

- Backward compatibility aliases (breaking change is acceptable for pre-1.0 per ADR-024)
- Configuration file support for format defaults
- Deprecation warnings for old names
