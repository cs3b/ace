# Update ace-* gems to use config.merge() instead of DeepMerger

## Objective

Reduce direct usage of `Ace::Config::Atoms::DeepMerger.merge()` in ace-* gems by using the new `config.merge()` API where configs are already represented as Config objects. This provides a cleaner API and encapsulates merge logic.

## Background: Analysis of DeepMerger Usage

After analyzing all DeepMerger.merge() calls in the codebase, they fall into these categories:

### Category 1: Config Model Internal (Keep as-is)
These are internal to ace-config and should NOT be changed:
- `ace-config/lib/ace/config/models/config.rb:67` - The `with()/merge()` method itself
- `ace-config/lib/ace/config/organisms/config_resolver.rb` - Internal resolver logic
- `ace-config/lib/ace/config/molecules/yaml_loader.rb` - Internal file loading

### Category 2: Raw Hash Merging (Legacy Pattern - Lower Priority)
These work with raw hashes, not Config objects. Refactoring would require larger changes to return Config objects from loading methods:
- `ace-test-runner/lib/ace/test_runner/molecules/config_loader.rb:63,70` - Working with symbolized hashes
- `ace-taskflow/lib/ace/taskflow/molecules/config_loader.rb:52` - Working with flat hashes
- `ace-nav/lib/ace/nav/molecules/config_loader.rb:58` - Working with raw YAML hashes
- `ace-support-core/lib/ace/core.rb:156,187` - Legacy backwards compat layer
- `ace-support-core/lib/ace/core/config_discovery.rb:94` - Legacy discovery

### Category 3: Merge Utility Methods (Candidates for Change)
These use DeepMerger as a utility but could use Config.merge() if converted to work with Config objects:
- `ace-search/lib/ace/search/molecules/preset_manager.rb:38` - Merging preset with options
- `ace-review/lib/ace/review/molecules/subject_extractor.rb:85` - Deep merge arrays utility
- `ace-git/lib/ace/git/molecules/config_loader.rb:23,42,45,73` - Multiple merge points

## Risk Assessment

### Technical Risks

- **Risk:** Changing return types from Hash to Config could break callers
  - **Probability:** High (if we change return types)
  - **Impact:** High
  - **Mitigation:** Keep existing hash-based APIs, only change where Config already used
  - **Rollback:** Revert individual gem changes

- **Risk:** Low benefit for high effort in raw-hash contexts
  - **Probability:** Medium
  - **Impact:** Low (wasted effort)
  - **Mitigation:** Only change where clear benefit exists

## Out of Scope

- ❌ Changing ace-config internals (ConfigResolver, YamlLoader)
- ❌ Changing ace-support-core legacy layer
- ❌ Refactoring all config loaders to return Config objects (future task)
- ❌ Test file changes (tests validating DeepMerger behavior)

## Recommendation

After analysis, this task may result in **no code changes** if the current raw-hash patterns are appropriate for each gem's architecture. The Config.merge() API is most valuable when:
1. You already have a Config object
2. You want to merge CLI options into loaded config
3. You want the merged result as a Config object

Most current usages are raw-hash merging before creating domain models, which is a valid pattern.

## Related: Ace::Core.get Refactor Opportunity

**Location:** `ace-support-core/lib/ace/core.rb:239-243`

The `Ace::Core.get` method currently duplicates the namespace-to-pattern logic that `ConfigResolver#resolve_namespace` now provides:

```ruby
# Current pattern (duplicated logic)
patterns = if file
  ["#{namespace_or_keys}/#{file}.yml", "#{namespace_or_keys}/#{file}.yaml"]
else
  ["#{namespace_or_keys}/*.yml", "#{namespace_or_keys}/*.yaml"]
end
config = resolver.resolve_for(patterns)
```

**Recommendation:** After this task, consider updating `Ace::Core.get` to use `resolve_namespace`:

```ruby
# Proposed (uses resolve_namespace for single-file case)
config = if file
  resolver.resolve_namespace(namespace_or_keys, filename: file)
else
  resolver.resolve_for(["#{namespace_or_keys}/*.yml", "#{namespace_or_keys}/*.yaml"])
end
```

Note: The glob pattern case (`*.yml`) still needs `resolve_for` since `resolve_namespace` doesn't support globs.

## Implementation Summary (2025-12-30)

All three DeepMerger usage cases were refactored to use `Config.merge()` for consistent merge strategy support.
This enables future per-key merge strategies via `_merge` directive (Task 157.13).

### Changes Made

#### 1. ace-search PresetManager (`ace-search/lib/ace/search/molecules/preset_manager.rb`)
```ruby
# Before
Ace::Config::Atoms::DeepMerger.merge(preset, options)

# After
Ace::Config::Models::Config.new(preset, source: "preset:#{preset_name}")
  .merge(options)
  .to_h
```

#### 2. ace-review SubjectExtractor (`ace-review/lib/ace/review/molecules/subject_extractor.rb`)
```ruby
# Before (in merge_typed_subject_configs)
subjects.reduce({}) do |merged, subject|
  config = resolve_single_subject(subject)
  Ace::Config::Atoms::DeepMerger.merge(merged, config, array_strategy: :coerce_union)
end

# After
initial_config = Ace::Config::Models::Config.new({}, merge_strategy: :coerce_union)
subjects.reduce(initial_config) do |acc, subject|
  config_hash = resolve_single_subject(subject)
  acc.merge(config_hash)
end.to_h
```
Also updated `deep_merge_arrays` private method to use Config.merge() internally.

#### 3. ace-git ConfigLoader (`ace-git/lib/ace/git/molecules/config_loader.rb`)
```ruby
# Before (in load and load_for_gem)
config_hash = DeepMerger.merge(global_diff_config, gem_diff_config)
config_hash = DeepMerger.merge(config_hash, instance_config)

# After
config_hash = Ace::Config::Models::Config.new(global_diff_config, source: "git_global")
  .merge(gem_diff_config)
  .merge(instance_config)
  .to_h
```
Also updated `extract_diff_config` internal merge to use Config.merge().

#### 4. ace-support-core Ace::Core.get (`ace-support-core/lib/ace/core.rb`)
Updated to use `resolve_namespace` for single-file case and `resolve_file` (instead of deprecated
`resolve_for`) for glob pattern case. This doesn't use Config.merge() since it's pure resolution,
not merging - the cascade merging happens inside ConfigResolver.
