# ace-retro Core Operations and Model

## Behavioral Specification

### User Experience
- **Input**: Retro content, references, type selection, task references
- **Process**: RetroManager organism handles all CRUD operations. Self-contained molecules — no dependency on external FolderMover/FieldUpdater (same pattern as ace-idea).
- **Output**: Retro objects created/loaded/updated with full metadata

### Expected Behavior

Build the complete core operations layer for retros, following ace-idea's proven architecture (ace-idea `v0.2.0`, task 291):

1. **RetroScanner**: Wraps DirectoryScanner for `.retro.md` files. Returns scan results with raw b36ts IDs.
2. **RetroResolver**: Wraps ShortcutResolver for raw retro IDs (no type marker, same as ideas). Suffix match on last 3 chars.
3. **RetroLoader**: Loads retro from directory, parses frontmatter + body, lists folder contents (future: reports, action-item files).
4. **RetroCreator**: Full creation with b36ts ID, slug generation, folder+file creation. Supports:
   - `--type`: Set retro type (standard, conversation-analysis, self-review)
   - `--task-ref`: Link retro to a task ID
   - `--move-to`: Place in specific folder on creation (only `_archive` supported)
5. **RetroMover**: Self-contained molecule (like `ace-idea/lib/ace/idea/molecules/idea_mover.rb`). Handles folder normalization via SpecialFolderDetector, archive chronological sub-paths via DatePartitionPath. **Must handle `Errno::EXDEV`** for cross-filesystem moves — try `File.rename`, rescue with `FileUtils.cp_r` + `FileUtils.rm_rf` (retro lesson from task 291).
6. **RetroManager**: Orchestrates create, show, list, update, move.

### Retro Lessons (from task 291)

- **Cross-platform `File.rename`**: RetroMover must handle `Errno::EXDEV` for cross-filesystem moves. Pattern: `File.rename` → rescue `Errno::EXDEV` → `FileUtils.cp_r` + `FileUtils.rm_rf`. See `ace-idea/lib/ace/idea/molecules/idea_mover.rb#atomic_move`.
- **`FieldArgumentParser.parse` expects array**: Use `parse([arg])` not `parse(arg)` when updating fields.
- **`gem_root` depth**: Document path depth with inline comment showing breakdown (e.g., `# lib/ace/retro/molecules/ → 4 levels up`).

### Interface Contract (Ruby API)

```ruby
manager = Ace::Retro::Organisms::RetroManager.new

# Create
retro = manager.create("Sprint Review",
                        type: "standard",
                        tags: ["sprint", "team"],
                        task_ref: "q7w")
# => Retro(id: "8ppq7w", status: "active", type: "standard", ...)

# Show
retro = manager.show("q7w")
# => Retro with full metadata + folder contents list

# List
retros = manager.list(status: "active", type: "standard", tags: ["sprint"])
# => Array of Retro objects

# Update
manager.update("q7w", set: { "status" => "done" }, add: { "tags" => "reviewed" })

# Move
manager.move("q7w", to: "archive")
# => Moves to _archive/2026-02/8ppq7w-sprint-review/
```

### Success Criteria

- [ ] RetroScanner finds all `.retro.md` files recursively, excluding `.s.md` (task) and `.idea.s.md` (idea) files
- [ ] RetroResolver resolves raw b36ts shortcuts (last 3 chars) with duplicate warnings
- [ ] RetroLoader parses frontmatter + body, enumerates additional files in retro folder
- [ ] RetroCreator generates raw b36ts ID, creates `{id}-{slug}/{id}-{slug}.retro.md`
- [ ] `--type` correctly sets retro type in frontmatter
- [ ] `--task-ref` stores linked task ID in frontmatter
- [ ] `--move-to` on create places retro in target folder immediately
- [ ] Archive moves organize by chronological sub-path (`_archive/YYYY-MM/`)
- [ ] RetroManager orchestrates all operations with config-driven root directory
- [ ] Retro model holds: id, status, title, type, tags, content, path, file_path, special_folder, created_at, task_ref, folder_contents
- [ ] Full test coverage at atoms/molecules/organisms/models layers
