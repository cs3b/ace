# Fix pr: array handling in ace-context - multiple PRs only shows one

## Objective

Fix bug where multiple PRs in array format (`pr: [75, 77]`) only shows one PR diff in output. This blocks the ability to review multiple PRs together.

## Problem

When a preset has `pr: [75, 77]` (YAML array), only one PR diff appears in output.

## Root Cause

`has_processed_section_content?` in context_loader.rb:822-829 does NOT check for `_processed_diffs`, causing `merge_contexts` to discard PR diffs when determining if a context has processed section content.

```ruby
# Current - missing _processed_diffs check:
def has_processed_section_content?(context)
  context.sections.any? do |_name, data|
    processed_files = data[:_processed_files] || data['_processed_files'] || []
    processed_commands = data[:_processed_commands] || data['_processed_commands'] || []
    processed_files.any? || processed_commands.any?  # <-- Missing _processed_diffs!
  end
end
```

## Technical Approach

### Architecture Pattern

**Pattern:** Consistency fix across helper methods that detect processed content

**Rationale:** The `_processed_diffs` key follows the same pattern as `_processed_files` and `_processed_commands` but was missed in several helper methods. The fix aligns all helper methods to check all three processed content types.

**Integration:** No new patterns introduced - simply extending existing pattern to include diffs.

### Technology Stack

- **No new dependencies required**
- **Ruby standard library only**
- **Follows existing ace-context patterns**

### Implementation Strategy

1. Fix primary detection method (`has_processed_section_content?`)
2. Fix secondary detection methods (`has_diffs_content?` in 2 locations)
3. Fix merge method to preserve `_processed_diffs` during section merging
4. Add ace-review comma syntax as enhancement
5. Add regression tests
