# Fix ace-git-commit path validation for deleted and renamed files

## Behavioral Specification

### User Experience
- **Input**: User runs `ace-git-commit scripts/` where `scripts/` contains deleted files
- **Process**: Tool validates paths, stages changes, generates commit message
- **Output**: Successful commit including the deleted files

### Expected Behavior

When a user specifies a path to `ace-git-commit`, the tool should accept:
1. Paths that exist on the filesystem (current behavior)
2. Paths that have git changes (deleted files, renamed files)

Currently, the tool rejects deleted paths with:
```
✗ Invalid path(s): scripts/
These paths do not exist. Please check the paths and try again.
```

After this fix, deleted and renamed paths will be recognized as valid.

### Interface Contract

```bash
# Deleted files - should work
git rm file.txt
ace-git-commit file.txt -i "remove file"
# Expected: Commit created successfully

# Directory with deleted files - should work
git rm scripts/*.sh
ace-git-commit scripts/ -i "cleanup scripts"
# Expected: Commit created successfully

# Renamed files (manual mv) - should work
mv old.txt new.txt
git add -A
ace-git-commit old.txt new.txt -i "rename file"
# Expected: Commit created successfully
```

**Error Handling:**
- Truly nonexistent paths (no git changes): Still rejected with current error message
- Git command failure: Falls back to filesystem-only validation (graceful degradation)

### Success Criteria

- [ ] Deleted files are accepted as valid paths
- [ ] Directories containing only deleted files are accepted
- [ ] Renamed file old paths are accepted (from `git status` R status)
- [ ] Truly nonexistent paths are still rejected
- [ ] Git errors are handled gracefully (fallback to filesystem check)
- [ ] All existing tests continue to pass

## Objective

Fix a usability bug where `ace-git-commit` cannot commit deleted files when specifying their paths directly, despite `git add` handling this correctly.
