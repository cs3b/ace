# Documentation strategy for gem publishing

## Behavioral Specification

### User Experience
- **Input**: Developers/agents reading ACE documentation to understand, install, and use gems
- **Process**: Clear navigation from vision → quick start → detailed usage, with no content duplication
- **Output**: Users can install and use any ACE gem within 5 minutes, understand the philosophy, and find detailed reference when needed

### Expected Behavior

The documentation should provide a clear hierarchy:

1. **Repository landing page (README.md)**: Concise overview, gem table, quick start - scannable in 2 minutes
2. **Vision document (docs/vision.md)**: Why ACE exists, core principles, the problem it solves
3. **Gem-level documentation**: Each gem has README.md (essential) + docs/usage.md (comprehensive)
4. **No duplication**: Each fact lives in exactly one place, other docs link to it

Documentation tiers by gem type:
- **Full tier** (ace-context, ace-review, ace-taskflow, ace-git-commit): Essential README + comprehensive usage.md
- **Standard tier** (ace-git, ace-lint, ace-docs, ace-nav, ace-search, ace-prompt): Essential README + standard usage.md
- **Minimal tier** (ace-support-*, ace-config): Concise README, optional usage.md

### Interface Contract

```
Repository docs structure:
├── README.md              # Landing page (≤150 lines ideally)
├── docs/
│   ├── vision.md          # Philosophy + principles (merged from philosophy + what-do-we-build)
│   ├── architecture.md    # Technical architecture
│   ├── tools.md           # CLI reference
│   └── ace-gems.g.md      # Gem development guide

Gem docs structure:
ace-*/
├── README.md              # 150-300 lines: install, quick start, features, config summary
├── CHANGELOG.md           # Version history
└── docs/
    └── usage.md           # Full CLI reference, configuration, examples, troubleshooting
```
