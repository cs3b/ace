# Add preset composition to ace-review (inspired by ace-context)

## Behavioral Specification

### User Experience

**Input**: Review preset YAML files that reference other presets via a `presets:` array at the root level
**Process**: The ace-review system loads and recursively merges referenced presets, validating for circular dependencies
**Output**: Fully composed preset with merged instructions, subjects, and configuration from all referenced presets

### Expected Behavior

Users can create base review presets (like `code.yml`) containing common instructions and configuration, then compose specific presets (like `code-pr.yml`, `code-wip.yml`) that extend the base with specialized subjects or overrides.

**Composition Example:**

Base preset (`code.yml`):
```yaml
description: "Base code review"
instructions:
  base: "prompt://base/system"
  context:
    sections:
      format:
        title: "Format Guidelines"
        files:
          - "prompt://format/detailed"
context: "project"
```

Composed preset (`code-pr.yml`):
```yaml
description: "Code PR review"
presets:
  - code
subject:
  context:
    sections:
      code_changes:
        title: "PR Changes"
        diffs:
          - "origin...HEAD"
```

When loading `code-pr.yml`, the system:
1. Detects the `presets: [code]` reference
2. Recursively loads `code.yml`
3. Merges the configurations (arrays concatenate+deduplicate, hashes deep merge, scalars last-wins)
4. Returns fully composed preset with both base instructions and specific subject

### Interface Contract

**CLI Interface:**
```bash
# Use composed preset (transparent to user)
ace-review --preset code-pr

# System internally:
# 1. Loads code-pr.yml
# 2. Detects presets: [code]
# 3. Loads code.yml
# 4. Merges configurations
# 5. Executes review with composed preset
```

**YAML Configuration Interface:**
```yaml
# Top-level preset composition
presets:
  - base-preset-name
  - another-preset-name

# Configuration follows standard ace-review preset structure
description: "Preset description"
instructions: { ... }
subject: { ... }
model: "model-name"
```

**Error Handling:**
- Circular dependency (A→B→A): Error with clear message showing dependency chain
- Missing preset reference: Error listing available presets
- Maximum nesting depth exceeded (>10): Error indicating depth limit
- Invalid YAML structure: Error with file location and parsing issue

**Edge Cases:**
- Preset with no references: Works normally (backward compatible)
- Multi-level composition (A→B→C): Handled recursively with depth limit
- Multiple base presets: Merged in order (first to last)
- Conflicting configurations: Last wins for scalars, merge for hashes, concat+dedup for arrays

## Objective

Enable DRY (Don't Repeat Yourself) preset configuration by allowing users to compose review presets from reusable base configurations, similar to how ace-context handles preset composition. This reduces duplication between similar presets (like pr.yml and wip.yml) that differ only in their subject definitions.

## Technical Research

### Architecture Pattern Analysis

**ace-context Implementation Study:**
- Uses `PresetManager#load_preset_with_composition` for recursive loading (lines 37-98)
- `PresetValidator` atom handles validation (circular detection, reference extraction)
- `merge_preset_data` method implements smart merging (lines 102-215)
- Section-level composition handled separately by `SectionProcessor`

**ace-review Current Implementation:**
- `PresetManager` loads single presets without composition
- No circular dependency detection
- Simple merge with defaults only
- Cache uses instance variable `@preset_cache`

**Integration Approach:**
- Adapt ace-context's composition pattern to ace-review's structure
- Create new `PresetValidator` atom following ace-review's ATOM architecture
- Update `PresetManager` molecule with composition methods
- Maintain backward compatibility with existing presets

### Merging Strategy

**Arrays (files, diffs, commands):**
```ruby
merged_array = (array1 + array2).uniq  # Concatenate and deduplicate
```

**Hashes (instructions, subject, context):**
```ruby
merged_hash = deep_merge_hash(hash1, hash2)  # Recursive merge
```

**Scalars (description, model, output_format):**
```ruby
merged_value = value2 || value1  # Last wins (nil-safe)
```

### Validation Strategy

**Circular Dependency Detection:**
- Track visited presets in Set during recursion
- Check if preset already in chain before loading
- MAX_DEPTH = 10 to prevent infinite loops

**Preset Existence:**
- Validate referenced presets exist before composition
- Provide helpful error with available presets list

**Reference Extraction:**
- Extract `presets` key from YAML (both string and symbol keys)
- Convert to array if single value provided
- Return empty array if key missing (backward compatible)

## Implementation Notes

### Key Design Decisions

**Composition at Root Level:**
- Top-level `presets:` array for preset-level composition
- Distinct from section-level `presets` in `instructions.context.sections`
- Clear separation of concerns

**Merging Strategy:**
- Arrays: Concatenate + deduplicate (order preserved from first to last)
- Hashes: Deep merge with recursive handling
- Scalars: Last wins (rightmost preset in chain)

**Cache Strategy:**
- Composed presets cached after full composition
- Cache key is preset name (not composition chain)
- Cache invalidation handled by instance lifecycle (per-execution)

**Error Handling:**
- Early validation before composition begins
- Clear error messages with context (preset chain, available presets)
- Graceful degradation when possible

### Potential Gotchas

**Section-Level vs Top-Level Presets:**
- Top-level `presets:` for preset composition (NEW)
- Section-level `presets` in instructions.context.sections (EXISTING)
- Both coexist without conflict

**Metadata Tracking:**
- `composed_from` metadata tracks composition chain
- Useful for debugging and understanding inheritance
- Stripped before caching to avoid confusion

**Performance Considerations:**
- Recursive loading has O(n*d) complexity (n presets, d depth)
- Cache mitigates repeated composition overhead
- MAX_DEPTH=10 prevents runaway recursion
