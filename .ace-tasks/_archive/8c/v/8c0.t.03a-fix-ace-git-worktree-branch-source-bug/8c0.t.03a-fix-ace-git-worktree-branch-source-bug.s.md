# Fix ace-git-worktree branch source bug

## Behavioral Specification

### User Experience
- **Input**: User runs `ace-git-worktree create --task 123` from a feature branch (e.g., `feature-x`)
- **Process**: System creates a new worktree with a task branch based on the CURRENT branch
- **Output**: New worktree created with branch `123-task-name` that has `feature-x` as its parent/start-point

### Expected Behavior

When a user creates a worktree from a non-main branch, the new worktree's branch should be based on the branch they're currently on, not the HEAD of the main worktree.

**Current (buggy) behavior:**
```bash
# User is on branch 'feature-x' in main worktree
ace-git-worktree create --task 123
# Creates: git worktree add <path> -b 123-branch
# Result: New branch based on HEAD of main worktree (often 'main'), NOT 'feature-x'
```

**Expected behavior:**
```bash
# User is on branch 'feature-x' in main worktree
ace-git-worktree create --task 123
# Creates: git worktree add <path> -b 123-branch feature-x
# Result: New branch based on 'feature-x' (current branch)
```

### Interface Contract

```bash
# Existing CLI (no change to interface)
ace-git-worktree create --task 123

# New optional flag for explicit source control
ace-git-worktree create --task 123 --source main
ace-git-worktree create --task 123 --source feature-x
ace-git-worktree create --task 123 --source HEAD~3

# Default behavior change
# Without --source: uses current branch (git rev-parse --abbrev-ref HEAD)
# With --source: uses specified ref
```

**Error Handling:**
- Invalid source ref: "Error: Source ref 'invalid-branch' does not exist"
- Detached HEAD without --source: "Warning: HEAD is detached. Using commit SHA as start-point."

**Edge Cases:**
- Detached HEAD state: Use current commit SHA as start-point
- Already on target branch: Create worktree pointing to existing branch (no -b)
- Source branch doesn't exist: Clear error message

### Success Criteria

- [ ] **Branch ancestry correct**: New worktree branch is based on current branch, not main worktree HEAD
- [ ] **Explicit source control**: `--source` flag allows specifying any valid git ref as start-point
- [ ] **Backward compatible**: Existing workflows work without modification (just with corrected behavior)
- [ ] **Clear error messages**: Invalid source refs produce helpful error messages
- [ ] **Detached HEAD handled**: Graceful handling when HEAD is detached

## Objective

Fix the bug where `ace-git-worktree create` uses the main worktree HEAD instead of the current branch when creating new worktree branches. This causes unexpected behavior when users work from feature branches and expect new worktrees to be based on their current work.

## Scope of Work

- **Bug Location**: `ace-git-worktree/lib/ace/git/worktree/molecules/worktree_creator.rb:323`
- **Root Cause**: `git worktree add <path> -b <branch>` without start-point uses main worktree HEAD
- **Fix**: Add current branch detection and pass as start-point to git worktree command

## Technical Analysis

### Root Cause

The bug is in `worktree_creator.rb:323`:
```ruby
result = Atoms::GitCommand.worktree("add", worktree_path, "-b", branch_name, timeout: @timeout)
```

This executes: `git worktree add <path> -b <branch>` without a start-point.

**Git behavior**: When no start-point is provided, git uses HEAD of the main worktree, not the current branch. This is documented git behavior but counterintuitive for users who expect the new branch to be based on their current working branch.

**Contrast with working code**: The `create_worktree_with_tracking` method (line 522-526) already does this correctly by passing the remote branch as start-point.

### Solution Approach

1. **Detect current branch**: Use `git rev-parse --abbrev-ref HEAD` to get current branch name
2. **Handle detached HEAD**: If detached, use `git rev-parse HEAD` to get commit SHA
3. **Pass as start-point**: Modify `create_worktree` to accept and use start-point parameter
4. **Add --source flag**: Allow explicit override via CLI

## Execution Steps

### Phase 1: Core Bug Fix

- [ ] Add `detect_current_branch` method to `Atoms::GitCommand`
  ```ruby
  # Returns branch name or commit SHA if detached
  def self.current_branch
    result = execute("rev-parse", "--abbrev-ref", "HEAD")
    if result[:success] && result[:output].strip != "HEAD"
      result[:output].strip
    else
      # Detached HEAD - return commit SHA
      sha_result = execute("rev-parse", "HEAD")
      sha_result[:success] ? sha_result[:output].strip : nil
    end
  end
  ```
