# Create ace-test package and consolidate testing documentation

## Behavioral Specification

### User Experience

- **Input**: Developer runs `ace-bundle project` or `ace-nav guide://testing-*`
- **Process**: Testing knowledge is discoverable via protocols; no 1104-line file in bundle
- **Output**: Lean bundle (~1000 lines), testing guides accessible via `ace-test` package

### Expected Behavior

**Current State Problem**:
- `docs/testing-patterns.md` is 1104 lines (52% of 2135-line bundle)
- Testing workflows scattered in `ace-taskflow` (create-test-cases, fix-tests)
- Testing guides scattered in `ace-test-runner`, `ace-docs`
- No single source of truth for testing best practices

**Desired State**:
1. New `ace-test` package exists (similar structure to `ace-handbook`)
2. All testing knowledge consolidated in one location
3. Testing resources discoverable via `guide://testing-*`, `agent://test*`, `wfi://create-test-cases` protocols
4. `docs/testing-patterns.md` deleted entirely
5. Project bundle reduced from 2135 to <1000 lines (~50% reduction)

  - **Answer**: Target **<1100 lines**. Testing-patterns.md contributes ~1103 lines to the 2137-line bundle. After removal, bundle will be ~1034 lines. Some overhead from other docs means realistic target is <1100 lines (~48% reduction).

- [x] **Agent Scope**: Are the 3 proposed agents (test, mock, profile-tests) sufficient?
  - **Answer**: Yes. These cover the main testing scenarios: running tests (test.ag.md), generating mocks (mock.ag.md), and profiling slow tests (profile-tests.ag.md). Additional agents can be added later.

- [x] **Guide Split**: Should testing-patterns.md be split into 8 guides, or fewer/more?
  - **Answer**: Split into **6 guides** (reduced from 8). Combine related topics:
    1. `testing-philosophy.g.md` - Pyramid, IO isolation (lines ~29-66)
    2. `test-organization.g.md` - Flat structure, naming (lines ~68-108)
    3. `mocking-patterns.g.md` - Git, HTTP, subprocess, ENV (lines ~109-520)
    4. `test-performance.g.md` - Targets, optimization, zombie mocks (lines ~397-1074)
    5. `testable-code-patterns.g.md` - Status codes, exceptions (lines ~521-680)
    6. `quick-reference.g.md` - TL;DR section (lines ~14-27)

- [x] **Tech Guides**: Confirm we want ALL technology-specific guides moved (including JS, Python, Rust)?
  - **Answer**: Yes. Move ALL from ace-test-runner/handbook/guides/testing/ and ace-test-runner/handbook/guides/test-driven-development-cycle/. This provides single source of truth.

- [x] **ace-handbook Pattern**: Is pure workflow package (no CLI) the right model?
  - **Answer**: Yes. Like ace-handbook, ace-test will have minimal lib/ (just version.rb), no executable, and focus on handbook/ content for protocol discovery.

## Objective

Reduce project bundle size by 50% while consolidating scattered testing knowledge into a single, discoverable package. This improves onboarding experience (leaner context) and provides a single source of truth for testing best practices.

- **System Behavior Scope**:
  - New `ace-test` package installed and discoverable
  - Protocol sources configured for guides, agents, workflows, templates
  - All testing knowledge accessible without bundle bloat

- **Interface Scope**:
  - Protocol discovery (`ace-nav guide://`, `agent://`, `wfi://`)

## Technical Approach

### Architecture Pattern
- **Pure Workflow Package**: Following ace-handbook pattern - minimal Ruby code, focus on handbook/ content
- **Protocol Discovery**: Using .ace-defaults/nav/protocols/ for guide, agent, wfi, tmpl sources
- **No Runtime Dependencies**: Gem only provides content for protocol resolution

### Technology Stack
- Ruby gem (gemspec with standard structure)
- YAML for protocol source configuration
- Markdown for all content (guides, workflows, agents)
