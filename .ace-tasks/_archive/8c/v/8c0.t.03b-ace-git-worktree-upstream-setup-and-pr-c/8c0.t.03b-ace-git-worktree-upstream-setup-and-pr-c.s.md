# ace-git-worktree: Upstream Setup and PR Creation

## Behavioral Specification

### User Experience
- **Input**: User runs `ace-git-worktree create --task 125` to create a worktree for a task
- **Process**: After worktree is created locally, the system automatically:
  1. Pushes the new branch to remote with upstream tracking (`-u` flag)
  2. Creates a draft PR on GitHub targeting the source branch
  3. Saves PR number and URL to task frontmatter for reference
- **Output**: User receives confirmation showing:
  - Worktree created at path
  - Branch pushed to remote with tracking
  - Draft PR created with link
  - Task file updated with PR metadata

### Expected Behavior

When a developer creates a task worktree, the tool should streamline the entire branch-to-PR workflow:

1. **Upstream Setup**: The new branch created in the worktree should be immediately pushed to the remote repository with upstream tracking configured. This ensures the branch exists on remote and is ready for collaboration.

2. **Draft PR Creation**: A draft PR should be automatically created on GitHub, targeting the source branch (the branch from which the worktree was created). This follows the pattern established by task 124's `--source` option - we now know the correct base branch for the PR.

3. **Frontmatter Integration**: The PR number and URL should be recorded in the task's YAML frontmatter under a `pr:` section, providing traceability between tasks and their associated pull requests.

4. **Config-Driven Defaults**: All new behaviors should be controlled by configuration with sensible defaults (enabled by default). CLI flags provide per-invocation overrides.

5. **Graceful Degradation**: If `gh` CLI is unavailable or not authenticated, the workflow should continue with a warning rather than failing entirely.

### Interface Contract

```bash
# Standard usage - all new features enabled by config defaults
ace-git-worktree create --task 125
# Creates worktree → pushes branch → creates draft PR → saves to task

# Explicit control via flags
ace-git-worktree create --task 125 --no-upstream
# Skip upstream setup (branch stays local)

ace-git-worktree create --task 125 --no-pr
# Skip PR creation (branch pushed but no PR)

ace-git-worktree create --task 125 --source main
# PR targets 'main' branch (from task 124's source option)

# Expected output
✓ Task worktree created successfully
  Worktree: /project/.ace-wt/task.125
  Branch: 125-upstream-setup-and-pr-creation
  Pushed: origin/125-upstream-setup-and-pr-creation
  PR: #456 (draft) - https://github.com/owner/repo/pull/456
  Task updated: pr.number=456
```

**Configuration (worktree.yml):**
```yaml
git:
  worktree:
    task:
      auto_setup_upstream: true     # Push new branch with -u
      auto_create_pr: true          # Create draft PR on GitHub
      pr_title_format: "{id} - {slug}"
      pr_base_branch: null          # null = use source branch
```

**Error Handling:**
- `gh` not installed: Warning "GitHub CLI not available, skipping PR creation", continue workflow
- `gh` not authenticated: Warning "GitHub CLI not authenticated, skipping PR creation", continue
- Push fails (network): Warning with error message, continue without PR
- PR creation fails: Warning with error message, task still created successfully

**Edge Cases:**
- Branch already exists on remote: Push should succeed (fast-forward) or warn if diverged
- PR already exists for branch: Report existing PR URL instead of creating duplicate
- Task frontmatter already has `pr:` section: Update/merge rather than overwrite

## Objective

Streamline the task-to-PR workflow by automating branch setup and PR creation. Currently, after creating a worktree, developers must manually push the branch and create a PR. This creates friction and delays the start of code review. By integrating these steps, developers get a complete development environment with PR ready for collaboration immediately after running a single command.

## Implementation Plan

### Technical Research Summary

**Current Workflow Analysis** (from `task_worktree_orchestrator.rb`):
- Steps 1-7: Task fetch, status update, metadata addition, commit, push (main branch)
- Step 8: Create worktree with new branch
- Step 9: Execute after-create hooks
- **Gap**: New branch exists only locally after worktree creation

**Key Integration Points**:
- `TaskPusher` molecule already handles `git push -u` operations
- `PrFetcher` molecule exists for fetching PR data (can reference for gh CLI patterns)
- `TaskStatusUpdater` molecule already has `add_worktree_metadata` method
- `WorktreeConfig` model has established pattern for config-driven defaults with accessors

**Source Branch Tracking** (from task 124):
- `worktree_result[:start_point]` contains the source branch/ref
- This becomes the PR base branch

### Architectural Approach

Extend the existing workflow in `TaskWorktreeOrchestrator` with three new steps after worktree creation:

1. **Step 9**: Push worktree branch to remote with upstream tracking
2. **Step 10**: Create draft PR via `gh pr create --draft`
3. **Step 11**: Update task frontmatter with PR metadata
4. **Step 12**: Execute hooks (renumbered from step 9)

Follow existing patterns:
- Config-driven defaults with `auto_*` boolean accessors
- CLI options pass through to options hash
- Non-blocking failures become warnings in `workflow_result[:warnings]`
- Result includes step details in `workflow_result[:steps_completed]`
