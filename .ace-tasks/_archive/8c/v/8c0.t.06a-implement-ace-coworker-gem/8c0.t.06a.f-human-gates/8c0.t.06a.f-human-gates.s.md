## Technical Approach

### Architecture Pattern

Following ATOM architecture with clear separation:
- **Atoms**: Pure functions for gate parsing (no I/O)
- **Molecules**: Gate state management with file I/O
- **Organisms**: Gate handling integrated into workflow executor
- **CLI**: Resume command with approval/rejection options

### Workflow YAML Schema for Gates

```yaml
steps:
  - name: implement
    instructions: |
      Execute wfi://work-on-task workflow.
      When complete, report results with: ace-coworker report <file>

  - name: review-gate
    gate: human
    prompt: "Review implementation before creating PR. Approve with: ace-coworker resume --approve"

  - name: create-pr
    instructions: |
      Create pull request for the changes.
```

**Gate step fields:**
- `gate`: Gate type (currently only "human" supported)
- `prompt`: Message displayed when workflow pauses (instructions for human reviewer)

### Job State Schema for Gates

```json
{
  "version": "1.0",
  "task_id": "228",
  "workflow": "task-completion",
  "status": "paused",
  "current_step": 3,
  "gate": {
    "step": "review-gate",
    "step_index": 3,
    "prompt": "Review implementation before creating PR",
    "waiting": true,
    "paused_at": "2026-01-28T01:35:00Z"
  },
  "steps": [
    {"name": "implement", "status": "completed"},
    {"name": "commit", "status": "completed"},
    {"name": "test", "status": "completed"},
    {"name": "review-gate", "status": "paused"},
    {"name": "create-pr", "status": "pending"}
  ]
}
```

**Gate state fields:**
- `step`: Gate step name
- `step_index`: Zero-based index in workflow
- `prompt`: Human-readable instruction
- `waiting`: Boolean indicating gate is awaiting approval
- `paused_at`: ISO timestamp when gate was reached

For rejection:

### CLI Interface

```bash
# Approve gate and continue workflow
ace-coworker resume --approve

# Reject gate with reason
ace-coworker resume --reject --reason "Need to fix X first"

# Check status shows gate state
ace-coworker status
# Output:
# Session: abc123
# Task: 228
# Status: paused
# Current step: 4/8 - review-gate [paused at gate]
#
# Gate prompt:
# Review implementation before creating PR
#
# To continue: ace-coworker resume --approve
# To reject: ace-coworker resume --reject --reason "..."
```

## Rejection Behavior

When a gate is rejected:
1. **State**: Status set to `rejected` (not `paused` or `running`)
2. **No auto-rewind**: Does NOT automatically re-execute previous step
3. **Manual resolution**: User must manually address the issue. Recovery options:
   - Edit job.json directly to reset step status to `pending`
   - Delete session and start fresh with `ace-coworker start --task <id>`
4. **Side effects**: Automated side effects from previous steps are NOT rolled back (commits, files written, etc.)

This conservative approach avoids unintended state corruption. Advanced recovery commands (`retry`, `goto`) are out of scope for MVP.

## Risk Assessment

### Technical Risks

- **Risk:** Gate state conflicts with retry logic from 234.04
  - **Probability:** Low
  - **Impact:** Medium
  - **Mitigation:** Gate steps have no verifications; retries apply only to instruction steps

- **Risk:** Resume command race condition with status changes
  - **Probability:** Low
  - **Impact:** Medium
  - **Mitigation:** Check job status before processing; fail gracefully if not at gate

- **Risk:** Workflow YAML parsing conflicts with new gate: field
  - **Probability:** Low
  - **Impact:** Low
  - **Mitigation:** Gate is additional field, backwards compatible with existing workflows

### Integration Risks

- **Risk:** Changes break 234.05 full workflow
  - **Probability:** Medium
  - **Impact:** High
  - **Mitigation:** Run full test suite; gate handling is additive to executor flow

- **Risk:** Event logging breaks with new event types
  - **Probability:** Low
  - **Impact:** Low
  - **Mitigation:** Follow existing event schema patterns; add events incrementally

## Out of Scope

- Multiple gate types (only "human" gate supported in MVP)
- Timeout gates (auto-approve after N minutes)
- Conditional gates (approve based on condition)
- `ace-coworker retry` command (recovery from rejection is manual in MVP)
- `ace-coworker goto --step <n>` command (manual rewind is future feature)
- Multi-session support (task 234.07)
- Gate approval via external API/webhook
