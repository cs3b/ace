## Technical Approach

### Architecture Pattern

**Key Insight: Agent is the Driver**

The ace-coworker CLI does NOT spawn subagents. The agent (Claude Code) does via the Task tool. ace-coworker's role is:
1. Provide instructions that guide the agent when to use Task tool
2. Log the delegation and subagent completion events
3. Accept the combined report (parent agent's summary of subagent work)

```
Parent agent invokes: /ace:coworker-do work-on task 228
    │
    ├── ace-coworker start --task 228
    │   └── Returns: step 1 instructions (may include Task tool pattern)
    │
    ├── Agent reads instructions, recognizes subagent pattern
    │   └── Agent spawns Task tool with isolated prompt
    │       └── Subagent executes, returns result
    │
    ├── Agent collects subagent output, writes combined report
    │   └── ace-coworker report combined-report.md
    │       └── Logs: subagent_delegated, subagent_completed events
    │
    └── ace-coworker advances to next step
```

### Subagent Instruction Pattern

Workflow instructions can include Task tool patterns:

```yaml
steps:
  - name: implement
    instructions: |
      Use the Task tool to delegate implementation to a subagent:

      Task prompt:
      """
      Implement task $task following wfi://work-on-task

      Working directory: $working_dir
      Report back: summary of changes, files modified, test results
      """

      After subagent returns, capture results and continue.
      Report with: ace-coworker report <file>

  - name: review
    instructions: ace-bundle wfi://review-pr
    # No subagent - direct execution
```

### Context Isolation Model

**Isolated (Subagent has its own):**
1. Instruction context - subagent receives only the Task tool prompt
2. Session scope - no access to parent's ace-coworker session state
3. Conversation history - fresh context for subagent

**Shared (Both can access):**
1. Working directory - same file system
2. Git state - same branch and repository
3. Environment variables - inherited

**Report Boundary:**
- Subagent output is captured by Task tool and returned to parent
- Parent agent writes combined report to ace-coworker
- Parent session log records delegation events

### Log Event Schema for Subagents

```jsonl
{"ts":"8or5ky","event":"step_started","step":"implement","step_index":0,"attempt":1}
{"ts":"8or5kz","event":"subagent_delegated","step":"implement","prompt_preview":"Implement task 228..."}
{"ts":"8or600","event":"subagent_completed","step":"implement","success":true}
{"ts":"8or601","event":"report_received","step":"implement","file":"reports/001-01-implement-report.md"}
```

## Context Isolation Definition

"Context isolation" means:
1. **Separate session scope**: Subagent operates with its own instruction context, not the parent's full workflow state
2. **No shared working directory requirements**: Subagent may use same working directory but does not inherit parent's internal state
3. **Report boundary**: Subagent output is captured as a discrete report file, not merged into parent's session log

**How subagent reports are captured:**
1. Parent workflow instruction tells agent to spawn Task tool with specific prompt
2. Task tool returns result to agent
3. Agent includes relevant output in its report via `ace-coworker report`
4. Parent session log records: `{"event": "subagent_completed", "step": "implement", "summary": "..."}`

**Not in scope for isolation:**
- File system isolation (subagent can modify same files)
- Environment variable isolation
- Git state isolation
