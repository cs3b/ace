## Technical Approach

### Architecture Pattern

Following ATOM architecture with clear separation:
- **Atoms**: Pure functions for error normalization, parsing, detection (no I/O)
- **Molecules**: File I/O operations for verification execution, retry management
- **Integration**: Verification and retry calls added to existing workflow executor

### Workflow YAML Schema Extension

```yaml
steps:
  - name: implement
    instructions: |
      Execute wfi://work-on-task workflow.
      When complete, report results with: ace-coworker report <file>
    verifications:
      - command: ace-test
        description: All tests pass
      - command: ace-lint --check
        description: No lint errors
    retries: 5
    on_repeated_failure: 3
```

**New fields:**
- `verifications`: Array of verification objects with `command` (shell command) and `description` (human-readable)
- `retries`: Maximum number of retry attempts (default: 3)
- `on_repeated_failure`: Stop after N consecutive identical errors (default: 3)

**Verification Execution Model:**
Verifications are **executable shell commands** (not prose descriptions). The `verification_executor.rb` molecule runs each command and checks exit codes:
- Exit code 0 = verification passed
- Non-zero exit code = verification failed (triggers retry if attempts remain)

This follows ACE's Unix philosophy: deterministic, predictable command output.

### Job State Schema Extension

```json
{
  "version": "1.0",
  "task_id": "228",
  "workflow": "work-on-task",
  "status": "in_progress",
  "current_step": 0,
  "steps": [{
    "name": "implement",
    "status": "in_progress",
    "attempts": 2,
    "last_error": "3 tests failed",
    "error_history": ["timeout", "3 tests failed", "3 tests failed"]
  }]
}
```

**New step fields:**
- `attempts`: Number of execution attempts (1-based, increments on each retry)
- `last_error`: Normalized error message from most recent failure
- `error_history`: Array of all error messages for this step

### Error Normalization Rules

The `ErrorNormalizer` atom normalizes errors for comparison:

```ruby
# Error types and their normalized forms:
# 1. Verification failures: first 400 chars + last 100 chars of output, trimmed
#    (captures both the beginning context and tail error signatures)
# 2. Exit code failures: "exit code {n}"
# 3. Timeout failures: "timeout after {n}s"

ErrorNormalizer.normalize(error_type:, message:, exit_code: nil, timeout: nil)
# => normalized string for comparison (max 500 chars)
```

### Repeated Failure Detection

The `RepeatedFailureDetector` atom counts consecutive identical errors:

```ruby
# Count consecutive identical errors from most recent
RepeatedFailureDetector.count_consecutive(
  error_history: ["timeout", "3 tests failed", "3 tests failed", "3 tests failed"],
  current_error: "3 tests failed"
)
# => 4 (current + 3 in history)

RepeatedFailureDetector.should_stop?(
  error_history: error_history,
  current_error: current_error,
  threshold: 3
)
# => true (4 >= 3)
```

## Error Comparison Rules

"Same error" is determined by **exact string match** of the normalized error message:
- Verification failures: the failing verification command output (first 400 chars + last 100 chars = max 500 chars, trimmed)
- Exit code failures: `"exit code {n}"`
- Timeout failures: `"timeout after {n}s"`

Repeated failure detection:
1. After each failure, compare `last_error` with previous entries in `error_history`
2. Count consecutive identical errors (from most recent backwards)
3. If count >= `on_repeated_failure`, transition to `failed` status

Example: `["timeout", "3 tests failed", "3 tests failed", "3 tests failed"]` → 3 consecutive same errors → workflow stops.

## Risk Assessment

### Technical Risks

- **Risk:** Verification commands may have unpredictable output formats
  - **Probability:** Medium
  - **Impact:** Medium
  - **Mitigation:** Use first 500 chars with trimming; document expected output formats

- **Risk:** Error normalization may not capture meaningful differences
  - **Probability:** Low
  - **Impact:** Medium
  - **Mitigation:** 500 char limit provides sufficient context; exact match is simple and predictable

- **Risk:** Retry loop could run indefinitely
  - **Probability:** Low
  - **Impact:** High
  - **Mitigation:** Always enforce retries limit; on_repeated_failure provides secondary safety

### Integration Risks

- **Risk:** Changes break existing workflow execution from 234.01/234.02
  - **Probability:** Medium
  - **Impact:** High
  - **Mitigation:** Run full test suite; verification is optional (backwards compatible)

- **Risk:** Event logging conflicts with 234.02 schema
  - **Probability:** Low
  - **Impact:** Low
  - **Mitigation:** Follow existing event schema patterns; use consistent field names
