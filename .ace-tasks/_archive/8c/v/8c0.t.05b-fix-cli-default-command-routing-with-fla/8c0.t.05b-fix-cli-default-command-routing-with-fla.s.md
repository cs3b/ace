# Fix CLI default command routing with flags

## Behavioral Specification

### User Experience
- **Input**: User runs CLI command with flags directly (e.g., `ace-git-commit -i "message"`)
- **Process**: CLI routes the command to the default subcommand with flags preserved
- **Output**: Command executes successfully with flags applied to the default command

### Expected Behavior

When users run ace-* CLI tools with flags but without explicitly specifying the subcommand, the CLI should:

1. Recognize that the flag is NOT a subcommand name
2. Prepend the default command automatically
3. Pass all arguments (including flags) to the default command

**Example flows:**
- `ace-git-commit -i "fix bug"` → routes to `commit -i "fix bug"` ✓
- `ace-git-commit --dry-run` → routes to `commit --dry-run` ✓
- `ace-prompt --enhance` → routes to `process --enhance` ✓
- `ace-docs` → routes to `status` (default) ✓

### Interface Contract

```bash
# These should all work (currently broken):
ace-git-commit -i "Create task 199"
ace-git-commit --intention "message" path/to/file
ace-prompt --enhance
ace-prompt -t 123

# These should continue working:
ace-git-commit --help          # Shows help (builtin)
ace-git-commit commit --help   # Shows commit help
ace-git-commit version         # Shows version
```

**Error Handling:**
- Unknown flags: Let dry-cli handle with appropriate error message
- Missing required arguments: Let dry-cli validate and report

**Edge Cases:**
- Flag that looks like command: `-i` should route to default, not be treated as command
- `--help` and `-h`: Should work directly (in BUILTIN_COMMANDS)
- `--version`: Should work directly (in BUILTIN_COMMANDS)

## Objective

Fix the CLI default command routing so users can use flags directly without specifying the subcommand name. This was broken during the dry-cli migration (Task 144) due to incorrect flag detection logic.

## Technical Approach

### Root Cause Analysis

The issue is in the `CLI.start` method's default command routing logic:

```ruby
# BROKEN (ace-git-commit line 53, ace-prompt line 60)
if args.empty? || (!known_command?(args.first) && !args.first.start_with?("-"))
```

When the first argument is a flag like `-i`:
- `known_command?("-i")` → false
- `"-i".start_with?("-")` → true
- `!true` → false
- Condition becomes `false || (true && false)` → false
- Default command NOT prepended → dry-cli fails

### Correct Pattern (from ace-taskflow)

```ruby
# CORRECT (ace-taskflow/molecules/command_router.rb:42-46)
if args.empty? || !known_commands.include?(args.first)
  [default] + args
else
  args
end
```

**Why `--help` still works**: Because `--help` IS in KNOWN_COMMANDS (via BUILTIN_COMMANDS), so no default prepending occurs.
