# Create E2E Tests for ace-llm

## Context

ace-llm is the LLM query tool providing provider routing, model selection, and response formatting. E2E tests require real API calls which are expensive and environment-dependent.

This is a P3 (low priority / deferred) task — real API calls required, high cost per run.

## Behavioral Specification

Create 1 scenario with ~2 test cases using the standalone goal-mode TC format.

### Scenario: TS-LLM-001-llm-query (deep)

LLM query operations with provider routing.

| TC | Goal | CLI Command | Verification |
|----|------|-------------|--------------|
| TC-001 | Basic query | `ace-llm query "simple prompt"` | Response received and formatted |
| TC-002 | Model selection | `ace-llm query --model haiku "prompt"` | Correct model used in response |

- Tags: `[deep, "use-case:llm"]`
- E2E justification: Real API calls, provider routing, response formatting

## Interface Contract

### Files to Produce

```
ace-llm/test-e2e/scenarios/
└── TS-LLM-001-llm-query/
    ├── scenario.yml           # mode: goal, tags: [deep, "use-case:llm"]
    ├── TC-001.runner.md       # Basic query
    ├── TC-001.verify.md       # Verify response format
    ├── TC-002.runner.md       # Model selection query
    └── TC-002.verify.md       # Verify model used
```

### CLI Execution

```bash
ace-test-e2e run ace-llm --provider claude:haiku
```

## Deliverables

- [ ] Scenario `TS-LLM-001-llm-query` created with 2 TCs (standalone goal-mode)
- [ ] scenario.yml has correct `mode: goal` and tags metadata
- [ ] Value-gate metadata populated in scenario.yml

## Success Criteria

- [ ] `ace-test-e2e run ace-llm --provider claude:haiku` discovers and executes the scenario
- [ ] Both TCs produce meaningful PASS/FAIL verdicts
- [ ] API call TC validates real response from LLM provider
