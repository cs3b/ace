# Tag System and CLI Filtering for E2E Tests

## Scope

Add a `tags` metadata field to `scenario.yml`, with CLI options to filter tests at discovery time. Tags are scenario-level metadata (not per-test-case). This is the infrastructure that 280.04 (grouping) and all future filtering depend on.

> **Dependency note**: 280.02 defines what the tier hierarchy means for goal-mode (runner config differences, verification depth per tier). This task provides the filtering mechanics; it does not define tier semantics.
> **Cutover mode**: Big-bang. Do not keep dual discovery paths for legacy standalone naming. New naming becomes canonical immediately.

## Behavioral Spec

### Tag Declaration

**In scenario.yml**:
```yaml
test-id: TS-LINT-001
title: Core Lint Pipeline
tags: [smoke, happy-path, use-case:lint]
mode: procedural           # NEW: "procedural" (default) or "goal"
execution-model: isolated  # NEW: "isolated" (default) or "sequential"
tool-under-test: ace-lint  # NEW: primary tool being tested
sandbox-layout:            # NEW: expected artifact directory structure
  output/: "Lint report output"
```

**Tag format rules**:
- Lowercase alphanumeric + hyphens: `smoke`, `happy-path`, `deep`
- Namespaced with colon for use-cases: `use-case:lint`, `use-case:commit`
- Tags are declared at scenario-level only

### Goal-Mode Discovery

When `mode: goal` is set in scenario.yml, the discoverer must find standalone goal-mode TC files alongside (or instead of) `TC-*.tc.md` files:

**Standalone goal-mode TC files** (pilot-validated format):
- `TC-*.runner.md` — runner instructions (headers may still use `Goal N`)
- `TC-*.verify.md` — verifier criteria
- `runner.yml.md` — runner agent system prompt / bundled context
- `verifier.yml.md` — verifier agent system prompt / bundled context

**Inline goal TCs** (simpler on-ramp):
- `TC-*.tc.md` with `mode: goal` in frontmatter — standard TC file with goal-mode sections

The discoverer should detect which format a scenario uses based on the `mode` field and presence of standalone goal-mode TC files in the scenario directory.

### CLI Filtering Interface

```bash
# Include by tags (OR logic within comma-separated list)
ace-test-e2e-suite --tags smoke
ace-test-e2e-suite --tags happy-path,smoke

# Exclude by tags
ace-test-e2e-suite --exclude-tags deep

# Optional package filter via positional argument
ace-test-e2e-suite ace-lint,ace-bundle --tags smoke

# Combined with existing flags
ace-test-e2e-suite --tags smoke --affected
ace-test-e2e-suite ace-lint,ace-bundle --tags happy-path

# Per-package run with tags
ace-test-e2e run ace-lint --tags smoke

# Dry-run to preview filtered set
ace-test-e2e-suite --tags smoke --dry-run
```

### Filtering Semantics

- **--tags**: Include scenarios that have ANY of the listed tags (OR)
- **--exclude-tags**: Exclude scenarios that have ANY of the listed tags (OR)
- **--tags + --exclude-tags**: Include first, then exclude (exclude wins)
- Filter applies at **discovery time** — excluded tests never reach setup/execution
- Untagged scenarios match no tag filters (excluded from `--tags` results)
