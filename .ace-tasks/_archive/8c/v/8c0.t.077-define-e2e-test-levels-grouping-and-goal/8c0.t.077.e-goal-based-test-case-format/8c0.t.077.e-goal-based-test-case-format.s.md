# Goal-Based Test Case Format

## Scope

Define two formats for goal-based test cases where agents plan their own approach to satisfy success criteria, rather than following procedural steps. The pilot (280.01) validated the standalone format at the smoke tier for a simple package; this subtask also introduces a simpler inline format as an on-ramp.

> **Dependency note**: The pilot validated format mechanics at smoke level (simple tool, 8 goals, single runner.yml.md). For deeper tiers and complex packages (ace-assign, ace-overseer), the runner.yml.md and verifier.yml.md content will need to scale â€” more complex sandbox setup, richer tool context, more thorough verification. 280.02 defines what each tier means; this task should document how the formats adapt accordingly (e.g., what a deep-tier runner.yml.md looks like vs a smoke-tier one).
> **Cutover mode**: Big-bang. Remove legacy standalone naming and legacy artifact/report conventions instead of maintaining compatibility aliases.

## Two Goal Formats

### Format 1 â€” Inline TC (`mode: goal`)

A simpler on-ramp: standard `TC-*.tc.md` files with `mode: goal` in frontmatter. The agent self-assesses against success criteria within a single execution.

```markdown
---
tc-id: TC-003
title: Generate commit message from staged changes
mode: goal
---

## Objective

Generate an intelligent commit message for staged Ruby file changes using ace-git-commit.

## Available Tools

- `ace-git-commit` â€” CLI tool for generating commit messages
- `git` â€” Standard git operations
- Standard shell tools (bash, fish)

## Success Criteria

- [ ] A git commit is created in the sandbox repository
- [ ] The commit message references the nature of the change (not generic "update files")
- [ ] The commit message follows conventional commit format or the repo's commit style
- [ ] `git log --oneline -1` shows the new commit

## Hints

- The sandbox has a git repo with staged changes (from setup)
- ace-git-commit can operate on staged changes with `--staged`
```

**Required sections**: Objective, Success Criteria, Available Tools
**Optional sections**: Hints, Constraints
**Prohibited**: `## Steps` section (contradicts goal-based approach)

### Format 2 â€” Standalone Goal-Mode TC Pair (pilot-validated)

Separate files for runner instructions and verifier criteria, bundled via agent prompt files. Validated by the ace-b36ts pilot (280.01, score 6/8).

**File naming**:
- `TC-{NNN}-{slug}.runner.md` â€” runner specification (`Goal N` heading allowed for readability)
- `TC-{NNN}-{slug}.verify.md` â€” verifier criteria
- `runner.yml.md` â€” runner agent system prompt (bundles all standalone goal-mode TCs)
- `verifier.yml.md` â€” verifier agent system prompt (bundles all criteria + artifacts)

**Runner file structure** (`TC-*.runner.md`):
```markdown
# Goal {N} â€” {Title}

## Goal
{1-3 sentences: what to accomplish}

## Workspace
Save artifacts to `results/tc/{NN}/`.

## Constraints
- Use only {tool} to perform the operation
- Do not hardcode values; discover via --help
```

**Verifier file structure** (`TC-*.verify.md`):
```markdown
# Goal {N} â€” {Title}

## Expectations
- File exists in `results/tc/{NN}/` with expected properties
- Content meets specific criteria (format, values, structure)

## Verdict
PASS if all expectations met; FAIL with evidence otherwise.
```

### Format Comparison

| Aspect | Format 1 (Inline TC) | Format 2 (Standalone) |
|--------|---------------------|----------------------|
| File naming | `TC-*.tc.md` | `TC-*.runner.md` + `TC-*.verify.md` |
| Execution model | Single agent, self-assessed | Dual agent (runner + verifier) |
| Verification | Agent self-evaluates criteria | Independent verifier inspects artifacts |
| Sections | Objective, Success Criteria, Available Tools | Goal, Workspace, Constraints + Expectations, Verdict |
| Context sharing | Per-TC isolated | Sequential across goals (shared agent context) |
| Bundling | N/A (loaded by TC parser) | `runner.yml.md` / `verifier.yml.md` |
| Best for | Simple goals, existing scenarios | Complex multi-goal scenarios, high-confidence needs |
| Pilot status | Not yet validated | Validated (280.01, 6/8 pass) |

## Execution Workflows

### Procedural Mode (unchanged)

Execute steps as written, verify outputs. No changes to existing behavior.

### Format 1 â€” Inline Goal Mode (self-assessed)

1. Read objective and success criteria from TC
2. Agent plans approach using available tools
3. Agent executes plan, adapting as needed
4. Agent self-evaluates against each success criterion
5. Report includes: approach taken, criteria met/unmet, observations

### Format 2 â€” Standalone Goal Mode (6-phase runner/verifier workflow)

1. **Setup**: Create sandbox, prepare `results/` directory structure per `sandbox-layout`
2. **Bundle runner prompt**: Combine `runner.yml.md` + all `TC-*.runner.md` into single prompt
3. **Execute runner**: Single agent receives all goal-mode TCs, executes sequentially with accumulated context, writes artifacts to `results/tc/{NN}/`
4. **Bundle verifier prompt**: Combine `verifier.yml.md` + all `TC-*.verify.md` + artifact references
5. **Execute verifier**: Separate agent evaluates artifacts against criteria, renders per-goal PASS/FAIL with evidence
6. **Report**: Generate YAML frontmatter report with `passed`/`failed` arrays, `score`, `verdict`
