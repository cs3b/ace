# Independent Verifier Agent Pattern

## Scope

Introduce a dual-agent pattern where a separate verifier agent independently examines sandbox state after test execution. The verifier runs IN the sandbox directory with full filesystem access — it inspects artifacts, runs commands, and evaluates against verify.md criteria. This reduces confirmation bias in agent self-assessment.

> **Dependency note**: The verifier architecture is tier-agnostic, but verifier.yml.md content (inspection thoroughness, what to check) scales with tier. A smoke verifier may do simple file-existence checks; a deep verifier for ace-assign may need to inspect git state, multi-file outputs, and workflow side-effects. 280.02 defines tier semantics that inform how verifier configs should differ.
> **Cutover mode**: Big-bang. TC-first report schema is canonical immediately; do not retain legacy `goal` schema aliases.

## Behavioral Spec

### Architecture

```
┌─────────────┐     ┌───────────────────┐     ┌──────────────┐
│   Runner     │────▶│     Sandbox       │◀────│   Verifier   │
│   Agent      │     │   (filesystem)    │     │    Agent     │
└─────────────┘     │                   │     └──────────────┘
       │            │  results/tc/01/       │            │
       │            │  results/tc/02/       │            │
       │            │  ...              │            │
       ▼            └───────────────────┘            ▼
  (no report —                              report.md
   artifacts only)                          - YAML frontmatter
                                            - per-goal verdicts
```

**Key difference from pilot**: The pilot (280.01) embedded artifacts as text in the verifier prompt for pragmatic reasons (ace-llm doesn't have filesystem access). The target design uses a CLI provider (e.g., claude with `--dangerously-skip-permissions`) so the verifier agent CAN inspect the sandbox directly.

### Verifier Context (What It Receives)

The verifier agent receives:
- **Sandbox directory path** — full filesystem access to inspect artifacts (ls, cat, wc, etc.)
- **`scenario.yml`** — scenario metadata, sandbox-layout, tool-under-test
- **`TC-*.verify.md` criteria** — what to check for each goal-mode TC (bundled via `verifier.yml.md`)

The verifier SHOULD be isolated from the runner session itself (no shared conversation/session memory). Beyond that, the verifier may inspect any sandbox artifacts and execution outputs needed to validate real impact.

### Verifier Workflow

1. **Receive context**: Sandbox path + scenario.yml + verify.md criteria
2. **Inspect sandbox**: Navigate to sandbox directory, examine artifacts using filesystem access (ls, cat, wc, file inspection, command execution)
3. **Evaluate against criteria**: For each TC, check verify.md expectations against actual artifacts
4. **Render verdicts**: Per-TC PASS/FAIL with evidence (what was found, what was expected)
5. **Generate report**: Write report with YAML frontmatter (passed/failed arrays, score, verdict)

### Result Categories

The verifier is the sole authority on pass/fail. When a TC fails, the failure is categorized as:

| Category | Meaning | Example |
|----------|---------|---------|
| `test-spec-error` | The spec itself is wrong or impossible | Token `70000` is invalid for ace-b36ts |
| `tool-bug` | The tool under test has a defect | Decode returns UTC instead of local timezone |
| `runner-error` | The runner agent misunderstood or failed | Artifacts written to wrong directory |
| `infrastructure-error` | Environment/setup failure | Tool not installed, sandbox not created |

### Isolation Mechanism

Runner and verifier are **two separate agent invocations** with no shared session context:

- **Runner session**: Receives `runner.yml.md` + all `TC-*.runner.md`, writes artifacts to sandbox
- **Verifier session**: Receives `verifier.yml.md` + all `TC-*.verify.md` + sandbox path, inspects and reports
- No shared conversation history or live session memory between runner and verifier
- Verifier can inspect anything persisted in the sandbox, including execution artifacts and logs

### Report Format

The verifier produces a report with YAML frontmatter (validated by pilot):

```yaml
---
scenario: TS-B36TS-001
runner-provider: claude:opus
verifier-provider: claude:opus
timestamp: 2026-02-24T00:04:24Z
score: 0.75
verdict: partial  # "pass" (all), "partial" (some), "fail" (none)
passed:
  - TC-001-help-survey
  - TC-002-encode-today
  - TC-004-error-behavior
failed:
  - tc: TC-003-decode-token
    category: test-spec-error
    evidence: "Token 70000 is not a valid ace-b36ts token"
  - tc: TC-007-roundtrip-pipeline
    category: tool-bug
    evidence: "Decode returns UTC (13:30) instead of local timezone (14:30)"
---

# Verification Report

## Goal 1 — Help Survey
**Verdict**: PASS
**Evidence**: File exists in results/tc/01/, contains observations about subcommands...

...
```

### CLI Interface

**Current state** (experiment workflow):
```bash
# Text-embedded simplification via ace-llm + ace-bundle
ace-llm --provider claude:opus --prompt "$(ace-bundle runner.yml.md)"
ace-llm --provider claude:opus --prompt "$(ace-bundle verifier.yml.md)"
```

**Target implementation**:
```bash
# Run with verification (dual-agent, sandbox-aware verifier)
ace-test-e2e run ace-b36ts TS-B36TS-001 --verify

# Run suite with verification
ace-test-e2e-suite --tags smoke --verify

# Verification is optional — default remains single-agent
ace-test-e2e run ace-lint TS-LINT-001  # no --verify, existing behavior
```

### Opt-In Design

- **Default**: Single-agent execution (existing behavior, unchanged)
- **--verify flag**: Enables dual-agent mode with sandbox-aware verifier
- Verification doubles LLM cost per test — appropriate for release gates, not pre-commit
- Recommended pairing: `ace-test-e2e-suite --verify` for release validation

- [ ] Failed goals categorized as test-spec-error, tool-bug, runner-error, or infrastructure-error
