# Research and Vision: E2E Test Levels, Grouping, and Goal-Based Execution

## Description

Establish the rationale, principles, and overall strategy for evolving the E2E test runner. This subtask captures the WHY before the other subtasks define the WHAT.

The current state: 33 E2E scenarios across 11 packages, still mostly treated as equal-weight procedural tests. No mechanism to run "just the smoke tests" or "only happy-path for changed packages." Test cases are step-by-step scripts - agents follow instructions rather than demonstrating capability.

**Gaps identified**:
- No filtering — running the full suite takes significant time/cost
- No way to request "fast confidence check" vs "thorough validation"
- Procedural TCs don't test agent adaptability — just instruction-following
- No independent verification — executor self-reports results

### Industry Patterns

- **Test pyramids** apply to E2E too: smoke (fast/cheap) → happy-path → deep (slow/expensive)
- **Goal-based testing** (from BDD/ATDD): define outcomes, not steps — better for agent evaluation
- **Tag-based filtering** (RSpec, pytest, JUnit): mature pattern for test selection
- **Dual-agent verification**: reduces confirmation bias in agent self-assessment

## Guiding Principles

1. **Incremental adoption** — Tags and groups layer onto existing scenarios without breaking anything
2. **Cost-aware execution** — smoke/standard/deep tiers map to real LLM API costs
3. **Agent capability testing** — Goal-mode tests evaluate problem-solving, not script-following
4. **Composable filtering** — Tags combine freely; groups are just curated tag sets
5. **Backward compatible** — Existing procedural TCs continue to work unchanged

## Vision Summary

Transform `ace-test-e2e` from "run everything" to "run what matters for this context":
- **Pre-commit**: `ace-test-e2e suite --tags smoke --affected` (30 seconds, changed packages only)
- **PR validation**: `ace-test-e2e suite --tags happy-path` (moderate coverage)
- **Release gate**: `ace-test-e2e suite` (full suite, all tiers)
- **Feature focus**: `ace-test-e2e suite --tags use-case:commit` (workflow-specific)

## Group Hierarchy and Runner/Verifier Differences

| Tier | Primary goal | Typical scope | Runner expectation | Verifier expectation |
|------|--------------|---------------|--------------------|----------------------|
| `smoke` | Fast confidence per package | One core workflow path | Minimal setup, deterministic commands, short runtime | Check high-signal artifacts (exit status, key output, required files) |
| `happy-path` | Standard feature confidence | Primary user workflows and normal inputs | Moderate setup, realistic workflow sequence, medium runtime | Validate workflow-level outputs and key side effects |
| `deep` | Risk-focused confidence | Error handling, edge cases, multi-step orchestration | Rich setup, complex sequences, longer runtime | Inspect detailed artifacts, multi-file side effects, recovery/error behavior |

Operational rules:
- Every package should expose at least one `smoke` scenario.
- `happy-path` should represent default user value paths for that package.
- `deep` should be reserved for expensive and risk-driven cases, not routine checks.
- Verifier depth should scale with tier so verification cost remains proportional.

## Pilot Validation (280.01)

The ace-b36ts pilot (280.01) validated the smoke/goal layer for a simple package:

- **Goal-based testing works** — agents plan approach from objectives, achieving 6/8 goals without procedural scripts
- **Dual-agent verification works** — runner/verifier split produced independent, credible assessments
- **Cost-tier "smoke" validated** — appropriate for the pilot's 8-goal scenario on a simple package
- **Sequential execution model is effective** — agents build on discovery (Goal 1) across subsequent goals
- **Tag-based metadata** — `scenario.yml` with tags, cost-tier, mode, execution-model validated as a workable schema
- **Report format** — YAML frontmatter with passed/failed arrays, score, verdict is parseable and actionable

### Remaining work

The pilot covered only one scenario at the smoke level for ace-b36ts (one of the simplest packages). This vision still needs to address:

- **Full group hierarchy** — how smoke → happy-path → deep tiers differ in goal complexity, agent context, and runner configuration for more complex packages (e.g., ace-assign, ace-review, ace-overseer)
- **Runner config per group** — how runner.yml.md / verifier.yml.md adapt to different tiers (e.g., deeper scenarios may need more tool access, longer timeouts, multi-step verification)
- **Scaling patterns** — what a happy-path or deep goal-based scenario looks like for packages with multi-step workflows, error recovery, and cross-package integration
- **Rewrite plan** — concrete guidance for converting existing procedural scenarios to goal-based at each tier level

## Procedural-to-Goal Rewrite Guidance

When converting existing procedural scenarios to goal-mode:

1. Preserve scenario intent and expected user value, but replace step scripts with explicit objectives.
2. Split each objective into independently verifiable criteria (files, outputs, exit codes, state changes).
3. Keep criteria objective and machine-checkable; avoid subjective language.
4. Add runner guidance that defines constraints (tools, sandbox, allowed assumptions), not command-by-command instructions.
5. Add verifier guidance that can confirm outcomes without relying on runner reasoning.
6. Map converted scenarios to tier tags (`smoke`, `happy-path`, `deep`) based on cost/risk profile.

Quality gate for conversion:
- Necessary: each criterion corresponds to a required behavior.
- Sufficient: combined criteria prove the objective was achieved.
- Independent: verifier can validate evidence from artifacts alone.
