# 6-Phase Goal-Mode Execution Pipeline

## Context

Task 280's experiment validated a 6-phase dual-agent pipeline for goal-mode E2E tests. The pilot (`experiment/phase-[a-f]`) correctly identified TC-003 as test-spec-error and TC-007 as tool-bug (roundtrip date mismatch). But `ace-test-e2e run` doesn't implement this pipeline — it delegates everything to the agent via skill invocation, producing false positives. The "big-bang cutover" from 280.05/280.06 changed naming/format but not the execution engine.

**Evidence**: `ace-test-e2e run ace-b36ts --provider claude:haiku` reports 8/8 PASS while TC-007 artifacts show exit code 1, date mismatch (2025-06-15 → 2025-06-14), and an ArgumentError in stderr.

## Scope

Implement the 6-phase pipeline in `ace-test-runner-e2e` for standalone goal-mode scenarios (`mode: goal` with `TC-*.runner.md` + `TC-*.verify.md` pairs). Procedural mode (`mode: procedural`) remains unchanged.

> **Cutover mode**: Big-bang. For goal-mode scenarios, the new pipeline replaces skill-invocation execution entirely. No fallback to old single-agent path.

### 6-Phase Architecture

```
Phase A (Ruby)          Phase B (Ruby)         Phase C (LLM)
┌──────────────┐   ┌───────────────────┐   ┌──────────────┐
│ Setup Sandbox │──▶│ Bundle Runner     │──▶│ Execute      │
│ .git, mise,   │   │ Prompt            │   │ Runner Agent │
│ .ace config   │   │ system + prompt   │   │ (artifacts)  │
└──────────────┘   └───────────────────┘   └──────┬───────┘
                                                   │
Phase F (Ruby)          Phase E (LLM)          Phase D (Ruby)
┌──────────────┐   ┌───────────────────┐   ┌──────┴───────┐
│ Generate      │◀──│ Execute Verifier  │◀──│ Bundle       │
│ Report        │   │ Agent (verdicts)  │   │ Verifier     │
│ YAML + MD     │   │ PASS/FAIL + evid  │   │ Prompt       │
└──────────────┘   └───────────────────┘   └──────────────┘
```

### Behavioral Spec

**User Experience**:
- **Input**: `ace-test-e2e run ace-b36ts --provider claude:haiku` (same CLI, no new flags for goal-mode)
- **Process**: For goal-mode scenarios, user sees the same progress output. Internally, the runner creates a proper sandbox, prepares prompts deterministically, runs the runner agent, collects artifacts, runs the verifier agent, and generates a report.
- **Output**: Accurate PASS/FAIL verdicts. TC-007 correctly reports FAIL with evidence of the roundtrip mismatch.

**Key behavioral difference from current**: The verifier ALWAYS runs for standalone goal-mode scenarios. The `--verify` flag becomes a no-op for goal-mode (it's always on). For procedural mode, `--verify` retains its current opt-in behavior.

### Phase A — Sandbox Setup (Ruby, deterministic)

Create an isolated environment ready for agent execution:

- `git init` (so ace-llm project-root discovery stops at sandbox)
- `mise.toml` with resolved worktree bin path, `PROJECT_ROOT_PATH={{ config_root }}`
- `.ace/llm/providers` symlink to worktree's provider configs
- `results/tc/{01..NN}/` directories per `sandbox-layout` in scenario.yml
- Verify tool-under-test is accessible: `{tool} --help`

Reference: `experiment/phase-a-setup-sandbox.wf.md`

### Phase B — Runner Prompt Preparation (Ruby, deterministic)

Bundle TC-*.runner.md files into a single runner prompt:

- **System prompt**: Rules for execution (execute goals in order, save artifacts, don't fabricate output)
- **User prompt**: Header from `runner.yml.md` + concatenated contents of all `TC-*.runner.md` files
- Use `ace-bundle` to expand `runner.yml.md` (already has `bundle:` frontmatter listing all runner files)
- Write to `sandbox/.cache/ace-e2e/runner-system.md` and `runner-prompt.md`

Reference: `experiment/phase-b-prepare-runner.wf.md`

### Phase C — Runner Execution (LLM)

Invoke runner agent via `ace-llm` with prepared prompts:

```bash
ace-llm {provider} \
  --cli-args dangerously-skip-permissions \
  --system .cache/ace-e2e/runner-system.md \
  --prompt .cache/ace-e2e/runner-prompt.md \
  --output .cache/ace-e2e/runner-output.md \
  --timeout {timeout}
```

Runner writes artifacts to `results/tc/{NN}/`. Runner does NOT self-evaluate — no PASS/FAIL from runner.

Reference: `experiment/phase-c-execute-runner.wf.md`

### Phase D — Verifier Prompt Preparation (Ruby, deterministic)

Collect runner artifacts and bundle with verification criteria:

- **System prompt**: Verifier rules (evaluate independently, cite evidence, follow output format)
- **User prompt**: Two parts concatenated:
  1. Directory tree + file contents of all `results/` artifacts (collected at runtime)
  2. Verification criteria from `verifier.yml.md` (bundled via ace-bundle, contains all `TC-*.verify.md`)
- Write to `sandbox/.cache/ace-e2e/verifier-system.md` and `verifier-prompt.md`

Reference: `experiment/phase-d-prepare-verifier.wf.md`

### Phase E — Verifier Execution (LLM)

Invoke verifier agent via `ace-llm` with prepared prompts:

```bash
ace-llm {provider} \
  --cli-args dangerously-skip-permissions \
  --system .cache/ace-e2e/verifier-system.md \
  --prompt .cache/ace-e2e/verifier-prompt.md \
  --output .cache/ace-e2e/verifier-output.md \
  --timeout {timeout}
```

Verifier renders per-TC PASS/FAIL verdicts with evidence. Verifier output follows the format from `verifier.yml.md`:
```
### Goal N — <title>
- **Verdict**: PASS | FAIL
- **Evidence**: <specific file/content citations>

Final line: **Results: X/N passed**
```

Reference: `experiment/phase-e-execute-verifier.wf.md`

### Phase F — Report Generation (Ruby, deterministic)

Parse verifier output into structured report:

- Count PASS/FAIL verdicts
- Compute score and overall verdict (pass/partial/fail)
- Generate `reports/report.md` with YAML frontmatter (`tcs-passed`, `tcs-failed`, `tcs-total`, `score`, `verdict`, `passed[]`, `failed[].tc/.category/.evidence`)
- Generate `metadata.yml` and `summary.r.md` compatible with existing report schema

Reference: `experiment/phase-f-generate-report.wf.md`

## Interface Contract

```bash
# Goal-mode scenario: automatically uses 6-phase pipeline
ace-test-e2e run ace-b36ts --provider claude:haiku
# → Phase A-F execute internally
# → Report: 6/8 (TC-003 FAIL, TC-007 FAIL) — matches experiment

# Procedural scenario: unchanged behavior
ace-test-e2e run ace-lint --provider claude:haiku
# → Existing skill-invocation path (no change)

# --verify flag: no-op for goal-mode (always on), opt-in for procedural
ace-test-e2e run ace-b36ts --verify  # same as without --verify for goal-mode
ace-test-e2e run ace-lint --verify   # triggers verifier for procedural (existing behavior)

# Suite: goal-mode scenarios use pipeline automatically
ace-test-e2e-suite --tags smoke --provider claude:haiku
```
