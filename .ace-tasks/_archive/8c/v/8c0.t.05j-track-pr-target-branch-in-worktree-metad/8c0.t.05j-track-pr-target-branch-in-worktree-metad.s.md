# Track PR target branch in worktree metadata

## Behavioral Specification

### User Experience
- **Input**: Developer runs `ace-git-worktree create --task 202.02` or `ace-git-worktree create --pr 26`
- **Process**: System determines the correct PR target branch by examining parent task's worktree metadata and saves it to the new worktree's metadata
- **Output**: Worktree created with `target_branch` field populated; PR creation later uses this saved target branch instead of defaulting to `main`

### Expected Behavior

When creating a worktree via `ace-git-worktree create`, the system should:

1. **Determine Parent Task**: Parse the task file to find the parent task reference
2. **Read Parent Worktree Branch**: Load the parent task's worktree metadata to get its branch name
3. **Save target_branch**: Write the parent's branch as `target_branch` in the new worktree's metadata
4. **Fallback to main**: If no parent task exists or parent has no worktree, default `target_branch` to `main`
5. **Display Transparency**: Show the determined target branch during worktree creation

When creating a PR via `ace-git-worktree` or `gh` cli:
1. Read `target_branch` from worktree metadata instead of defaulting to `main`
2. Use the saved target branch for the PR

### Interface Contract

```bash
# CLI Interface - Worktree Creation
ace-git-worktree create --task 202.02
# Output:
# Creating worktree for task 202.02...
# Parent task: 202 - Rename Support Gems
# Target branch: 202-rename-support-gems-and-executables-for-naming-consistency
# Worktree created: ../ace-task.202.02

ace-git-worktree create --pr 26
# Similar behavior - determines parent from PR

# Worktree Metadata (YAML)
worktree:
  branch: 202.02-rename-ace-config-to-ace-support-config
  target_branch: 202-rename-support-gems-and-executables-for-naming-consistency  # NEW
  path: "../ace-task.202.02"
  created_at: '2026-01-11 13:00:09'
  updated_at: '2026-01-11 13:00:09'

# CLI Interface - PR Creation
gh pr create --base $(yq eval '.worktree.target_branch' .ace-taskflow/task.yml)
# Or ace-git-worktree pr create (if implemented)
# Reads target_branch from metadata, uses it for --base flag
```

**Error Handling:**
- Parent task has no worktree metadata: Warn user and default to `main`
- Parent task file not found: Warn user and default to `main`
- Invalid task hierarchy: Warn user and default to `main`

**Edge Cases:**
- Orchestrator task (no parent): `target_branch` defaults to `main`
- Parent worktree not yet created: Default to `main`, warn user
- Multiple levels of nesting (grandparent tasks): Use immediate parent's branch

## Objective

PRs for subtasks of orchestrator tasks should target the orchestrator branch for proper integration, not main. This ensures:
- Sequential merging of subtask PRs into the orchestrator branch
- The orchestrator branch accumulates all subtask changes before final merge to main
- Proper dependency management between related subtasks
