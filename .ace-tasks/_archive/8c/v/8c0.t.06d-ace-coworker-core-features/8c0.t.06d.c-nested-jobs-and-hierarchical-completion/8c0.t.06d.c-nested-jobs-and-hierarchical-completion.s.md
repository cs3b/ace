# Nested Jobs and Hierarchical Completion

## Overview

Enable hierarchical job structure where verification is a separate nested job, and parent jobs complete only when all children are done.

## Objective

Replace the verification-as-property model with verification-as-job:
- Jobs can have nested sub-jobs (010.01, 010.02, etc.)
- Verification is just another job (not a frontmatter field)
- Failed verification → inject fix job + new verify job
- Parent job completes only when all nested jobs complete
- Clear audit trail of work → verify → fix → verify cycles

## Key Insight

> Verification is just another job. Worker shouldn't verify own work.

This creates explicit, traceable workflows where you can see exactly what happened.

## Example Structure

```
jobs/
├── 010-implement.j.md           ← status: done
│   ├── 010.01-verify.j.md       ← status: failed (tests failed)
│   ├── 010.02-fix-tests.j.md    ← status: done (injected after failure)
│   └── 010.03-verify.j.md       ← status: done (injected, passed)
├── 020-document.j.md            ← status: pending (blocked until 010.xx done)
```

## Technical Approach

### Job Numbering

```ruby
module Ace::Coworker::Atoms
  class JobNumbering
    # Parse job number into components
    # "010.02" → { parent: "010", index: 2, depth: 1 }
    # "010" → { parent: nil, index: 10, depth: 0 }
    def self.parse(number)
      parts = number.split(".")
      {
        parent: parts.length > 1 ? parts[0..-2].join(".") : nil,
        index: parts.last.to_i,
        depth: parts.length - 1,
        full: number
      }
    end

    # Generate next sibling number
    # "010.02" → "010.03"
    def self.next_sibling(number)
      parsed = parse(number)
      next_index = format("%02d", parsed[:index] + 1)
      parsed[:parent] ? "#{parsed[:parent]}.#{next_index}" : next_index
    end

    # Generate first child number
    # "010" → "010.01"
    def self.first_child(number)
      "#{number}.01"
    end

    # Check if child of parent
    # "010.02".child_of?("010") → true
    def self.child_of?(child, parent)
      child.start_with?("#{parent}.")
    end
  end
end
```

### Workflow Executor Changes

```ruby
def advance
  current = current_job
  return if current.nil?

  # Check if current has incomplete children
  children = jobs_for_parent(current.number)
  incomplete_children = children.reject { |j| j.status == "done" }

  if incomplete_children.any?
    # Can't advance parent - show next incomplete child
    next_child = incomplete_children.first
    display_job(next_child)
  else
    # All children done (or no children) - mark parent done, advance
    mark_done(current)
    advance_to_next_sibling_or_parent
  end
end

def jobs_for_parent(parent_number)
  @jobs.select { |j| JobNumbering.child_of?(j.number, parent_number) }
end
```

### Verification as Job Pattern

Instead of `verification: ace-test` in frontmatter, create a verify job:

```markdown
<!-- 010.01-verify.j.md -->
---
status: pending
---

## Verify Implementation

Run tests to verify the implementation:

```bash
ace-test
```

**Pass criteria:** All tests pass (exit 0)

**On failure:** Report what failed, a fix job will be added.
```

### Typical Workflow

1. Agent completes `010-implement.j.md`
2. `ace-coworker advance` → shows `010.01-verify.j.md`
3. Agent runs verification, tests fail
4. Agent reports failure: `ace-coworker report failure.md`
5. Orchestrator (or agent) injects fix job: `ace-coworker add --after 010.01 --name fix-tests`
6. Creates `010.02-fix-tests.j.md` and `010.03-verify.j.md`
7. Agent works through 010.02, 010.03
8. When 010.03 passes, all of 010.xx are done
9. `ace-coworker advance` → moves to `020-document.j.md`
