# Implement Path-Based Configuration Splitting in ace-git-commit

## Idea Source

## Behavioral Specification

### User Experience

**Input**: User runs `ace-git-commit` on staged files spanning multiple packages with different commit configurations (e.g., files in `ace-docs/` and `ace-handbook/`)

**Process**:
1. System detects staged files belong to different configuration scopes
2. Displays detected scopes and file counts
3. Generates separate commits sequentially, each using its package's config
4. Shows progress for each commit group

**Output**:
- Multiple commits created, each adhering to its package's commit conventions
- Clear summary showing commits created per scope

### Expected Behavior

When staged files span multiple packages with different configurations (via distributed `{package}/.ace/git/commit.yml` or centralized `paths:` rules):

1. **Auto-split (default)**: Groups files by effective config, creates sequential commits
2. **Single commit (`--no-split`)**: Uses project-level config for all files
3. **Single scope (no split needed)**: Behaves exactly as today

If any commit in the sequence fails, all commits in this operation are rolled back.

### Interface Contract

```bash
# Auto-split when different configs detected (default)
ace-git-commit .ace-taskflow/ ace-docs/
# Output:
# Detected 2 configuration scopes:
#   - .ace-taskflow (.ace-taskflow/.ace/git/commit.yml): 3 file(s)
#   - project default (.ace/git/commit.yml): 2 file(s)
# [1/2] Committing .ace-taskflow changes...
# abc1234 chore(taskflow): update task specifications
# [2/2] Committing ace-docs changes...
# def5678 docs: update API reference
# Created 2 commits successfully.

# Force single commit
ace-git-commit .ace-taskflow/ ace-docs/ --no-split
# Output:
# Note: Multiple config scopes detected but --no-split specified.
# Using nearest common configuration.
# ghi9012 feat: update specs and documentation
```

**New CLI Option:**
- `--no-split` - Force single commit even when multiple configs detected

**Configuration Format** - Two complementary approaches:

#### Approach 1: Centralized `paths` rules (recommended for visibility)

In `.ace/git/commit.yml`:
```yaml
git:
  model: glite  # default for files not matching any path rule

  paths:
    # Named path scopes with glob patterns
    ace-support:
      glob: 'ace-support-*/**'
      model: gflash
      conventions:
        format: conventional

    taskflow-specs:
      glob: '.ace-taskflow/**'
      conventions:
        format: simple  # no feat: prefix for task specs

    handbook:
      glob: 'ace-handbook/**'
      conventions:
        format: simple
```

#### Approach 2: Distributed config (co-located with package)

```
{package}/.ace/git/commit.yml
```

Example: `.ace-taskflow/.ace/git/commit.yml` applies to all files in `.ace-taskflow/`

#### Combined (both work together)

- **Distributed config has precedence** - if `.ace/git/commit.yml` exists in any parent folder, use it
- Centralized `paths` rules are fallback when no distributed config found
- This allows project-wide rules via `paths:`, with package-specific overrides via distributed config

**Error Handling:**
- Config loading fails: Falls back to project-level config with warning
- Staging fails for a group: Aborts entire operation, no commits
- Commit fails (hook rejection): Rolls back all commits created in this operation
- LLM generation fails: Rolls back and reports error

### Success Criteria

- [ ] Files in different packages with distinct configs produce separate commits
- [ ] Each commit uses its package's specific configuration (model, conventions, etc.)
- [ ] `--no-split` flag forces single commit behavior
- [ ] Failed commits trigger rollback of all commits in the operation
- [ ] Single-scope commits (no split needed) work exactly as before
- [ ] Clear output shows detected scopes, progress, and results

## Objective

Enable consistent commit standards across different packages in a monorepo by automatically splitting commits when staged files span packages with different configurations.

### Deliverables

#### New Components (ace-support-config)

| Layer | Component | Purpose |
|-------|-----------|---------|
| Atom | `PathRuleMatcher` | Match file paths against glob patterns from `paths:` config |
| Molecule | `FileConfigResolver` | Resolve config for a file: traverse up for distributed, then check `paths:` rules |
| Model | `ConfigGroup` | Files sharing same effective config |

**Resolution logic in `FileConfigResolver`**:
1. Start from file's directory, traverse up toward project root
2. At each level, check for `.ace/git/commit.yml` -> if found, return it (distributed wins)
3. If no distributed config found, load project `.ace/git/commit.yml`
4. Check `paths:` section, match file against globs -> apply first matching rule
5. Fall back to project default if no match

#### New Components (ace-git-commit)

| Layer | Component | Purpose |
|-------|-----------|---------|
| Molecule | `CommitGrouper` | Group staged files by config |
| Molecule | `SplitCommitExecutor` | Execute sequential commits with rollback |
| Model | `CommitGroup` | Group of files for one commit |
| Model | `SplitCommitResult` | Result with rollback support |

## Configuration Cascade (Updated)

**Key insight**: Distributed config (nearest `.ace/` in parent) has **precedence** over centralized `paths` rules.

**Resolution for a file** (e.g., `.ace-taskflow/v.0.9.0/tasks/foo.md`):

1. **CLI options** (highest priority)
2. **Distributed config**: Traverse up from file's directory toward project root
   - Check `.ace-taskflow/.ace/git/commit.yml` -> **if found, use it and stop**
   - This means package-local config always wins
3. **Centralized `paths` rules**: If no distributed config found, check `.ace/git/commit.yml`
   - Match file against glob patterns in `paths.*.glob`
   - First matching rule applies its overrides
4. **Project-level default**: `.ace/git/commit.yml` (base config, no paths match)
5. **User-level**: `~/.ace/git/commit.yml`
6. **Gem defaults**: `.ace-defaults/git/commit.yml`

**Grouping logic**: Files are grouped by their **effective config hash**. Different configs = separate commits.
