# Specify GitHub repo name, description, and topics

## Behavioral Specification

### User Experience
- **Input**: Approved naming policy and selected slug
- **Process**: Maintainer gets an ordered GitHub settings checklist that can be executed without ambiguity
- **Output**: Repository metadata reflects ACE brand and ADE category while preserving link continuity

### Expected Behavior
This subtask defines the exact GitHub-side changes required:
- Repository name/slug update decision path
- Repository description/headline/tagline copy
- Topics and social-preview alignment
- Post-rename redirect and link-resolution verification checks

### Interface Contract
```text
Required GitHub output checklist:
- Repo slug target:
- Repo description:
- Profile/README headline line:
- Topics list:
- Redirect verification checks:
```

## Acceptance Criteria

- [x] Checklist includes exact field-level updates and expected final values
- [x] Redirect checks cover old URL resolution and clone URL continuity
- [x] Failure/rollback conditions are specified

### Defaults
- **Description max length**: 350 characters (GitHub limit)
- **Topics limit**: Max 20 topics, prioritize discoverability terms
- **Redirect grace period**: GitHub provides automatic redirects indefinitely for renamed repos; verification checks should confirm this within 24h of rename

## Verification Plan

### Checklist Completeness
- Every GitHub metadata field (name, description, topics, social preview) has an explicit target value
- Redirect verification checks cover: old HTTPS URL, old clone URL (SSH + HTTPS), old API endpoint
- Rollback conditions are defined (what triggers reverting the rename)

### Copy Quality
- Description fits within 350-character GitHub limit
- Headline and tagline are distinct (not just copies of each other)
- Topics list includes both brand terms and category/discovery terms

### Negative Check
- Checklist does not include actual `gh api` or `gh repo rename` execution commands (spec only)

## Deliverables

- [x] GitHub metadata update checklist

### 4. Redirect Verification Checklist

After renaming from `ace-meta` to `ace`, verify the following within 24 hours:

| Check | Method | Expected Result |
|-------|--------|----------------|
| Old HTTPS URL resolves | Visit `https://github.com/cs3b/ace-meta` in browser | 301 redirect to `https://github.com/cs3b/ace` |
| Old HTTPS clone works | `git clone https://github.com/cs3b/ace-meta.git /tmp/ace-clone-test` | Clone succeeds via redirect |
| Old SSH clone works | `git clone git@github.com:cs3b/ace-meta.git /tmp/ace-ssh-test` | Clone succeeds via redirect |
| Old API endpoint resolves | `curl -s https://api.github.com/repos/cs3b/ace-meta` | Returns repo JSON (may show new name) |
| New HTTPS URL works | Visit `https://github.com/cs3b/ace` | Repository loads correctly |
| New clone URL works | `git clone https://github.com/cs3b/ace.git /tmp/ace-new-test` | Clone succeeds |
| Existing local remote works | `git fetch origin` from existing checkout | Fetch succeeds (git follows redirects) |
| GitHub Pages (if enabled) | Visit pages URL | Pages serve correctly |

### 5. Rollback Conditions

Revert the repository name back to `ace-meta` if any of these occur within 48h of rename:

| Condition | Severity | Action |
|-----------|----------|--------|
| Old HTTPS clone fails (no redirect) | Critical | Immediate revert via GitHub Settings |
| GitHub API returns 404 for old slug | Critical | Immediate revert |
| `gem push` or CI/CD pipeline fails due to homepage URL validation | High | Revert, fix gemspec URLs first, then re-rename |
| Old SSH clone fails | High | Investigate; revert if not resolved within 4h |
| Search indexing drops significantly | Medium | Monitor for 1 week; revert only if unrecoverable |

**Rollback procedure**: GitHub Settings → Repository name → change back to `ace-meta`. GitHub will then redirect `ace` → `ace-meta` automatically.
