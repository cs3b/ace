# Add context.base support to ace-context for generic base content handling

## Behavioral Specification

### User Experience
- **Input**: Context file with `context.base` field in frontmatter (e.g., `context.base: "prompt://base/system"`)
- **Process**: ace-context automatically resolves the base protocol, loads the referenced content, and uses it as the primary document body
- **Output**: Generated output file contains base content first, followed by processed sections

### Expected Behavior

When users specify a `context.base` field in their context configuration, ace-context should:

1. **Detect the base field** in the `context` section of frontmatter
2. **Resolve the protocol** (e.g., `prompt://base/system` → actual file path)
3. **Load the base content** from the resolved path
4. **Use base as primary content** - the base content becomes the main document body
5. **Append processed sections** after the base content
6. **Handle missing base gracefully** - work normally if base field is absent

This makes `base` a first-class citizen alongside `sections`, `files`, `diffs`, and `presets` in the context configuration.

### Interface Contract

**Configuration Format:**
```yaml
context:
  base: "prompt://base/system"  # Protocol-based reference
  sections:
    section_name:
      files:
        - "prompt://focus/scope/tests"
```

**Supported Base Formats:**
- Protocol references: `prompt://path/to/file`
- Relative paths: `./path/to/file.md`
- Absolute paths: `/full/path/to/file.md`

**Processing Order:**
1. Load base content (if specified)
2. Process sections
3. Output: `[base content]\n\n[processed sections]`

**Error Handling:**
- Base file not found: Clear error message with file path
- Invalid protocol: Error with supported protocol list
- Missing base field: Continue normally (base is optional)
- Empty base content: Warning but continue processing

**Edge Cases:**
- Base without sections: Output only base content
- Sections without base: Current behavior (sections only)
- Base with empty string: Treat as missing, continue normally

### Success Criteria

- [ ] **Base Field Detection**: ace-context detects `context.base` in frontmatter and processes it
- [ ] **Protocol Resolution**: `prompt://` protocols are resolved to actual file paths
- [ ] **Content Loading**: Base content is loaded and becomes primary document body
- [ ] **Proper Ordering**: Base content appears first, followed by sections
- [ ] **Backward Compatibility**: Existing configs without `base` continue to work
- [ ] **ace-review Integration**: When ace-review uses `context.base`, system.prompt.md contains both base and sections
- [ ] **Error Messages**: Clear, actionable error messages for missing files or invalid protocols

### Validation Questions

- [ ] **Protocol Support**: Should we support only `prompt://` or also other protocols like `file://`, `wfi://`?
- [ ] **Base Location**: Should base content come before or after the context metadata header?
- [ ] **Multiple Bases**: Should we support multiple base files or just one?
- [ ] **Base Caching**: Should base content be cached separately from sections?

## Objective

Enable generic base content handling in ace-context so that any tool (ace-review, ace-docs, etc.) can compose a primary document body with additional processed sections. This eliminates the need for tool-specific base handling and provides a consistent pattern across the ACE ecosystem.

## Implementation Plan

### Technical Research

#### Current Architecture Analysis

**ace-context Processing Flow:**
1. `load_template` (context_loader.rb:381-465) - Parses frontmatter and config
2. `process_template_config` (context_loader.rb:769+) - Processes files, commands, diffs
3. `load_from_preset_config` (context_loader.rb:504-549) - Handles sections processing
4. `resolve_protocol` (context_loader.rb:920-929) - Uses ace-nav for protocol resolution

**Protocol Resolution:**
- Already implemented via `resolve_protocol` method
- Uses `ace-nav` CLI to resolve protocol references
- Returns resolved file path or nil if resolution fails
- Supports any protocol that ace-nav recognizes (prompt://, wfi://, etc.)

**Section Processing:**
- Sections are processed by `SectionProcessor` molecule
- Each section can contain files, commands, diffs, or inline content
- Sections are formatted as separate blocks in output
- Currently no concept of "base" content that comes before sections

#### Implementation Approach

**Where to Add Base Handling:**
1. **Primary location**: `load_from_preset_config` method (line 504-549)
   - This is where presets are processed and sections are handled
   - Add base handling before section processing
   - Store base content in context for later formatting

2. **Alternative for template files**: `load_template` method (line 381-465)
   - For direct template file loading (non-preset usage)
   - Similar base handling logic as presets

**Content Composition Strategy:**
- Load base content as primary document body (context.content)
- Process sections normally (stored in context.sections)
- During formatting, prepend base content before sections
- Use existing section formatter with base content integration

**Error Handling Approach:**
- Check if base field exists before processing
- Resolve protocol using existing `resolve_file_reference` method
- If resolution fails, add error to context metadata
- If file doesn't exist, add error message
- Continue processing sections even if base fails (graceful degradation)

**Add new private method `process_base_content`:**
```ruby
def process_base_content(context, context_config, options)
  base_ref = context_config['base'] || context_config[:base]
  return unless base_ref

  # Resolve protocol reference
  resolved_path = resolve_file_reference(base_ref)
  unless resolved_path
    context.metadata[:base_error] = "Failed to resolve base reference: #{base_ref}"
    warn "Warning: Failed to resolve base reference: #{base_ref}" if options[:debug]
    return
  end

  # Check if file exists
  unless File.exist?(resolved_path)
    context.metadata[:base_error] = "Base file not found: #{resolved_path}"
    warn "Warning: Base file not found: #{resolved_path}" if options[:debug]
    return
  end

  # Load base content
  base_content = File.read(resolved_path).strip
  if base_content.empty?
    warn "Warning: Base file is empty: #{resolved_path}" if options[:debug]
  end

  # Store base content as primary content
  context.content = base_content
  context.metadata[:base_path] = resolved_path
  context.metadata[:base_ref] = base_ref
end
```

**Changes to `load_template` method (line 381-465):**

### Risk Analysis

**Technical Risks:**

1. **Protocol Resolution Failure**
   - Risk: ace-nav fails to resolve prompt:// protocols
   - Impact: Base content not loaded, empty output
   - Mitigation: Graceful error handling, clear error messages
   - Rollback: Remove base field processing, use sections only

2. **Content Composition Conflicts**
   - Risk: Base content interferes with section formatting
   - Impact: Malformed output, parsing issues
   - Mitigation: Clear separation with `\n\n`, test various combinations
   - Rollback: Revert section_formatter changes

3. **Backward Compatibility**
   - Risk: Existing configs break with new base handling
   - Impact: Tools using ace-context fail
   - Mitigation: Base field is optional, existing behavior unchanged
   - Rollback: Feature flag to disable base processing

**Performance Risks:**

1. **Additional File I/O**
   - Risk: Loading base files adds latency
   - Impact: Slower ace-context processing
   - Mitigation: Base files are typically small (<50KB)
   - Rollback: Cache base content per protocol reference

**Integration Risks:**

1. **ace-review Preset Migration**
   - Risk: Preset structure change breaks existing workflows
   - Impact: Users need to update their presets
   - Mitigation: Update all example presets, document migration
   - Rollback: Support both `instructions.base` and `instructions.context.base` temporarily

### Rollback Strategy

**If base processing fails:**
1. Remove `process_base_content` method calls
2. Revert section_formatter changes
3. Remove base field from ace-review presets
4. All configs work as before (sections only)

**If integration tests fail:**
1. Add feature flag: `enable_base_field` (default: false)
2. Enable only for specific presets during testing
3. Gradually roll out after validation

**If performance degrades:**
1. Add caching layer for base content by protocol reference
2. Lazy load base content only when needed
3. Add size limits for base files
