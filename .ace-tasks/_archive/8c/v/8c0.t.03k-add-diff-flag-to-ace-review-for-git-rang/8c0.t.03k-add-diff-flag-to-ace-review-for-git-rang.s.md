# Add --diff flag to ace-review for Git range-based code reviews

## Description

Add a `--diff` flag to the `ace-review` CLI that accepts a Git diff range (e.g., `main..HEAD`, `commit_sha`, `sha1..sha2`). This flag will automatically identify changed files from the Git diff and construct the review context from the diff output, similar to how the `--pr` flag provides context for PR reviews.

Currently, `ace-review` requires users to manually specify files or rely on the current working directory. The `--diff` flag will enable deterministic, precise scoping of code reviews to specific Git changes, improving both manual and automated workflows.

## Acceptance Criteria

### CLI Behavior
- [ ] `ace-review --diff <range>` accepts Git diff range syntax (e.g., `main..HEAD`, `abc123`, `abc123..def456`)
- [ ] Invalid Git ranges (non-existent SHAs, invalid syntax) produce clear error messages
- [ ] `--diff` flag can be combined with `--subject` to provide custom review context
- [ ] `--diff` flag precedence is clearly defined when used with other file selection flags (e.g., `--files`)

### Git Integration
- [ ] Automatically executes `git diff --name-only <range>` to identify changed files
- [ ] Retrieves actual diff content via `git diff <range>` for review context
- [ ] Validates Git repository accessibility and provides appropriate error handling
- [ ] Works correctly with various Git range formats (commit SHA, branch names, ranges)

### Review Context Generation
- [ ] Review subject is automatically set/augmented to reflect the Git range (e.g., "Review changes from `main..HEAD`")
- [ ] Diff content is properly integrated into the LLM prompt context
- [ ] Generated prompts are cached using `PromptCacheManager` in `.cache/ace-review/sessions/{operation}-{timestamp}/`
- [ ] Cached metadata includes the Git range used for the review

### Documentation
- [ ] `--diff` flag usage is documented in `ace-review/docs/usage.md`
- [ ] Examples demonstrate various Git range formats and use cases
- [ ] Flag interactions and precedence rules are clearly explained
- [ ] Relevant workflow instructions are updated

### Testing
- [ ] Unit tests cover Git range parsing and validation
- [ ] Integration tests verify end-to-end workflow with various Git ranges
- [ ] Edge cases tested: empty diffs, large diffs, merge commits, initial commits
- [ ] Flag combination scenarios are tested (--diff with --subject, --files, etc.)

## Implementation Notes

### Architecture Components

1. **CLI Update** (`lib/ace/review/commands/cli.rb`)
   - Add `--diff` option to Thor CLI definition
   - Accept string argument for Git range

2. **Git Interaction Layer** (new Molecule/Organism)
   - Parse and validate Git diff range syntax
   - Execute `git diff --name-only <range>` for file list
   - Execute `git diff <range>` for diff content
   - Leverage utilities from `ace-git-commit` or direct `git` commands

3. **Context Generation** (existing Organism)
   - Integrate diff content into `user.prompt.md` generation
   - Implement flag precedence logic
   - Handle large diffs (chunking/summarization if needed)

4. **Caching** (existing `PromptCacheManager`)
   - Store generated prompts with Git range metadata
   - Standard cache path: `.cache/ace-review/sessions/{operation}-{timestamp}/`

### Flag Precedence Rules
- `--diff` defines the scope of files to review
- If `--subject` is not provided, auto-generate from Git range
- If `--subject` is provided, use it but include Git range context
- `--diff` should take precedence over implicit file selection (cwd)
- Define behavior when `--diff` is combined with `--files` (error or merge?)

### Error Handling Scenarios
- Invalid Git range syntax
- Non-existent commit SHAs
- Repository not accessible or not a Git repo
- Empty diff (no changes in range)
- Detached HEAD or other Git states

### Large Diff Considerations
- Monitor LLM context window limits
- Consider chunking strategies for very large diffs
- Potential integration with `ace-context` for context management
- May need to prioritize or summarize certain file types

### Testing Strategy
- Use `ace-test` for all test execution (NOT `bundle exec rake test`)
- Create test fixtures with sample Git repositories
- Test various range formats: `HEAD~1`, `main..HEAD`, `abc123..def456`, `abc123`
- Verify prompt cache structure and metadata
- Test integration with existing `ace-review` features
