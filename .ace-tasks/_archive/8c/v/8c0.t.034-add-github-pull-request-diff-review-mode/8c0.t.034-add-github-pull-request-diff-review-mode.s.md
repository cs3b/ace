# Add GitHub Pull Request diff review mode to ace-review

## Behavioral Specification

### User Experience
- **Input**: PR identifier (number, URL, or `owner/repo#number` format), optional review preset, optional `--post-comment` flag
- **Process**: ace-review fetches PR diff via `gh` CLI, analyzes code changes with LLM using standard review presets, saves review locally, optionally posts formatted review as PR comment
- **Output**: Cached review file in `.cache/ace-review/sessions/pr-review-{timestamp}/`, terminal summary of findings, optional GitHub PR comment, clear success/error messages

### Expected Behavior

When a developer or AI agent uses `ace-review --pr <PR_IDENTIFIER>`, they experience:

1. **Seamless PR Access**: Provide PR number, URL, or `owner/repo#number` format - ace-review handles diff retrieval automatically via `gh` CLI
2. **Standard Review Flow**: Same LLM-powered analysis as file-based reviews, supporting all existing presets (security, architecture, etc.)
3. **Local Caching**: Review results saved to standard cache location for debugging and reference
4. **Optional GitHub Integration**: `--post-comment` flag publishes review directly to the PR
5. **Clear Feedback**: Informative messages for authentication, network issues, permissions, or PR states (draft, closed, merged)

### Interface Contract

```bash
# Basic PR diff review (save locally only)
ace-review --pr 123
ace-review --pr https://github.com/owner/repo/pull/123
ace-review --pr owner/repo#123

# Review with specific preset
ace-review --pr 123 --preset security
ace-review --pr 123 --preset architecture

# Review and post comment to GitHub
ace-review --pr 123 --post-comment
ace-review --pr 123 --preset security --post-comment

# Dry run (no comment posting, just show what would be posted)
ace-review --pr 123 --post-comment --dry-run

# Expected outputs:
# Success: "✓ PR #123 review completed. Results saved to .cache/ace-review/sessions/pr-review-20251116-140530/"
# Success with comment: "✓ Review posted to PR #123: https://github.com/owner/repo/pull/123#issuecomment-123456"
# Error: "✗ Failed to fetch PR: GitHub authentication required. Run 'gh auth login'"
# Error: "✗ PR #999 not found in owner/repo"
# Warning: "⚠ PR #123 is in draft state"
```

**Error Handling:**
- [No `gh` CLI installed]: Error with installation instructions (`brew install gh` or link to docs)
- [Not authenticated with GitHub]: Error directing to `gh auth login` or `gh auth status`
- [PR not found]: Clear message indicating invalid PR number/URL for the repository
- [Insufficient permissions]: Error explaining required GitHub permissions (repo read access minimum)
- [Network/API failures]: Retry with exponential backoff (3 attempts), clear failure message with suggestion to check connectivity
- [Rate limiting]: Pause with informative message about rate limit status and reset time

**Edge Cases:**
- [Draft PRs]: Reviewable with warning note in output and review comment
- [Closed/merged PRs]: Reviewable but `--post-comment` blocked with clear error message
- [Large diffs]: Handle pagination from `gh` CLI, warn if diff exceeds LLM context limits (suggest file filtering)
- [Private repos]: Ensure token has appropriate access, clear error if not
- [Cross-repo PRs]: Support forks with proper owner/repo identification
- [Empty diffs]: Handle gracefully with informative message (no changes to review)

### Validation Questions

- [ ] **Comment Format**: Should we support GitHub's suggestion format (`suggestion` code blocks) for inline code comments, or only PR-level summary comments?
- [ ] **Update Strategy**: When re-reviewing same PR, should we replace existing review comment, append to it, or create new comment?
- [ ] **Diff Scope**: Should we support reviewing specific file paths within a PR (e.g., `--pr 123 --files "src/**/*.rb"`)?
- [ ] **Authentication Fallback**: Should we support direct `GITHUB_TOKEN` env var if `gh` CLI is unavailable or preferred?
- [ ] **Concurrent Reviews**: How do we handle multiple reviews of same PR running simultaneously (different presets)?
- [ ] **Review Comment Metadata**: Should we add metadata to comments (preset used, ace-review version, timestamp)?

## Technical Approach

### Architecture Pattern

**Integration Strategy**: Extend existing ace-review architecture with new GitHub PR molecule layer

The implementation follows ace-review's ATOM pattern:
- **Atoms**: Pure functions for gh CLI execution (`gh_cli_executor.rb`)
- **Molecules**: PR-specific operations (`pr_identifier_parser.rb`, `gh_pr_fetcher.rb`, `gh_comment_poster.rb`)
- **Organisms**: Orchestration via existing `review_manager.rb`
- **CLI**: New options in `cli.rb` for PR mode

**Key Design Decisions**:
1. **Molecule Layer**: Create new PR-specific molecules that integrate with existing review pipeline
2. **CLI Integration**: Extend existing CLI with `--pr` and `--post-comment` flags
3. **Subject Extraction**: Modify `subject_extractor.rb` to detect and handle PR identifiers
4. **Session Naming**: Use `pr-review-{timestamp}` pattern to distinguish PR reviews from file reviews
5. **Error Handling**: Centralized error handling with clear, actionable messages

**Integration with Existing Architecture**:
- PR diff extraction flows through existing `subject_extractor.rb` → `review_manager.rb` pipeline
- PR metadata (state, draft status) added as context via `context_extractor.rb`
- Comment posting happens after LLM execution, independent of core review logic
- All existing presets work with PR mode without modification

### Technology Stack

**Primary Dependencies**:
- **GitHub CLI (`gh`)**: External dependency for GitHub API integration
  - Version: 2.0+ (widely available, stable API)
  - Installation: `brew install gh` (macOS), apt/yum packages (Linux)
  - Authentication: Handles OAuth and token management
  - Commands: `gh pr diff`, `gh pr view`, `gh pr comment`

**Ruby Libraries** (already in ace-review):
- `Open3`: For safe subprocess execution (existing, no new dependency)
- `YAML`, `JSON`: For parsing gh CLI output (existing)
- `FileUtils`, `Time`: For session management (existing)

**Integration Points**:
- **ace-context**: Existing integration for content aggregation (no changes needed)
- **ace-llm-query**: Existing integration for LLM execution (no changes needed)
- **ace-nav**: Existing integration for prompt resolution (no changes needed)

**Security Considerations**:
- Use `Open3.capture3` with array-form arguments (prevent command injection)
- Validate PR identifiers before passing to `gh` CLI
- Handle authentication tokens securely via `gh` CLI (no direct token handling)
- Sanitize user input in comments (escape markdown special characters)

**Performance Implications**:
- Network latency: `gh pr diff` typically 1-3 seconds depending on PR size
- Rate limiting: GitHub API limits (5000 req/hour authenticated, 60/hour unauthenticated)
- Diff caching: First call fetches, subsequent calls use cached session files
- No impact on existing file-based review performance

### Tool Selection

| Criteria | gh CLI | Octokit.rb | GitHub API Direct | Selected |
|----------|--------|------------|-------------------|----------|
| **Authentication** | Managed by gh | Manual OAuth | Manual tokens | gh CLI |
| **Installation** | Simple (brew/apt) | Gem dependency | No install | gh CLI |
| **Maintenance** | GitHub-maintained | Community | DIY | gh CLI |
| **Error Messages** | Clear, actionable | Generic | Raw HTTP | gh CLI |
| **Learning Curve** | Low (CLI user-friendly) | Medium (API knowledge) | High (API expertise) | gh CLI |
| **Integration** | Shell execution | Ruby API | HTTP requests | gh CLI |

**Selection Rationale**:
- **gh CLI** provides the best developer experience and simplest integration
- Users likely already have `gh` installed for GitHub workflows
- Authentication handled externally (no token management in ace-review)
- Clear error messages and status codes from `gh` CLI
- Future-proof: GitHub maintains compatibility guarantees

**Dependency Analysis**:
- **New external dependency**: `gh` CLI (user must install)
- **No new gem dependencies**: Uses existing Ruby stdlib (Open3, JSON, YAML)
- **Compatibility**: Works with all Ruby versions ace-review supports (2.7+)
- **Conflict risk**: None (gh CLI is standalone tool)

## Test Case Planning

### PR Identifier Parsing Tests

**Happy Path Scenarios**:
1. Parse PR number in current repo context: `"123"` → `{owner: "user", repo: "repo", number: 123}`
2. Parse full GitHub URL: `"https://github.com/owner/repo/pull/456"` → `{owner: "owner", repo: "repo", number: 456}`
3. Parse qualified reference: `"external/project#789"` → `{owner: "external", repo: "project", number: 789}`

**Edge Cases**:
1. PR number with leading zeros: `"0042"` → normalized to `42`
2. GitHub Enterprise URL: `"https://github.company.com/team/app/pull/10"`
3. URL with query parameters: `"https://github.com/owner/repo/pull/5?tab=files"`
4. Very large PR numbers: `"999999999"`

**Error Conditions**:
1. Invalid URL format: `"https://example.com/pr/123"` → error
2. Negative PR number: `"-5"` → error
3. Non-numeric PR reference: `"abc"` → error
4. Empty string: `""` → error
5. Malformed qualified ref: `"owner/repo/123"` (missing #) → error

## Risk Assessment

### Technical Risks

- **Risk**: gh CLI not installed on user systems
  - **Probability**: Medium
  - **Impact**: High (feature unusable)
  - **Mitigation**: Clear installation instructions in error message, README prerequisites section
  - **Rollback**: Feature is additive; users without gh CLI can still use existing ace-review features

- **Risk**: gh CLI API changes in future versions
  - **Probability**: Low
  - **Impact**: Medium (breakage in future updates)
  - **Mitigation**: Test against multiple gh versions, use stable JSON output format, monitor gh release notes
  - **Rollback**: Pin to known working gh CLI version in documentation

- **Risk**: GitHub API rate limiting impacting usability
  - **Probability**: Medium (for frequent users)
  - **Impact**: Medium (temporary service disruption)
  - **Mitigation**: Detect rate limit errors, show clear message with reset time, suggest authenticated token
  - **Rollback**: N/A (GitHub API limitation, not code issue)

- **Risk**: Large PR diffs exceeding LLM context limits
  - **Probability**: Medium
  - **Impact**: Medium (incomplete reviews)
  - **Mitigation**: Warn users about large diffs, suggest file filtering, document context limit guidance
  - **Rollback**: Users can fall back to reviewing specific files

### Integration Risks

- **Risk**: Breaking existing subject extraction functionality
  - **Probability**: Low
  - **Impact**: High (existing workflows broken)
  - **Mitigation**: Add PR detection before existing logic (early return pattern), comprehensive regression tests
  - **Monitoring**: Full test suite execution, manual testing of existing presets
  - **Rollback**: Revert subject_extractor.rb changes, feature disabled

- **Risk**: Session directory conflicts with existing reviews
  - **Probability**: Low
  - **Impact**: Low (minor confusion)
  - **Mitigation**: Use distinct `pr-review-{timestamp}` naming pattern
  - **Monitoring**: Verify session directory creation in tests

- **Risk**: Comment posting errors leaving reviews incomplete
  - **Probability**: Medium
  - **Impact**: Low (review saved locally, just not posted)
  - **Mitigation**: Comment posting happens after review completion, clear error messages, suggest manual posting
  - **Monitoring**: Test comment posting error scenarios

### Performance Risks

- **Risk**: Network latency for gh CLI commands
  - **Mitigation**: Implement timeout (30s default), retry logic, clear feedback
  - **Monitoring**: Log gh CLI execution time in verbose mode
  - **Thresholds**: Warn if fetch takes >10s, timeout at 30s

- **Risk**: Subprocess execution overhead
  - **Mitigation**: Use efficient Open3 API, minimal subprocess calls (batch metadata+diff if possible)
  - **Monitoring**: Benchmark PR review vs file review performance
  - **Thresholds**: PR review should complete within 2x file review time

### Security Risks

- **Risk**: Command injection via malicious PR identifiers
  - **Probability**: Low (requires crafted input)
  - **Impact**: High (arbitrary command execution)
  - **Mitigation**: Strict input validation, array-form subprocess arguments, whitelist-based identifier parsing
  - **Rollback**: Input sanitization layer before gh CLI execution

- **Risk**: Sensitive data in review comments
  - **Probability**: Low (LLM-dependent)
  - **Impact**: Medium (information disclosure)
  - **Mitigation**: Document best practices, dry-run mode for preview, user review before posting
  - **Rollback**: Delete comment via GitHub UI if posted in error

## Validation Questions Decisions

### Comment Format
**Decision**: Start with PR-level summary comments only
- Inline suggestion format deferred to future enhancement
- Simpler implementation, faster delivery
- Can iterate based on user feedback

### Update Strategy
**Decision**: Create new comment on each review
- Append strategy considered but adds complexity
- Users can delete old comments manually if desired
- Clear audit trail of review iterations

### Diff Scope
**Decision**: Support full PR diff initially, file filtering via existing `--subject` syntax
- Example: `ace-review --pr 123 --subject 'files: ["lib/**/*.rb"]'`
- Leverages existing subject extraction capabilities
- No new syntax needed

### Authentication Fallback
**Decision**: gh CLI only for MVP
- Direct GITHUB_TOKEN support deferred
- Reduces complexity, leverages gh's auth management
- Can add env var fallback in future if requested

### Concurrent Reviews
**Decision**: Each review creates separate session directory
- Timestamp-based naming prevents conflicts
- No special handling needed
- Works naturally with existing architecture

### Review Comment Metadata
**Decision**: Include preset, model, timestamp in comment header
- Aids debugging and audit trails
- Follows existing review session metadata pattern
- Example:
  ```markdown
  ## Code Review - ace-review

  **Preset**: security
  **Model**: google:gemini-2.5-flash
  **Generated**: 2025-11-16 14:30:00 UTC

  [Review content...]
  ```
