# Allow ace-overseer work-on to accept ordered multi-task --task lists

## Behavioral Specification

### User Experience
- **Input**: User runs `ace-overseer work-on` with multiple `--task` values either repeated (`--task 288 --task 287`) or comma-separated (`--task 288,287`), optionally mixing both forms.
- **Process**: Command validates all task references up front, preserves positional order, expands each task to task/subtask sequence when needed, then prepares one assignment session for ordered execution.
- **Output**: User receives clear progress output and a prepared assignment whose task execution order matches the input order; invalid inputs fail with actionable error messages.

### Expected Behavior
`ace-overseer work-on` must support multi-task list input through `--task` and interpret both repeated and comma-separated syntax as equivalent ordered input. The command must preserve sequence exactly as provided, including mixed task/subtask references. When tasks include orchestrators with subtasks, the generated execution list must expand in sequence without reordering unrelated entries. All provided refs must be validated before side effects. If multiple refs are provided with a preset that only supports single `taskref`, command must fail with a clear message instructing use of a `taskrefs` preset.

### Interface Contract
```bash
# Supported equivalent forms
ace-overseer work-on --task 288 --task 287 --preset work-on-tasks
ace-overseer work-on --task 288,287 --preset work-on-tasks
ace-overseer work-on --task 288,287.01 --task 300 --preset work-on-tasks

# Expected behavior contract
# - Input order preserved
# - Validation before worktree/tmux/assignment side effects
# - One assignment run with ordered taskrefs for multi-task preset
```

**Error Handling:**
- Multiple refs with single-task preset: fail with explicit preset-capability error.
- Invalid task reference token: fail with token-specific validation message.
- Task not found in taskflow: fail with task-specific not-found message before side effects.

**Edge Cases:**
- Mixed repeated and comma forms in one command preserve left-to-right order.
- Explicit subtask refs remain explicit; orchestrator refs expand to subtasks in-place.

### Success Criteria
- [ ] **Multi-Input Equivalence**: Repeated `--task` and comma-separated `--task` forms produce the same ordered task sequence.
- [ ] **Order Guarantee**: Task and subtask execution order in generated assignment matches user-provided order.
- [ ] **Safe Validation**: Any invalid/missing reference aborts before worktree provisioning, tmux changes, or assignment creation.
- [ ] **Preset Guarding**: Multi-task input with non-`taskrefs` preset fails with actionable guidance.

### Validation Questions
- [ ] **Duplicates**: Should duplicate task refs be preserved as-is or deduplicated?
- [ ] **Cross-release refs**: Should mixed qualified refs across releases be accepted in one call?
- [ ] **Partial continuation**: Should future behavior ever allow best-effort execution mode?

## Objective
Enable users to start multi-task assignment flows directly from `ace-overseer work-on` using natural list-style `--task` input while preserving explicit execution sequencing.
