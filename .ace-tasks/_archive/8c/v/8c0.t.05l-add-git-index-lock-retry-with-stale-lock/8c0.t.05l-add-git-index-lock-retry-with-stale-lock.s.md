# Add Git Index Lock Retry with Stale Lock Cleanup

## Behavioral Specification

### User Experience
- **Input**: Any git operation via ace-git tools (ace-git-commit, ace-git status, etc.)
- **Process**: When encountering `index.lock` errors, the system silently retries with exponential backoff, automatically cleaning stale locks
- **Output**: Either successful git operation or clear error message after all retries exhausted

### Expected Behavior

When working with multiple git worktrees and using ace-git tools frequently, users and agents encounter `Unable to create .git/index.lock: File exists` errors. This happens when:
- Previous git operations were interrupted (Ctrl+C, crashes, timeouts)
- Multiple concurrent operations across worktrees contend for the same lock
- Agents are blocked mid-operation leaving orphan lock files

The system should:
1. Detect index.lock errors from git command stderr
2. Automatically clean stale lock files (older than 60 seconds)
3. Retry with exponential backoff (50ms → 100ms → 200ms → 400ms)
4. Silently handle transient lock contention without user intervention
5. Only surface errors if all retries fail

### Interface Contract

```bash
# No change to CLI interface - behavior is transparent
ace-git-commit -i "message"

# Current behavior (broken):
# ✗ Failed to stage changes
# Error: fatal: Unable to create '.git/index.lock': File exists.

# New behavior (graceful):
# [Silent retries with backoff]
# [Stale lock auto-cleaned if >60 seconds old]
# ✓ Changes staged and committed
# OR after max retries:
# ✗ Failed to stage changes after 4 retries
# Error: Git index locked by another process
```

**Configuration** (`.ace/git/config.yml`):
```yaml
lock_retry:
  enabled: true                   # Enable/disable retry behavior
  max_retries: 4                  # Maximum retry attempts
  initial_delay_ms: 50            # Initial backoff delay
  stale_cleanup: true             # Auto-remove stale locks
  stale_threshold_seconds: 60     # Age threshold for stale locks
```

**Error Handling:**
- Lock error detected: Silent retry with backoff
- Stale lock found: Auto-remove and retry
- Fresh lock (another process): Retry with backoff, fail after max attempts
- Non-lock errors: Pass through immediately (no retry)

**Edge Cases:**
- Lock file exists but is fresh (<60s): Retry only, don't remove
- Lock file in worktree: Handle worktree-specific lock paths
- Permission denied on lock removal: Surface error clearly
- Config disables retry: Fail immediately as before

## Risk Assessment

### Technical Risks
- **Risk**: Removing lock owned by active git process
  - **Probability**: Low (60s threshold is generous)
  - **Impact**: Medium (could corrupt index)
  - **Mitigation**: Only remove locks older than 60 seconds
  - **Rollback**: Disable stale_cleanup in config

### Integration Risks
- **Risk**: Retry delays slow down operations
  - **Probability**: Low (max ~750ms total)
  - **Impact**: Low (barely noticeable)
  - **Mitigation**: Exponential backoff keeps fast path fast
  - **Monitoring**: Total retry time logged on final failure
