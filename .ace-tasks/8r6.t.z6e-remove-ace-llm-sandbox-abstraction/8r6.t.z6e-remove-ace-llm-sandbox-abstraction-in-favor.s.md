# Remove ace-llm sandbox abstraction in favor of presets

## Objective

Remove the generic `sandbox` execution knob from `ace-llm` so callers rely on provider presets for execution policy and use `working_dir` plus `subprocess_env` for runtime context. This keeps provider-specific CLI behavior in provider config instead of hidden translation logic inside `ace-llm`.

## Behavioral Specification

### User Experience

- Input:
  - Developers call `Ace::LLM::QueryInterface.query` or the `ace-llm` CLI with a provider/model, prompt, optional preset, optional `working_dir`, and optional extra `cli_args`.
  - Callers that need repo or sandbox context pass that context through `working_dir` and `subprocess_env`.
  - Provider execution policy such as read-only, auto-edit, yolo, or other CLI mode selection is expressed through provider presets or explicit `cli_args`.
- Process:
  - `ace-llm` resolves provider/model and preset configuration without maintaining an additional sandbox-policy abstraction.
  - The process runs from the requested working directory and inherits the provided environment overrides.
  - Fallback attempts keep using the same call-site working directory and environment while recomputing provider-specific preset-derived options.
- Output:
  - Users get the same LLM response formats as today.
  - Invalid use of removed sandbox configuration surfaces as a clear migration error.
  - E2E runner behavior remains controlled by sandbox directory setup outside `ace-llm`.

### Expected Behavior

`ace-llm` must treat execution location and execution policy as separate concerns. It must support running provider CLIs from a specified directory and with injected environment variables, but it must not expose a generic top-level sandbox abstraction that overlaps with presets. Provider-specific execution flags must come from preset configuration or explicit raw CLI arguments only.

When existing callers or configs still reference sandbox behavior, the system must fail in a way that points them to the supported mechanisms: `working_dir`, `subprocess_env`, preset configuration, or explicit `cli_args`.

## Interface Contract

### Developer API

```ruby
Ace::LLM::QueryInterface.query(
  "role:e2e-runner",
  prompt,
  working_dir: "/tmp/e2e-sandbox",
  subprocess_env: {"PROJECT_ROOT_PATH" => "/tmp/e2e-sandbox"},
  preset: "ro"
)
```

Expected behavior:
- `working_dir` controls process cwd only.
- `subprocess_env` controls environment injection only.
- preset or `cli_args` control provider-specific execution flags.
- `sandbox:` is not accepted as a supported caller-facing control.

### CLI / Config Behavior

- Provider presets in `.ace/llm/config.yml` remain the canonical place for provider-specific execution policy.
- Any documented examples that currently describe sandbox policy via `sandbox` move to preset or `cli_args` examples.
- E2E execution keeps passing sandbox path via `working_dir` and env vars, not via an `ace-llm` sandbox option.

### Error Handling

- If a caller passes `sandbox` input, the system returns a clear error explaining the supported replacement.
- If preset-derived CLI flags are incompatible with a fallback provider, fallback must recompute the target provider’s preset options rather than forwarding the previous provider’s flags.

### Edge Cases

- Fallback from one CLI provider to another preserves `working_dir` and `subprocess_env` but does not preserve provider-specific CLI flags.
- Callers with no preset and no `cli_args` still execute successfully when the provider’s defaults are sufficient.

## Success Criteria

- `ace-llm` no longer exposes a generic sandbox abstraction as part of its active execution contract.
- Provider execution policy is sourced only from presets or explicit `cli_args`.
- E2E runner and similar callers express sandbox location only through `working_dir` and environment variables.
- Documentation and tests make the separation between cwd/env and provider policy explicit.

## Validation Questions

- `sandbox` is removed immediately with a hard error; no deprecation alias period.
- Are there any non-E2E callers relying on `sandbox` today that need migration examples in usage docs?
- Should preset names standardize around execution intent labels like `ro`, `auto-edit`, and `yolo` across providers, or remain provider-local?

## Vertical Slice Decomposition (Task/Subtask Model)

- Slice type: standalone task
- Slice outcome: `ace-llm` callers use presets for provider policy and `working_dir`/`subprocess_env` for runtime context without a parallel sandbox abstraction.
- Advisory size: medium
- Context dependencies: `ace-llm` query interface, provider preset config, E2E runner integration points, and usage docs for API surface changes.

## Verification Plan

### Unit / Component Validation

- Query interface accepts `working_dir` and `subprocess_env` without requiring sandbox-specific options.
- Fallback recomputes provider-specific preset options for each attempted provider.
- Deprecated or removed sandbox input produces the documented migration signal.

### Integration / E2E Validation

- E2E pipeline invokes `ace-llm` with sandbox cwd and env vars only.
- A CLI-provider E2E scenario still runs from the prepared sandbox after sandbox abstraction removal.

### Failure / Invalid Path Validation

- Passing sandbox-specific input yields a clear message describing the supported replacement.
- Provider fallback does not leak one provider’s CLI flags into another provider.

## Scope of Work

- Included:
  - `ace-llm` public execution contract cleanup for sandbox-related inputs.
  - migration of internal callers to `working_dir` and `subprocess_env` where needed.
  - usage/docs updates for the supported execution model.
- Excluded:
  - redesign of provider preset formats beyond what is necessary to express current execution policy.
  - unrelated E2E scenario fixes not caused by sandbox contract confusion.
  - broader CLI API redesign outside sandbox-related execution controls.

## Deliverables

### Behavioral Specifications

- clear runtime contract separating cwd/env handling from provider execution policy
- documented hard-removal migration guidance for sandbox usage
- preserved fallback semantics with provider-specific preset recomputation

### Validation Artifacts

- updated tests covering the supported contract
- usage examples showing preset plus `working_dir` usage
- one focused E2E scenario demonstrating sandbox-root execution without `sandbox`

## Out of Scope

- introducing new provider policy abstractions beyond presets
- hiding provider CLI behavior behind additional generic execution knobs
- changing unrelated task, review, or assignment workflows
