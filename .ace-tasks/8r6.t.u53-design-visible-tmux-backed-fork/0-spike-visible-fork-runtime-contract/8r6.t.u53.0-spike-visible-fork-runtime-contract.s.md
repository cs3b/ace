# Spike ACE tmux runtime visibility and recording contract

## Objective

Validate the end-state contract for ACE-managed tmux runtime visibility and recording, including which current visible-fork concepts survive, which recording concepts are required, and how `ace-assign`, `ace-tmux`, and provider CLI execution boundaries should relate before additional implementation subtasks are drafted.

## Behavioral Specification

### User Experience

- A designer or implementer can answer, from one spike outcome, how a visible fork run should appear in tmux, how recorded manual takeover activity should be captured for later inspection, and what fallback behavior users should experience when visible mode is unavailable.
- The spike produces a concept inventory that makes later implementation subtasks stable instead of speculative.
- The spike resolves whether existing launcher, metadata, and tmux behaviors survive intact, are extended, or should be superseded.

### Expected Behavior

1. The spike validates one coherent end-state scenario for visible fork execution from `ace-assign fork-run` through operator inspection in tmux.
2. The spike validates one coherent end-state scenario for ACE-managed recording from session start through later inspection of pane artifacts after manual takeover work.
3. The spike explicitly identifies which existing concepts remain authoritative, including assignment-state completion and the current provider-backed launcher fallback.
4. The spike explicitly identifies the minimum new concepts required, including runtime tmux state inventory, fork-window placement behavior, and recording artifact scope.
5. The spike produces a concept inventory table showing concept status as kept, changed, new, or rejected.
6. The spike defers further subtask drafting until the inventory makes later decomposition stable.

### Interface Contract

- **Spike output contract**
  - Updated parent task concept inventory and scope notes
  - Clear go/no-go statement for these behavioral contracts:
    - optional visible fork mode
    - sibling `-fork` window naming and placement
    - shared auto-grid pane surface
    - generic `ace-tmux state` runtime inventory
    - optional ACE-managed recording
    - `.ace-local/tmux` artifact contract
    - assignment-state-authoritative completion
- **Decision contract**
  - The spike does not implement code.
  - The spike may refine task specs and usage draft content if the validated contract differs from the initial draft.

### Success Criteria

- [ ] One validated end-state scenario exists for visible fork execution.
- [ ] One validated end-state scenario exists for ACE-managed recording of manual takeover activity.
- [ ] The spike identifies which current concepts survive unchanged, which are extended, and which are newly introduced.
- [ ] The spike leaves no unresolved behavioral ambiguity for window naming, placement, fallback, recording scope, or source-of-truth completion semantics.
- [ ] The parent task can be safely decomposed into implementation subtasks after the spike.

### Validation Questions

- Do any currently assumed visible-fork concepts become unnecessary once recording and runtime inspection are evaluated together?
- Does the proposed `ace-tmux state` surface need to expose only effective recording state plus artifact paths, or also the scope that enabled recording?

## Vertical Slice Decomposition (Task/Subtask Model)

- **Slice type**: subtask spike
- **Slice outcome**: validated runtime contract and concept inventory for visible fork execution and ACE-managed recording
- **Advisory size**: small
- **Context dependencies**: parent task draft, current `ace-assign` fork path, `ace-tmux` runtime gaps, ACE-managed pane launch behavior, provider CLI command-planning behavior

## Verification Plan

### Unit / Component Validation

- Review the spike output and confirm each end-state concept maps cleanly to one subsystem responsibility.
- Confirm the recording contract stays evidence-oriented and does not claim semantic learning behavior.

### Integration / E2E Validation

- Trace the single validated scenario from `ace-assign fork-run --launch-mode tmux-visible` to a visible pane in `<current-window>-fork` and back to assignment-state completion.
- Trace the single validated scenario from recorded `ace-tmux` session startup to later inspection of pane artifacts under `.ace-local/tmux/`.

### Failure / Invalid Path Validation

- Confirm the spike explicitly covers the no-tmux fallback path and the premature-pane-exit diagnostic path.
