# Spike visible tmux-backed fork execution contract

## Objective

Validate the end-state contract for visible tmux-backed fork execution in `ace-assign`, including which visible-fork concepts survive, how provider CLI and tmux boundaries relate, and what persisted linkage and diagnostic behavior visible runs require before additional implementation subtasks are drafted.

## Behavioral Specification

### User Experience

- A designer or implementer can answer, from one spike outcome, how a visible fork run should appear in tmux and what fallback behavior users should experience when visible mode is unavailable.
- A designer or implementer can answer, from one spike outcome, what visible-run metadata and diagnostics `ace-assign` must own directly versus what should be consumed later from the sibling tmux inspectability surface.
- The spike produces a concept inventory that makes later implementation subtasks stable instead of speculative.
- The spike resolves whether existing launcher, metadata, and tmux behaviors survive intact, are extended, or should be superseded.

### Expected Behavior

1. The spike validates one coherent end-state scenario for visible fork execution from `ace-assign fork-run` through operator inspection in tmux.
2. The spike explicitly identifies which existing concepts remain authoritative, including assignment-state completion and the current provider-backed launcher fallback.
3. The spike explicitly identifies the minimum new concepts required, including fork-window placement behavior, visible-run linkage metadata, and visible-pane diagnostics.
4. The spike explicitly defers generic `ace-tmux state`, recording, and artifact contracts to sibling task `8r6.t.xeu`.
5. The spike produces a concept inventory table showing concept status as kept, changed, new, or rejected.
6. The spike defers further subtask drafting until the inventory makes later decomposition stable.
7. The spike locks the operator-facing diagnostic contract for premature pane exit to include pane-tail output plus any already-available generic inspectability references without owning their schema.

### Interface Contract

- **Spike output contract**
  - Updated parent task concept inventory and scope notes
  - Clear go/no-go statement for these behavioral contracts:
    - optional visible fork mode
    - sibling `-fork` window naming and placement
    - shared auto-grid pane surface
    - assignment-state-authoritative completion
    - visible-run linkage metadata
    - premature-pane-exit diagnostics with pane tail plus generic inspectability references when present
- **Decision contract**
  - The spike does not implement code.
  - The spike may refine task specs and usage draft content if the validated contract differs from the initial draft.
  - The spike treats generic tmux inspectability and recording contracts as out of scope and owned by sibling task `8r6.t.xeu`.

### Success Criteria

- [ ] One validated end-state scenario exists for visible fork execution.
- [ ] The spike identifies which current visible-fork concepts survive unchanged, which are extended, and which are newly introduced.
- [ ] The spike leaves no unresolved behavioral ambiguity for window naming, placement, fallback, pane-exit diagnostics, linkage metadata, or source-of-truth completion semantics.
- [ ] The spike leaves generic tmux inspectability and recording concerns cleanly assigned to sibling task `8r6.t.xeu`.
- [ ] The parent task can be safely decomposed into implementation subtasks after the spike.
