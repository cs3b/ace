# Spike visible fork runtime contract

## Objective

Validate the end-state contract for visible tmux-backed fork execution, including which current concepts survive, which new concepts are required, and how `ace-assign`, `ace-tmux`, and provider CLI execution boundaries should relate before additional implementation subtasks are drafted.

## Behavioral Specification

### User Experience

- A designer or implementer can answer, from one spike outcome, how a visible fork run should appear in tmux, how operators inspect it, and what fallback behavior users should experience when visible mode is unavailable.
- The spike produces a concept inventory that makes later implementation subtasks stable instead of speculative.
- The spike resolves whether existing launcher, metadata, and tmux behaviors survive intact, are extended, or should be superseded.

### Expected Behavior

1. The spike validates one coherent end-state scenario for visible fork execution from `ace-assign fork-run` through operator inspection in tmux.
2. The spike explicitly identifies which existing concepts remain authoritative, including assignment-state completion and the current provider-backed launcher fallback.
3. The spike explicitly identifies the minimum new concepts required, including runtime tmux state inventory and fork-window placement behavior.
4. The spike produces a concept inventory table showing concept status as kept, changed, new, or rejected.
5. The spike defers further subtask drafting until the inventory makes later decomposition stable.

### Interface Contract

- **Spike output contract**
  - Updated parent task concept inventory and scope notes
  - Clear go/no-go statement for these behavioral contracts:
    - optional visible fork mode
    - sibling `-fork` window naming and placement
    - shared auto-grid pane surface
    - generic `ace-tmux state` runtime inventory
    - assignment-state-authoritative completion
- **Decision contract**
  - The spike does not implement code.
  - The spike may refine task specs and usage draft content if the validated contract differs from the initial draft.

### Success Criteria

- [ ] One validated end-state scenario exists for visible fork execution.
- [ ] The spike identifies which current concepts survive unchanged, which are extended, and which are newly introduced.
- [ ] The spike leaves no unresolved behavioral ambiguity for window naming, placement, fallback, or source-of-truth completion semantics.
- [ ] The parent task can be safely decomposed into implementation subtasks after the spike.

### Validation Questions

- Do any currently assumed concepts become unnecessary once live provider execution and tmux runtime inspection are evaluated together?
- Does the proposed `ace-tmux state` surface need to be generic for all packages from day one, or only generic enough to avoid `ace-assign`-specific shape leakage?
