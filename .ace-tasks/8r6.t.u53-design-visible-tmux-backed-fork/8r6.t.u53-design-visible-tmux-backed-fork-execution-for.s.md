# Design ACE tmux runtime visibility and recording for forked and manual work

## Objective

Define a broader ACE-managed tmux runtime contract that makes forked `ace-assign` subtree work optionally visible in tmux and optionally records manual takeover activity for later inspection, while keeping assignment state as the authoritative execution state and leaving learning/synthesis out of scope.

## Behavioral Specification

### User Experience

- A user driving an assignment can opt into a visible fork mode and watch forked subtree work in tmux instead of treating it as a hidden background subprocess.
- When visible mode is active inside tmux, the current task window gains a sibling fork window named from the current window with a `-fork` suffix.
- Every active visible fork for that task session appears in the same fork window as its own pane, so the operator can see parallel fork activity in one place.
- Finished panes remain visible so the operator can inspect the last terminal state and output without reopening logs elsewhere.
- Users can start ACE-managed tmux sessions or windows with recording enabled when they want to capture manual takeover activity for later review.
- Recording can also be enabled or disabled explicitly in ACE-managed tmux session, window, and pane presets, with lower scopes overriding higher ones.
- Recorded panes store raw evidence under `.ace-local/tmux/` so operators can inspect what happened after switching from agent-driven to manual work.
- Users and higher-level tooling can query `ace-tmux` for live session, window, and pane state, including effective recording status, the scope that enabled it, and artifact locations, rather than shelling directly to raw `tmux` commands.

### Expected Behavior

1. `ace-assign fork-run` supports an optional visible launch mode instead of replacing the current provider-backed path.
2. Visible mode uses tmux only when a suitable tmux session is active; otherwise the command falls back to the current launcher rather than failing the assignment flow.
3. Visible mode creates or reuses a dedicated sibling fork window named `<current-window>-fork`, positioned next to the current window.
4. The fork window accumulates panes for active forked subtrees and re-applies an auto-grid layout as panes are added.
5. The assignment queue state remains the source of truth for subtree completion and failure; tmux runtime state serves only for visibility, diagnostics, and operator navigation.
6. `ace-tmux` exposes runtime state for sessions, windows, and panes in a reusable interface consumable by `ace-assign`, `ace-overseer`, and operators.
7. ACE-managed tmux sessions, windows, and panes can opt into recording through preset fields and CLI convenience flags, with effective recording behavior inheriting from session to window to pane unless explicitly overridden.
8. Recording remains disabled by default and applies only to ACE-managed panes created through `ace-tmux`.
9. Recorded panes persist raw evidence and manifest metadata under `.ace-local/tmux/` for later inspection, but the task does not promise semantic interpretation, redaction, or automated learning from those logs.
10. Visible mode records enough tmux metadata to reconnect assignment subtrees to their live or completed pane surfaces, including any recording artifacts when recording is enabled.
11. When a visible fork pane exits before assignment state reaches a terminal subtree state, operator-facing diagnostics include both recent pane tail output and any available recording artifact paths without changing assignment state as the source of truth.

### Interface Contract

- **`ace-assign` CLI contract**
  ```bash
  ace-assign fork-run --assignment 8r6.t.u53@010 --launch-mode tmux-visible
  ace-assign fork-run --assignment 8r6.t.u53@010
  ```
- **`ace-assign` behavior contract**
  - Default launch path remains the current provider-backed fork launcher.
  - `--launch-mode tmux-visible` requests tmux-backed visible execution.
  - If visible execution is unavailable, `ace-assign` falls back to the default launcher and reports the downgrade unless quiet mode suppresses it.
- **`ace-tmux` CLI contract**
  ```bash
  ace-tmux state
  ace-tmux state --format json
  ace-tmux state --session dev
  ace-tmux start --record
  ace-tmux window --record
  ```
- **`ace-tmux state` output contract**
  - Reports ACE-managed tmux sessions, windows, and panes in human and JSON forms.
  - For recording, reports the effective enabled state, the `source_scope` that set it (`session`, `window`, `pane`, or CLI override), and any artifact path or manifest path when recording is enabled.
  - Reports pane/session/window identifiers and enough liveness metadata to reconnect visible assignment panes to runtime state.
- **`ace-tmux` recording contract**
  - Recording is opt-in and off by default.
  - Recording config can be set at session, window, or pane scope in ACE-managed presets, with lower scopes overriding higher ones.
  - `ace-tmux start --record` and `ace-tmux window --record` are convenience flags that request recording through the same inheritance contract instead of defining a separate recording mode.
  - Recording applies only to ACE-managed panes launched through `ace-tmux`.
  - Recorded artifacts live under `.ace-local/tmux/`.
- **Fork-window contract**
  - Window name: `<current-window>-fork`
  - Placement: immediately after the current tmux window
  - Layout: auto-grid/tiled for all visible fork panes in that window
  - Pane lifecycle: panes remain visible after success or failure
- **Assignment metadata contract**
  - Visible runs retain assignment-scoped fork metadata and additionally persist tmux session/window/pane identifiers for operator inspection and recovery.
- **Recording artifact contract**
  - Recorded sessions persist manifest metadata plus raw evidence files for the recorded pane surface.
  - The recorded artifact contract is for later human or workflow inspection only; it does not claim to distinguish every semantic action inside every TUI program.

### Success Criteria

- [ ] Operators can choose a visible fork mode for `ace-assign` without losing the current default execution path.
- [ ] Visible fork execution groups all active fork panes into a single sibling `-fork` window for the active task window.
- [ ] `ace-assign` still determines subtree completion and failure from assignment state, not pane presence alone.
- [ ] `ace-tmux` exposes reusable runtime state for sessions, windows, and panes, including effective recording status, source scope, and artifact locations when applicable.
- [ ] Operators can enable recording for ACE-managed panes and later inspect raw session evidence for manual takeover activity under `.ace-local/tmux/`.
- [ ] Recording remains optional, off by default, and ACE-managed only.
- [ ] Visible runs leave enough persisted metadata for later status, diagnostics, and recovery workflows.
- [ ] Draft usage guidance exists for the new `ace-assign` and `ace-tmux` command surface.
- [ ] Prematurely exited visible panes surface both pane-tail diagnostics and recording artifact references when available.
- [ ] The task does not promise automatic learning, redaction, or semantic interpretation of captured logs.
