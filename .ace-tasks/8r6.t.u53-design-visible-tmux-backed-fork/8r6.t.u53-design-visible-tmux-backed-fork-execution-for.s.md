# Design visible tmux-backed fork execution for `ace-assign`

## Objective

Define the visible tmux-backed execution contract for forked `ace-assign` subtree work so operators can watch fork activity live in tmux without changing the current default launcher semantics or replacing assignment state as the source of truth.

## Behavioral Specification

### User Experience

- A user driving an assignment can opt into a visible fork mode and watch forked subtree work in tmux instead of treating it as a hidden background subprocess.
- When visible mode is active inside tmux, the current task window gains a sibling fork window named from the current window with a `-fork` suffix.
- Every active visible fork for that task session appears in the same fork window as its own pane, so the operator can see parallel fork activity in one place.
- Finished panes remain visible so the operator can inspect the last terminal state and output without reopening logs elsewhere.
- When visible execution is requested outside a suitable tmux session, the user still gets the existing provider-backed fork execution path rather than a hard failure.
- Visible runs persist enough tmux identifiers to reconnect a subtree to generic tmux inspectability surfaces defined by sibling task `8r6.t.xeu`, but this task does not define those inspectability contracts itself.

### Expected Behavior

1. `ace-assign fork-run` supports an optional visible launch mode instead of replacing the current provider-backed path.
2. Visible mode uses tmux only when a suitable tmux session is active; otherwise the command falls back to the current launcher rather than failing the assignment flow.
3. Visible mode creates or reuses a dedicated sibling fork window named `<current-window>-fork`, positioned next to the current window.
4. The fork window accumulates panes for active forked subtrees and re-applies an auto-grid layout as panes are added.
5. The assignment queue state remains the source of truth for subtree completion and failure; tmux runtime state serves only for visibility, diagnostics, and operator navigation.
6. Visible mode records enough tmux metadata to reconnect assignment subtrees to their live or completed pane surfaces, but the structure of generic tmux state and any recording artifacts is defined by sibling task `8r6.t.xeu`.
7. When a visible fork pane exits before assignment state reaches a terminal subtree state, operator-facing diagnostics include recent pane tail output and any generic inspectability references already made available by the tmux inspectability surface, without changing assignment state as the source of truth.

### Interface Contract

- **`ace-assign` CLI contract**
  ```bash
  ace-assign fork-run --assignment 8r6.t.u53@010 --launch-mode tmux-visible
  ace-assign fork-run --assignment 8r6.t.u53@010
  ```
- **`ace-assign` behavior contract**
  - Default launch path remains the current provider-backed fork launcher.
  - `--launch-mode tmux-visible` requests tmux-backed visible execution.
  - If visible execution is unavailable, `ace-assign` falls back to the default launcher and reports the downgrade unless quiet mode suppresses it.
- **`ace-tmux` recording contract**
  - Generic tmux inspectability and recording contracts are explicitly out of scope for this task and are owned by sibling task `8r6.t.xeu`.
  - This task may consume those surfaces only as already-defined dependencies.
- **Fork-window contract**
  - Window name: `<current-window>-fork`
  - Placement: immediately after the current tmux window
  - Layout: auto-grid/tiled for all visible fork panes in that window
  - Pane lifecycle: panes remain visible after success or failure
- **Assignment metadata contract**
  - Visible runs retain assignment-scoped fork metadata and additionally persist tmux session/window/pane identifiers for operator inspection and recovery.
- **Cross-task contract**
  - If generic tmux inspectability later exposes state or artifact references for visible panes, `ace-assign` may surface those references in diagnostics without redefining their schema here.

### Success Criteria

- [ ] Operators can choose a visible fork mode for `ace-assign` without losing the current default execution path.
- [ ] Visible fork execution groups all active fork panes into a single sibling `-fork` window for the active task window.
- [ ] `ace-assign` still determines subtree completion and failure from assignment state, not pane presence alone.
- [ ] Visible runs leave enough persisted metadata for later status, diagnostics, and recovery workflows.
- [ ] Draft usage guidance exists for the `ace-assign` visible-launch command surface.
- [ ] Prematurely exited visible panes surface actionable pane-tail diagnostics and may include generic inspectability references when that sibling surface exists.
- [ ] The task does not define or own generic `ace-tmux state`, recording inheritance, or artifact schema behavior.
