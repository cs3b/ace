# Define CLI usage and failure behavior

## Behavioral Specification

### User Experience
- **Input**: Normal and invalid CLI invocations for exact mode.
- **Process**: The command communicates defaults, output destination, and failure reasons clearly.
- **Output**: Predictable success output and actionable error messages.

### Expected Behavior
The exact-mode CLI must be easy to understand from examples alone. Errors must tell the user what happened and what to do next. When exact mode succeeds with unresolved artifacts, the command still succeeds while making the unresolved state obvious in output.

### Interface Contract
```bash
ace-compressor compress docs/architecture.md --mode exact
ace-compressor compress assets/chart.png --mode exact
```

**Error Handling:**
- Missing input path: exit non-zero and print usage guidance.
- Unsupported file type: exit non-zero and recommend Markdown/plain-text input.

**Edge Cases:**
- Mixed supported and unsupported inputs fail loudly rather than partially succeeding silently.
- Unresolved non-text items in otherwise supported documents remain warnings within successful output.

### Success Criteria
- [ ] Success-path examples are unambiguous.
- [ ] Invalid input errors are actionable.
- [ ] Mixed-input behavior is explicit.
- [ ] Success with unresolved artifacts is distinguishable from hard failure.

### Validation Questions
- [x] Mixed supported and unsupported explicit inputs fail non-zero.
- [x] Unresolved artifacts inside supported docs do not convert success into failure.
- [x] CLI examples anchor the eventual README and usage docs.
- [x] Error guidance recommends exact next steps.
