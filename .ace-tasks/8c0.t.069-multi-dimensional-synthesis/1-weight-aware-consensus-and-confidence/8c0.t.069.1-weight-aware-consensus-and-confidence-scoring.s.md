# Extend Provider-Side Thinking Controls in ace-llm

## Goal

Expand and normalize provider-side thinking control in `ace-llm` so explicit effort selection remains a provider concern, while `ace-review` stays focused on preset intent and prompt composition.

## Current State

The current implementation already supports explicit thinking suffixes on fully qualified targets and preserves `@ro`, `@rw`, and `@yolo`, but it is still an early slice:

- provider-specific thinking overlays exist only for a subset of providers
- the failure/compatibility story is stronger than before, but not yet broadly documented as a stable contract
- review presets can use these targets, but future provider coverage is still uneven
- the architecture lesson is clear: runtime/effort knobs should live in `ace-llm`, not in review preset orchestration

## Behavioral Spec

### Input -> Process -> Output

**Input**: A fully qualified LLM target such as `provider:model:thinking@preset`.

**Process**: `ace-llm` resolves the provider/model/preset as before, then applies provider-specific thinking overlays as an additional runtime layer without changing the review-side preset architecture.

**Output**: Explicit thinking control is predictable, provider-owned, and extensible across providers.

### Interface Contract

Required contract:

- preserve `@ro`, `@rw`, and `@yolo`
- preserve existing provider/model resolution semantics when no explicit thinking suffix is provided
- support explicit thinking suffixes on fully qualified targets
- keep provider-specific mapping data in `ace-llm`
- fail clearly when a requested provider/thinking combination is unsupported
- let higher-level callers such as `ace-review` consume the feature without owning the mapping rules

### Defaults

- provider-owned thinking control is the default architecture
- review presets may reference thinking-aware model targets, but should not define provider runtime logic themselves
- partial provider coverage is acceptable during rollout if unsupported combinations fail clearly

## Success Criteria

- [ ] The future contract keeps effort/thinking behavior inside `ace-llm`
- [ ] Provider coverage can expand without changing `ace-review` preset architecture
- [ ] Unsupported provider/thinking combinations fail explicitly
- [ ] Existing `@ro`, `@rw`, and `@yolo` behavior remains stable
- [ ] Documentation and future specs describe thinking control as provider-owned

## Verification Plan

- Review parser/loader/query responsibilities against this contract
- Confirm no review-side preset design requires provider-specific runtime logic
- Confirm future provider expansion can happen in `ace-llm` alone

## Negative Check

- Do NOT move thinking controls into `ace-review` config
- Do NOT require alias-only or provider-only shorthand to carry the same explicit guarantee as fully qualified targets unless deliberately specified later
