# Connect Reviewer Pipeline to Execution Chain

## Goal

Wire Reviewer objects through the full execution pipeline so that downstream components (MultiModelExecutor, FeedbackSynthesizer) have access to reviewer metadata (weight, critical, focus). This is pure plumbing — no behavioral change to output yet.

## Current State

- `Reviewer.from_preset_config(config)` exists and correctly parses reviewer YAML into Reviewer objects with weight/critical/focus attributes
- `ReviewManager` calls `from_preset_config` but only extracts `model` strings for the `models:` array passed to MultiModelExecutor
- `MultiModelExecutor.execute(models:, ...)` accepts a flat `models:` array of strings — Reviewer objects are discarded
- `FeedbackSynthesizer.synthesize(report_paths:, ...)` receives report file paths with no reviewer metadata attached

## Behavioral Spec

### Input → Process → Output

**Input**: A multi-reviewer preset config containing `reviewers:` array with weight/critical/focus metadata.

**Process**: ReviewManager creates Reviewer objects and threads them through MultiModelExecutor and FeedbackSynthesizer, maintaining the association between each reviewer and its execution result.

**Output**: Each stage of the pipeline can access the originating Reviewer object for any given result. No change to final user-visible output.

### Interface Contract

**MultiModelExecutor** accepts Reviewer objects alongside (or instead of) model strings:

- New parameter: `reviewers:` (array of Reviewer objects) — optional, coexists with `models:` for backward compatibility
- When `reviewers:` is provided, executor uses `reviewer.model` for LLM dispatch and tags each result with its Reviewer
- Result structure gains `reviewer:` key per model result (nil when using legacy `models:` path)

**FeedbackSynthesizer** receives reviewer metadata with report paths:

- New parameter shape: `report_paths:` may be paired with reviewer info (e.g., array of `{path:, reviewer:}` hashes)
- Backward compatible: plain string paths still work (reviewer metadata is nil)
- Reviewer metadata is available when building FeedbackItems but not yet used (069.b and 069.c consume it)

**ReviewManager** threads Reviewers through the chain:

- Creates Reviewer objects via `from_preset_config`
- Passes them to MultiModelExecutor alongside model strings
- Forwards reviewer-tagged results to FeedbackSynthesizer

**ReviewOptions** gains reviewer awareness:

- Can carry `reviewers:` from preset config resolution
- Exposes reviewers to ReviewManager during execution setup

### Defaults

- When no `reviewers:` config exists (legacy presets), behavior is identical to current code
- Single-reviewer presets work unchanged
- Reviewer weight defaults to 1.0, critical to false, focus to nil (existing Reviewer defaults)

## Success Criteria
