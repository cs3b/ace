# Formalize the Explicit Preset Contract and Authoring Model

## Goal

Define the single-file preset contract as the primary way to author review behavior in `ace-review`, so new review modes can be added without recreating hidden reviewer, pipeline, or partition layers.

## Current State

The simplified direction already proved that explicit preset files are the easiest review-definition unit, but the contract is still implicit:

- package defaults and repo-level `.ace/review` overrides can drift apart
- authoring expectations are spread across config, code, and examples
- validation rules for what a preset must contain are not the main documented product surface
- future preset growth could accidentally reintroduce indirection unless the contract is written down clearly

## Behavioral Spec

### Input -> Process -> Output

**Input**: A preset file that defines a review mode in one place.

**Process**: `ace-review` resolves the preset from package defaults or `.ace/review`, validates the expected shape, and executes the resulting prompt/model configuration directly.

**Output**: A preset is understandable end-to-end from a single file, with cascade behavior documented and validation failures surfaced clearly.

### Interface Contract

The primary preset contract should be documented and validated around these top-level concerns:

- `description`
- `instructions`
- `bundle`
- `subject`
- `model` or `models`

Required behavior:

- the preset file is the main authoring unit
- package defaults and repo-level overrides must use the same contract
- model targets may include `@ro`, `@rw`, and `@yolo`
- authoring guidance must prefer readable duplication over hidden composition
- presets without optional advanced features still remain fully valid

### Defaults

- one-file explicit presets are the default architecture
- config cascade remains supported, but contract drift is treated as a product risk
- hidden lane/pipeline composition is not assumed in the baseline contract

## Success Criteria

- [ ] The preset contract is documented as the primary review-definition surface
- [ ] Validation expectations are clear for both package defaults and `.ace/review` overrides
- [ ] New presets can be authored without adding reviewer or pipeline layers
- [ ] Model target guidance explicitly supports `@ro`, `@rw`, and `@yolo`
- [ ] Contract guidance prefers explicit self-contained presets over indirection

## Verification Plan

- Review the parent task and this subtask for a single consistent preset contract
- Confirm the documented contract covers both package defaults and repo-level `.ace/review`
- Confirm future preset growth can be explained without introducing hidden composition layers

- Do NOT make lane-mode behavior part of the baseline preset contract (that belongs in 069.2)
