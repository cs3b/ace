# Critical Finding Surfacing and Focus-Area Grouping

## Goal

Surface findings from `critical: true` reviewers prominently and group feedback by `focus` area when metadata is available, making multi-dimensional review output easier to scan and act on.

## Current State

- Reviewer model has `critical` (boolean) and `focus` (string) attributes — neither is used downstream
- FeedbackItem has no `critical_source` or `focus` fields
- Synthesized feedback is listed in arbitrary order with no grouping
- All findings from all reviewers are presented identically regardless of source criticality

## Behavioral Spec

### Input → Process → Output

**Input**: FeedbackItems with reviewer associations carrying critical/focus metadata (available after 069.0).

**Process**:
1. Tag FeedbackItems that originate (wholly or partly) from `critical: true` reviewers
2. Sort output so critical-source findings appear first
3. When `focus` metadata is available, group remaining findings by focus area

**Output**: Feedback listing where critical findings are surfaced at the top, and remaining items are grouped by focus area with section headers.

### Interface Contract

**FeedbackItem** gains two fields:

- `critical_source` — Boolean, true when any contributing reviewer has `critical: true`. Default: false.
- `focus` — String (nilable), the focus area of the primary contributing reviewer. Default: nil.
- Both serialized in `to_h` when present
- Immutable after creation (consistent with existing pattern)

**FeedbackSynthesizer** output ordering:

- When reviewer metadata is available:
  1. Critical-source items first, ordered by priority (critical > high > medium > low)
  2. Remaining items grouped by `focus` value (nil-focus items last, ungrouped)
- When no reviewer metadata: order unchanged (backward compatible)

**Focus-area grouping** in feedback listing:

- Items sharing the same `focus` value are grouped together
- Each group is preceded by a focus label (e.g., the focus string itself)
- Items with nil focus are listed after all grouped items
- Grouping is advisory — it affects presentation order, not the FeedbackItem data model

### Defaults

- `critical_source` defaults to false (no special treatment unless reviewer is critical)
- `focus` defaults to nil (no grouping, listed after grouped items)
- When no reviewer metadata is present, output order and format are unchanged

## Success Criteria

- [ ] FeedbackItem has `critical_source` attribute (boolean, default false)
- [ ] FeedbackItem has `focus` attribute (string, nilable, default nil)
- [ ] Findings from `critical: true` reviewers are tagged with `critical_source: true`
- [ ] Critical-source findings appear first in synthesized output
- [ ] Findings are grouped by `focus` area when metadata is available
- [ ] Nil-focus findings appear after grouped findings
- [ ] No change to output when reviewer metadata is absent (backward compatible)
- [ ] All existing tests pass

## Verification Plan

### Critical Surfacing Check
- Test: 3 findings — one from critical reviewer, two from non-critical → critical finding listed first
- Test: Multiple critical findings → ordered by priority within critical group
- Test: No critical reviewers → order unchanged

### Focus Grouping Check
- Test: Findings with focus "security", "quality", nil → grouped as [security items], [quality items], [nil items]
- Test: All findings share same focus → single group, no visual difference
- Test: No focus metadata → order unchanged from current behavior

### Backward Compatibility Check
- Test: FeedbackItem created without critical_source/focus → valid, defaults applied
- Test: Legacy preset (no reviewer metadata) → output identical to current
- Run full test suite — zero regressions

### Negative Check
- Do NOT change consensus logic (that's 069.b)
- Do NOT modify confidence scoring (that's 069.b)
