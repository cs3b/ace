# Add Optional Narrow Reviewer Lanes as a Secondary Extension

## Goal

Define a future opt-in lane mode for `ace-review` that can sharpen broad presets when needed, without replacing the explicit preset model as the default architecture.

## Current State

The original weighted-synthesis task treated reviewer metadata and reviewer pipelines as the primary path. The work in `ace-improve-review` showed the opposite should be true:

- broad explicit presets are the baseline path
- simple model-output synthesis is the baseline execution model
- hidden reviewer decomposition adds real complexity costs
- narrower internal lanes may still be useful later for special cases, but only if they stay explicit, optional, and subordinate to the preset file

## Behavioral Spec

### Input -> Process -> Output

**Input**: A normal explicit preset file, optionally extended with a lane declaration in the same file.

**Process**: `ace-review` executes the baseline preset as usual when no lanes are declared. When lanes are declared, the preset fans out into a small set of narrower prompts or model groups, then merges their outputs back into the standard synthesis/report flow.

**Output**: Lane mode sharpens certain review modes when explicitly requested, while presets without lanes remain simple and unchanged.

### Interface Contract

If lane mode is added, it must follow these rules:

- lanes are declared directly in the preset file, not in a separate hidden pipeline catalog
- a preset without lanes remains fully valid and unchanged
- lane execution is opt-in per preset
- synthesis still produces the standard review output shape
- lane metadata is advisory and local to that preset definition
- lane mode must not require repo users to understand a second architecture just to author normal presets

A future lane declaration should be small and explicit, for example a `lanes:` section inside the preset file, not a separate required reviewer registry.

### Defaults

- no lanes by default
- baseline model synthesis remains the default
- any future lane metadata is subordinate to the preset, never the primary product surface

## Success Criteria

- [ ] The future lane-mode direction is explicitly secondary to the preset-centric baseline
- [ ] A preset without lanes remains the canonical default path
- [ ] Any future lane declaration is colocated with the preset definition
- [ ] Lane-mode synthesis still lands in the standard review output flow
- [ ] The design does not reintroduce mandatory reviewer/pipeline/partition architecture

## Verification Plan

- Check that this subtask can be explained without changing the parent task's baseline architecture
- Confirm that lane-mode semantics are opt-in, local, and preset-owned
- Confirm the design leaves normal preset authoring simple

## Negative Check

- Do NOT restore weighted reviewer consensus as the default execution model
- Do NOT require separate reviewer catalogs, partition builders, or tool-review integrations
