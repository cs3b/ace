## Review Questions (Pending Human Input)

### [HIGH] Critical Implementation Questions

- [ ] **Conflict with existing `skill` semantics**: The Phase model already has a `skill` attribute used for sub-phase enrichment (via `SkillAssignSourceResolver`), not for direct execution. The spec proposes `skill: ace-review-run` for direct workflow execution in `fork_mode: workflow`. This creates a semantic conflict: does `skill` mean "decompose into sub-phases" or "execute this workflow directly"?
  - **Research conducted**: `assignment_executor.rb` `enrich_skill_declared_sub_phases` resolves skill metadata to generate child phases. The `skill` key is consumed and replaced with `source_skill` when sub-phases are generated.
  - **Suggested default**: `fork_mode: workflow` could skip skill enrichment and use skill reference for direct execution
  - **Why needs human input**: Fundamental semantic decision about when `skill` means "decompose" vs "execute directly"

- [ ] **How does ForkSessionLauncher invoke a skill (not `/ace-assign-drive`)?**: The spec says child agent "executes the referenced skill end-to-end" but doesn't specify the mechanism. Currently `ForkSessionLauncher` always sends `/ace-assign-drive` as the prompt. For `fork_mode: workflow`, should it send `/<skill-name>` instead?
  - **Research conducted**: `ForkSessionLauncher.launch` hardcodes `"/ace-assign-drive #{scoped_assignment}"` as the query prompt
  - **Suggested default**: Send `/<skill-name> <args>` as the query prompt for workflow forks
  - **Why needs human input**: Invocation protocol affects child agent behavior and skill argument passing

- [ ] **Who writes the `fork_result` YAML?**: The spec shows a structured `fork_result:` block in the phase report. Current fork results are written by child's `ace-assign-drive` completion. A standalone skill (e.g., `ace-review-run`) has no knowledge of phase files.
  - **Research conducted**: Child agents write reports via `ace-assign-drive` → `finish_phase`. Standalone skills don't know about phase files.
  - **Suggested default**: The launcher wraps the skill execution and writes the structured result
  - **Why needs human input**: Determines whether skills need to be fork-aware or infrastructure handles it transparently

- [ ] **Driver workflow changes explicitly out-of-scope but required**: The spec says driver skill changes are out of scope, yet the driver must detect `fork_mode: workflow` phases and handle their results differently. How does the driver display the summary format?
  - **Research conducted**: Drive workflow is separate from library code
  - **Suggested default**: ForkRun command handles summary formatting (library side), transparent to driver
  - **Why needs human input**: Scope boundary decision — may need to be in-scope or a follow-up task

# Enhanced FORK Delegation for Complete Workflow Execution

## Behavioral Specification

### User Experience
- **Input**: Assignment authors mark phases with `context: fork` and a new `fork_mode: workflow` attribute (default: `fork_mode: prompt` for current behavior). The child task receives a complete workflow skill reference to execute autonomously.
- **Process**: When the driver encounters a `fork_mode: workflow` phase, it launches a child agent that executes the referenced skill end-to-end (loads context, runs workflow, produces output). The driver waits for completion and receives only the consolidated result summary.
- **Output**: Driver task sees a clean result report from the forked workflow. No intermediate noise (context loading, step-by-step progress) leaks into the driver's conversation.

### Expected Behavior

Today's FORK mechanism (`context: fork`) launches a child agent with a prompt (via `/ace-assign-drive`). The child executes within the assignment's phase structure, reporting back through the phase file.

Enhanced FORK adds `fork_mode: workflow` which changes the delegation pattern:
1. Child agent receives a complete skill/workflow reference (e.g., `skill: ace-review-run`)
2. Child agent executes the full workflow independently — loads its own context, runs all steps
3. Child agent writes results to a designated output location (e.g., session file, report file)
4. Driver receives a structured result summary: status, output path, key findings count
5. Driver's conversation stays clean — only the summary appears, not the child's full execution trace

The existing `fork_mode: prompt` (current default) behavior is unchanged.

### Interface Contract

```yaml
# Phase file with enhanced FORK
# 020-run-review.ph.md
---
name: run-review
status: pending
context: fork
fork_mode: workflow           # NEW: delegates complete workflow
skill: ace-review-run         # skill to execute
fork_result_format: summary   # "summary" (default) | "full"
---

# Instructions for the forked agent
Run code review on the current PR using the code-deep preset.
```

```yaml
# Result written by child to phase report
---
fork_result:
  status: done
  output_path: .ace-local/review/sessions/2026-03-06-abc123/
  summary: "3 critical, 5 medium findings across 12 files"
  duration_seconds: 45
---
```

```bash
# Driver sees in its conversation (not the full review trace):
# Phase 020 (fork:workflow) completed:
#   Status: done
#   Output: .ace-local/review/sessions/2026-03-06-abc123/
#   Summary: 3 critical, 5 medium findings across 12 files
```

**Error Handling:**
- Forked workflow fails: Phase marked as `failed` with error message, driver continues to next phase or stops per assignment config
- Forked workflow times out: Phase marked as `failed` with timeout error, last-message captured
- Skill not found: Phase fails immediately with clear "skill not found" error
- Fork result file missing/corrupt: Driver logs warning, treats as failed with "no result captured"

**Edge Cases:**
- `fork_mode: workflow` without `skill:` key: Falls back to `fork_mode: prompt` behavior with warning
- `fork_mode: prompt` (existing behavior): Unchanged — child runs within assignment phase structure
- Nested forks (workflow fork launches another fork): Supported — each fork is independent
- `fork_result_format: full`: Driver receives complete output instead of summary (opt-in verbose mode)

## Objective

Improve ace-assign's FORK mechanism to support delegating complete workflows to child tasks that execute autonomously and report only results back. This reduces driver task noise and enables efficient parallel processing of multi-step operations like code reviews, test runs, and documentation generation.

## Scope of Work

- **User Experience Scope**: Assignment authors can mark phases for complete workflow delegation; drivers see clean result summaries
- **System Behavior Scope**: Phase model gains `fork_mode` attribute; ForkSessionLauncher handles workflow delegation; result capture and summary formatting
- **Interface Scope**: Phase frontmatter gains `fork_mode` and `fork_result_format` keys; fork result structured format
