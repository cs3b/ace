# Replace hardcoded `providers.cli_args` with ace-llm `@preset` suffixes

## Behavioral Specification

### User Experience
- **Input**: No user action required -- this is an internal cleanup. Consumers of ace-assign, ace-task, ace-retro, ace-idea, and ace-test-runner-e2e continue using the same CLI commands.
- **Process**: Developer removes per-package `providers.cli_args` / `doctor.cli_args` / `plan.cli_args` config sections, adds `@yolo`/`@ro` preset suffixes to model strings in config, and removes Ruby code that reads/merges config-based cli_args.
- **Output**: All packages use ace-llm's native `@preset` system for permission flags. No more duplicated, error-prone provider-specific CLI arg mappings.

### Expected Behavior

When a package forks a CLI agent (e.g., ace-assign launching Claude Code), the permission flags (like `--dangerously-skip-permissions`) are resolved by ace-llm's preset system from the `@yolo` or `@ro` suffix on the model string, instead of being looked up from a per-package `providers.cli_args` config section.

The `@preset` suffix on model strings (e.g., `claude:opus@yolo`) tells ace-llm to load the corresponding preset YAML from `.ace-defaults/llm/presets/{provider}/{preset}.yml`, which contains the correct CLI flags for that provider. This eliminates the need for each consumer package to maintain its own mapping of provider-to-flags.

Doctor commands in ace-task, ace-retro, and ace-idea pass the model string with `@yolo` suffix directly, removing the `provider_cli_args()` and `normalize_provider_cli_args()` methods that currently hardcode legacy aliases.

Plan commands in ace-task pass the model string with `@ro` suffix, removing the per-provider `--permission-mode plan` / `--sandbox read-only` / `--approval-mode plan` mapping.

### Interface Contract

No external interface changes. All existing CLI commands continue to work identically. The change is internal: how permission flags are resolved.

**Config format change (internal):**

Before:
```yaml
execution:
  provider: "claude:opus"
providers:
  cli_args:
    claude: "dangerously-skip-permissions"
    codex: "--sandbox danger-full-access"
```

After:
```yaml
execution:
  provider: "claude:opus@yolo"
# No providers.cli_args section needed
```

**Error Handling:**
- Invalid preset suffix (e.g., `@nonexistent`): ace-llm already handles this with a clear error message
- Missing preset YAML file: ace-llm raises descriptive error at query time

**Edge Cases:**
- Caller passing explicit `cli_args` AND model has `@preset` suffix: explicit cli_args override preset cli_args (existing `first_non_nil` behavior in `QueryInterface`). No current caller does this after cleanup, but the behavior is safe.
- `.ace/` project overrides that add custom `providers.cli_args`: These would become dead config. Not harmful, just ignored.

## Objective

Multiple packages duplicate provider-specific CLI permission flags in their own `providers.cli_args` config sections. ace-llm already has a native preset system (`@yolo`, `@ro`, `@rw`) that maps these flags per-provider. The duplication is error-prone, hard to maintain, and caused a crash (provider not in active list + hardcoded args mismatch). This task consolidates all permission flag resolution into ace-llm's `@preset` system.

## Scope of Work

- **User Experience Scope**: No user-facing changes. All CLI commands work identically.
- **System Behavior Scope**: Internal change to how CLI permission flags are resolved for forked agent sessions and doctor/plan commands.
- **Interface Scope**: Config file format changes (model strings gain `@preset` suffix, `cli_args` sections removed). No CLI or API changes.
