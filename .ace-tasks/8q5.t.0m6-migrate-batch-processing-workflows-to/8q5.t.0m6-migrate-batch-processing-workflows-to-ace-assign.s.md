# Unify ace-assign creation and generic batch fan-out

## Behavioral Specification

### User Experience
- **Input**: Users create assignments through `/as-assign-create` using presets, task refs, explicit step requests, or natural-language intent, and use `/as-assign-run-in-batches` for generic per-item fan-out instead of bespoke batch workflows.
- **Process**: The create workflow interprets intent, resolves phases from catalog data and skill metadata, persists a hidden normalized assignment spec under `.ace-local/assign/jobs/`, creates the assignment through `ace-assign create`, and optionally continues into `/as-assign-drive`.
- **Output**: Users get one consistent create-and-drive experience, hidden provenance for generated specs, and one reusable batch interface that replaces tool-specific batch workflow wrappers.

### Expected Behavior

This redesign replaces the current split between public `prepare`, `start`, and file-driven `create` with one public smart-create surface plus a generic batch wrapper:

1. **Public creation is unified** under `/as-assign-create`; `/as-assign-prepare` and `/as-assign-start` are removed from the public skill surface.
2. **Generated job specs remain real YAML** but are internal-only provenance artifacts written under `.ace-local/assign/jobs/`.
3. **Intent resolution is advisory, not rigid**: explicit user step requests win, while the catalog and recipe rules provide ordering, prerequisite, and default guidance.
4. **Skill-backed phases can expand from workflow metadata** when `assign.source` exists, so high-level phases such as task work still materialize into concrete sub-phases.
5. **Generic batch fan-out becomes reusable** through `/as-assign-run-in-batches`, and custom batch workflow wrappers such as E2E batch orchestration are retired once migrated.

This parent task is an orchestrator. It does not describe one implementation slice; it coordinates four vertical slices:
- `8q5.t.0m6.0` validates the hidden-spec architecture end to end
- `8q5.t.0m6.1` promotes create/drive as the only public creation flow
- `8q5.t.0m6.3` teaches smart create to resolve explicit intent into phases
- `8q5.t.0m6.2` introduces the generic fan-out workflow and migrates E2E batch usage

### Interface Contract

```bash
# Smart create from a known preset/task flow
/as-assign-create work-on-task --taskref 123

# Smart create from explicit requested steps
/as-assign-create "run tests, reorganize commits, push to remote"

# Smart create and immediately continue into drive
/as-assign-create "work on task 123 and create a PR" --run

# Generic fan-out for repeated work
/as-assign-run-in-batches "Run E2E scenario {{item}}" --items TS-001,TS-002 --run
```

**Error Handling:**
- Unresolvable explicit step: create returns a clear error listing the unmatched phrase and the closest known catalog/skill candidates.
- Invalid batch request with no items: batch workflow refuses creation with an actionable `--items` requirement message.
- `--run` with a successfully created assignment but no workable phase: create reports creation success and the empty-queue reason instead of silently failing.

**Edge Cases:**
- File-based CLI usage remains supported through `ace-assign create FILE`; smart parsing is not added to the CLI itself.
- Explicit user ordering wins unless it violates a hard sequencing rule such as review-before-apply or reorganize-before-push.
- Batch child phases fork by default, but `--sequential` disables fan-out for ordered or stateful work.
- If a batch instruction omits `{{item}}`, the generated child instruction prepends a deterministic `Target item:` line instead of requiring placeholder syntax.

## Objective

Replace the current fragmented ace-assign creation story with one coherent user path and one generic repeated-item orchestration primitive. The result should let users ask for assignments in the language of intent or explicit steps, keep the normalized spec as hidden provenance instead of a user-managed middle file, and remove tool-specific batch wrappers once the generic fan-out path exists.

## Scope of Work

- **User Experience Scope**: Users create assignments via `/as-assign-create`, optionally continue straight into `/as-assign-drive`, and use `/as-assign-run-in-batches` for repeated per-item work.
- **System Behavior Scope**: Smart create persists hidden normalized specs, resolves intent through catalog and skill metadata, and routes generic fan-out through reusable assign phases.
- **Interface Scope**: Public skill surface (`create`, `drive`, `run-in-batches`), batch workflow replacement, and hidden-spec provenance under `.ace-local/assign/jobs/`.
