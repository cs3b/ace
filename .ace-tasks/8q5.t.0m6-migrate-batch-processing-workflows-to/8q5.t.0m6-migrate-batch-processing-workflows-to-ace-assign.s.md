# Migrate Batch Processing Workflows to ace-assign

## Behavioral Specification

### User Experience
- **Input**: Users who previously used batch workflow files (e.g., `e2e/run-batch.wf.md`) now use ace-assign presets or job.yaml configs to achieve the same batch operations.
- **Process**: Each batch workflow's capabilities are expressed as an ace-assign preset with phases representing individual batch items. Phases can optionally use `context: fork` for parallel execution (leveraging enhanced FORK from task 0ki).
- **Output**: Same batch results as before — multiple items processed in sequence or parallel — but driven through ace-assign's standardized assignment/phase model instead of ad-hoc workflow files.

### Expected Behavior

Batch workflows today are ad-hoc workflow instruction files that loop over a set of items (tasks, tests, files) and execute an action on each. This migration:

1. **Identifies** all batch processing workflows across the codebase
2. **Creates** equivalent ace-assign presets or job templates for each
3. **Maps** each batch item to an ace-assign phase (sequential by default, `context: fork` for parallel)
4. **Updates** consumers (skills, agents) to invoke ace-assign instead of batch workflows
5. **Removes** the migrated batch workflow files

Key batch workflows to migrate:
- `e2e/run-batch.wf.md` — runs multiple E2E tests (maps to phases with fork)
- Any other discovered batch patterns in workflow-instructions directories

Each migrated workflow becomes an ace-assign preset in the appropriate gem's catalog, with phases that reference the single-item skill (e.g., `ace-e2e-run` for each test scenario).

### Interface Contract

```yaml
# Before (batch workflow): /ace-e2e-run-batch with ad-hoc looping
# After (ace-assign preset): ace-assign job.yaml

# Example job.yaml for batch E2E tests
assignment:
  name: batch-e2e-tests
  description: Run multiple E2E test scenarios

phases:
  - name: run-scenario-1
    skill: ace-e2e-run
    context: fork
    instructions: |
      Run E2E scenario TS-001
  - name: run-scenario-2
    skill: ace-e2e-run
    context: fork
    instructions: |
      Run E2E scenario TS-002
```

```bash
# Before
# /ace-e2e-run-batch (agent executes batch workflow)

# After
ace-assign create job.yaml
# => Assignment created: batch-e2e-tests (3 phases, 3 forkable)
```

**Error Handling:**
- Batch workflow has no ace-assign equivalent: Document as a gap, create TODO for manual migration
- Consumer references removed workflow: Broken reference detected during migration, fixed immediately
- Partial migration (some consumers not updated): Not acceptable — migration is atomic per workflow

**Edge Cases:**
- Batch workflow with dynamic item discovery (e.g., "find all pending tasks"): Requires a "prepare" phase that generates the phase list dynamically before execution
- Batch workflow with inter-item dependencies: Phases execute sequentially (no fork) with dependency ordering
- Batch workflow used by zero consumers: Still remove, but note in migration log

## Objective

Remove batch processing workflows from individual gems and centralize batch operations in ace-assign. This simplifies the workflow system by using ace-assign's phase model and FORK capabilities as the single mechanism for "run N things in sequence or parallel." The immediate target is `e2e/run-batch.wf.md` and any similar batch patterns discovered during inventory.

## Scope of Work

- **User Experience Scope**: Users transition from batch workflow invocations to ace-assign preset/job invocations for multi-item operations
- **System Behavior Scope**: Batch workflow files migrated to ace-assign presets/jobs, consumers updated, original files removed
- **Interface Scope**: ace-assign job.yaml templates for batch operations, updated skill references

### Deliverables

#### Behavioral Specifications
- Inventory of all batch processing workflows
- Migration mapping (old workflow -> new ace-assign equivalent)
- Updated consumer references

#### Validation Artifacts
- Equivalence tests comparing old and new batch execution results
- Clean grep for removed workflow references

## Out of Scope

- Implementation Details: File structures, code organization, technical architecture
- Technology Decisions: Tool selections, library choices, framework decisions
- Changes to ace-assign core (covered by task 0ki for FORK enhancements)
- Non-batch workflow migrations
- Modular preset redesign (see task 8q3zd9)
