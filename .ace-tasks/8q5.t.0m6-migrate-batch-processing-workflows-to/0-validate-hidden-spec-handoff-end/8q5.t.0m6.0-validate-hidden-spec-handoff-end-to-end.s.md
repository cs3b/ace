# Validate hidden spec handoff end to end

## Behavioral Specification

### User Experience
- **Input**: A user invokes `/as-assign-create` with one proven smart-create path, such as `work-on-task --taskref 123` or an equivalent simple intent phrase.
- **Process**: The create workflow generates a normalized assignment spec under `.ace-local/assign/jobs/`, calls `ace-assign create` with that hidden spec, and shows the same creation summary users already expect from the CLI path.
- **Output**: Users see a created assignment and first phase instructions, while the runtime retains a persistent hidden spec path for provenance.

### Expected Behavior

This tracer validates the new architecture before broader cleanup:

1. Smart create can choose one supported path and render a real YAML spec without exposing it as the user's primary artifact.
2. The rendered spec lives under `.ace-local/assign/jobs/` and is retained after creation.
3. `ace-assign create FILE` remains the deterministic runtime boundary and does not need natural-language parsing.
4. Assignment metadata references the retained spec location so later status/debug flows can trace the source configuration.
5. The tracer documents which concepts survive into the end-state and which legacy concepts should be removed in later slices.

This slice is intentionally narrow: it proves the hidden-spec handoff with one end-to-end path before expanding the public surface or generic fan-out behaviors.

### Interface Contract

```bash
/as-assign-create work-on-task --taskref 123

# Expected output:
# Assignment: work-on-task-123 (<id>)
# Created from hidden spec: .ace-local/assign/jobs/<timestamp>-work-on-task-123.yml
# Phase 010: ...
```

**Error Handling:**
- If smart create cannot render the hidden spec, no assignment is created and the user gets a concrete rendering failure.
- If the hidden spec is rendered but `ace-assign create` rejects it, the CLI error is surfaced unchanged.

**Edge Cases:**
- Re-running the same request produces a new hidden spec file rather than mutating the previous one.
- The hidden spec path is internal provenance; users are not required to edit or move it.

### Success Criteria

- [x] **Hidden spec path exists**: One smart-create flow writes a normalized spec under `.ace-local/assign/jobs/`.
- [x] **Runtime handoff works**: The created hidden spec is accepted by `ace-assign create` without requiring CLI changes.
- [x] **Provenance survives creation**: Assignment metadata continues to point at the retained spec path after creation completes.
- [x] **Tracer output is reusable**: This slice documents the retained concepts that later slices must preserve.

### Validation Questions

- [x] **Should the spec be persisted or in-memory only?** -> Persist it under `.ace-local/assign/jobs/`.
- [x] **Should the tracer validate every create path?** -> No; validate one end-to-end path first and broaden later.
