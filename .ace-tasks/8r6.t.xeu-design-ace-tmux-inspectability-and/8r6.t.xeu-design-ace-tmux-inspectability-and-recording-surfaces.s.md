# Design ACE tmux inspectability and recording surfaces

## Objective

Define the generic `ace-tmux` inspectability and recording contract so operators and higher-level ACE tools can query ACE-managed tmux runtime state, enable recording for ACE-managed panes, and inspect persisted evidence independently of any single consumer such as visible fork execution.

## Behavioral Specification

### User Experience

- A user or higher-level ACE tool can query `ace-tmux` for live ACE-managed tmux sessions, windows, and panes without shelling directly to raw `tmux` commands.
- Operators can inspect a stable human and JSON runtime view that identifies sessions, windows, panes, liveness hints, and any persisted recording artifacts for ACE-managed panes.
- Users can enable recording for ACE-managed tmux sessions, windows, or panes through preset fields, with lower scopes overriding higher ones.
- Users can also request recording through CLI convenience flags on ACE-managed tmux entrypoints instead of manually editing presets for one-off sessions.
- Recording remains opt-in, off by default, and scoped to ACE-managed panes rather than arbitrary foreign tmux surfaces.
- Generic inspectability and recording work independently of visible fork execution, though visible fork may consume these surfaces as a client.

### Expected Behavior

1. `ace-tmux state` exposes ACE-managed tmux runtime state for sessions, windows, and panes in reusable human and JSON forms.
2. Runtime state reports enough identifiers and liveness hints for operators and higher-level tools such as `ace-overseer` and `ace-assign` to reconnect work to tmux surfaces.
3. ACE-managed tmux sessions, windows, and panes can opt into recording through preset fields, with lower scopes overriding higher ones.
4. `ace-tmux start --record` and `ace-tmux window --record` act as convenience entrypoints into the same recording model rather than creating a separate recording mode.
5. Recording remains disabled by default and applies only to ACE-managed panes created through `ace-tmux`.
6. Recorded panes persist raw evidence and manifest metadata under `.ace-local/tmux/` for later human or workflow inspection.
7. Runtime state reports the effective recording state, the `source_scope` that enabled it, and any available artifact or manifest paths when recording is enabled.
8. The inspectability surface does not promise semantic interpretation, redaction, or automated learning from captured logs.
9. The inspectability surface is generic and does not depend on visible fork windows or `ace-assign`-specific behaviors.

### Interface Contract

- **`ace-tmux` CLI contract**
  ```bash
  ace-tmux state
  ace-tmux state --format json
  ace-tmux state --session dev
  ace-tmux start --record
  ace-tmux window --record
  ```
- **`ace-tmux state` output contract**
  - Reports ACE-managed tmux sessions, windows, and panes in human and JSON forms.
  - Reports pane/session/window identifiers and enough liveness metadata to reconnect ACE-managed surfaces to runtime state.
  - For recording, reports the effective enabled state, the `source_scope` that set it (`session`, `window`, `pane`, or CLI override), and any artifact path or manifest path when recording is enabled.
- **`ace-tmux` recording contract**
  - Recording is opt-in and off by default.
  - Recording config can be set at session, window, or pane scope in ACE-managed presets, with lower scopes overriding higher ones.
  - `ace-tmux start --record` and `ace-tmux window --record` are convenience flags that request recording through the same inheritance contract instead of defining a separate mode.
  - Recording applies only to ACE-managed panes launched through `ace-tmux`.
  - Recorded artifacts live under `.ace-local/tmux/`.
- **Consumer contract**
  - Other tools such as `ace-overseer` and `ace-assign` may consume this generic surface but must not redefine its schema in their own task specs.

### Success Criteria

- [ ] `ace-tmux` exposes reusable runtime state for ACE-managed sessions, windows, and panes in both human and JSON forms.
- [ ] Runtime state includes identifiers, liveness hints, effective recording status, `source_scope`, and artifact references when applicable.
- [ ] Operators can enable recording for ACE-managed panes through preset inheritance or CLI convenience flags.
- [ ] Recording remains optional, off by default, and ACE-managed only.
- [ ] Operators can later inspect raw recording evidence under `.ace-local/tmux/`.
- [ ] The contract is generic enough for `ace-overseer`, `ace-assign`, and direct operator use without depending on visible fork semantics.
- [ ] Draft usage guidance exists for the new `ace-tmux` command and configuration surface.
- [ ] The task does not promise semantic interpretation, redaction, or learning from captured logs.
