# Spike ACE tmux runtime inventory and recording contract

## Objective

Validate the end-state contract for generic ACE-managed tmux runtime inspectability and recording, including which concepts are required for reusable `ace-tmux state` output, how recording inheritance and provenance should work, and what artifact contract later consumers can rely on before additional implementation subtasks are drafted.

## Behavioral Specification

### User Experience

- A designer or implementer can answer, from one spike outcome, what `ace-tmux state` must report for sessions, windows, panes, liveness, provenance, and artifacts.
- A designer or implementer can answer, from one spike outcome, how recording inheritance and override scope work across ACE-managed session, window, pane, and CLI entrypoints.
- The spike produces a concept inventory that makes later implementation subtasks stable instead of speculative.
- The spike resolves whether existing `ace-tmux` launcher and model concepts survive intact, are extended, or should be superseded for a reusable inspectability surface.

### Expected Behavior

1. The spike validates one coherent end-state scenario for generic runtime inspection through `ace-tmux state`.
2. The spike validates one coherent end-state scenario for ACE-managed recording from session start through later inspection of pane artifacts after manual takeover work.
3. The spike explicitly identifies which existing concepts remain authoritative and which new concepts are required for reusable runtime state.
4. The spike explicitly identifies the minimum new concepts required, including runtime tmux state inventory, recording artifact scope, recording inheritance, and `source_scope` provenance reporting.
5. The spike produces a concept inventory table showing concept status as kept, changed, new, or rejected.
6. The spike defers further subtask drafting until the inventory makes later decomposition stable.
7. The spike confirms the inspectability surface is generic and not owned by visible-fork-specific behavior.

### Interface Contract

- **Spike output contract**
  - Updated parent task concept inventory and scope notes
  - Clear go/no-go statement for these behavioral contracts:
    - generic `ace-tmux state` runtime inventory
    - optional ACE-managed recording
    - recording override and inheritance behavior across session, window, pane, and CLI entrypoints
    - `.ace-local/tmux` artifact contract
    - generic session/window/pane identifiers and liveness hints
    - `source_scope` recording provenance
- **Decision contract**
  - The spike does not implement code.
  - The spike may refine task specs and usage draft content if the validated contract differs from the initial draft.
  - The spike treats visible-fork launch and fork-window behavior as out of scope and owned by sibling task `8r6.t.u53`.

### Success Criteria

- [ ] One validated end-state scenario exists for generic tmux runtime inspection.
- [ ] One validated end-state scenario exists for ACE-managed recording of manual takeover activity.
- [ ] The spike identifies which current inspectability concepts survive unchanged, which are extended, and which are newly introduced.
- [ ] The spike leaves no unresolved behavioral ambiguity for runtime inventory, recording scope, recording source-scope reporting, artifact layout, or consumer reuse boundaries.
- [ ] The parent task can be safely decomposed into implementation subtasks after the spike.
