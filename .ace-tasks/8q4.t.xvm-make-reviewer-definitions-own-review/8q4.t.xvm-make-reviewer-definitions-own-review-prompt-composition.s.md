# Make reviewer definitions own review prompt composition

## Behavioral Specification

### User Experience
- **Input**: Review authors define review intent in reviewer files, not in presets. A reviewer can own a rich prompt bundle with base prompt, focus sections, format requirements, and review guidelines.
- **Process**: `ace-review` resolves a preset, selects a pipeline, loads reviewer definitions, composes the final review prompt from reviewer-owned prompt sections, and executes the selected reviewers.
- **Output**: Presets become orchestration-only configuration. Reviewers become the canonical source of what each review lane checks and how it frames feedback.

### Expected Behavior

`ace-review` must transfer review prompt ownership from presets to reviewers.

Reviewer definitions must support structured prompt composition rather than only a single-line `focus` field plus optional freeform text. A reviewer must be able to define:
- a base prompt reference
- ordered prompt sections
- focus bundles for scope, languages, architecture, and quality concerns
- formatting requirements
- tone and communication guidelines

Presets must become thin orchestration wrappers. A review preset may define:
- subject
- context or bundle references
- pipeline selection
- minor runtime overrides that do not redefine review intent

A review preset must not define the actual review instructions. In particular, presets must not remain the source of truth for:
- `instructions`
- `prompt_composition`
- focus section bundles such as review scope, format guidance, or review guidelines

The runtime must be able to execute at least one canonical preset with zero preset-owned review instructions. Reviewer prompt ownership is complete only when removing review instructions from presets does not break review execution.

The phased presets `code-valid`, `code-fit`, and `code-shine` may remain as user-facing entry points, but they must delegate review intent to reviewer definitions rather than embedding prompt bundles themselves.

`pr-risk-based` must also follow this rule because it is the assignment-facing preset.

`pr-risk-based` must be shipped in `ace-review/.ace-defaults/review/presets/` as part of this migration. The current repo-local `.ace/review/presets/pr-risk-based.yml` may remain only as a project override if it differs intentionally from the shipped default.

The runtime contract is:
- reviewer YAML owns the semantic prompt composition tree
- preset YAML never owns review semantics
- `PromptComposer` or its replacement may internally normalize reviewer prompt sections into the existing `base/format/focus/guidelines` execution shape, but that internal normalization must be invisible at the preset layer

### Interface Contract

```yaml
# reviewer-owned prompt composition
.ace-defaults/review/reviewers/correctness.yml

provider: llm-review-fast
weight: 1.0
critical: true

prompt:
  base: "prompt://base/system"
  sections:
    review_focus:
      title: "Correctness Focus"
      description: "Logic, contracts, and missing behavior"
      files:
        - "prompt://focus/languages/ruby"
        - "prompt://focus/architecture/atom"
        - "prompt://focus/quality/correctness"
    format_guidelines:
      title: "Format Guidelines"
      description: "Structured findings output"
      files:
        - "prompt://format/detailed"
    review_guidelines:
      title: "Review Guidelines"
      description: "Tone and evidence expectations"
      files:
        - "prompt://guidelines/tone"
        - "prompt://guidelines/icons"
```

```yaml
# thin orchestration preset
.ace-defaults/review/presets/pr-risk-based.yml

description: "PR review with risk-based lane selection"
pipeline: narrow-risk-based
subject:
  type: diff
  ranges:
    - "origin/main...HEAD"
```

```bash
# expected usage remains simple
ace-review --preset pr-risk-based --auto-execute
ace-review --preset code-valid --auto-execute
```

**Error Handling:**
- If a selected reviewer has no prompt definition, review setup fails with an actionable error naming the reviewer.
- If a preset still references removed preset-owned instruction keys after migration, resolution fails with a migration-focused error rather than silently mixing the old and new paths.
- If reviewer prompt composition references missing prompt files, review setup fails with the missing prompt URI listed.

**Edge Cases:**
- Reviewers may share common prompt fragments, but the ownership must still remain with reviewers via references, not by reintroducing preset-owned composition.
- Compatibility wrappers such as `code-valid` may select different reviewers or pipelines, but they must not override reviewer-owned focus bundles.
- Runtime overrides may append non-semantic execution hints, but they must not replace reviewer prompt sections.
- If a repo-local preset shadows a shipped default, the repo-local preset may override subject or pipeline wiring, but it must still obey the no-preset-owned-instructions rule.

- [ ] Reviewer definitions support structured prompt composition with `base` plus named prompt sections.
- [ ] At least one shipped reviewer owns the full review-focus, formatting, and guideline prompt bundle needed for execution.

- [x] Runtime overrides: only non-semantic execution overrides are allowed above reviewer prompt composition.

## Objective

Close the architectural gap left by `8q4.t.ssx` by making reviewer definitions the only owner of review prompt composition and reducing presets to orchestration-only configuration.

## Scope of Work

- **User Experience Scope**: Review authors edit reviewer files to change review intent. Preset authors only choose subject, context, and pipeline.
- **System Behavior Scope**: Prompt composition is built from reviewer definitions, not preset instruction bundles.
- **Interface Scope**: Reviewer YAML schema, preset YAML schema, preset validation rules, and assignment review preset compatibility.
- **Consumer Packages**:
  - `ace-review`: reviewer schema, preset validation, prompt composition resolution
  - `ace-assign`: assignment presets that reference `pr-risk-based`

### Deliverables

#### Behavioral Specifications
- Reviewer prompt schema with `base` and named prompt sections
- Preset schema constraints that forbid review instructions in presets
- Runtime validation behavior for migrated and non-migrated configs

#### Validation Artifacts
- Usage contract showing thin preset plus rich reviewer example
- Tests proving execution works after removing preset-owned instructions

### Ownership Transfer

- **New source of truth**: reviewer definitions under `reviewers/*.yml`
- **Must not remain in old layer**:
  - preset `instructions`
  - preset `prompt_composition`
  - preset-owned focus bundles
  - preset-owned formatting or review guideline bundles
- **Allowed to remain in presets**:
  - subject
  - context references
  - pipeline selection
  - non-semantic runtime overrides that do not define review intent

## Defaults

- Canonical assignment-facing preset: `pr-risk-based`
- Compatibility wrappers retained: `code-valid`, `code-fit`, `code-shine`
- Backward compatibility mode for shipped presets: no silent fallback to preset-owned instructions after migration
- Shared prompt material is referenced from reviewers through prompt URIs, not copied into presets
- No CLI flag or operating-mode changes are part of this task; existing `ace-review` invocation modes continue unchanged

## Out of Scope

- ❌ Reworking lane-selection policy or reviewer catalog semantics from `8q4.t.ssx`
- ❌ Reintroducing broad preset-owned prompt bundles such as `code-deep`
- ❌ Adding skeptic/referee or new adjudication flows
- ❌ Redesigning `ace-llm` preset-qualified model targeting
