# Migrate batch 2 tested packages to the fast/feat/e2e testing model

## Objective

Apply the validated `ace-b36ts` pilot taxonomy to the remaining tested packages that were not covered by batch 1, keeping one explicit migration subtask per package.

## Behavioral Specification

### User Experience

- Every package assigned to batch 2 gets an explicit migration task.
- Batch 2 covers the lower-level support and handbook-integration packages omitted from batch 1 because of the one-character child-slot limit.
- Package tasks still describe `test/fast/`, `test/feat/`, and `test/e2e/` expectations explicitly instead of relying on implicit migration rules.

### Expected Behavior

1. Run subtask `8r9.t.i06.0` first to refresh all batch 2 child specs from the pilot outputs and current repo state.
2. Execute one batch 2 package migration subtask at a time.
3. Use explicit package tasks for all remaining tested packages.
4. Keep `ace-test <package>` defaulting to `fast` and `ace-test <package> feat` as the deterministic feature layer.
5. Keep `ace-test-e2e <package>` as the only scenario E2E entrypoint.
6. Hand completion to sibling cleanup task `8r9.t.i07` once all batch 2 package migrations are done.

### Interface Contract

- `ace-test <package>`
- `ace-test <package> feat`
- `ace-test <package> all`
- `ace-test-e2e <package>`

### Success Criteria

- [ ] Every package assigned to batch 2 has one explicit migration subtask.
- [ ] Batch 2 package tasks state whether they are fast-only, fast+feat, or include real `test/e2e` ownership.
- [ ] The refresh subtask runs before batch 2 migration implementation begins.
- [ ] No batch 2 child task teaches `ace-test <package> e2e`.
- [ ] Combined with `8r9.t.i05`, all non-pilot tested packages are covered exactly once.
