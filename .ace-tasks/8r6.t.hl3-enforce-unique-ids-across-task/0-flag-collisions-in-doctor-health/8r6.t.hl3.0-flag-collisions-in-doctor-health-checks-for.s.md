# Flag collisions in doctor health checks for tasks and ideas

## Objective

Ensure `ace-task doctor` and `ace-idea doctor` make duplicate persisted IDs visible as explicit health-check failures so maintainers can detect identifier corruption before continuing work.

## Behavioral Specification

### User Experience

- A user running `doctor` sees duplicate-ID problems called out directly instead of silently passing or treating them as ambiguous state.
- A user can run focused frontmatter checks and still receive duplicate-ID findings.
- A maintainer has enough file-path context in the output to locate each colliding item without manual repository forensics.

### Expected Behavior

1. `ace-task doctor` reports duplicate full task IDs as errors.
2. `ace-task doctor` duplicate-ID coverage includes both top-level tasks and subtasks.
3. `ace-idea doctor` reports duplicate idea IDs as errors.
4. Duplicate-ID findings appear in normal doctor runs and `--check frontmatter`.
5. A repository with no duplicate IDs keeps its current healthy-path experience.
6. Duplicate-ID findings are treated as integrity failures, not low-signal warnings.

### Interface Contract

- **CLI entrypoints**
  - `ace-task doctor`
  - `ace-task doctor --check frontmatter`
  - `ace-idea doctor`
  - `ace-idea doctor --check frontmatter`
- **Output contract**
  - Duplicate-ID findings identify the duplicate ID value.
  - Duplicate-ID findings include enough path context to locate every collision involved.
  - The command exits as unhealthy when duplicate-ID errors are present.
- **Out-of-contract behavior**
  - This task does not add a new `duplicates` check flag.
  - This task does not define auto-fix behavior for duplicates.

### Success Criteria

- [ ] Duplicate top-level task IDs fail `ace-task doctor`.
- [ ] Duplicate subtask IDs fail `ace-task doctor`.
- [ ] Duplicate idea IDs fail `ace-idea doctor`.
- [ ] `--check frontmatter` includes duplicate-ID detection for both packages.
- [ ] Healthy repositories retain their expected passing doctor experience.

## Validation Questions

- Path reporting should optimize for operator usefulness rather than strict formatting aesthetics; the important contract is that collisions are locatable.
- Duplicate-ID detection belongs to the existing doctor/frontmatter surface and should not require a separate mode.

## Vertical Slice Decomposition

- **Slice type**: standalone subtask
- **Observable outcome**: doctor users can detect duplicate IDs across task and idea data with no new command surface
- **Advisory size**: medium
- **Distinct scope**: health-check behavior only; no create-time collision handling

## Validation Plan

- **Primary checks**
  - Verify duplicate top-level task IDs are reported as errors.
  - Verify duplicate subtask IDs are reported as errors.
  - Verify duplicate idea IDs are reported as errors.
  - Verify `--check frontmatter` surfaces the same class of duplicate-ID failures.
- **Failure path**
  - Verify doctor does not silently pass when duplicates exist.
- **Success-criterion mapping**
  - Top-level task detection: create two colliding task fixtures and run `ace-task doctor`.
  - Subtask detection: create colliding subtask IDs and run `ace-task doctor`.
  - Idea detection: create two colliding idea fixtures and run `ace-idea doctor`.
  - Focused-check coverage: run `--check frontmatter` for both packages and confirm duplicate detection still triggers.
