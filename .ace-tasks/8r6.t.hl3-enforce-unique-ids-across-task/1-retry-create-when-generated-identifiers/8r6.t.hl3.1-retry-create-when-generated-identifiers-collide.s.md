# Retry create when generated identifiers collide

## Objective

Ensure `ace-task create` and `ace-idea create` never report a successful create result for a newly persisted item whose ID is not unique, while preserving the existing normal-path UX for non-collision cases.

## Behavioral Specification

### User Experience

- A user running create receives one successful result with a unique ID or a clear failure message.
- Users do not have to manually resolve transient create-time identifier collisions when the tool can safely retry.
- Normal create usage remains familiar when no collision occurs.

### Expected Behavior

1. `ace-task create` verifies the persisted task ID is unique before printing success.
2. `ace-idea create` verifies the persisted idea ID is unique before printing success.
3. When a create-time collision occurs, the command retries automatically with a fresh generated ID rather than treating the duplicate persisted result as success.
4. Success output is emitted only after a unique created item exists.
5. If uniqueness cannot be established after a bounded retry limit, the command fails with a clear error.
6. `--dry-run` behavior remains unchanged.
7. For `ace-idea create`, “same ID, different folder slug” is not an acceptable successful create outcome for this scope.

### Interface Contract

- **CLI entrypoints**
  - `ace-task create ...`
  - `ace-idea create ...`
- **Success contract**
  - Success output corresponds to one created item whose persisted ID is unique.
- **Failure contract**
  - Retry exhaustion returns a clear error rather than pretending creation succeeded.
- **Stability contract**
  - No new create flags are introduced by this task.
  - Existing non-collision create flows remain behaviorally unchanged from a user’s perspective.

### Success Criteria

- [ ] `ace-task create` retries when a generated ID collides.
- [ ] `ace-idea create` retries when a generated ID collides.
- [ ] Retry exhaustion fails clearly for both packages.
- [ ] `--dry-run` behavior remains unchanged.
- [ ] Standard non-collision create behavior remains unchanged for both packages.

## Validation Questions

- The draft intentionally leaves retry implementation details unspecified; only the observable contract matters here.
- Cleanup semantics after failed retry attempts should preserve the user-facing guarantee of no successful duplicate result.

## Vertical Slice Decomposition

- **Slice type**: standalone subtask
- **Observable outcome**: create commands either succeed with unique IDs or fail clearly after bounded retries
- **Advisory size**: medium
- **Distinct scope**: create-time behavior only; no doctor duplicate reporting

## Validation Plan

- **Primary checks**
  - Force a create-time ID collision for `ace-task create` and verify the final successful item has a different unique ID.
  - Force a create-time ID collision for `ace-idea create` and verify the final successful item has a different unique ID.
  - Verify standard non-collision create flows still succeed as expected.
  - Verify `--dry-run` output remains unchanged.
- **Failure path**
  - Verify retry exhaustion produces a clear failure for both packages.
- **Success-criterion mapping**
  - Task create retry behavior: run a forced collision scenario and confirm only a unique successful result remains.
  - Idea create retry behavior: run a forced collision scenario and confirm only a unique successful result remains.
  - Retry exhaustion: force repeated collisions and confirm clear failure behavior.
  - Dry-run stability: compare behavior before and after the change using representative dry-run invocations.
