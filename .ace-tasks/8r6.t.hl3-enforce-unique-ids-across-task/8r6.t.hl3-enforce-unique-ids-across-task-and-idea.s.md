# Enforce unique IDs across task and idea lifecycle commands

## Objective

Make ID uniqueness an explicit integrity contract for `ace-task` and `ace-idea` so users can trust doctor health checks and create flows not to silently tolerate duplicate persisted identifiers.

## Behavioral Specification

### User Experience

- A maintainer can run health checks and immediately learn whether duplicate IDs exist anywhere relevant to task or idea operations.
- A user creating a task or idea either gets one successfully created item with a unique ID or a clear failure, never a “success” that leaves duplicate persisted state.
- The work is clearly split into one doctor-focused slice and one create-focused slice so implementation can be planned and verified cleanly.

### Expected Behavior

1. The work is represented as one orchestrator task with two child tasks:
   - `8r6.t.hl3.0` covers duplicate-ID detection in doctor flows.
   - `8r6.t.hl3.1` covers uniqueness guarantees in create flows.
2. The parent is complete only when both child tasks are implemented, verified, and shown to preserve normal healthy-path behavior.
3. Duplicate IDs are treated as integrity failures rather than advisory warnings.
4. Create commands must not report success for a newly created item whose `id` is not unique after persistence.
5. The parent task defines the combined acceptance contract, while the children define the precise behavior and verification details for their individual slices.

### Interface Contract

- **Task decomposition contract**
  - Orchestrator: `8r6.t.hl3`
  - Child tasks:
    - `8r6.t.hl3.0` -> duplicate-ID detection in `doctor`
    - `8r6.t.hl3.1` -> uniqueness guarantees in `create`
- **Public surface contract**
  - `ace-task doctor`
  - `ace-idea doctor`
  - `ace-task create`
  - `ace-idea create`
- **Change boundary**
  - No new command names are introduced by this parent scope.
  - Child tasks may refine behavior of existing commands but must preserve the basic CLI entrypoints.

### Success Criteria

- [ ] Two child tasks are drafted and linked under `8r6.t.hl3`.
- [ ] Child `8r6.t.hl3.0` defines duplicate-ID detection behavior for both `ace-task` and `ace-idea`.
- [ ] Child `8r6.t.hl3.1` defines unique-create behavior for both `ace-task` and `ace-idea`.
- [ ] Each child task includes explicit bundle context, success criteria, and at least one failure-path verification scenario.
- [ ] The parent makes it unambiguous that implementation planning happens later and that this draft is behavior-first.

## Validation Questions

- Duplicate-ID auto-fix behavior is intentionally out of scope for this draft and should not be assumed by implementers.
- The existing CLI entrypoints stay in place; the intended change is behavior of current commands, not introduction of new interfaces.

## Vertical Slice Decomposition (Task/Subtask Model)

- **Slice type**: orchestrator with two independent end-to-end subtasks.
- **Outcome**: a single integrity-focused scope covering both duplicate detection and create-time uniqueness guarantees.
- **Advisory size**: medium
- **Context dependencies**: `ace-task` doctor/create flows, `ace-idea` doctor/create flows, existing task and idea test surfaces.
- **End-state coherence**
  - Child `8r6.t.hl3.0` makes duplicates visible and actionable through health checks.
  - Child `8r6.t.hl3.1` prevents new duplicate IDs from being reported as successful create results.
  - The parent introduces no extra runtime behavior beyond coordinating those two durable outcomes.
