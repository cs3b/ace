# Replace skill references with workflow instructions in ace-assign

## Behavioral Specification

### User Experience
- **Input**: Assignment presets and catalog phases currently reference skills by name (e.g., `skill: ace-github-pr-update`). Agents receive these references via `ace-assign status` output.
- **Process**: When a driver/fork agent reaches a phase, it reads the status output to determine what action to take. Currently the `Skill:` label is informational and easy to ignore. After the change, `Workflow:` and `Action:` labels provide a prescriptive directive, and the instructions body itself starts with the `ace-bundle wfi://` call.
- **Output**: Phase files contain `workflow: wfi://path` in frontmatter. Status output shows `Workflow: wfi://path` and `Action: read and run ace-bundle wfi://path`. Instructions start with the workflow load command. Agents can't miss the action.

### Expected Behavior

Assignment phases currently use `skill: <name>` frontmatter to reference a skill that wraps a workflow instruction. Skills are thin indirection: SKILL.md files just say `read and run ace-bundle wfi://...`. This indirection causes agents to ignore skills and follow prose instructions instead, producing wrong results (as observed with phase 150 in assignment 8q4toe).

After the change:
- Presets and catalog phases use `workflow: wfi://path` instead of `skill: name`
- Phase instructions start with `read and run ace-bundle wfi://path` as the first line
- `ace-assign status` shows `Workflow:` and `Action:` for the current phase
- Sub-phases for fork phases are explicit in presets (not derived from skill metadata)
- Existing completed assignments with `skill:` in phase files still render correctly (backward compat)

### Interface Contract

```bash
# Status output for a phase with workflow:
ace-assign status --assignment <id>
# Current Phase: 150 - update-pr-desc
# Workflow: wfi://github/pr/update
# Action: read and run `ace-bundle wfi://github/pr/update`
#
# Instructions:
# read and run `ace-bundle wfi://github/pr/update`
# Update PR description with final implementation summary.

# Status JSON output
ace-assign status --format json
# phase object includes: {"workflow": "wfi://github/pr/update"} instead of {"skill": "ace-github-pr-update"}

# Backward compat: old phase files with skill: still show
# Skill: ace-github-pr-update
```

**Phase file format (before):**
```yaml
---
name: update-pr-desc
status: pending
skill: ace-github-pr-update
---

Update PR description with final implementation summary.
```

**Phase file format (after):**
```yaml
---
name: update-pr-desc
status: pending
workflow: wfi://github/pr/update
---

read and run `ace-bundle wfi://github/pr/update`

Update PR description with final implementation summary.
```

**Error Handling:**
- Phase with unknown workflow path: agent runs `ace-bundle wfi://path` which fails with actionable error from ace-bundle
- Phase with both `skill:` and `workflow:`: `workflow:` takes precedence in status display

**Edge Cases:**
- `ace-onboard` skill has no wfi:// path (uses `ace-bundle project` preset): instructions say `run ace-bundle project` directly
- Fork phases with `workflow: wfi://task/work`: sub-phases are explicit in preset, not derived from workflow frontmatter

## Objective

Assignment phases reference skills as a thin indirection to workflow instructions. This indirection causes agents to ignore the skill and follow prose instructions instead, producing wrong results. Replacing `skill:` with `workflow:` and embedding the `ace-bundle wfi://` call directly in instructions makes the action explicit and impossible to miss.

Triggered by: assignment 8q4toe phase 150 (`update-pr-desc`) where the driver agent ignored `skill: ace-github-pr-update` and wrote a poor PR description by following prose instructions instead of invoking the workflow.

## Scope of Work

- **User Experience Scope**: Assignment creation and driving — agents see explicit workflow directives in phase status and instructions
- **System Behavior Scope**: Phase model, file parser, executor, status command, preset/catalog definitions, drive workflow
- **Interface Scope**: `ace-assign status` output format, phase file frontmatter format, preset YAML format
