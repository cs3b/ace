# Address missing ace-support-models and quick-start onboarding docs

## Objective

Reduce onboarding friction so fresh installations follow quick-start without manual dependency fixes for provider configuration, and so onboarding docs accurately describe the required first-run dependency state.

## Behavioral Specification

### User Experience

- A user follows quick-start steps and can run provider-related commands without immediate runtime `LoadError` on first setup.
- Quick-start documentation remains explicit about required components for a successful first run.

### Expected Behavior

1. `ace-llm` declares the runtime dependency needed for `ace/support/models`, so a fresh environment can complete the relevant quick-start path to provider listing without manual `Gemfile` patching.
2. Quick-start and onboarding documentation list the required packages and commands for a successful first provider-related run.
3. The onboarding behavior addresses the dependency/documentation portion of issue [#276](https://github.com/cs3b/ace/issues/276).
4. Any remaining one-time setup actions are clearly documented in docs and aligned to first-run expectations.

### Interface Contract

- **User-facing contract**
  - Quick-start sequence provides all required inputs so provider checks work in a new project.
- **Error behavior**
  - If setup still requires explicit action, the documentation surfaces exact required manual steps and rationale.
  - Fresh installs must not fail at runtime with `cannot load such file -- ace/support/models` during the documented provider-listing path.

### Success Criteria

- [ ] Fresh setup flow remains runnable without runtime `ace/support/models` load failure.
- [ ] Quick-start wording no longer implies an incomplete first-run dependency state.
