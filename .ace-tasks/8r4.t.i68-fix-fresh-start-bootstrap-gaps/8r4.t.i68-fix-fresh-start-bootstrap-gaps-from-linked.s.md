# Fix fresh-start bootstrap gaps from linked issues

## Objective

Create a single, release-ready onboarding fix scope that addresses the three known fresh-start blockers in one orchestrated set of tasks: bootstrap artifact hygiene, onboarding docs/guidance, and commit-flow provider compatibility.

## Behavioral Specification

### User Experience

- A new user following the quick-start flow sees a project that starts with stable local-hygiene behavior for ACE artifacts.
- A fresh project can run `ace-config init`, inspect generated guidance, and later run commit/message generation without avoidable onboarding friction.
- After merge, all three upstream regressions are closed without ad-hoc manual setup.

### Expected Behavior

1. The work is delivered as a parent task with three child tasks covering onboarding, setup completeness, and provider compatibility.
2. Subtasks remain independently verifiable and can be completed and verified out of order where safe.
3. The parent scope explicitly connects to upstream GitHub issues [#276](https://github.com/cs3b/ace/issues/276), [#278](https://github.com/cs3b/ace/issues/278), and [#279](https://github.com/cs3b/ace/issues/279).
4. PRs for implementation are tied to the task references and close the linked issues on merge.

### Interface Contract

- **Task decomposition contract**
  - Orchestrator: `8r4.t.i68` with three child tasks.
  - Child tasks:
    - `8r4.t.i68.0` -> [#279](https://github.com/cs3b/ace/issues/279)
    - `8r4.t.i68.1` -> [#276](https://github.com/cs3b/ace/issues/276)
    - `8r4.t.i68.2` -> [#278](https://github.com/cs3b/ace/issues/278)
- **PR contract**
  - PR description must include task references and closing statements:
    - `Fixes task 8r4.t.i68` (or equivalent task link format)
    - `Closes #276`, `Closes #278`, `Closes #279`

### Success Criteria

- [ ] Three child tasks are drafted and linked under `8r4.t.i68`.
- [ ] Each child task has clear behavioral expectations and verifiable acceptance criteria.
- [ ] Each linked issue (#276, #278, #279) is referenced in the task content and mapped to one child task.
- [ ] PR process requirement is present: task reference and explicit issue closure lines.

## Validation Questions

- Should the child tasks execute in strict order, or can subtasks be worked in parallel by independent workers?
- What is the target minimum verification level for each subtask before marking done (manual spot-check vs full automated run)?

## Vertical Slice Decomposition (Task/Subtask Model)

- **Slice type**: orchestrator with independent delivery subtasks.
- **Outcome**: one coherent onboarding reliability release covering all three current blockers.
- **Advisory size**: medium
- **Context dependencies**: `ace-support-config`, `ace-llm`, `ace-llm-providers-cli`, `ace-git-commit`, onboarding docs.
