# Fix fresh-start bootstrap gaps from linked issues

## Objective

Create a single, release-ready onboarding fix scope that addresses the three known fresh-start blockers in one orchestrated set of tasks: bootstrap artifact hygiene, onboarding docs/guidance, and commit-flow provider compatibility.

## Behavioral Specification

### User Experience

- A new user following the quick-start flow sees a project that starts with stable local-hygiene behavior for ACE artifacts.
- A fresh project can run `ace-config init`, inspect generated guidance, and later run commit/message generation without avoidable onboarding friction.
- After merge, all three upstream regressions are closed without ad-hoc manual setup.

### Expected Behavior

1. The work is delivered as a parent task with three child tasks covering onboarding, setup completeness, and provider compatibility.
2. Child tasks may be implemented and verified in parallel, but parent completion is blocked until all three child tasks are done and verified.
3. The parent scope explicitly owns upstream GitHub issue [#276](https://github.com/cs3b/ace/issues/276), while child tasks own [#278](https://github.com/cs3b/ace/issues/278) and [#279](https://github.com/cs3b/ace/issues/279) directly.
4. The issue-closure contract is enforced by task ownership: the parent closes [#276](https://github.com/cs3b/ace/issues/276), child `8r4.t.i68.0` closes [#279](https://github.com/cs3b/ace/issues/279), and child `8r4.t.i68.2` closes [#278](https://github.com/cs3b/ace/issues/278).
5. Child PRs may exist during implementation, but they must reference their child task and parent task without using GitHub issue-closing keywords.

### Interface Contract

- **Task decomposition contract**
  - Orchestrator: `8r4.t.i68` with three child tasks.
  - Child tasks:
    - `8r4.t.i68.0` -> [#279](https://github.com/cs3b/ace/issues/279)
    - `8r4.t.i68.1` -> no direct GitHub issue ownership
    - `8r4.t.i68.2` -> [#278](https://github.com/cs3b/ace/issues/278)
- **PR contract**
  - Final parent PR description must include task references and closing statements:
    - `Fixes task 8r4.t.i68`
    - `Closes #276`
    - `Closes #278`
    - `Closes #279`
  - Any child PR must include:
    - `Implements task 8r4.t.i68.x`
    - `Part of task 8r4.t.i68`
    - No GitHub issue-closing keywords

### Success Criteria

- [ ] Three child tasks are drafted and linked under `8r4.t.i68`.
- [ ] Each child task has clear behavioral expectations, explicit bundle context, and verifiable acceptance criteria.
- [ ] Each linked issue (#276, #278, #279) is referenced in the task content and mapped to a concrete delivery path.
- [ ] The PR process requirement is unambiguous: one parent PR closes issues, child PRs do not.
- [ ] Issue `#276` remains owned and closed by parent task `8r4.t.i68`.

## Validation Questions

- Child tasks may be worked in parallel by independent workers, but the parent cannot be closed until all three child tasks pass their verification plans.
- Minimum verification level is subtask-specific command coverage plus one failure-path check for each child before marking done.

## Vertical Slice Decomposition (Task/Subtask Model)

- **Slice type**: orchestrator with independent delivery subtasks.
- **Outcome**: one coherent onboarding reliability release covering all three current blockers.
- **Advisory size**: medium
- **Context dependencies**: `ace-support-config`, `ace-llm`, `ace-llm-providers-cli`, `ace-git-commit`, onboarding docs.
- **End-state coherence**
  - Child `8r4.t.i68.0` delivers init/bootstrap hygiene and guidance updates.
  - Child `8r4.t.i68.1` delivers dependency and docs completeness for first-run provider setup.
  - Child `8r4.t.i68.2` delivers Claude CLI capability-aware commit-flow behavior.
  - The parent introduces no behavior beyond coordinating these three durable end-state changes.

## Verification Plan

  - Task acceptance checklist covers expected behavior for each subtask.
- **Integration checks**
  - Validate behavior across a fresh bootstrap scenario, a fresh provider-listing scenario, and a commit message generation scenario.

  - Child task existence and linkage: verified by task tree and parent-child references.
