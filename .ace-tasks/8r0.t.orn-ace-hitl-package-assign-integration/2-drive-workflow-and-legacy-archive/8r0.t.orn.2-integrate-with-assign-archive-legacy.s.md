# Integrate ace-hitl with ace-assign

## Objective

Make `ace-assign` use the new `ace-hitl` package as its standard human-judgment pause mechanism, then archive the superseded gate-state-machine task once the replacement path is clearly documented.

## Behavioral Specification

### User Experience

- **Input**: An `ace-assign` agent encounters a blocking ambiguity that requires human judgment.
- **Process**: The agent creates a HITL item through `ace-hitl`, fails the current step with a standard HITL stall message, and exits. A human answers through the HITL item. On retry, the agent reads the answer and proceeds.
- **Output**: The assignment flow pauses cleanly without gate-state-machine complexity, and the human/agent interaction is tracked in `.ace-hitl/`.

### Expected Behavior

1. `drive.wf.md` gets a dedicated HITL section that explains:
   - when HITL should be used
   - how to create the item via `ace-hitl create`
   - how to fail the step with `ace-assign fail --message "HITL: <id> <path>"`
   - how the human answers and resumes
   - how the agent archives the item after using the answer
2. `ace-assign status` keeps the stall reason readable and, when the reason begins with `HITL:`, provides helper guidance that points users to `ace-hitl show` or the stored path.
3. The integration must continue using existing fail/retry mechanics and must not add gate phases, resume approval commands, or assignment-level gate state.
4. After the replacement direction is in place, task `8c0.t.070` is archived with a clear superseded-by reference to `8r0.t.orn`.

### Interface Contract

Agent-side creation and fail pattern:

```bash
ace-hitl create "Need product decision" --question "Should retries be visible?" --assignment <id> --step <number> --step-name <name> --resume "/as-assign-drive <id>"
ace-assign fail --message "HITL: <hitl-id> <hitl-path>"
```

Human-side answer flow:

```bash
ace-hitl show <hitl-id>
ace-hitl update <hitl-id> --answer "Yes, show retries in the user-facing output."
/as-assign-drive <assignment-id>
```

### Success Criteria

- [ ] `ace-assign` workflow docs use `ace-hitl` rather than package-internal helper names
- [ ] The fail/status interaction is aligned around the standard HITL message format
- [ ] No gate-state-machine behavior is reintroduced
- [ ] Legacy task `8c0.t.070` is archived with an explicit superseded note
