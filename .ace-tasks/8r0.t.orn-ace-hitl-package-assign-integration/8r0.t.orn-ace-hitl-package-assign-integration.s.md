# ace-hitl Package and ace-assign Integration

## Objective

Create a dedicated `ace-hitl` package as the ACE domain owner for human-in-the-loop items, then integrate it with `ace-assign` as the standard way agents pause for human judgment and resume after an answer. This replaces the earlier narrow plan that centered HITL primitives in `ace-support-items`.

Why: the intended product direction is a reusable HITL package with its own CLI and item-store semantics, not just a pair of low-level helper classes. `ace-support-items` should remain generic infrastructure. HITL-specific file contracts, lifecycle rules, and command behavior should live in `ace-hitl`.

## Behavioral Specification

### User Experience

- **Input**: A human or agent needs to create, inspect, answer, or archive a HITL item representing a blocking clarification or decision.
- **Process**: `ace-hitl` manages the `.ace-hitl/` file-backed store through a dedicated CLI and library layer. `ace-assign` agents use that package when work is blocked on human judgment.
- **Output**: HITL questions and answers are tracked in one canonical store with a consistent CLI, auditable markdown files, and a standard `ace-assign` stall/resume pattern.

### Expected Behavior

1. A new top-level gem/package `ace-hitl/` exists in the monorepo, is registered in `Gemfile`, and has a root wrapper `bin/ace-hitl`.
2. `ace-hitl` owns the `.ace-hitl/` item format, item lifecycle, and CLI surface for `list`, `show`, `create`, and `update`.
3. Each HITL item represents one clarification/decision thread with file layout:
   - `.ace-hitl/<id>-<slug>/<id>-<slug>.hitl.s.md`
4. Item lifecycle for v1 is:
   - `pending -> answered -> archived`
5. `ace-hitl update --answer "..."` writes the `## Answer` section and marks the item answered.
6. `ace-assign` drive guidance tells agents when to create a HITL item, how to fail with a standard HITL message, how to detect an answered item on retry, and how to archive it after consuming the answer.
7. The legacy human approval gate design in task `8c0.t.070` is archived only after the package-first replacement direction is specified and the integration path is documented.

### Interface Contract

CLI surface for `ace-hitl` v1:

```bash
ace-hitl list [--status pending|answered|archived] [--kind clarification|decision|approval] [--tags tag1,tag2] [--in next|all|archive]
ace-hitl show REF [--path] [--content]
ace-hitl create "Which auth strategy?" --question "JWT or sessions?" --question "Refresh token storage?" --assignment 8qr5kx --step 020 --step-name implement-auth --resume "/as-assign-drive 8qr5kx"
ace-hitl update t.abc --answer "Use JWT with server-side refresh tokens." --set status=answered
ace-hitl update t.abc --move-to archive
```

Item file contract:

```markdown
---
id: <b36ts>
status: pending
title: <question summary>
kind: clarification
created_at: "<iso8601>"
answered_at:
assignment: <id>
step: "<number>"
step_name: <name>
requester: <agent-or-human-name>
tags: []
---

# <title>

## Context
<what was happening>

## Questions
1. <question 1>
2. <question 2>

## Resume
<how to continue>

## Answer
<!-- answer goes here -->
```

For `ace-assign`, the standard stall contract is:

```bash
ace-assign fail --message "HITL: <id> <path>"
```

The `ace-assign status` output must surface the stall reason and keep the item reference/path legible to the user.

- Create the package directory, gemspec, version file, executable entry points, wrapper, and baseline docs

### Subtask 1: Implement ace-hitl Item Store and CLI (medium)
- Build the item-store manager, resolver/loader/scanner/update flows, and `list/show/create/update` commands
- Implement answer writing and answered-state handling
- Observable outcome: humans and agents can work with HITL items entirely through `ace-hitl`

### Subtask 2: Integrate ace-hitl with ace-assign (small)
- Update the drive workflow and align fail/status behavior around the HITL package
- Archive the superseded gate-state-machine task once the replacement path is documented
- Observable outcome: `ace-assign` has a clear HITL interaction pattern and the legacy design is retired

## Consumer Packages / Interfaces Affected

- `ace-hitl` becomes the domain package and public CLI for HITL items
- `ace-assign` consumes the `ace-hitl` package and its item contract at the workflow/CLI behavior layer
- `ace-support-items` remains shared infrastructure and may be reused internally by `ace-hitl`, but does not own HITL semantics
- `.ace-tasks` is affected by archiving task `8c0.t.070` as superseded
