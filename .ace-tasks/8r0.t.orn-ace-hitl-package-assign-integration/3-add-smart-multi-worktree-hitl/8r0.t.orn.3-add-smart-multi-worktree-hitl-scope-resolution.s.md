# Add Smart Multi-Worktree HITL Scope Resolution

## Objective

Make `ace-hitl` discovery commands behave like a blocker-resolution tool rather than a fleet dashboard: local-first inside a task worktree, project-wide from the main checkout, with an explicit scope override when users want a different view.

Why: `ace-hitl create` is naturally scoped to the current worktree, but humans answering HITL items may operate from the main checkout. The read path needs smart defaults so both blocked agents and human operators get the expected view without extra flags.

## Behavioral Specification

### User Experience

- **Input**: A human or agent runs `ace-hitl list` or `ace-hitl show` from either a task worktree or the main checkout.
- **Process**: `ace-hitl` detects the current repository context and resolves HITL items with a smart default scope. Users can override that scope explicitly.
- **Output**: In task worktrees, users see the HITL items relevant to that worktree first. In the main checkout, users see pending HITL items across worktrees by default. Item lookup remains predictable and prints the resolved location.

### Expected Behavior

1. `ace-hitl list` gains an explicit `--scope current|all` option.
2. Without `--scope`, `ace-hitl list` uses smart defaults:
   - in a linked task worktree: behave as `--scope current`
   - in the main checkout: behave as `--scope all`
3. Without an explicit `--status`, `ace-hitl list` defaults to `pending` items only.
4. Existing filters (`--kind`, `--tags`, `--in`) continue to apply after scope resolution.
5. `ace-hitl show` gains the same `--scope current|all` option.
6. Without `--scope`, `ace-hitl show` resolves refs local-first:
   - first search the current worktree scope
   - if not found and the invocation is using smart scope, retry across all worktrees
7. When `ace-hitl show` resolves an item outside the current worktree, the output must make the resolved path/worktree explicit so the user understands where the item lives.
8. `ace-hitl create` and `ace-hitl update` remain current-worktree scoped in this slice. No cross-worktree write behavior is introduced.
9. `ace-hitl` documentation must describe the distinction from `ace-overseer`: `ace-overseer status` is a global dashboard, while `ace-hitl` is local-first with a global operator view from main.

### Interface Contract

```bash
ace-hitl list [--status pending|answered|archived] [--kind clarification|decision|approval] [--tags tag1,tag2] [--in next|all|archive] [--scope current|all]
ace-hitl show REF [--path] [--content] [--scope current|all]
```

Examples:

```bash
# In a task worktree: defaults to current scope + pending
ace-hitl list

# In the main checkout: defaults to all worktrees + pending
ace-hitl list

# Explicit project-wide scan from any location
ace-hitl list --scope all

# Explicit local-only lookup
ace-hitl show t.abc --scope current

# Smart lookup: local first, then global fallback
ace-hitl show t.abc
```

Error handling:

- Invalid `--scope` values return a clear CLI error.
- `ace-hitl show REF --scope current` must fail clearly if the item is not present in the current scope; it must not silently widen the search.
- If the same ref is ambiguous across worktrees during an all-scope lookup, `show` must fail with a disambiguation error that includes candidate paths.

Edge cases:

- Running from the main checkout with no HITL items returns an empty result consistent with existing ACE item tools.
- Running from a non-task linked worktree still uses current scope unless explicitly widened.
- Smart fallback for `show` applies only when scope was implicit, not when `--scope` was provided.

### Success Criteria

- [ ] `ace-hitl list` supports smart default scope resolution and explicit `--scope`
- [ ] `ace-hitl list` defaults to pending items when `--status` is omitted
- [ ] `ace-hitl show` resolves local-first with all-scope fallback only for implicit smart lookup
- [ ] Resolved output makes cross-worktree item location legible
- [ ] No cross-worktree write semantics are added to `create` or `update`
- [ ] Usage docs explain the smart-default model and explicit override

## Vertical Slice Decomposition

Single behavioral slice:
- **Outcome**: `ace-hitl` read-path behavior is worktree-aware and operator-friendly
- **Advisory size**: small
- **Context**: align with `ace-overseer`'s worktree awareness, but keep `ace-hitl` local-first because it is a blocker-resolution tool
