# Documentation Overhaul - Core User Tools

## Objective

Rewrite documentation for ace-bundle, ace-task, ace-review, ace-git-commit, and the root README.md to create a user-first landing experience. Transform reference-heavy READMEs into landing pages with hero GIFs, taglines, and quick start guides. Create tutorial-style getting-started.md files. Delete usage.md files (CLI reference stays in --help only).

This is the PRIORITY orchestrator. The root README subtask (subtask 0) must be completed first to establish the landing page template that all other packages -- across all orchestrators -- will follow.

Research basis: Diataxis framework, clig.dev, top OSS README patterns (gum, lazygit, httpie, ripgrep).

## Behavioral Specification

### User Experience
- **Input**: Existing package READMEs, --help output, current docs/usage.md files for ace-bundle, ace-task, ace-review, ace-git-commit, and root README.md
- **Process**: Rewrite root README first (establishes template), then rewrite each package README as landing page, create getting-started tutorial, record demo GIF, delete usage.md
- **Output**: Per package: rewritten README.md, new docs/getting-started.md, new docs/demo.gif, deleted docs/usage.md. Plus: rewritten root README.md.

### Expected Behavior
When a first-time visitor opens any package README on GitHub, they should:
1. See a clear tagline explaining the tool's value proposition (not implementation)
2. See a hero GIF demo (<=30s) showing the tool's core value
3. Find a Quick Start section to get running in <2 minutes
4. Find feature bullets framed as benefits
5. Find links to getting-started.md tutorial and --help for CLI reference
6. NOT find --help output, full config schemas, or API docs in the README

### Interface Contract

Each package gets these files:

README.md structure:
- One-line tagline (value proposition)
- Demo GIF
- Why section (2-3 sentences)
- Quick Start (install + primary usage)
- Features (benefit-oriented bullets)
- Documentation links
- Part of ACE footer

docs/getting-started.md structure:
- Prerequisites
- Installation
- First real task walkthrough
- What to try next

Root README.md structure:
- Project tagline and logo
- Hero GIF showing overall workflow
- What is ACE (2-3 sentences)
- Tool categories with one-line descriptions
- Quick Start (install + first command)
- Links to individual package READMEs

### Success Criteria
- [ ] Root README follows landing page template and establishes pattern for all packages
