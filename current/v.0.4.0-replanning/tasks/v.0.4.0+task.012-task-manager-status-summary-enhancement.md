# Task Manager Status Summary Enhancement

## Behavioral Specification

### User Experience
- **Input**: User runs any task-manager listing command (`list`, `all`, `recent`, `next`)
- **Process**: System dynamically counts task statuses from current release task files and displays summary
- **Output**: One-line status summary at top of output showing task distribution (e.g., "Status: 3 draft, 12 pending, 5 in_progress, 8 done, 2 blocked (30 total)")

### Expected Behavior
The task-manager should automatically prepend a status summary to all listing commands, providing immediate visibility into task distribution across different states. The summary should be dynamically generated by scanning task files in the current release context and counting status field values.

### Interface Contract
```bash
# Enhanced listing commands with status summary
task-manager list       # Shows: "Status: X draft, Y pending... (N total)" + normal list
task-manager all        # Shows: "Status: X draft, Y pending... (N total)" + normal list  
task-manager recent     # Shows: "Status: X draft, Y pending... (N total)" + normal list
task-manager next       # Shows: "Status: X draft, Y pending... (N total)" + normal list

# Example output format:
# Status: 3 draft, 12 pending, 5 in_progress, 8 done, 2 blocked (30 total)
# [normal task listing follows]
```

### Success Criteria

- [ ] Status summary appears on first line of all task-manager listing commands
- [ ] Status counts are dynamically determined from actual task files (not hardcoded)
- [ ] Status counts reflect current release context only (same as task-manager scope)
- [ ] Summary format is consistent and readable across all commands
- [ ] Performance impact is minimal even with hundreds of task files
- [ ] Unknown status values are handled gracefully and included in summary

### Validation Questions

- [ ] Should the summary be alphabetically sorted by status name?
- [ ] How should empty status values or missing status fields be handled?
- [ ] Should there be a way to disable the summary for users who don't want it?
- [ ] What happens if no tasks exist in the current release?

## Risk Assessment

### Technical Risks
- **Risk:** Performance impact on large task collections
  - **Probability:** Medium
  - **Impact:** Low
  - **Mitigation:** Implement efficient counting algorithm with early exit strategies
  - **Rollback:** Remove status summary functionality

- **Risk:** Inconsistent status counting across commands
  - **Probability:** Low
  - **Impact:** Medium
  - **Mitigation:** Centralize counting logic in single molecule
  - **Rollback:** Fix inconsistencies through unified molecule usage

### Integration Risks
- **Risk:** Breaking existing CLI output parsing
  - **Probability:** Low
  - **Impact:** Medium
  - **Mitigation:** Prepend summary to preserve existing output structure
  - **Monitoring:** Test all CLI command outputs after changes

### Performance Risks
- **Risk:** Slowing down CLI commands with extra file parsing
  - **Mitigation:** Reuse already-loaded task data for counting
  - **Monitoring:** Time CLI command execution before and after
  - **Thresholds:** Maximum 50ms additional latency acceptable
